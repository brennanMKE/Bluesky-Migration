## Description

When a user signs in to an account whose status is `takendown` (or `suspended`), the React Native client routes to a dedicated screen at `Bluesky-ReactNative/src/screens/Takendown.tsx` that explains the suspension, links to the moderation policy, and includes an in-app appeal submission UI. The SwiftUI client has no equivalent — a takendown account currently has no surface and lands on a broken MainTabView the same way #0094 (deactivated) does.

## Steps to reproduce

1. Have (or simulate) a Bluesky account in takendown / suspended state.
2. Sign in to the SwiftUI app with that account.
3. Observe what the app does.

## Expected behavior

When `SessionManager` detects the session is on a `takendown` (or `suspended`) account, the app routes (above MainTabView) to a `TakendownView` that:

- Displays the suspension reason / policy violation summary.
- Links to the Bluesky moderation policy / community guidelines.
- Provides an "Appeal" form: textarea + submit → calls the Ozone appeal endpoint.
- Shows a "Sign out" / "Switch account" affordance.

## Actual behavior

No takendown/suspended handling. The session is treated as normal and the user lands on MainTabView with most API calls failing.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Takendown.tsx`.
- Status detection: see #0094 — `getSession.status` covers both deactivated and takendown.
- Appeal endpoint: `com.atproto.moderation.createReport` with `reasonType = appeal` (or the Ozone-specific appeal lexicon — confirm against RN).
- File to create: `BlueskyKit/Sources/BlueskyAuth/TakendownView.swift`; gate from `RootView` alongside the `DeactivatedView` from #0094.
- The appeal endpoint may already be available via the existing `BlueskyModeration` types — reuse before adding lexicons.

## Root cause

The plumbing landed by #0094 already detected `.takendown` and `.suspended` on the `Account` value, but `RootView` only branched on `.deactivated`, and there was no SwiftUI surface for the takendown holding screen. RN's `screens/Takendown.tsx` carries the explanatory copy, an inline ToS link, and an in-app appeal flow that calls `com.atproto.moderation.createReport` with `tools.ozone.report.defs#reasonAppeal` against the user's own DID — none of which existed on the Swift side.

## Fix

1. **`BlueskyAuth/TakendownView`** (new) — Mirrors `screens/Takendown.tsx`:
   - Default state: SF Symbol logo, "Your account has been suspended" headline, ToS-linked explanation paragraph (markdown `AttributedString`), primary "Sign Out" button, ghost "Appeal Suspension" button.
   - Appeal state: title flips to "Appeal suspension"; multiline `TextEditor` with `MAX_REPORT_REASON_GRAPHEME_LENGTH = 2000` enforced via `reason.count` (Swift's grapheme-cluster count matches RN's `countGraphemes` semantics for our purposes); inline "current / max" character counter; "Submit Appeal" primary, "Cancel" ghost.
   - Success state: form replaced by "Your appeal has been submitted. If your appeal succeeds, you will receive an email."
   - Errors render inline using `ATError.xrpc` server messages or a generic fallback.
   - Lives in BlueskyAuth (Layer 2). No BlueskyUI dep; styling matches `DeactivatedView` (`.quaternary.opacity(0.5)` field background, `.borderedProminent` primary, plain ghost).
2. **`BlueskyAuth/SessionManager.submitTakendownAppeal(reason:)`** — POSTs to `com.atproto.moderation.createReport` directly via `URLSession`, using the access JWT and the labeler proxy header `did:plc:ar7c4by46qjdydhdevvrndac#atproto_labeler` (equivalent to RN's `BLUESKY_MOD_SERVICE_HEADERS`). Body is encoded by a private `TakendownAppealBody` with `reasonType = tools.ozone.report.defs#reasonAppeal` and a `subject = com.atproto.admin.defs#repoRef` containing the current account's DID.
3. **`Bluesky-SwiftUI/ContentView.RootView`** — Adds an `else if` branch immediately after the `.deactivated` branch. When `currentAccount?.status` is `.takendown` *or* `.suspended`, render `TakendownView`. The existing `task(id: …)` / `currentAccount` re-evaluation handles route-out on sign-out.

`AccountStatus.takendown` / `.suspended`, `Account.status` plumbing, and the four-state RootView gate were already in place from #0094 — this issue did not need any further `BlueskyCore` or session-restore changes.

## Gotchas

- The `atproto-proxy` header is mandatory: without it, the user's PDS rejects the report (a takendown account can't act on its own behalf via the plain `createReport` route). RN ships the same constant under `BLUESKY_MOD_SERVICE_HEADERS`. The shared `NetworkClient` only auto-applies the chat proxy header for `chat.bsky.*` lexicons, so the appeal call goes through `URLSession` directly — same pattern as `activateAccount`.
- The submission uses the **access** JWT, not the refresh JWT (mirrors the activate-account constraint surfaced in #0094). A takendown user with an App Password should still be able to submit, since the appeal report does not require a "main password" scope.
- `tools.ozone.report.defs#reasonAppeal` is the canonical Ozone appeal reason. We hard-code the literal — there is no Swift constant in `BlueskyCore/Moderation.swift` for it (the existing `CreateReportRequest` is reused for ad-hoc moderation reports, and the Ozone lexicon currently has only this one appeal value).
- Both `.takendown` and `.suspended` route to `TakendownView`. RN's `screens/Takendown.tsx` uses identical copy for both ("Your account has been suspended") — we follow that rather than splitting into two screens.
- `String.count` in Swift returns grapheme cluster count, so the character counter naturally matches RN's `countGraphemes(reason)`. No external grapheme library needed.
