## Description

The repost, like, and other action buttons in the post action bar use the `PostView.viewer` state that was captured when the feed was last loaded. If the user leaves the feed open for more than a few seconds and acts on a post from another client (or the post's state is mutated remotely), the local state can be stale. The toggle logic (`viewer?.repost != nil`) then performs the wrong operation — e.g., reposting a post that was already reposted from another device, or undoing a repost that no longer exists on the server.

## Steps to reproduce

1. Open the feed on two devices (or in a simulator and the web app) signed in to the same account.
2. On device A, repost a post.
3. Wait a few seconds without refreshing device B.
4. On device B, tap the repost button on the same post — the local state still shows it as not reposted, so it tries to repost again.

## Expected behavior

Before executing a toggle action, the app should refresh the post's viewer state from the server if the cached data is older than a short threshold (a few seconds). The toggle then operates on the freshly confirmed state.

## Actual behavior

The action is applied immediately against the stale `PostView` snapshot captured at feed load time, potentially performing a duplicate or contradictory operation.

## Notes

This affects all viewer-state toggles: repost, unrepost, like, unlike, and any future mute/block actions. The fix likely belongs in `FeedStore` or `FeedViewModel` — fetch a fresh `PostView` (e.g., via `app.bsky.feed.getPosts`) when the post was last seen more than a configurable threshold ago, then apply the toggle against the refreshed state.

## Root cause

`FeedView.actions(for:vm:)` decided between `like` vs `unlike` (and `repost` vs the Repost/Quote sheet) by reading `vm.posts.first(where:...).viewer` — i.e. the most recent local snapshot for that post. That snapshot is only refreshed when the feed itself is re-fetched, so any state change made from another device (or by the labelers/PDS asynchronously) between feed loads was invisible to the toggle decision. The toggle then performed the wrong operation: re-creating a like/repost record that already existed, or trying to `deleteRecord` a record the server had already cleared.

## Fix

Introduced a freshness-checked toggle path in `FeedStore`:

1. Added `viewerStateFreshnessWindow = 5s` plus a per-URI `lastViewerStateCheck: [ATURI: Date]` map (`@ObservationIgnored`). Every successful feed fetch stamps `now` for all returned posts so the common "tap right after the feed loads" path needs no extra round-trip.
2. Added `freshenedPost(for:)` which calls `app.bsky.feed.getPosts?uris=...` when the cached snapshot is older than the window, writes the fresh `PostView` back into `posts`, and falls back to the local snapshot on network error so toggles never hard-fail offline.
3. New `toggleLike(post:)` and `toggleRepost(post:)` on `FeedStoring` / `FeedViewModel`. They claim the corresponding in-flight slot synchronously, freshen, then dispatch to `performLike` / `performUnlike` / `performRepost` / `performUnrepost`. The dispatch helpers are private and assume the caller already owns the slot.
4. Refactored the existing `like` / `unlike` / `repost` / `unrepost` public methods to claim the slot themselves and then delegate to the same `perform*` helpers — this avoids double-claim when toggle is the entry point but keeps the explicit operations callable (the Repost confirmation dialog still calls `vm.repost`).
5. The `perform*` helpers also call `freshenedPost` before deciding to create or delete a record, so direct callers (Repost dialog, future composer flows) also benefit. `performUnlike` / `performUnrepost` now also reset local `viewer.like` / `viewer.repost` to `nil` when the server says the record is already gone — keeps the UI in sync without firing a redundant `deleteRecord`.
6. Added `GetPostsResponse` (`{ posts: [PostView] }`) to `BlueskyCore/Feed.swift`.
7. `FeedView.onLike` is now a one-liner that hands the freshest local post to `vm.toggleLike`; `onRepost` keeps the Repost/Quote chooser when the local snapshot says not-yet-reposted, otherwise routes through `vm.toggleRepost` (which will itself unrepost or skip after refresh).

This also addresses the closely related issue #0053 (like state reverts after tap): the new toggle path serializes through `likeInFlight` end-to-end and the `freshenedPost` write-back replaces the post atomically, so a background feed refresh that arrives mid-tap can no longer clobber the optimistic update. Leaving #0053 open for a separate confirmation pass per the workflow note.

## Files changed

- `BlueskyKit/Sources/BlueskyCore/Feed.swift` — added `GetPostsResponse`.
- `BlueskyKit/Sources/BlueskyFeed/FeedStore.swift` — new `toggleLike` / `toggleRepost`, `freshenedPost`, `lastViewerStateCheck` map, `perform*` private helpers; existing public methods refactored to delegate.
- `BlueskyKit/Sources/BlueskyFeed/FeedViewModel.swift` — exposed `toggleLike` / `toggleRepost`.
- `BlueskyKit/Sources/BlueskyFeed/FeedView.swift` — `onLike` / `onRepost` use the new toggle methods.

## Gotchas

- `app.bsky.feed.getPosts` accepts up to 25 URIs in a single `uris` array; the current implementation passes a single URI per tap. If/when bulk-freshen becomes worthwhile (e.g. on app foreground) this can batch.
- The 5s freshness window is intentionally short. A longer window would reduce round-trips at the cost of letting more stale taps through; a shorter window would refresh nearly every tap. 5s lines up with the typical feed-fetch round-trip plus a small grace period.
- `freshenedPost` writes the fresh `PostView` back into `posts` via `updatePost`, which means a tap-without-stale-state can briefly cause a re-render with updated counts. That is the desired behaviour — the user's like count should reflect any other-device activity that happened in the background.
