## Description

The React Native composer toolbar includes a GIF button (`SelectGifBtn`) that opens a Tenor-backed GIF search dialog. The selected GIF is attached to the post as an external embed (`app.bsky.embed.external` pointing at the Tenor URL with thumbnail). The SwiftUI composer toolbar has only image and video pickers — no GIF support.

Tracked separately from #0034 (which lists video picker / link card preview / threading / drafts) because GIF is a distinct media path with its own search UI and embed type.

## Steps to reproduce

1. Open the SwiftUI composer.
2. Look for a GIF button in the media toolbar.
3. Observe there is none.

## Expected behavior

A GIF button next to the image / video buttons in the composer media toolbar. Tapping opens a sheet:

- Search bar with debounced Tenor query.
- Trending GIFs section when query is empty.
- Grid of animated GIF previews; tap to select.
- Selected GIF appears in the composer's embed area with a "Remove" affordance.

On post creation, the GIF is attached as an external embed (`app.bsky.embed.external` with `uri`, `title`, `description`, and `thumb` blob).

## Actual behavior

No GIF button. Only image and video pickers exist.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/com/composer/gifs/SelectGifBtn.tsx` and the GIF search dialog implementation.
- Tenor API: requires a Tenor API key. RN reads it from the build env. SwiftUI side would need an equivalent config slot in `Bluesky-SwiftUI/Config.swift` or `Build.xcconfig`.
- Embed type: `app.bsky.embed.external` (already supported in `BlueskyCore` for link cards). Reuse — no new lexicons needed.
- File to edit: `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift`. Add a `GIFPickerSheet.swift` view + a small Tenor client.

## Root cause

The SwiftUI composer never modeled GIF media. RN exposes `SelectGifBtn`
in the toolbar (`view/com/composer/photos/SelectGifBtn.tsx`), backed by
`useTenorFeaturedGifsQuery` / `useTenorGifSearchQuery` in
`state/queries/tenor.ts`, and posts the chosen GIF as
`app.bsky.embed.external` via `lib/api/resolve.ts#resolveGif`. Our
composer only had image and video paths — no GIF button, no Tenor
client, no GIF-aware embed builder, and no `selectedGIF` state on the
view model. As a result there was no way for a user to attach a GIF
from the SwiftUI client.

## Fix

**Tenor client (`BlueskyComposer/TenorClient.swift`).** New module-local
client that calls Bluesky's GIF proxy
(`https://gifs.bsky.app/tenor/v2/featured` /
`/tenor/v2/search`) — mirroring RN's `state/queries/tenor.ts`. The
proxy is the canonical entry point Bluesky's RN client uses; it
forwards to Tenor with the org's API key, so this side of the port
needs no Tenor key of its own (see Gotchas). Query parameters match RN
exactly: `client_key=bluesky-{ios,macos}`, `limit=30`,
`contentfilter=high`,
`media_filter=preview,gif,tinygif`, plus a BCP-47 `locale`. Response is
decoded into a small `TenorGif` value type with helpers for the embed
URL (`gifURL`), the picker thumbnail (`thumbURL` → `tinygif`), and the
preview blob URL (`previewURL` → `preview` falling back to `tinygif`).
The client is a `nonisolated final class : Sendable` rather than an
`actor` because there is no shared mutable state — every call is a
stateless `URLSession` request. Modeling it as an actor would have
forced UI callers off MainActor for no benefit.

**GIF picker sheet (`BlueskyComposer/GIFPickerSheet.swift`).** New view
that mirrors RN's `GifSelectDialog` (`components/dialogs/GifSelect.tsx`):

- Search bar at top with magnifier icon, debounced 500ms (matching
  RN's `useThrottledValue(undeferredSearch, 500)`).
- Empty query → trending / featured grid.
- Non-empty query → search results.
- Two-column `LazyVGrid` of square `AsyncImage` cells loading the
  static `tinygif` thumbnail. Tapping a cell fires `onSelect(gif)` and
  dismisses the sheet.
- Loading / empty / error states with a Retry button.
- A `Cancel` toolbar button to dismiss without selecting.

**Composer state (`ComposerViewModel`, `ComposerStore`).**

- `ComposerViewModel.selectedGIF: TenorGif?` — the picked GIF, mutually
  exclusive with images / video / link card. New `setGIF(_:)` clears
  the competing slots; `removeGIF()` clears just the GIF. The
  `visibleLinkURL` getter now also returns nil when a GIF is selected,
  so the link card preview stops fighting for the same render slot.
- `ComposerStoring.post(...)` and `ComposerStore.post(...)` gained a
  `selectedGIF: TenorGif?` parameter. When non-nil and no images /
  video are attached, the store calls `buildGIFExternalEmbed(_:)`
  which: (1) appends `?hh=H&ww=W` to the canonical gif URL (matching
  RN's `resolveGif`), (2) downloads the static `preview` bytes, (3)
  uploads them via `com.atproto.repo.uploadBlob`, (4) builds an
  `EmbedExternal { uri, title=alt, description="ALT: <alt>", thumb }`.
  The `"ALT: "` prefix matches RN's
  `createGIFDescription` default-alt sentinel
  (`lib/gif-alt-text.ts`). The GIF + quote case wraps the external
  embed in `recordWithMedia`, mirroring RN's behavior when a quote
  post carries an external embed. A failure to fetch / upload the
  preview surfaces a non-fatal error to the user via the existing
  `errorMessage` channel — the post does not go out without its
  embed, matching RN's `resolveGif` failure path.

**Composer UI (`ComposerSheet`).**

- A `gifButton` (Label "GIF" + `photo.stack` symbol) was added to the
  media toolbar between the image and video buttons on both iOS and
  macOS. The button hides when an image is attached, when a video is
  attached, or when a GIF is already selected — and the image / video
  buttons hide when a GIF is selected. This matches RN's mutual
  exclusion.
- A `gifPreview` view renders the selected GIF as a square
  `AsyncImage` thumbnail (200pt tall) with a black "GIF" badge
  overlay and an `xmark.circle.fill` remove button. Animated playback
  in the preview is intentionally deferred (see Gotchas). RN renders
  the GIF playback inline; we match the static-thumbnail-with-GIF-
  overlay shape.
- `showGIFPicker: Bool` drives the sheet presentation: a sheet on
  iOS, a sized sheet on macOS. RN doesn't use a popover for the GIF
  picker — it's tall enough that a sheet is the right primitive on
  both platforms.

The view-model resets `selectedGIF` (alongside `text`, `images`,
`additionalPosts`, `attachedVideo`, `selectedLabels`,
`threadgateAllow`, `quotesEnabled`, …) on a successful post.

## Gotchas

- **No Tenor API key is needed; RN uses the Bluesky GIF proxy.** The
  issue spec assumed Tenor would be called directly and asked for an
  `Info.plist` slot. RN's `state/queries/tenor.ts` actually points at
  `https://gifs.bsky.app/tenor/v2/...` — the bsky proxy injects the
  org's Tenor key server-side. Matching RN, the SwiftUI client also
  hits the proxy and ships no API key. The default `TenorClient`
  base URL is `https://gifs.bsky.app`; override it via
  `TenorClient(baseURL:)` if you need a different proxy host. There
  is therefore no `TENOR_API_KEY` config slot in `Build.xcconfig` or
  the app's `Info.plist`. If Bluesky ever flips RN to call Tenor
  directly, this is the seam where a key would slot in: pass it
  through as a custom `URLRequest` header in `TenorClient.fetch(...)`.
- **Animated thumbnails in the picker are deferred.** SwiftUI's
  `AsyncImage` renders the static first frame of an animated GIF, not
  a continuous animation, and the picker's two-column grid would need
  a custom `ImageIO` / `CGImageSource` cell to animate every visible
  result. That's a non-trivial component (memory pressure on the
  scrolling grid is the hard part) and is parity-acceptable as a
  follow-up: RN ships animated thumbnails; we ship static ones for
  now and animate the chosen GIF in the composer's preview area
  (also static today — same follow-up). File a #01XX issue if user
  testing flags this as a regression rather than a polish item.
- **GIF preview area shows a static frame.** Same root cause as the
  picker. The composer's `gifPreview` renders the static
  `tinygif` thumbnail with a "GIF" badge overlay so the embed type is
  unambiguous. Once the chosen GIF actually posts, downstream feed
  rendering (Bluesky-side) animates it as expected — only the
  composer-local preview is static.
- **Post is rejected if the preview blob upload fails.** RN's
  `resolveGif` similarly bails the GIF attachment when the preview
  image can't be fetched. We surface the failure via
  `errorMessage` and return early without posting, so the user can
  retry. We do not silently post a `thumb`-less external embed; the
  embed without a thumbnail looks broken in feed and is worse UX
  than an error.
- **GIF + images mutual exclusion enforced in two places.** The
  toolbar hides conflicting buttons; `setGIF(_:)` defensively also
  clears `images` and `attachedVideo`. RN encodes the same invariant
  via its embed-draft state machine. Two layers of enforcement keeps
  programmatic state-setting (e.g. a future "share to Bluesky" intent
  that pre-fills a GIF) from creating an impossible state.
- **Search field uses `Search Tenor` placeholder copy.** RN dynamically
  switches between `Search Tenor` and `Search KLIPY` based on a
  feature flag (`KlipyGifProviderEnable`). KLIPY isn't ported (it's
  a separate provider entirely; RN gates it behind a remote feature
  flag), so the placeholder is fixed at "Search Tenor". A follow-up
  can wire this if KLIPY ships to Apple platforms.
- **Pre-existing `BlueskyKitTests` build failure.** The test target
  still fails to compile against `BlueskyEnvironment`'s
  `bookmarks` / `pathMonitor` signature drift (noted in #0097 /
  #0098). New code in this issue is exercised at runtime via the
  composer; no unit tests were added because they couldn't compile
  alongside the existing failure. The library `swift build` and the
  macOS workspace build both pass.
