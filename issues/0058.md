# 0058 — Audit and eliminate `try?` to prevent silent failures

| | |
|---|---|
| **Status** | open |
| **Module** | All |
| **Platform** | All |
| **First seen** | 2026-04-30 |

## Description

`try?` swallows thrown errors and converts them to `nil`, which causes failures to disappear without any log, telemetry, or surfaced UI state. This is an anti-pattern that hides real bugs (decoding failures, network errors, persistence failures) and makes them effectively undebuggable. Every existing use of `try?` in the Swift code under `../BlueskyKit/` and `../Bluesky-SwiftUI/` should be audited and replaced with explicit error handling.

## Expected behavior

Errors are either:
- handled with `do/try/catch` and logged via the project logger, or
- propagated with `try` (when the caller can deal with them), or
- intentionally turned into `nil` only with a clear comment justifying why a failure is acceptable in that exact spot — and even then, logged at `debug` level so it can be observed.

No code path silently discards a thrown error.

## Actual behavior

`try?` is used in multiple places, hiding errors. Failures present as missing data, blank screens, or "nothing happens" with no diagnostic trail.

## Steps to reproduce

1. From either Swift repo root, run:
   ```
   rg --type swift '\btry\?' ../BlueskyKit ../Bluesky-SwiftUI
   ```
2. Each match is a candidate site for review.

## Notes

Audit approach:

1. Enumerate every `try?` site in `../BlueskyKit/Sources/` and `../Bluesky-SwiftUI/`.
2. For each site, classify:
   - **Replace with `do/try/catch`** — log the error and decide on a sensible fallback. Default choice for I/O, decoding, persistence, and networking.
   - **Replace with `try`** — propagate to caller when the function already throws or can be made to throw.
   - **Keep `try?` with justification** — only when failure truly is expected and benign (e.g., best-effort cleanup); add a one-line comment explaining why and add a `debug` log.
3. Pay special attention to:
   - `JSONDecoder().decode(...)` — decoding failures should never be silent.
   - `FileManager` / SwiftData / Keychain operations — persistence failures must surface.
   - XRPC / `NetworkClient` calls — network errors must be observable.
4. Consider adding a SwiftLint or custom build-script rule to flag new `try?` introductions after the audit so the anti-pattern does not creep back in.

Track sub-fixes by module if the audit produces a long list — small focused PRs per module are preferable to one sweeping change.
