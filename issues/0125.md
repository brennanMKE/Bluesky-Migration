## Description

`ListDetailScreen` jumps straight to a tab strip (Members / Feed) with no header. The React Native reference (`Bluesky-ReactNative/src/screens/ProfileList/`) renders a full header above the tabs:

- List avatar (square thumbnail, ~64pt).
- List name (large title).
- Purpose badge (Curate / Moderation).
- Description text.
- Creator handle row ("by @alice.bsky.social", tappable to profile).
- Member count + subscriber count.
- **Primary action**: Subscribe / Unsubscribe (for mod lists) or Pin / Unpin (for curate lists).
- **Secondary actions menu** (ellipsis): Mute list, Block list (mod lists), Share list (copy link / sheet), Report list, Edit list (covered separately in #0127), Delete list (covered separately in #0127).

Without a header, users have no context about what list they're looking at and no way to subscribe/manage.

## Steps to reproduce

1. Tap into any list from `ListsScreen`.
2. Observe the screen opens directly to tabs with no header above them.

## Expected behavior

`ListDetailScreen` renders the full header above the tab strip. Action buttons are wired to the existing `ListsViewModel` mutations (mute, subscribe) and to a new edit/delete flow for #0127.

## Actual behavior

Just tabs.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/ProfileList/index.tsx` — the `Header` and `AboutSection` components.
- File to edit: `BlueskyKit/Sources/BlueskyLists/ListDetailScreen.swift`.
- Pairs with #0126 (About tab) and #0127 (edit/delete).

## Root cause

`ListDetailScreen` was scaffolded with just a `Picker(.segmented)` for Members
/ Feed and never grew the header that RN's `ProfileList/Header.tsx` renders
above the tabs. The data model also lacked the bits the header needs — viewer
state (muted / blocked) and `listItemCount` — so even if the view existed it
would have nothing to drive the Subscribe/Unsubscribe label or the member
count line.

## Fix

- **Lexicon**: extended `ListView` (`BlueskyCore/Graph.swift`) with the
  optional `listItemCount: Int?` and `viewer: ListViewerState?` fields from
  `app.bsky.graph.defs#listView`. Added a new `ListViewerState` value type
  carrying `muted` and `blocked` (AT-URI of the listblock record).
- **Mutations**: added `muteList()` / `unmuteList()` to `ListDetailStore` /
  `ListDetailStoring` and exposed them on `ListDetailViewModel` as
  `subscribeMute()` / `unsubscribeMute()`. The store optimistically flips
  `viewer.muted` so the header button label updates immediately and rolls
  back on failure.
- **Header**: rewrote `ListDetailScreen.body` to put a header above the tab
  strip — 64pt rounded avatar, large title with a "Curate" / "Moderation"
  purpose badge (matching RN's pill copy), tappable creator row, member
  count line, full description, primary action button, and an ellipsis
  More-options menu.
- **Primary action**: `Subscribe` (curate-list muted-accounts confirmation)
  and `Unmute` for mod lists, wired to `muteActorList` / `unmuteActorList`.
  Curate-list `Pin to home` is rendered disabled for now — it depends on the
  saved-feeds preferences plumbing tracked by #0060; the slot is in place so
  the UI doesn't shift when that lands.
- **More-options menu**: Share list (`ShareLink` to
  `https://bsky.app/profile/<handle>/lists/<rkey>`), Copy link, Unmute (when
  applicable), and Report list. Report opens `BlueskyModeration.ReportDialog`
  with the list's strongRef as the subject and posts to
  `com.atproto.moderation.createReport`. Edit / Delete are intentionally
  omitted — those land with #0127.
- **Profile navigation**: added `onProfileTap: ((DID) -> Void)?` to
  `ListsScreen` (public) and `ListDetailScreen` (internal) so the host shell
  (`MainTabView`) can wire the creator-row tap to push a `ProfileScreen`. The
  callback is optional so the existing call site continues to compile —
  wiring lands as a follow-up app-shell change.
- **Package graph**: added `BlueskyModeration` as a dependency of
  `BlueskyLists` so the header can present `ReportDialog` directly. Both are
  feature-layer modules and the import does not introduce a cycle.

## Files changed

- `BlueskyKit/Sources/BlueskyCore/Graph.swift` — new `ListViewerState`,

- `BlueskyKit/Sources/BlueskyLists/ListsStore.swift` —
  `ListDetailStoring.muteList/unmuteList` + optimistic store impls.
- `BlueskyKit/Sources/BlueskyLists/ListDetailViewModel.swift` —
  `subscribeMute/unsubscribeMute`.
- `BlueskyKit/Sources/BlueskyLists/ListDetailScreen.swift` — header view,

- `BlueskyKit/Sources/BlueskyLists/ListsScreen.swift` — propagates the new

- `BlueskyKit/Package.swift` — `BlueskyLists` now depends on
  `BlueskyModeration`.

## Gotchas

- Curate-list Pin/Unpin is **disabled** in this change. RN reads the saved
  feeds preference to decide which label to show and to mutate the entry;
  the SwiftUI `ListDetailScreen` does not yet receive a preferences feed
  (that wiring is part of #0060). The button is rendered disabled so the
  layout matches RN now and only the action wires up later.
- `ListView.viewer` is decoded but the SwiftUI app does not currently fetch
  the field with credentials in every code path — older fixtures will
  decode `viewer = nil` and the header falls back to "Subscribe". This is
  the correct behavior when no viewer is signed in.
- Profile-tap is delivered as a callback rather than an in-module
  `NavigationLink(value:)` because `BlueskyLists` cannot import
  `BlueskyProfile` (the latter is the heavier feature module). The host
  shell needs to pass `onProfileTap: { feedProfileDID = $0 }` to
  `ListsScreen` to complete the loop; the closure is optional so existing
  call sites still compile.
