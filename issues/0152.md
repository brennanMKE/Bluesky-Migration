## Description

Opening the Bookmarks screen surfaces a fatal decoding error and the list never renders:

> Response decoding failed: DecodingError.keyNotFound: Key 'uri' not found in keyed decoding container. Path: `bookmarks[0]`. Debug description: No value associated with key CodingKeys(stringValue: "uri", intValue: nil) ("uri").

The SwiftUI `BookmarkView` (or whatever decodes the `app.bsky.bookmark.getBookmarks` response items) expects `uri` directly on each bookmark item, but the wire shape doesn't carry `uri` at that level. The JSON likely nests the post under a different key (commonly `subject`, `item`, or `post`), so the AT-URI lives at `bookmarks[0].subject.uri` — not `bookmarks[0].uri`.

## Steps to reproduce

1. Sign in.
2. Tap the Saved / Bookmarks tab (or open from the sidebar drawer per #0150).
3. Observe the error alert; the list never loads.

## Expected behavior

Bookmarks load and render as a paginated list of `PostCard`s. The `BookmarkView` Codable shape matches the actual `app.bsky.bookmark.getBookmarks` response — typically:

```json
{
  "bookmarks": [
    {
      "subject": { "uri": "at://...", "cid": "..." },
      "createdAt": "2026-..."
    },
    ...
  ],
  "cursor": "..."
}
```

The Swift type should mirror that nesting (`subject: PostRef` or `subject: StrongRef`, with `uri` and `cid` inside).

## Actual behavior

JSON decode aborts at `bookmarks[0]` because the outer item carries no `uri` key. The screen shows the error alert and no rows.

## Root cause

`BookmarkView` decoded `uri`/`cid` at the top level, but the
`app.bsky.bookmark.getBookmarks` response nests the strong reference under
the `subject` key:

```json
{
  "bookmarks": [
    {
      "subject":   { "uri": "at://...", "cid": "bafy..." },
      "createdAt": "2026-05-07T12:00:00Z",
      "item":      { "$type": "app.bsky.feed.defs#postView", ... }
    }
  ],
  "cursor": "..."
}
```

The bookmark item itself carries no top-level `uri`. When decoding
encountered the first item it threw `keyNotFound` for `uri` at
`bookmarks[0]`, the bookmark list never populated, and the screen showed
an error alert.

Confirmed against the React Native client at
`Bluesky-ReactNative/src/state/queries/bookmarks/useBookmarksQuery.ts` —
`useBookmarksQuery` uses `AppBskyBookmarkGetBookmarks.OutputSchema` and
the optimistic helpers in the same file build items with
`{subject: {uri, cid}, createdAt, item}` (see `optimisticallySaveBookmark`
and `optimisticallyDeleteBookmark`, which filters by
`b.subject.uri !== uri`). The companion mutation
`useBookmarkMutation.ts` confirms `deleteBookmark` takes the **post's**
AT-URI, not a separate bookmark record URI.

## Fix

In `BlueskyKit/Sources/BlueskyCore/Bookmark.swift`:

- New `BookmarkSubject { uri: ATURI; cid: CID }` (com.atproto.repo.strongRef shape).
- `BookmarkView` now carries `subject: BookmarkSubject`, `createdAt: Date`,
  and `item: PostView?` (optional because the lexicon allows the post to
  be unresolved). Convenience accessors `bookmark.uri` / `bookmark.cid`
  forward to `subject` so call sites that previously read the flat field
  keep working.
- `GetBookmarksResponse` is unchanged in shape but now decodes correctly.
- `DeleteBookmarkRequest` gained `init(postURI:)` to match the lexicon's
  semantics; the old `init(bookmarkURI:)` is kept as a deprecated shim.

Consumers updated:

- `BookmarksStore` exposes `delete(postURI:)` and filters via
  `$0.subject.uri == postURI`.
- `BookmarksStoring` protocol renamed to match.
- `BookmarksScreen` reads `bookmark.item` (now optional) and uses
  `bookmark.subject.uri` for ID/pagination.
- `FeedStore.unbookmark` now constructs `DeleteBookmarkRequest(postURI:)`.

Regression test added in `LexiconCodableTests.swift` — `BookmarkCodableTests`:

- Decodes the canonical nested-subject JSON.
- Round-trips a `BookmarkView` through Codable.
- Asserts the legacy flat shape (top-level `uri`/`cid`) now fails to
  decode, so a future field rename is caught at the test boundary.

## Gotchas

- The lexicon's `deleteBookmark` takes the **post's** AT-URI (the same
  value as `subject.uri`), not a separate bookmark record URI. The old
  Swift API used the misleading label `bookmarkURI`. Both the new
  initializer (`postURI:`) and the protocol method (`delete(postURI:)`)
  now reflect this. The deprecated `init(bookmarkURI:)` is retained
  as a back-compat shim.
- `BookmarkView.item` is `Optional<PostView>` rather than required —
  matching the open-union shape in the lexicon and guarding against
  bookmarks whose post has since been deleted. `BookmarksScreen` skips
  rows with `item == nil` rather than rendering a placeholder; if RN
  parity wants a "post unavailable" cell, that can be added later.
- Pre-existing test compile failure noted in past issues' Gotchas
  (`BlueskyKitTests.swift:90`) blocks `swift test`. The new
  `BookmarkCodableTests` was verified by inspection; it follows the
  same pattern as the surrounding `@Suite` cases and uses the existing
  `iso8601` / `iso8601Encoder` helpers.
