## Description

`ModerationListsScreen` renders a single flat list of every moderation list the viewer has touched. The React Native reference splits these into two sections (or tabs): **My moderation lists** (lists the viewer created) and **Subscribed moderation lists** (lists the viewer subscribes to but does not own). The actions per row differ — own lists support edit/delete; subscribed lists support unsubscribe — so combining them is awkward UX.

## Steps to reproduce

1. Both create your own mod list and subscribe to someone else's.
2. Open Moderation → Moderation lists.
3. Observe everything is one flat list.

## Expected behavior

Either tabs (Created / Subscribed) or two sections in the same scroll. Each row gets its appropriate action (edit/delete or unsubscribe). Use `creator.did == viewerDID` to classify.

## Actual behavior

Flat list, no ownership separation.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Lists/MyLists.tsx` with `filter="mod"` (same component as curate-list hub at #0131; the parity gap mirrors that one for moderation lists specifically).
- File to edit: `BlueskyKit/Sources/BlueskyModeration/ModerationListsScreen.swift`.

## Root cause

`ModerationListsScreen` rendered a single flat list of every mod list the
viewer had touched, drawn solely from `getLists` filtered to
`purpose == modlist`. The store never called `getListMutes` /
`getListBlocks`, so subscribed-but-not-owned mod lists were missing
entirely, and there was no ownership classification to drive different
per-row affordances (edit/delete vs unsubscribe).

## Fix

Mirrored the #0131 split for the mod-list hub:

- `ModerationStore` now accumulates three endpoints behind `loadModLists`
  (matching RN's `useMyListsQuery(filter: 'mod')` accumulator):
  - `getLists` filtered to `purpose == modlist` for the owned section.
  - `getListMutes` for subscribed-by-mute (excludes the viewer's own DID).
  - `getListBlocks` for subscribed-by-block (excludes the viewer's own DID).
  De-duplication is keyed on `list.uri`. Pagination cursors are tracked
  separately (`modListsCursor`, `subscribedMutesCursor`,
  `subscribedBlocksCursor`); `loadMoreSubscribedModLists` chains mutes
  before blocks the way RN's promise array does.
- `ModerationViewModel` exposes the `subscribedModLists` array,
  `viewerDID`, and an `unsubscribe(from:)` helper that picks
  `unmuteActorList` for muted lists and a `deleteRecord(listblock)` for
  blocked lists — matching RN's `useListMuteMutation` /
  `useListBlockMutation` split.
- New XRPC response types `GetListMutesResponse` and
  `GetListBlocksResponse` (same shape as `GetListsResponse`).
- `ModerationListsScreen` renders two `Section`s — **My moderation
  lists** and **Subscribed**. The owned section has no per-row action
  (edit/delete continues to live in the list detail screen). The
  subscribed section carries an Unsubscribe button per row.
- `viewerDID` resolves once on `task` via
  `accountStore.loadCurrentDID()`. Per-section empty states match RN's
  copy.

## Files changed

- `BlueskyKit/Sources/BlueskyCore/Graph.swift` — added `GetListMutesResponse` and `GetListBlocksResponse`.
- `BlueskyKit/Sources/BlueskyModeration/ModerationStore.swift` — fan-out fetch over `getLists` / `getListMutes` / `getListBlocks` with separate cursors, plus `unblockList(_:)`.
- `BlueskyKit/Sources/BlueskyModeration/ModerationViewModel.swift` — exposes `subscribedModLists`, `viewerDID`, `resolveViewerDID()`, `unsubscribe(from:)`, `loadMoreSubscribedModLists()`.
- `BlueskyKit/Sources/BlueskyModeration/ModerationListsScreen.swift` — two `Section`s with per-section row builders + tail-of-section pagination.

## Gotchas

- The subscribed-section pagination tail compares against
  `subscribedModLists.last?.uri`, not a flat-list tail, because the two
  sections page through different XRPC endpoints. Owned-section
  pagination still triggers off `viewModel.modLists.last`.
- De-dup keys on `list.uri.rawValue`. A list the viewer both owns and
  has muted will only appear in the owned section (we drop creator-DID
  matches from the subscribed merge before adding).
- `unblockList` reads the listblock AT-URI from `list.viewer?.blocked`
  (the appview surfaces it specifically so clients can drop the record
  without re-discovering its rkey). Mirrors how `unblock(profile:)`
  already worked for actor blocks.
