## Description

The Home feed currently has only two tabs — Following and Discover — rendered in a fixed two-tab strip. The React Native reference uses a horizontal scroller with at least four built-in tabs (Following, Mentions, Discover, Popular With Friends) plus the user's pinned custom feeds appended to the right. The user can swipe between feeds horizontally and the strip scrolls to keep the active tab visible.

This is a follow-up to the iOS parity audit in #0068.

## Scope

- **Convert the tab strip to a horizontal `ScrollView`** with `LazyHStack` containing one tab button per feed.
- **Built-in tabs in this order**: Following, Mentions, Discover, Popular With Friends. Match RN labels exactly.
- **User-pinned feeds** appended to the right, ordered by the user's saved-feed pin order from `app.bsky.actor.getPreferences`.
- **Selected-tab indicator**: a 2pt bottom underline in the brand color (matches RN). The full strip-background pill currently used in SwiftUI is dropped.
- **Auto-scroll the strip** to keep the active tab visible when switching tabs (e.g. via swipe gesture on the underlying feed pager).
- **Swipe-to-switch** the underlying feed pager — left/right swipe between adjacent tabs — using `TabView` with `.page` style or a manual `DragGesture`.

## Implementation notes

- Each tab content is a separate `FeedView` (or equivalent) bound to its feed identifier (`following`, `mentions`, `discover`, `popular-with-friends`, or a custom feed AT-URI).
- The Mentions feed is a server-side filter on the home timeline; check whether it has its own lexicon or is just a client-side filter.
- "Popular With Friends" — confirm the lexicon name (RN may use `app.bsky.unspecced.getPopular` or similar).
- Pinned-feed list comes from the same `SavedFeedsStore` used by #0073.
- Adding/removing pinned feeds in the My Feeds screen (#0073) should update this strip live.

## Root cause

The Home `FeedView` shipped with a fixed two-button `FeedSwitcherView`
(Following + Discover) plus a filter dropdown wedged into the same row.
There was no concept of "user-pinned feeds also live in the home tab
strip", no horizontal scroller, and no swipe-to-switch pager — every
new feed required tapping out to the My Feeds screen first.

## Fix

Replaces `FeedSwitcherView` with a horizontally-scrollable `LazyHStack`
of `HomeFeedTab` rows. Built-in tabs are the four-up Following ·
Mentions · Discover · Popular With Friends, followed by every saved
feed the viewer has pinned (`SavedFeed.pinned == true && type ==
"feed"`) in their preference-saved order. Pinned feeds whose URI matches
a built-in are skipped so the strip never shows duplicate Discover tabs.
The active tab carries a 2pt brand-color underline and the scroller
auto-centers it via `ScrollViewReader.scrollTo(_:anchor: .center)` on
every selection change.

Underneath the strip, `FeedView` now hosts a per-tab pager. iOS uses
`TabView(.page(indexDisplayMode: .never))` so left/right swipes step
between adjacent tabs with native gesture handling and per-page state
retention. macOS uses a `ZStack` of all feed containers with the active
one on top via `.opacity(...)` plus `.allowsHitTesting`/`.accessibilityHidden`
gates, which preserves per-tab scroll state across switches without
attempting page-style swipe (unsupported by macOS `TabView`).

`FeedView` now owns its own `SavedFeedsStore` instance and listens for
a new `Notification.Name.savedFeedsChanged`. `MyFeedsScreen` posts that
notification after a successful `putPreferences`, so pinning or
unpinning a feed in #0073 updates the home strip live without needing
to share an explicit store reference up the dependency tree. Per-tab
scroll position is retained because each tab's `FeedViewModel` lives in
the existing `FeedViewModelCache` and is created lazily on first
appearance — no view tear-down on tab change.

The filter dropdown that previously sat next to the strip is dropped at
this position with an inline `// TODO #0017` referencing the existing
filter feature; when the toggle controls return they should live under
each tab's overflow menu rather than the global header (RN parity).

## Files changed

- `BlueskyKit/Sources/BlueskyFeed/FeedSwitcherView.swift` — replaces the
  fixed two-tab implementation with a `HomeFeedTab` model and a
  scrollable strip that auto-centers the active tab. Adds AT-URI
  constants for the four built-in feeds.
- `BlueskyKit/Sources/BlueskyFeed/FeedView.swift` — rewires the home
  view around a per-tab pager (`TabView` `.page` on iOS, `ZStack` on
  macOS). Owns a `SavedFeedsStore` for the pinned-feed list and listens
  for `.savedFeedsChanged`. Drops the filter dropdown with a TODO.
- `BlueskyKit/Sources/BlueskyFeed/FeedViewModel.swift` — exposes the
  view model's `selection` publicly so `FeedView` can route the
  scroll-to-top notification only to the active tab's container.
- `BlueskyKit/Sources/BlueskyFeed/MyFeedsScreen.swift` — posts
  `.savedFeedsChanged` after `putPreferences` so the home strip
  re-loads pinned feeds without further glue.

## Gotchas

- **Mentions feed sourcing — third-party generator, not a lexicon.**
  AT Protocol has no first-party lexicon for "posts that mention the
  viewer's DID" and the React Native reference does not include a
  Mentions tab in the home strip at all (RN derives the home strip
  entirely from pinned saved feeds — see `usePinnedFeedsInfos`). The
  issue still asks for it as a built-in. The implementation routes it
  to the community-maintained
  `at://did:plc:wzsilnxf24ehtmmc3gssy5bu/app.bsky.feed.generator/mentions`
  generator (`flicknow / mentions`), which is on the AT Proto known
  authed-only list. The URI lives as a single constant on `HomeFeedTab`
  so it's a one-line swap if the generator goes down or a first-party
  lexicon ever lands. Errors from the generator surface in the per-tab
  error view; the existing FeedStore retry path covers them.
- **Popular-With-Friends sourcing — the bsky.app `with-friends` feed
  generator.** Same shape: routes to
  `at://did:plc:z72i7hdynmk6r22z27h6tvur/app.bsky.feed.generator/with-friends`
  via the standard `app.bsky.feed.getFeed` lexicon, since the
  `app.bsky.unspecced.getPopular` lexicon mentioned in the issue is
  unrelated (it returns popular *posts*, not a personalised
  popular-with-friends feed; that's a feed generator). Authed-only —
  logged-out viewers see an error message rather than a "Coming soon"
  placeholder; if RN parity for the empty-state copy matters more, the
  per-tab `errorView` is the place to swap.
- **Filter toggles moved to a TODO under #0017.** The
  hide-replies / hide-reposts dropdown that previously sat next to the
  strip is dropped — RN doesn't have it at this position — and the
  `FeedFilter` plumbing is left intact (each per-tab `FeedViewModel`
  still wires its filter from the cached value) so the controls can
  reappear under each tab's per-feed overflow menu without a backend
  rework. A `// TODO #0017` in `FeedView.body` flags the position.
- **iOS `TabView(.page)` pre-instantiates every page.** That means each
  tab's `.task(id:)` fires on first render and creates its
  `FeedViewModel` + `FeedStore` even if the user never swipes to it.
  Following reuses the boot-time pre-warmed timeline store so it's
  free; Mentions / Discover / Popular-With-Friends each start their own
  `loadInitial` in parallel. This is the cost of getting instant
  swipes; if startup network pressure is a concern, the eager-load
  could be gated behind a "first-swipe-only" flag, but the typical
  saved-feeds set is small and AT Proto rate limits accommodate the
  fan-out.
- **Pinned feed labels show the rkey until metadata resolves.** The
  strip uses the AT-URI's last path component as the tab label fallback
  because `FeedView` doesn't yet wire a `SuggestedFeedsStore.resolve`
  call to populate display names. This is a tiny visual papercut on
  first launch only — the My Feeds screen does resolve, and the next
  visit to that screen primes the resolution. A follow-up could push
  the resolved `GeneratorView` map into a shared store if the FOUC
  becomes annoying.
- **No Bluesky-SwiftUI app target changes were needed.** All wiring
  lives in the BlueskyKit / BlueskyFeed module — `MainTabView` already
  hosts `FeedView` for the iOS Home tab and the existing top bar
  (#0072) sits above it; the new strip slots into `FeedView`'s body
  unchanged.
