## Description

The React Native client has a dedicated **Thread Preferences** screen at Settings → Content & Media → Thread preferences (`Bluesky-ReactNative/src/screens/Settings/ThreadPreferences/`). It has:

- **Sort replies** — radio: **Hot** (engagement), **Top** (likes), **Newest**, **Oldest**, **Random**.
- **Group threads** — toggle: tree-style (grouped by sub-conversation) vs flat.
- **Hide replies by muted accounts**, **Hide replies by people you don't follow** — toggles.

These preferences ride in `app.bsky.actor.defs#threadViewPref` inside `app.bsky.actor.putPreferences`. The SwiftUI app has no Thread Preferences screen and the thread view (per audit feed) doesn't honor these prefs anyway.

## Steps to reproduce

1. Look in Settings for a "Thread preferences" entry.
2. Observe there is none; no way to choose reply sort or grouping.

## Expected behavior

A new `ThreadPreferencesScreen.swift` accessible from `ContentSettingsScreen` (or a new "Threads" hub). Fields:

- Sort replies (radio): Hot / Top / Newest / Oldest / Random.
- Group threads (toggle).
- Hide replies by muted accounts (toggle).
- Hide replies by people you don't follow (toggle).

State persists via preferences endpoint and is consumed by `ThreadView` (cross-cuts with #0140 / reply sort dropdown).

## Actual behavior

Screen does not exist.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Settings/ThreadPreferences.tsx`.
- Lexicon: `app.bsky.actor.defs#threadViewPref` (`sort: 'oldest' | 'newest' | 'most-likes' | 'random' | 'hotness'`, `prioritizeFollowedUsers: bool`).
- File to create: `BlueskyKit/Sources/BlueskySettings/ThreadPreferencesScreen.swift`.

## Root cause

The Thread Preferences screen had no SwiftUI counterpart. `ContentSettingsScreen` exposed only the Autoplay / External Embeds toggles, and `BlueskyCore` modeled neither the `app.bsky.actor.defs#threadViewPref` shape on the read side (`GetPreferencesResponse` had no `threadView` field) nor an encoder on the write side (`PutPreferencesRequest` had `adultContentEnabled`, `savedFeeds`, `interests`, and `birthDate` initializers but nothing that produced a `threadViewPref` payload). As a result there was no way to plumb the user's reply-sort + prioritize-followed-users choice through `getPreferences` / `putPreferences`.

## Fix

- **`BlueskyCore/Moderation.swift`** — added `ThreadViewPref` value type (`sort: String`, `prioritizeFollowedUsers: Bool`) with a `defaultPref` matching the appview default (`sort: "hotness"`, `prioritizeFollowedUsers: true`). Extended `GetPreferencesResponse`'s decoder to look for the `app.bsky.actor.defs#threadViewPref` item and surface it on a new optional `threadView: ThreadViewPref?` property; the inner decoder helper learned `sort` and `prioritizeFollowedUsers` keys. Added `PutPreferencesRequest(threadView:)` initializer that wraps the pref into the standard preferences-array payload, with a private `_ThreadViewPref` encoder writing `$type`, `sort`, `prioritizeFollowedUsers`.
- **`BlueskySettings/ThreadPreferencesScreen.swift`** — new file. Form-style screen with two sections:
  1. **Sort replies** — five tappable rows (Hot, Top, Newest, Oldest, Random) showing a checkmark on the active row. Mapped to lexicon values `hotness`, `most-likes`, `newest`, `oldest`, `random`.
  2. **Other** — *Prioritize people you follow* `Toggle`.
  Backed by `ThreadPreferencesViewModel` (`@Observable @MainActor`) that calls `getPreferences` on `.task` to seed `sort` + `prioritizeFollowedUsers`, and writes a `threadViewPref` via `putPreferences` on every change with optimistic update + rollback on failure. Errors surface as a one-shot SwiftUI `alert`.
- **`BlueskySettings/ContentSettingsScreen.swift`** — added an optional `network` parameter and a new "Threads" section containing a `NavigationLink` row "Thread preferences" (`bubble.left.and.bubble.right` icon) that pushes `ThreadPreferencesScreen`. The parameter is optional so the existing `#Preview` blocks compile unchanged; the row is hidden when no network is wired.
- **`BlueskySettings/SettingsScreen.swift`** — pass `network` through to `ContentSettingsScreen` so the Thread preferences row is reachable in production builds.

## Gotchas

- **Five-option radio vs. RN's three-option radio.** The live RN screen (`ThreadPreferences.tsx`) currently exposes only `top` / `oldest` / `newest` and collapses anything else to `top` via `normalizeSort`. The issue spec called for the full lexicon set (Hot / Top / Newest / Oldest / Random) and the `app.bsky.actor.defs#threadViewPref` schema admits all five, so we surface all five. RN's older flag set still round-trips: writing `hotness` or `random` from SwiftUI is a no-op for RN reads (RN's `normalizeSort` will fall back to "top" UI-side) but the value persists on the server unchanged.
- **Tree view + muted/non-followed reply hiding deferred.** The live RN screen also has a "Tree view" toggle (`lab_treeViewEnabled`, a labs flag) and the issue notes mention "Hide replies by muted accounts" / "Hide replies by people you don't follow". Those ride in different / unmodeled prefs so they were skipped per the bail rule. They should be filed as a #0116 follow-up if/when needed.
- **Cross-cuts with #0140 (reply-sort dropdown in `ThreadView`).** `GetPreferencesResponse.threadView` is now public, so when #0140 lands it can read the user's `sort` choice from `BlueskyKit`-level preferences and pre-select the matching dropdown option without re-doing the parsing.
- **Optimistic write on every change.** Unlike RN's debounced `useThreadPreferences` (2-second leading+trailing debounce), SwiftUI writes immediately on tap/toggle. Acceptable for now because the screen has only six controls and rapid toggling is unlikely; revisit if telemetry shows excessive `putPreferences` traffic.
