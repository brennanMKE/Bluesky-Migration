## Description

`ThreadView` flattens replies with `flattenThread()` and renders them in a single column at uniform indentation. The React Native client renders a tree:

- **Indentation** per nesting level (RN uses a `TREE_INDENT` constant).
- **Vertical connector lines** on the left edge of each row, joining a reply to its parent above.
- **"Show N more replies"** expanders for collapsed branches deeper than the default depth.
- **Avatar clustering**: the thinner left rail uses smaller avatars when nested.

Without this rendering, multi-branch threads collapse into a flat sequence and the conversation structure is lost.

## Steps to reproduce

1. Open a thread with replies that themselves have replies.
2. Compare to the same thread in RN.
3. Observe SwiftUI shows everything flat.

## Expected behavior

`ThreadView` recursively renders the reply tree:

- Each row's left edge has a vertical connector line drawn from its parent's row, indented by depth × `TREE_INDENT`.
- Branches deeper than ~2 levels collapse to a "Show N more replies" button; tapping expands inline.
- Indentation caps at a sensible max (RN uses ~6 levels) before flattening to keep readability on narrow screens.

## Actual behavior

`flattenThread()` returns a flat array; `ForEach` renders rows with no indentation, no connectors, no expanders.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/com/post-thread/ThreadItemTreePost.tsx` and the line-drawing helpers.
- File to edit: `BlueskyKit/Sources/BlueskyFeed/ThreadView.swift` — keep recursion via `AnyView` (already used to break self-reference). Add a `depth: Int` parameter to the recursive renderer and use it to drive horizontal padding + connector overlay.
- Connector lines: a `Rectangle().frame(width: 2)` overlay at the leading edge with absolute positioning works for SwiftUI; alternative is a custom `Shape`.

## Root cause

`ThreadView.flattenThread` walked the focal-post → ancestors → direct replies and emitted a flat `[FeedViewPost]` array — direct children of the focal only, with no awareness of nested replies, no per-row depth, and no indent/connector chrome. The view rendered each row through `PostCard` at uniform indent, so a tree of replies decoded by `app.bsky.feed.getPostThread` (with `depth=6` already on the wire) collapsed into a single column. There was also no `expandedNodes` state on `ThreadViewModel` to gate deep branches, and no SwiftUI counterpart to RN's `ThreadItemTreePostOuterWrapper` rails or the `ThreadItemReadMore` "Show N more replies" expander.

## Fix

- **`BlueskyKit/Sources/BlueskyFeed/ThreadViewModel.swift`** — added `expandedNodes: Set<ATURI>` plus a `toggleExpanded(_:)` mutator. The set is the source of truth for which deep subtrees the user has chosen to reveal past the indentation cap. Per-session only — re-loads keep the user's expansions until the screen is dismissed.
- **`BlueskyKit/Sources/BlueskyFeed/ThreadView.swift`** —
  - Added a `ThreadTree` constants enum mirroring RN's `TREE_INDENT` (= `tokens.space.lg` = 16), `TREE_AVI_WIDTH` (= 24), `REPLY_LINE_WIDTH` (= 2), and `TREE_VIEW_BELOW_DESKTOP` (= 6) `MAX_DEPTH` cap. `indentUnit` is `TREE_INDENT + TREE_AVI_WIDTH/2` exactly as RN computes per-rail width.
  - Replaced `flattenThread` with `linearRows(for:)` (ancestors + focal, flat) plus a recursive `replyRow(_:depth:focalURI:)` walker that returns `AnyView` to break the self-referential `some View` recursion (matches the existing `AnyView` pattern used elsewhere in this file).
  - The recursive walker takes a 1-based `depth: Int`. Indentation is `min(depth, MAX_DEPTH) × indentUnit`. At the cap, further nested replies stop indenting and a "Show N more replies" button appears; tapping it adds the parent URI to `viewModel.expandedNodes` and the subtree renders inline (still flat at the cap).
  - When the server reported more children than were hydrated (`post.replyCount > nestedReplies.count`), a "Show N more replies" leaf appears that pushes a new `ThreadView` for the parent so the user can drill in for the missing branch — the SwiftUI analogue of RN's `ThreadItemReadMore` link to `/post/<rkey>`.
  - The walker iterates `viewModel.sortedReplies` for the depth-1 entry (preserving the #0140 sort), and `tp.replies` for deeper levels (the appview returns each branch in its own intra-branch order).
  - Added a private `TreeReplyRow` view that lays out the parent-chain rails on the left (`depth - 1` thin `Rectangle`s tinted `theme.colors.border`) followed by a `PostCard`. The row's own connector hooking into its parent's rail is drawn as a SwiftUI `Path` (`ConnectorCornerShape`) — a vertical stem that curves into a horizontal stub via a quarter `addQuadCurve`. This mirrors RN's `borderLeftWidth + borderBottomWidth + borderBottomLeftRadius` corner without depending on platform-specific border drawing.
  - The "Show N more replies" button reuses `Spacing.lg`-derived indent math so the affordance lines up with the rail it descends from.

## Gotchas

- **Avatar size parity is partial.** RN uses `TREE_AVI_WIDTH = 24` for nested rows (vs 42 for the linear focal/ancestor rows). The SwiftUI tree row keeps `PostCard`'s built-in 44-pt avatar so we don't fork the card into "tree" and "linear" variants — the `treeAvatarSize` constant feeds the connector geometry, not the avatar itself. Shrinking the avatar would require either a new `PostCard` "compact" mode or a bespoke tree row body; deferred so the issue can land without a `PostCard` rewrite.
- **`MAX_DEPTH = 6` is the desktop cap.** RN ships two values — `TREE_VIEW_BELOW = 4` (mobile) and `TREE_VIEW_BELOW_DESKTOP = 6` (desktop / wide). Since the SwiftUI app is currently desktop-first (Bluesky-SwiftUI on macOS), we use 6. If/when iPhone-narrow-width rendering needs the lower bound, switch on `horizontalSizeClass`.
- **Two flavors of "Show N more replies".** When the data is at the cap and the model has more *loaded* children, tapping expands inline (toggles `expandedNodes`). When the data is hydrated *but the server promised more* (`replyCount > replies.count`), tapping pushes a new `ThreadView` rooted at that parent so the deeper subtree gets its own `getPostThread` fetch. Mirrors RN's `ThreadItemReadMore` which always links out (no inline expand path) — we expand inline at the cap because we already have the subtree client-side.
- **`AnyView` is required.** SwiftUI's `some View` opaque return types cannot be self-referential, so the recursive `replyRow` returns `AnyView`. The pattern was already used in this file for similar reasons; no new performance concern beyond what RN's tree already incurs.
- **Connector approximation.** The corner is a `Path` using `addQuadCurve` rather than RN's `borderLeftWidth + borderBottomLeftRadius`. The visual difference at 6-pt radius and 2-pt stroke is below noticeable; using a `Shape` keeps the rendering identical across iOS and macOS without relying on platform-specific border drawing semantics.
