## Description

When multiple images are attached to a post, the React Native composer's `Gallery` lets the user drag-and-drop them to reorder before posting. The SwiftUI composer renders a fixed 2-column grid (`imageGrid`) with no reorder affordance. The order is effectively the pick order, and the user has to remove + re-pick to change it.

## Steps to reproduce

1. Open the SwiftUI composer.
2. Attach 4 images.
3. Long-press / drag any thumbnail to reorder.
4. Observe the gesture is ignored.

## Expected behavior

The image grid supports reorder via a drag interaction. On iOS, long-press to lift, then drag to a new slot; on macOS, drag-and-drop. Reordering updates the post's image array so embed order matches what the user sees.

For accessibility, an explicit "Move left" / "Move right" affordance (or up/down arrows in the per-image action sheet) is acceptable as a parallel path.

## Actual behavior

Images stay in pick order. No drag, no reorder.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/com/composer/photos/Gallery.tsx`.
- File to edit: `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` (the `imageGrid` ForEach around lines 317‚Äì338).
- Implementation: `LazyVGrid` with `.onDrag` / `.onDrop` and `UTType.image` payload, or wrap in a custom `OrderedImageGrid` view that tracks gesture-based offsets.

## Root cause

The composer's image grid was a plain `LazyVGrid` with no drag affordance and no per-cell reorder action. The image array on `ComposerViewModel` only exposed `addImage` / `removeImage`, so the on-screen order matched the pick order with no way to permute it before posting. Embed order at post time follows the array, so users had to remove and re-pick images to change order.

## Fix

Two parallel reorder paths, in line with the issue's accessibility note:

1. **Drag-and-drop** ‚Äî every grid cell is now both an `.onDrag` source (carrying its UUID via `NSItemProvider`) and an `.onDrop` target wired to a new `ImageReorderDropDelegate`. The delegate swaps the dragged cell into the hovered cell's slot live during the drag, with a `scaleEffect(1.05)` + `opacity(0.7)` lift on the source. A second catch-all `.onDrop` on the grid container clears the drag state if the user drops outside any cell, so the lift effect always resets.
2. **Context menu** ‚Äî each cell exposes "Move Left", "Move Right", "Edit Alt Text", and "Remove" via `.contextMenu`. Move actions are disabled at the array boundaries so the affordance is self-describing.

`ComposerViewModel` gained `moveImage(from:to:)` (SwiftUI `move(fromOffsets:toOffset:)` semantics with bounds-clamping), plus `moveImageEarlier(id:)` / `moveImageLater(id:)` convenience swappers used by the context menu.

Draft persistence was already array-driven, so the new order naturally round-trips through `DraftsStore` without further changes.

## Gotchas

- `.onDrag` has no native "drag cancelled" callback, so a stray drop in empty space wouldn't normally clear `draggedImageID`. The catch-all `ImageReorderCancelDelegate` on the grid container handles that case.
- The drop delegate matches by the bound `draggedID`, not by reading the `NSItemProvider` payload ‚Äî async provider reads would defer the swap past `dropEntered`, defeating the live-preview reorder.
- `moveImage(from:to:)` follows SwiftUI's `move(fromOffsets:toOffset:)` convention where a destination beyond the source means "after the destination element"; the helper subtracts one to compensate so callers can pass a simple "swap into this slot" index.
