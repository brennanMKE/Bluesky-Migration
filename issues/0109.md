# 0109 — Message thread has no scroll-to-bottom button or new-messages divider

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskyMessages |
| **Platform** | All |
| **First seen** | 2026-05-06 |

## Description

When a user scrolls up through history in a message thread and a new message arrives, the React Native client:

1. **Does not auto-scroll** if the user is reading older messages — instead it shows a floating **"Jump to newest"** pill (FAB-style) anchored to the bottom-right.
2. Renders a **"New messages"** centered divider above the first unread message so it's clear where you left off.

The SwiftUI implementation auto-scrolls to the bottom on every new message regardless of where the user is reading, which yanks them out of context. There's no FAB and no divider.

## Steps to reproduce

1. Open a long thread; scroll up several screens into history.
2. Have a new message arrive.
3. Observe the SwiftUI app jumps you back to the bottom.

## Expected behavior

- Track scroll position (a `ScrollView` `GeometryReader` proxy or `scrollPosition(id:)` API).
- If the user is at-bottom (within ~50pt), auto-scroll to bottom on new message — current behavior.
- If the user is scrolled up, do **not** auto-scroll. Instead, show a floating "Jump to newest" pill in the bottom-right corner (small rounded background, chevron-down icon, optional unread count). Tapping the pill scrolls to bottom and dismisses itself.
- Above the first message that arrived since the user last marked-read, render a centered **"New messages"** divider (1pt line + small text label).

## Actual behavior

Always auto-scrolls. No FAB. No divider.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Messages/components/MessagesList.tsx` and `NewMessagesPill.tsx`.
- File to edit: `BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift` (lines ~62–88 — replace the unconditional `proxy.scrollTo(latest)` with an at-bottom check).
- `MessageThreadViewModel` may need a `lastReadMessageID` to anchor the divider.
