## Description

When a user scrolls up through history in a message thread and a new message arrives, the React Native client:

1. **Does not auto-scroll** if the user is reading older messages — instead it shows a floating **"Jump to newest"** pill (FAB-style) anchored to the bottom-right.
2. Renders a **"New messages"** centered divider above the first unread message so it's clear where you left off.

The SwiftUI implementation auto-scrolls to the bottom on every new message regardless of where the user is reading, which yanks them out of context. There's no FAB and no divider.

## Steps to reproduce

1. Open a long thread; scroll up several screens into history.
2. Have a new message arrive.
3. Observe the SwiftUI app jumps you back to the bottom.

## Expected behavior

- Track scroll position (a `ScrollView` `GeometryReader` proxy or `scrollPosition(id:)` API).
- If the user is at-bottom (within ~50pt), auto-scroll to bottom on new message — current behavior.
- If the user is scrolled up, do **not** auto-scroll. Instead, show a floating "Jump to newest" pill in the bottom-right corner (small rounded background, chevron-down icon, optional unread count). Tapping the pill scrolls to bottom and dismisses itself.
- Above the first message that arrived since the user last marked-read, render a centered **"New messages"** divider (1pt line + small text label).

## Actual behavior

Always auto-scrolls. No FAB. No divider.

## Root cause

`MessageThreadScreen.messageScrollView` had a single unconditional `.onChange(of: viewModel.messages.count)` handler that always called `proxy.scrollTo(last.id, anchor: .bottom)`. There was no notion of where the user was currently parked in the scroll view, so any incoming message yanked them back to the latest. There was also no FAB and no divider — the screen had no UI at all to surface "you have unread messages above your current scroll position".

## Fix

Three pieces of plumbing were added.

1. **At-bottom tracking.** A zero-height sentinel row is appended to the bottom of the `LazyVStack`, and a `GeometryReader` reports its `maxY` (in a named scroll coordinate space) up via a `BottomSentinelOffsetKey` `PreferenceKey`. A second `PreferenceKey` reports the scroll viewport height. The screen compares the two: if the sentinel sits within `Self.atBottomThreshold` (50pt) of the viewport bottom, `isAtBottom` is true. This avoids depending on iOS-17-only `scrollPosition(id:)` and works on the project's iOS 18 / macOS 15 baseline.

2. **Conditional auto-scroll.** `handleMessageCountChange` replaces the unconditional scroll. New behavior:
   - If the count went down (deletion / older-page no-op), do nothing.
   - If `isAtBottom` (or the new last message was sent by the viewer), scroll to the bottom sentinel — RN parity: own messages always pull the viewer down so the input doesn't appear to swallow what was sent.
   - Otherwise, leave the position alone and call `viewModel.noteNewUnread(messageID:)` so the divider/pill appear.

3. **Pill + divider UI.**
   - `JumpToNewestPill` is a small chevron-down capsule with `.regularMaterial` background, optional unread-count badge, and a soft shadow. Anchored to the bottom-right of the message scroll view inside a `ZStack`, with an opacity+slide transition.
   - `NewMessagesDivider` is a thin accent-tinted horizontal line with a centered "New messages" label. Rendered inline inside the `LazyVStack` directly above the bubble whose `id == viewModel.firstUnreadID`, so it lives in the message stream as a real row (not an overlay).
   - `MessageThreadViewModel` gains `firstUnreadID`, `unreadCount`, `noteNewUnread(messageID:)`, and `clearUnread()`. The screen calls `clearUnread()` when the user scrolls back to the bottom or taps the pill.

## Gotchas

- The implementation deliberately uses `GeometryReader` + `PreferenceKey` instead of iOS 17's `scrollPosition(id:)`. The latter would have been simpler but reports identifier transitions, not pixel offsets, so a 50pt at-bottom threshold cannot be expressed cleanly. The chosen approach also works identically on macOS, which `scrollPosition(id:)` historically has rough edges on.
- `lastAutoScrolledMessageID` guards against double-firing the scroll for the same final message ID. Without it, an optimistic-then-replaced send (where the count changes but the trailing ID stays stable) could re-trigger animation while the viewer was reading older content.
- Own-sent messages always scroll the viewer to the bottom regardless of `isAtBottom`. This matches RN's behavior in `MessagesList.tsx`'s `onSendMessage` path — sending implies "I want to see what I just sent". The unread divider is also cleared in that case so the viewer doesn't see "New messages" above their own outgoing bubble.
- The pill is placed inside the same `ZStack` as the scroll view, not the screen-level `VStack`, so it sits above the message list but below the compose bar — same z-ordering as RN's `NewMessagesPill` which renders above the keyboard sticky input.
- Unread count is currently inferred locally from message-count deltas (count it when a new message arrives while scrolled away). The conversation lexicon also exposes a server-side unread count, which is wired through `ConvoView` for the inbox list but is not used here — local tracking is sufficient for the in-thread divider/pill, and matches RN's `setNewMessagesPill({show: true, …})` which is also a purely local signal.
