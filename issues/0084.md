## Description

The profile info row (stats and URL) has four small differences from the React Native reference. Bundling because they all touch the same row layout and ship as one focused PR.

This is a follow-up to the iOS Profile parity audit in #0070.

## Scope

### 1. Stats order

- **Today**: `359 Following · 286 Followers · 1,424 Posts` — Following first.
- **RN**: `286 followers · 359 following · 1.4K posts` — Followers first.
- **Fix**: swap Following and Followers in the stats row.

### 2. Label casing

- **Today**: title-case ("Following", "Followers", "Posts").
- **RN**: lower-case ("following", "followers", "posts").
- **Fix**: lowercase the labels.

### 3. Count abbreviation

- **Today**: full-precision counts with group separators (`1,424`).
- **RN**: abbreviated counts (`1.4K`, `2.3M`, etc.).
- **Fix**: format counts using a compact-style number formatter (`NumberFormatter.Style.decimal` with `usesGroupingSeparator = false` plus a manual K/M/B suffix helper, or `Measurement`-based formatting). Threshold: show full precision below 1,000; `1.4K` between 1K–999K; `1.4M` above 1M.

### 4. URL display strips protocol

- **Today**: `https://brennan.sstools.co` rendered with the protocol prefix.
- **RN**: `brennan.sstools.co` — protocol stripped for display, but the link still opens with the full URL.
- **Fix**: strip the protocol prefix when rendering, retain it when handling the tap.

## Implementation notes

- Code lives in `BlueskyKit/Sources/BlueskyProfile/ProfileScreen.swift` (or its header view).
- The compact-number formatter belongs in `BlueskyUI` — extract `CompactNumberFormatter.string(from: Int)` and reuse it on follower/following counts wherever they appear in the app (also used in feed action bar repost/like counts under #0076).
- URL protocol stripping: a tiny helper `URL.displayString` that drops `https://` / `http://` and the trailing `/` if present.

## Acceptance

- Stats row reads `<followers> followers · <following> following · <posts> posts`.
- Counts are abbreviated as K/M for >= 1,000.
- URL renders without the protocol prefix; tapping still navigates to the full URL.
- iOS Simulator and macOS builds pass.

## Root cause

Four independent polish gaps in the SwiftUI Profile screen against the RN reference, all rooted in `ProfileHeaderView.statsRow` and the description text rendering. (1) The stats row laid out `Following · Followers · Posts`, inverting RN's followers-first order. (2) Labels were title-cased ("Following") rather than RN's lowercase ("following"). (3) Counts were `String(format)`-style integers with group separators (`1,424`) instead of compact ("1.4K") — the formatter for it already shipped in `BlueskyUI` as `CompactNumberFormatter` (#0076) but the profile stat view never picked it up. (4) The profile description rendered as a plain `Text(desc)` with no URL detection, so a bio containing `https://brennan.sstools.co` showed the full string verbatim, schema and all.

## Fix

All changes ship in `BlueskyKit` and are picked up by the app via the local Swift package — no `Bluesky-SwiftUI` change was needed.

1. **Stats order, casing, abbreviation.** `ProfileHeaderView.statsRow` reorders to `followers · following · posts` and lowercases the labels. `statView(count:label:)` now formats the count with `CompactNumberFormatter.string(from:)` so 1,424 reads "1.4K" and small counts (< 1,000) keep full precision per the formatter's existing thresholds.
2. **URL protocol stripping.** New `URL+DisplayString.swift` in `BlueskyUI` adds a `URL.displayString` computed property (and a sibling `String.urlDisplayString`) that drops `http://` / `https://` and a single trailing `/`. `ProfileHeaderView` now feeds the description through a new `profileDescriptionAttributed(_:)` that runs `NSDataDetector` over the raw text, replaces each URL match's display text with `displayString`, colors the run in `theme.colors.link`, and pins the underlying link to the full URL so taps still navigate correctly. Plain (non-URL) runs of the description stay untouched.

## Files changed

- `BlueskyKit/Sources/BlueskyProfile/ProfileHeaderView.swift` — stats row reorder + lowercase labels, count via `CompactNumberFormatter`, new `profileDescriptionAttributed(_:)` URL-detecting renderer.
- `BlueskyKit/Sources/BlueskyUI/URL+DisplayString.swift` (new) — `URL.displayString` and `String.urlDisplayString` helpers.

## Gotchas

- **`URL.displayString` lives in `BlueskyUI`** alongside `CompactNumberFormatter` and `RelativeTimeFormatter`. Reuse from any feature module that needs the same protocol-stripped display treatment (e.g. external link cards, embed previews).
- **Profile descriptions don't ship rich-text facets** in `ProfileDetailed`, so URL detection runs locally with `NSDataDetector`. If the lexicon is later extended to expose description facets, swap the detector pass for a facet-driven render the way `RichTextView` already does.
- **Trailing slash trim is single-character only** — multi-trailing-slash inputs (`https://example.com//`) keep one slash. That mirrors RN's behaviour and avoids accidentally collapsing path segments.
