## Description

The Notifications screen maps a notification's `reason` string to a human-readable phrase ("liked your post", "followed you", etc.). The current mapping covers six reasons; the AT Proto notification API surfaces several more, and unrecognized reasons fall through a `default` case that prints the raw `reason` string. Most visibly, `subscribed-post` notifications show the literal text `"subscribed-post"` instead of a real label.

## Steps to reproduce

1. Subscribe to another user's posts (notifications-on toggle on a profile).
2. Wait for that user to publish a new post.
3. Open the Notifications tab and find the row for that activity.

## Expected behavior

Every reason returned by `app.bsky.notification.listNotifications` is mapped to a localized, human-readable phrase that matches the React Native client's wording. At minimum, all reasons defined in `Bluesky-ReactNative/src/state/queries/notifications/types.ts` should be handled:

| Reason | Suggested label |
|---|---|
| `like` | "liked your post" *(already mapped)* |
| `repost` | "reposted your post" *(already mapped)* |
| `follow` | "followed you" *(already mapped)* |
| `mention` | "mentioned you" *(already mapped)* |
| `reply` | "replied to your post" *(already mapped)* |
| `quote` | "quoted your post" *(already mapped)* |
| `like-via-repost` | "liked your repost" |
| `repost-via-repost` | "reposted your repost" |
| `feedgen-like` | "liked your custom feed" |
| `starterpack-joined` | "joined via your starter pack" |
| `subscribed-post` | "new post from {actor}" (special: actor name leads, not the verb) |
| `contact-match` | "is in your contacts" |
| `verified` | "verified you" |
| `unverified` | "removed your verification" |

Reference: `Bluesky-ReactNative/src/view/com/notifications/NotificationFeedItem.tsx` (around lines 100–170 has the full set with translations).

## Actual behavior

`NotificationsScreen.swift:202-212`:

```swift
private var reasonText: String {
    switch group.reason {
    case "like":    return "liked your post"
    case "repost":  return "reposted your post"
    case "follow":  return "followed you"
    case "mention": return "mentioned you"
    case "reply":   return "replied to your post"
    case "quote":   return "quoted your post"
    default:        return group.reason
    }
}
```

`subscribed-post` falls through to `default` and the raw string is rendered.

## Steps to fix

1. Extend the `switch` in `reasonText` to cover the reasons listed above.
2. Add corresponding cases to `reasonIcon` (NotificationsScreen.swift:158) and the icon-color helper (NotificationsScreen.swift:178). RN uses distinct icons (heart, repeat, person-plus, at-sign, etc.) — pick SF Symbols equivalents.
3. `subscribed-post` is structurally different from the others (the actor *led* the post; there's no verb against the user). Phrase it as "new post" with the actor as subject rather than object — see RN's `Trans>New posts from …</Trans>` pattern.
4. Add a unit test (or a SwiftUI preview gallery) that fans out across every reason so future regressions are visible.

## Root cause

`NotificationsScreen.GroupedNotificationRow` had three small switches
keyed on `group.reason` — `reasonText`, `iconName`, and `iconColor` —
each covering only the original six reasons (`like` / `repost` /
`follow` / `mention` / `reply` / `quote`). Every other reason
(`subscribed-post`, `like-via-repost`, `repost-via-repost`,
`feedgen-like`, `starterpack-joined`, `contact-match`, `verified`,
`unverified`) fell through to a `default` that returned the raw AT
Proto string, so users saw labels like "subscribed-post" or
"feedgen-like" in plain text instead of a human-readable phrase.

## Fix

Extended all three switches in
`BlueskyKit/Sources/BlueskyNotifications/NotificationsScreen.swift` to
cover every reason listed in the issue's mapping table:

- **`reasonText`** — added "liked your repost", "reposted your repost",
  "liked your custom feed", "joined via your starter pack", "shared a
  new post", "is in your contacts", "verified you", "removed your
  verification".
- **`iconName`** — `bell.badge.fill` for `subscribed-post`,
  `person.2.fill` for `starterpack-joined`,
  `person.crop.circle.badge.checkmark` for `contact-match`,
  `checkmark.seal.fill` for `verified`, `xmark.seal.fill` for
  `unverified`. `like-via-repost` and `feedgen-like` share the
  `heart.fill` glyph; `repost-via-repost` shares
  `arrow.2.squarepath`.
- **`iconColor`** — pink for any like variant, green for any repost
  variant, blue for follow / starterpack-joined / contact-match /
  verified, orange for `unverified`, accent for `subscribed-post`.

The `subscribed-post` phrasing ("shared a new post") fits the existing
row layout — the actor name already leads on its own line above
`reasonText`, so the verb-after-actor pattern reads naturally without
restructuring the row.

## Gotchas

- RN's `subscribed-post` copy is "new post from {actor}", which leads
  with the actor name. The SwiftUI row already separates the actor
  summary onto its own line above the verb, so "shared a new post"
  reads correctly without changing the row structure. If a future row
  redesign drops the actor-led line, this label needs to be reworded.
- The `iconColor` switch deliberately has two `case … return .blue`
  arms because `verified` is logically distinct from the
  follow-family even though it shares the colour — splitting the
  cases keeps later tint tweaks (e.g. making `verified` a different
  blue) a one-line change.
- The fallback `default:` arms are kept for forward-compatibility
  with reasons the AT Proto API may add — they fall through to the
  generic bell glyph and the raw reason string, which is still
  better than crashing on an unknown case.
