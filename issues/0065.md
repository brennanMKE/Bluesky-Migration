# 0065 — Module 14 gate (iOS): settings persistence on iPhone

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskySettings |
| **Platform** | iOS |
| **First seen** | 2026-05-05 |

## Description

Once the iOS build is restored (#0061), the Module 14 settings-persistence gate must be re-run on a real iPhone. Most settings store via `UserDefaultsPreferencesStore` and are platform-agnostic, but the Appearance behavior diverges from macOS — on iOS the app does not yet defer to the system appearance the same way, and the Settings UI surfaces some controls that don't apply on macOS (haptics, in-app browser, etc.). Each persistence path needs to round-trip through a force-quit + relaunch cycle on iOS.

This issue split off from #0037, which now focuses on the macOS gate.

## Steps to reproduce

1. Resolve #0061 so the iOS build is green.
2. Install the app on a registered iPhone (or run in iOS Simulator) and sign in.
3. Run the live checks below.

## Live checks (must all pass)

- **iOS appearance.** Toggle iOS-wide appearance (Control Center → brightness long-press, or Settings → Display & Brightness → Light/Dark) and confirm Bluesky updates immediately. If iOS provides an in-app override picker, confirm the choice persists across relaunch and that "Use System" returns to the system setting.
- **Font size.** Drag the Font Size slider; force-quit Bluesky from the app switcher; relaunch; confirm the slider is still at the chosen value and text in the feed reflects it.
- **Notifications toggles.** Settings → Notifications: toggle Likes / Reposts / Follows / Mentions / Replies / Quotes. Force-quit and relaunch. Confirm each toggle persists.
- **Content & Media + Accessibility.** Toggle at least one option in each pane (e.g. autoplay video, reduce motion). Force-quit and relaunch. Confirm both survive.
- **Languages.** Change the post-language selection. Force-quit and relaunch. Confirm it persists.
- **Haptics (iOS-only).** If a haptics toggle exists, flip it; confirm haptic feedback on actions changes accordingly; relaunch and confirm the toggle state survived.
- **In-app browser preference (iOS-only).** If the "open links in app" toggle exists, flip it, tap an external link, confirm it opens in the configured surface (SFSafariViewController vs. system Safari); relaunch and confirm the toggle persists.

## Expected behavior

All seven checks pass. iOS-specific controls are present where appropriate and store correctly via `UserDefaultsPreferencesStore`. iOS appearance behavior matches the user expectation (follow system or honor in-app override).

## Actual behavior

Cannot be validated until #0061 is resolved.
