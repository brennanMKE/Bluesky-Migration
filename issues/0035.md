## Description

The Module 12 gate — confirm posting with plain text, image attachments, and @mentions all work end-to-end — has not been validated against the live API.

## Steps to reproduce

1. Open the composer and type a post with plain text. Submit and confirm it appears in your feed.
2. Attach one or more images. Submit and confirm the post appears with images.
3. Type an @mention. Confirm the autocomplete overlay appears. Select a suggestion. Submit and confirm the mention is a live link in the post.

## Expected behavior

All three post types submit successfully and appear correctly in the feed.

## Actual behavior

Not validated. See also issue 0034 for missing composer features.

## Live testing observations (2026-04-30)

**Text posting works** — plain text posts submit successfully and appear in the feed. ✓

**Image picker missing** — there is no visible button to attach an image in the composer UI on macOS. The `PhotosPicker` added in issue #0034 may not be rendering on macOS (it's iOS-only via `#if os(iOS)`), leaving no way to attach images on macOS.

**Composer text not cleared after posting** — reopening the composer after a successful post shows the previous post's text still in the field. The draft-restore logic from #0034 may be restoring the draft even after a successful post, or `clearDraft()` is not being called correctly on submit.

## Incomplete reason

Validation gate — image attachment on macOS and draft clearing after post need fixes before this gate can pass.

## Notes

### 2026-05-05 — Code blockers addressed (BlueskyKit `c826300`)

Both code-side blockers identified in the 2026-04-30 testing notes have been fixed in `BlueskyKit`:

1. **macOS image picker added.** `ComposerSheet.mediaToolbar` now exposes an "Add image" button on macOS (parallel to iOS) backed by an `NSOpenPanel` (`pickImagesMac()` in `ComposerSheet.swift`). Allows multi-selection up to the 4-image cap, accepts JPEG/PNG/HEIC/GIF/WebP, and infers MIME type from file extension via `mimeType(forImageExtension:)`.

2. **Draft no longer leaks across sessions after a successful post.** `ComposerViewModel.post()` now clears `text`, `images`, `additionalPosts`, `attachedVideo`, link state, and mention state immediately after `store.didPost == true`. Previously `clearDraft()` removed the UserDefaults key, but the sheet's `.onChange(of: text)` / `.onDisappear { saveDraft() }` hooks then re-persisted the just-posted text on the way out, so the next composer open restored the old post.

Both BlueskyKit (`swift build`) and the macOS app (`xcodebuild ... -destination 'platform=macOS'`) build clean.

### What still requires live human validation (cannot auto-validate)

The Module 12 gate is fundamentally a hands-on validation against the live AT Proto API. Bailing on the gate itself; please run the following manually and update this issue:

- Plain-text post submits and appears in the feed (was confirmed working 2026-04-30; re-confirm after the draft-clear change).
- Image post on **macOS**: open composer, click "Add image", pick 1–4 images, optionally add alt text, submit, confirm the post appears in the feed with images.
- Image post on **iOS** (regression check): existing iOS `PHPicker` flow still works.
- @mention autocomplete: type `@` plus a few characters, confirm suggestion overlay, select one, confirm the posted mention is a live link to that profile.
- Draft clearing: post successfully, immediately reopen the composer, confirm the text field is empty (not the previous post's body).
- Reply draft scoping: open a reply composer, type text without posting, dismiss; reopen the same reply — draft restores. Open the root composer — text field is empty (drafts are keyed per-`replyTo.uri`).

Once those pass end-to-end, mark this issue resolved with the BlueskyKit commit `c826300` plus any follow-up commits.
