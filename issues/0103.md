## Description

The React Native composer lets a video attachment carry both alt text *and* a captions track (WebVTT). RN's `SubtitleDialogBtn` opens a `SubtitleDialog` where the user can upload a `.vtt` file (or paste captions inline) and the captions blob is uploaded alongside the video and referenced from `app.bsky.embed.video.captions`.

The SwiftUI composer supports alt text on videos but has no captions UI. A SwiftUI-posted video cannot carry captions, which fails accessibility parity with RN.

## Steps to reproduce

1. Open the SwiftUI composer, attach a video.
2. Look for a "Captions" / "Subtitles" affordance next to the alt-text editor.
3. Observe there is none.

## Expected behavior

Per video, in addition to the alt-text editor:

- A "Captions" button that opens a sheet.
- The sheet supports: pick a `.vtt` file from Files, or paste inline VTT text into a `TextEditor`.
- Captions count appears next to the button (e.g. "Captions (en)" once attached).
- On post submit, the captions text is uploaded as a blob and referenced in `app.bsky.embed.video.captions[]` with the language code (default to the post's primary language).

## Actual behavior

No captions UI. Videos post with alt text only.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/com/composer/videos/SubtitleDialog.tsx`.
- Lexicon: `app.bsky.embed.video#caption` (`lang: string`, `file: blob`). Confirm `BlueskyCore`'s `EmbedVideo` has the `captions` field; if not, add it.
- File to edit: `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` (the video preview block around lines 340‚Äì369), and `ComposerViewModel.swift` (extend `uploadVideoBlob` to also upload captions blobs).

## Root cause

The `VideoAttachment` model only carried `data` / `mimeType` / `altText`; there was no field for a WebVTT track and no UI surface to attach one. `ComposerStore.post` always built `EmbedVideo` with `captions: nil` and `alt: nil`, so even the alt-text field on the model went unused on the wire. The video preview block in `ComposerSheet.swift` rendered a plain "Video attached" placeholder with only a remove button ‚Äî no alt-text editor, no captions affordance.

## Fix

- `VideoAttachment` (in `ComposerStore.swift`): added `captionsText: String?` (raw VTT) and `captionsLang: String` (defaulting to `"en"`). Mirrors RN's `media.video.captions[]` deferred-upload model.
- `ComposerStore.post`: when `attachedVideo` carries non-blank captions text, encodes UTF-8 bytes and uploads via `network.upload(lexicon: "com.atproto.repo.uploadBlob", ..., mimeType: "text/vtt")`, then attaches a single-entry `[VideoCaption(lang:file:)]` to `EmbedVideo`. Also wires `video.altText` through to `EmbedVideo.alt` (was previously dropped on the floor). Captions upload failures are best-effort (logged, don't fail the post) ‚Äî matches RN's per-file `try/catch` in `uploadCaptions`.
- `ComposerSheet.swift` video preview block: now stacks the placeholder over an affordance row carrying both an alt-text button and a captions button. Alt text uses the same compact popover the image cell uses ("Describe this video‚Ä¶" `TextField` + Done). Captions opens a new `VideoCaptionsSheet` (sheet on iOS, popover on macOS ‚Äî matches the labels/threadgate/GIF picker pattern).
- New `VideoCaptionsSheet` private view: `Form` with a language picker, a "Select caption file (.vtt)" button (`.fileImporter` accepting `.vtt`/`.text`/`.plainText`), a multiline monospaced `TextEditor` for inline paste, and a "Remove captions" destructive button when content is present. Save/Cancel toolbar items. Reads imported files behind a security-scoped resource scope; falls back to `isoLatin1` on a UTF-8 decode failure.
- Affordance labels reflect state: `Alt text` / `Alt text ‚úì`, and `Captions` / `Captions (en)`.

## Gotchas

- This issue closes the cliff that #0101 left open. #0101 added `VideoAttachment.altText` so the require-alt-text gate could apply uniformly to videos, but the composer still shipped no UI for editing it ‚Äî the field was effectively unreachable from the user's side. The video preview block now ships both editors (alt text + captions) in a single affordance row, so a user can satisfy the alt-text-required gate on a video and attach captions in the same flow. The previous `EmbedVideo(... alt: nil ...)` write has also been corrected to thread `video.altText` through, so the alt actually lands on the post record.
- `BlueskyCore.EmbedVideo` already had the `captions: [VideoCaption]?` field; no Layer 0 change was needed.
- RN's `SubtitleDialog` shows the captions list inline next to the alt-text editor only on web (`IS_WEB`). On native, RN hides captions entirely behind a hint that says "Captions are added on the web client". We diverge: SwiftUI exposes captions on both iOS and macOS because the `.fileImporter` API works equivalently on both ‚Äî there's no native-platform reason to hide it. (Flagging the divergence per CLAUDE.md.)
- The captions upload uses the same `text/vtt` MIME the AT Proto reference clients use. The `.fileImporter` accepts `text/plain` as a fallback for hosts that mislabel the MIME, mirroring RN's web `endsWith('.vtt')` permissiveness.
- A single track is supported for now (RN caps at `MAX_NUM_CAPTIONS = 1`). The lexicon array shape leaves room to add multi-language support later without a record migration.
