## Description

The React Native moderation hub includes a **Muted Words & Tags** entry that opens a screen (or dialog) where users curate a list of strings to filter from feeds. Each entry can target tag-only or text-anywhere matches, and can target specific timeframes (forever / 24h / 7d / 30d) and contexts (content vs tags). The SwiftUI moderation surface has no equivalent — there is no way to mute a specific word, hashtag, or domain.

This is one of the most-used moderation tools in RN and its absence is a significant gap.

## Steps to reproduce

1. Open the SwiftUI Moderation hub.
2. Look for "Muted Words" / "Muted Tags".
3. Observe neither row exists.

## Expected behavior

A new "Muted words & tags" row in the Moderation hub. Tapping opens `MutedWordsScreen.swift`:

- Add new entry: text field, target picker (Content + tags / Tags only / Content only), duration picker (Forever / 24h / 7d / 30d).
- List of existing entries with edit and delete actions.
- Persists via `app.bsky.actor.putPreferences` under `app.bsky.actor.defs#mutedWordsPref`.
- Feed-side filtering: `FeedViewModel` (or a higher-up filter) drops posts matching any non-expired muted word.

## Actual behavior

No screen, no preference handling, no filtering.

## Notes

- RN reference: `Bluesky-ReactNative/src/components/dialogs/MutedWords.tsx` and `state/queries/preferences/muted-words.ts`.
- Lexicon: `app.bsky.actor.defs#mutedWord` and `#mutedWordsPref`.
- Files to create: `BlueskyKit/Sources/BlueskyModeration/MutedWordsScreen.swift`. Filter helper to add to `BlueskyFeed`.

## Root cause

The Moderation hub had no entry point for managing muted words and tags, and `BlueskyCore` did not model `app.bsky.actor.defs#mutedWord` / `#mutedWordsPref`. `GetPreferencesResponse` ignored the `mutedWordsPref` items returned by `app.bsky.actor.getPreferences`, and `PutPreferencesRequest` had no initializer to write them back. RN ships the same UX as a modal dialog (`MutedWords.tsx`); the SwiftUI app needed an equivalent screen.

## Fix

- **Lexicon (BlueskyCore).** Added `MutedWord` (`id`, `value`, `targets`, `actorTarget`, `expiresAt`) plus `isExpired(now:)`. Extended `GetPreferencesResponse` with a `mutedWords: [MutedWord]` field, parsing the `mutedWordsPref` body alongside the existing `savedFeedsPrefV2` (both share an `items:` JSON key — a custom decoder picks the right helper based on `$type`). Added `PutPreferencesRequest(mutedWords:)` initializer that writes the full list back via a private `_MutedWordsPref` encoder.
- **New screen.** `BlueskyModeration/MutedWordsScreen.swift` is a `Form`-based settings-style screen pushed from the moderation hub. Top section is the add form: text field, Duration picker (Forever / 24 hours / 7 days / 30 days), Mute-in picker (Text & tags / Tags only — emits `["tag", "content"]` or `["tag"]` to mirror RN's `surfaces`), and an "Exclude users you follow" toggle. Submit creates a `MutedWord` and appends. Below, a list of existing entries (reverse-chronological to match RN) shows the value, target ("in text & tags" / "in tags"), an expiry pill ("Expires in 1 day" via `RelativeDateTimeFormatter`, or "Expired"), and an "Excludes users you follow" hint. Trailing destructive swipe removes the entry.
- **View model.** `MutedWordsViewModel` (`@MainActor @Observable`) loads via `getPreferences`, persists via `putPreferences(mutedWords:)`. `add(_:)` and `remove(_:)` perform optimistic local updates with rollback on failure; the alert surfaces server errors.
- **Hub linkage.** Added a "Muted Words & Tags" `NavigationLink` row to `ModerationScreen` in the Content section.

## Gotchas

- **Feed-side filter wiring is deferred.** This issue intentionally stops at "screen persists muted words". Plumbing the list into `FeedViewModel` (or a higher filter) so muted posts disappear from the timeline is a follow-up — track separately. The lexicon types and `mutedWords` accessor on `GetPreferencesResponse` are now in place to make that wiring cheap.
- **`items:` key collision.** Both `savedFeedsPrefV2` and `mutedWordsPref` use `items:` in their JSON shape. The `Item` decoder helper now keys off `$type` to choose between the saved-feed helper and the muted-word helper before decoding the array.
- **Server-issued `id`s.** New entries are sent without an `id`; the appview will assign one on next `getPreferences`. Removal falls back to matching on `value + targets` when no `id` is present.
