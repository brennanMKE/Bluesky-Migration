## Description

The top-right of the macOS SwiftUI app shows three icons packed into a single rounded capsule: a compose square (pencil), a list-with-stars glyph, and a `>>` double-chevron. The grouping reads as "one button" until you look closely; the icons are unequally sized, the capsule has a thin border that makes them look like a segmented control, and there's no spacing between them.

There is no Mac RN reference. The iOS RN reference (`Bluesky-ReactNative`, see attachment) places the equivalent affordances individually:

- **Hamburger** at the very leading edge of the top bar (opens the sidebar drawer).
- **Butterfly logo** centered.
- **Hash (`#`)** at the trailing edge (opens "My Feeds" — see #0073).

The SwiftUI macOS top bar is conflating multiple concepts into one capsule, and the `>>` glyph in particular has no obvious purpose for the user — it's likely the macOS "expand sidebar" / "show inspector" affordance that has visually merged with the action cluster.

## Steps to reproduce

1. Run the SwiftUI app on macOS.
2. Look at the top-right toolbar of the Home (or any tab) screen.

## Expected behavior

Either:

- **Match the RN-iOS top bar layout** even on macOS: hamburger leading, wordmark centered, `#` trailing — three independent buttons with normal toolbar spacing, no capsule wrapper. The `>>` (sidebar expand) button should be a system `ToolbarItem(placement: .navigation)` if needed and should be visually separated from the action cluster.
- Or, if a custom action cluster is intentional on macOS, give each icon: a clear label / tooltip, equal sizing, separator gaps, and remove the capsule outline. And replace `>>` with the standard SF Symbol for the sidebar control (`sidebar.left` or `sidebar.right`).

The middle icon (list-with-stars) is presumably "My Feeds" / saved feeds — that's the `#` button on iOS. Confirm and align the icon to the RN equivalent or use the same `#` glyph.

## Actual behavior

Three icons in a single capsule with no labels, unclear purpose, and a `>>` affordance that doesn't match any RN entry-point.

## Notes

- iOS top-bar pattern is `BlueskyTopBar` (added in #0072). macOS uses a different toolbar path — find it in `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` (the `macOSSidebarLayout` or similar).
- Tooltips on every macOS toolbar button are tracked separately by #0009 (already resolved); confirm that issue's helper covers these too.
- Pairs with #0150 (sidebar drawer parity) — once the sidebar is the proper drawer affordance, the `>>` here may not be needed at all.

## Root cause

The macOS branch of `MainTabView.tabContent(.home)` declared the **compose** and **Saved Feeds** buttons as two consecutive `ToolbarItem(placement: .primaryAction)` instances with no `id` and no tooltips. macOS Sequoia's toolbar coalesces same-placement, anonymous toolbar items into a single visually grouped capsule, which is why the screenshot shows the `square.and.pencil` and `list.star` icons sharing one rounded outline. The trailing `>>` glyph is the system-injected NavigationSplitView column-visibility toggle (it lives in the `.navigation` slot the system manages), not something the app rendered — so it's correctly its own slot already.

The Saved Feeds icon (`list.star`) also drifted from the iOS RN reference, which uses `#` (`number`) for the equivalent My Feeds entry point (#0072 / #0073). And neither macOS button had a `.help()` tooltip, breaking the parity established in #0009.

## Fix

Refactored the macOS branch of the Home toolbar in `MainTabView.swift`:

- **Compose** is now `ToolbarItem(id: "bsky.compose", placement: .primaryAction)` with `.help("New post")`.
- **My Feeds** is now `ToolbarItem(id: "bsky.myfeeds", placement: .primaryAction)` with `Image(systemName: "number")` (matches RN-iOS) and `.help("My Feeds")`.
- Each toolbar item now carries a stable `id`, which prevents SwiftUI's anonymous-item coalescing on macOS Sequoia and gives each button its own slot. Tooltips surface the button purpose on hover.
- The system NavigationSplitView sidebar toggle (the `>>` glyph) is left in the `.navigation` slot SwiftUI manages — adding our own sidebar control would duplicate the system affordance.
- Bonus: the macOS Profile-tab overflow `Menu` toolbar item also picks up `.help("More")` so hover tooltips are consistent across the macOS toolbar.

The `Saved Feeds` action still resolves to the existing `SavedFeedsScreen` destination (no behavior change beyond the icon).

## Gotchas

- `Saved Feeds` and `My Feeds` are nominally the same destination (`SavedFeedsScreen`) — RN-iOS calls the entry point "My Feeds" and uses `#` as its glyph, which is what we now show on macOS. If a separate `MyFeedsScreen` (per #0073) ever lands on macOS, this binding can be redirected without further toolbar changes.
- The `>>` chevron in the original screenshot is the SwiftUI NavigationSplitView column-visibility toggle, not an app-defined button. There's no public API in macOS 15.6 SwiftUI to restyle that glyph; it stays as the system control. Pairs with #0150 if the sidebar is later replaced with a hand-rolled drawer.
- macOS 15.6 deployment target — `ToolbarSpacer` (macOS 26) is not available, so the `id:` differentiation is the right tool to break SwiftUI's auto-coalescing.
