## Description

In a group chat, the React Native message thread shows the sender's display name (or handle) above the first bubble in a run from each participant. SwiftUI's `MessageBubble` (`MessageThreadScreen.swift` ~lines 163–186) renders the bubble alone, with no per-sender attribution. In a 5-person group thread, every "other" bubble looks the same; the user can't tell who said what without tapping into the avatar.

In direct (1:1) threads the omission is fine — there's only one "other" sender. The bug is specifically the group case.

## Steps to reproduce

1. Open a group chat in the SwiftUI app.
2. Scroll through a section with messages from multiple participants.
3. Observe that no sender display-name labels appear above the bubbles.

## Expected behavior

For group convos:

- Above the first bubble in each run from a non-self sender, render a small label: "**Display Name** @handle.bsky.social".
- Avatar appears beside the first bubble of each run, hidden on subsequent bubbles in the run (avatar clustering, RN already does this via `ChatItem`).
- For runs of self-bubbles, no sender label.

For direct convos this stays as today — no sender label.

## Actual behavior

No sender labels at all. Group thread looks like a flat sequence of left-aligned bubbles.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Messages/components/MessageItem.tsx` — search for `showAvatar` / `showName` clustering logic.
- Convo type detection: `MessageThreadViewModel` already loads a `ConvoWithDetails` discriminated union (per the 2026-04-26 RN drift note in `Progress.md`). Use the `.group` case to gate the sender-label rendering.
- File to edit: `BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift`.

## Root cause

`MessageBubble` rendered a flat HStack: avatar gutter Spacer + a single VStack
containing only the (optional image stack and) text bubble. The thread loop
walked `viewModel.messages` with no awareness of the previous message's
sender, so there was no way to identify the start of a "run" from a particular
participant. As a result, group threads gave no per-sender attribution at all.

Separately, the dispatch referenced a `ConvoWithDetails.group` case for group
detection, but no such type exists in BlueskyKit yet (see #0104 gotchas) — so
group detection had to be inferred from `members.count > 2`, the same rule
the conversation list row uses.

## Fix

`BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift`:

- The thread `ForEach` now walks `Array(viewModel.messages.enumerated())` so
  each bubble can examine the previous message's sender.
- Added screen-level helpers: `isGroup` (using the `members.count > 2` rule
  per #0104), `isFirstInRun(at:)` (true when the previous message's sender
  DID differs from the current one), and `senderProfile(for:)` (member
  lookup by DID against the freshest convo we have — `viewModel.convo` if
  loaded, else the init-time `convo`).
- `MessageBubble` gained `isGroup`, `isFirstInRun`, and `senderProfile`
  parameters (defaulted to preserve existing call sites). It now:
  - Reserves a fixed avatar gutter (24pt + 6pt) on every non-self bubble in
    a group so subsequent bubbles in the run line up under the first.
  - Renders the sender's avatar in that gutter only on the first bubble of
    the run.
  - Renders a small two-part label above the first bubble of the run:
    `**Display Name** @handle.bsky.social`, using `Text` concatenation so the
    first run can be bolded while sharing the secondary foreground style.
  - Suppresses both the avatar and the header for self-runs and for direct
    (1:1) convos — preserving today's behavior in those cases.

## Gotchas

- The dispatch's `ConvoWithDetails.group` discriminator does not exist in
  BlueskyKit yet (already noted in #0104). Group detection therefore mirrors
  the conv-list row: `members.count > 2`. When the discriminator is later
  added, both call sites should switch to `kind == .group` together.
- RN's `MessageItem` also enforces a 5-minute time gap inside the
  cluster-boundary check (`CLUSTERED_MESSAGE_THRESHOLD_MS`). This implementation
  intentionally only uses the sender-DID transition because we do not yet
  surface inline timestamps or date dividers in the SwiftUI thread; once those
  ship, the run check should adopt the same time threshold so very-spaced
  messages from the same sender re-show their attribution.
- RN places the avatar at the *bottom* of each cluster (paired with
  metadata like the message timestamp). We place it at the *top* of the run
  to pair visually with the new sender header above the bubble. If/when the
  thread renders per-message metadata at the bottom of clusters, this should
  flip to match RN.
