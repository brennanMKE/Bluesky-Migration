# 0010 — `getFeed` XRPC call missing required `feed` query parameter

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskyFeed |
| **Platform** | All |
| **First seen** | 2026-04-29 |

## Description

Switching to a custom feed (e.g. "What's Hot") results in a blank feed. The server rejects the `app.bsky.feed.getFeed` XRPC call with `InvalidRequest` because the `feed` query parameter is absent from the outgoing HTTP request, even though the URI is available locally.

## Steps to reproduce

1. Launch the app and sign in.
2. Navigate to a custom or algorithmic feed (e.g. What's Hot).
3. Observe that no posts load.

## Expected behavior

Posts from the selected custom feed are loaded and displayed.

## Actual behavior

The feed is blank. The server returns:

```
fetch error: BlueskyCore.ATError.xrpc(code: "InvalidRequest", message: "Invalid app.bsky.feed.getFeed params: Missing required key \"feed\"")
```

## Error log

```
task fired, selection=feed(uri: "at://did:plc:z72i7hdynmk6r22z27h6tvur/app.bsky.feed.generator/whats-hot")
creating FeedViewModel for feed(uri: "at://did:plc:z72i7hdynmk6r22z27h6tvur/app.bsky.feed.generator/whats-hot")
calling loadInitial
fetch start, reset=true
calling getFeed uri=at://did:plc:z72i7hdynmk6r22z27h6tvur/app.bsky.feed.generator/whats-hot
fetch error: BlueskyCore.ATError.xrpc(code: "InvalidRequest", message: "Invalid app.bsky.feed.getFeed params: Missing required key \"feed\"")
loadInitial returned, posts=0, error=The operation couldn't be completed. (BlueskyCore.ATError error 2.)
```

## Notes

The feed URI is received by `getFeed` (visible in the local log line `calling getFeed uri=…`) but is not being appended to the outgoing XRPC URL as the `feed` query parameter. Check the `getFeed` implementation in `BlueskyNetworking` — the `feed` parameter is likely being passed to the function but not included when building the `URLComponents` query items. The AT Protocol expects `GET /xrpc/app.bsky.feed.getFeed?feed=<uri>`.

The log also shows the timeline `.task` firing twice on the same view lifecycle, causing one `getTimeline` request to be cancelled (`NSURLErrorDomain Code=-999`) before the second succeeds. This race may be a related regression from issue 0002 and warrants a separate look.
