## Description

The Bluesky-SwiftUI app target does not build for iOS. A `PhotosPickerItem` error in `BlueskyKit/Sources/BlueskyComposer/ComposerViewModel.swift` is the immediate symptom, but a full iOS build audit is needed — most development has been done on macOS, so other iOS-only paths may have drifted out of sync (PhotosUI imports, `#if os(iOS)` gating, `UIKit` vs. `AppKit` shims, deployment-target API availability, etc.).

This issue blocks #0030 (push notification routing) because that work requires installing the app on a real iOS device.

## Steps to reproduce

1. Open `Bluesky.xcworkspace` in Xcode.
2. Select the `Bluesky-SwiftUI` scheme and an iOS Simulator (or device) destination.
3. Build (⌘B).

## Expected behavior

iOS build succeeds with no errors. App can be installed and run on an iOS Simulator and on a registered iPhone.

## Actual behavior

Build failure related to `PhotosPickerItem` in `ComposerViewModel.swift`. Other iOS-specific errors may surface once that one is fixed.

## Notes

Investigation should run:

```
xcodebuild -workspace ../Bluesky-SwiftUI/Bluesky.xcworkspace \
  -scheme Bluesky-SwiftUI \
  -configuration Debug \
  -destination 'generic/platform=iOS Simulator' \
  build 2>&1 | grep -E '(error|warning):' | head -50
```

…and then audit:

- All `import PhotosUI` sites — confirm they are wrapped in `#if os(iOS)` blocks (or `#if canImport(PhotosUI)`).
- All `PhotosPickerItem` references — same gating.
- `import UIKit` / `import AppKit` boundaries — anything used unconditionally that should be platform-gated.
- `UIImage` vs. `NSImage` shims (`PlatformImage` typealias if any).
- `aps-environment` entitlement and Push Notifications capability — recently added at the App ID; confirm the entitlement file includes it once the build is clean.
- Any deployment-target API availability (`@available`) gaps.

Suspect file from initial report: `BlueskyKit/Sources/BlueskyComposer/ComposerViewModel.swift` line 182 (`attachVideo(_ item: PhotosPickerItem)`). Looks correctly gated by `#if os(iOS)` on inspection — the actual error may be elsewhere or a downstream effect.

## Root cause

`ComposerViewModel.attachVideo(_:)` takes a `PhotosPickerItem` parameter, but the
type is declared by the PhotosUI/SwiftUI cross-import overlay, which only loads
when **both** `PhotosUI` and `SwiftUI` are imported in the same file. The
existing `#if os(iOS)` gate imported only `PhotosUI`, so during iOS module
emission Swift could not resolve `PhotosPickerItem` even though the function
body was hidden behind the same gate (the signature itself still requires the
type to be visible). The macOS build was unaffected because the macOS path
uses `NSOpenPanel` and never references `PhotosPickerItem`.

## Fix

Added `import SwiftUI` inside the existing `#if os(iOS)` block in
`ComposerViewModel.swift` so the cross-import overlay loads on iOS. The fix is
a single line; no other iOS gating drift was found. iOS Simulator, macOS, and
`swift build` all build cleanly afterwards.

## Files changed

- `BlueskyKit/Sources/BlueskyComposer/ComposerViewModel.swift` — added `import SwiftUI` to the `#if os(iOS)` import block so the PhotosUI/SwiftUI overlay resolves `PhotosPickerItem`.

## Gotchas

`PhotosPickerItem` looks like a plain PhotosUI type but is actually defined in
the cross-import overlay — any future file that references it must import
**both** `PhotosUI` and `SwiftUI`. ViewModels often skip `import SwiftUI`, so
this is easy to repeat. `ComposerSheet.swift` and `MessageThreadScreen.swift`
already import both and are fine.
