## Description

The React Native moderation hub is divided into three labeled sections, each with specific rows. The SwiftUI `ModerationScreen` uses different (and fewer) sections and skips several rows.

**RN section structure:**

- **Moderation tools**: Interaction settings → Muted words & tags → Moderation lists → Muted accounts → Blocked accounts → Verification settings.
- **Content filters**: Adult content toggle (gated by age assurance) → per-label visibility prefs.
- **Advanced**: Subscribed labelers (with availability status + cleanup action).

**SwiftUI current state**: two sections ("Account Actions", "Content") containing only Mutes, Blocks, Mod Lists, and (under Content) Adult-content / Filters. Missing rows: **Interaction settings**, **Muted words & tags** (filed separately as #0132), **Verification settings**, **Advanced / labelers** section with subscribed-labeler list.

## Steps to reproduce

1. Open the SwiftUI Moderation hub.
2. Compare to RN's hub.
3. Observe missing rows and different section grouping.

## Expected behavior

Restructure `ModerationScreen` to mirror RN's three sections and rows:

- **Moderation tools**: Interaction settings (link to new screen — see #0138), Muted words & tags (#0132), Moderation lists (existing), Muted accounts (existing), Blocked accounts (existing), Verification settings (link to new screen).
- **Content filters**: Adult content + per-label preferences (existing — but per #0136 should be per-labeler).
- **Advanced**: Subscribed labelers list with each labeler's name + availability ("Unavailable" badge if the labeler service is down) + cleanup action ("Remove unavailable").

## Actual behavior

Two sections; missing Interaction settings, Verification, Muted words, and Advanced/labelers list.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Moderation/index.tsx`.
- File to edit: `BlueskyKit/Sources/BlueskyModeration/ModerationScreen.swift` and `ModerationViewModel.swift` (load subscribed labelers via `app.bsky.labeler.getServices`).

## Root cause

`ModerationScreen` predated several of the RN rows: it grouped the few rows it had into ad-hoc "Account Actions" / "Content" sections that didn't line up with RN's "Moderation tools" / "Content filters" / "Advanced" structure, and skipped Interaction settings, Verification settings, and the Advanced/labelers list entirely. `ModerationStore` loaded basic preferences but never read the `app.bsky.actor.defs#labelersPref` body, so subscribed-labeler DIDs weren't surfaced and there was no path to call `app.bsky.labeler.getServices` for display names + availability.

## Fix

- **Lexicon (BlueskyCore).** Added `subscribedLabelerDIDs: [DID]` to `GetPreferencesResponse`, parsing the `labelersPref.labelers[].did` array via a new `LabelerEntryHelper` and an `Item` decoder branch keyed off `$type`. Added `PutPreferencesRequest(subscribedLabelerDIDs:)` initializer that emits a fully-formed `app.bsky.actor.defs#labelersPref` for the cleanup write-back. `LabelerView` and `GetLabelerServicesResponse` already existed from prior work.
- **`ModerationStore`.** Extended `ModerationStoring` with `subscribedLabelers: [LabelerView]`, `subscribedLabelerDIDs: [DID]`, `unavailableLabelerDIDs: [DID]`, and `isLoadingLabelers: Bool`. `loadPreferences()` now captures the DIDs from `getPreferences`. New `loadSubscribedLabelers()` reads the prefs, then resolves the DIDs via `app.bsky.labeler.getServices?dids=...` and diffs requested vs returned DIDs to populate `unavailableLabelerDIDs` (mirrors RN's `useMyLabelersQuery`). New `removeUnavailableLabelers()` writes the trimmed list back via `putPreferences` (mirrors RN's `useRemoveLabelersMutation`).
- **`ModerationViewModel`.** Added passthroughs for the four new state properties and the two new methods.
- **`ModerationScreen` restructure.** Replaced the two-section layout with RN's three sections in row order:
  - **Moderation tools**: Interaction settings (placeholder sheet → bsky.app, awaiting #0138) → Muted words & tags → Moderation lists → Muted accounts → Blocked accounts → Verification settings (placeholder sheet → bsky.app).
  - **Content filters**: Adult-content toggle (calls `setAdultContent(enabled:)`) → "Configure content filters" navigation link to the existing `ContentFilterSettingsScreen`.
  - **Advanced**: A destructive "Remove unavailable" row appears only when `unavailableLabelerDIDs` is non-empty. Subscribed labelers render as `NavigationLink`s into the existing `LabelerProfileScreen`. Empty-state shows "No subscribed labelers"; while the fetch is running we show a `ProgressView` row. Unresolved DIDs render as a separate row group with an "Unavailable" caption so users see what the cleanup will drop.
- **Initial load.** `.task` and `.refreshable` both call `loadPreferences()` then `loadSubscribedLabelers()` so the hub stays warm on revisit.

## Gotchas

- **Interaction settings + Verification settings are placeholders.** Both rows exist for parity with RN's row order, but tapping opens a sheet that points the user at bsky.app. #0138 will replace the Interaction settings sheet with a real screen; the Verification screen is bundled into this issue's scope per the spec, but a dedicated screen is still TBD — the placeholder is the documented interim state.
- **`getServices` DIDs encoding.** `dids` is encoded as a comma-joined query value here. The appview accepts both the comma form and the repeated-param form; the `NetworkClient.get` interface only carries `[String: String]`, so comma is the lighter path. If a labeler DID happens to contain a comma we'll need to switch to repeated params, but DIDs in the wild don't.
- **Adult-content age-assurance is not modeled.** RN gates the toggle behind the AA flag (`aa.flags.adultContentDisabled`) and shows an admonition for under-18 users. The SwiftUI client doesn't ship age assurance yet, so the toggle is unconditionally visible. This is a pre-existing divergence and is left for a future AA pass.
- **Per-label preferences moved off the hub.** RN renders the per-label rows (`porn`, `sexual`, `graphic-media`, `nudity`) inline below the adult-content toggle when AA is "Full". To keep the hub scannable on Apple platforms we link to the existing `ContentFilterSettingsScreen` instead, where the same prefs are already managed. #0136 calls for migrating those to per-labeler — keeping the hub link-only avoids fighting that work.
