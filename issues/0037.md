## Description

The Module 14 gate — confirm every setting persists after app restart and takes immediate visual effect — has not been validated.

## Steps to reproduce

1. Change the theme (light → dark → dim) and confirm the app UI updates immediately.
2. Change the font size and confirm text updates.
3. Force-quit and relaunch. Confirm the theme and font size settings are preserved.
4. Toggle notification preferences and confirm they survive restart.

## Expected behavior

All settings are saved via `SettingsViewModel`/`UserDefaultsPreferencesStore` and applied on next launch.

## Actual behavior

Not validated. Some settings (theme) require wiring into the environment for immediate effect.

## Live testing observations (2026-04-30)

**Appearance settings are broken** — the theme/appearance controls in Settings do not work correctly. The app should defer to the macOS system appearance setting (light/dark) rather than managing its own theme toggle. The custom theme switcher adds complexity without benefit on macOS where the OS already handles this at the system level. The recommended fix is to remove the in-app theme picker and use `.preferredColorScheme(nil)` or remove the modifier entirely so the app follows the system setting automatically.

## Incomplete reason

Validation gate — appearance settings need to be fixed to follow system appearance before this gate can pass.

## Notes

**2026-05-05 — Appearance blocker resolved in code; gate still needs hands-on validation.**

Code change (BlueskyKit `93314f7`):

- Removed the Light/Dark/Dim picker from `AppearanceSettingsScreen`. It now shows a short note that Bluesky follows the macOS system appearance.
- Removed `themeVariant`, `setTheme(_:)`, the `settings.themeVariant` UserDefaults key, and the `BlueskyUI` import from `SettingsViewModel`. `BlueskyTheme.Variant` still lives in `BlueskyUI` for design-system use; nothing in the app reads it now.
- `Bluesky_SwiftUIApp` already applies `.adaptiveBlueskyTheme()` at the root, which selects `.light`/`.dark` from `@Environment(\.colorScheme)`. No `.preferredColorScheme(...)` is set on `RootView`, so the app inherits the OS appearance. Switching System Settings → Appearance now flips the app immediately.

What the user still needs to verify (live, on macOS):

1. Open System Settings → Appearance. Toggle Light → Dark → Light. Confirm Bluesky's UI flips to match each time without relaunching.
2. Open Settings → Appearance in Bluesky. Confirm the "Theme" section shows the explanatory text (no picker), and the Font Size slider still appears.
3. Drag the Font Size slider. Force-quit Bluesky (Cmd-Q or Activity Monitor). Relaunch. Open Settings → Appearance. Confirm the slider is still at the previously chosen value.
4. Open Settings → Notifications. Toggle Likes/Reposts/Follows/Mentions/Replies/Quotes. Force-quit and relaunch. Confirm each toggle is still in the state you left it.
5. Open Settings → Content & Media and Settings → Accessibility. Change at least one toggle in each (e.g. autoplay video, reduce motion). Force-quit and relaunch. Confirm both survive.
6. Open Settings → Languages. Change the post-language selection. Force-quit and relaunch. Confirm the selection survives.

If all six pass, mark this issue resolved (set Status to `resolved`, add a `**Closed**` row, and run `python3 issues/generate.py`). If any setting fails to persist, file a follow-up issue against `UserDefaultsPreferencesStore`/`SettingsViewModel` with the exact key that was lost.
