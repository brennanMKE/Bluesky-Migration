## Description

Tapping a notification row (a like, reply, mention, quote, or subscribed-post) navigates to a screen titled "Thread" whose entire body is a single centered line of debug text:

> Thread: at://did:plc:2izaurqsrmguegexlfoc7pps/app.bsky.feed.post/3mllevupzac2j

There is no post, no parent chain, no replies, no compose bar — just the raw AT-URI rendered as a label. The placeholder is what `BlueskyProfile.ThreadPlaceholder` (or an equivalent stub view) renders when `BlueskyProfile` was scaffolded with a stand-in for the real `ThreadView` to avoid a circular `BlueskyProfile → BlueskyFeed` dependency (see Progress.md notes from 2026-04-24 — "ThreadView in the profile context uses a `ThreadPlaceholder` stub to avoid a circular dependency").

The notifications tab is hitting the same stub. The real `ThreadView` from `BlueskyFeed` exists and works (the Home feed → tap a post → see thread path is functional per #0039 / #0045 / #0054). The notifications tab's `navigationDestination` for thread URIs is wired to the placeholder, not the real view.

This regresses / extends #0062 ("Tapping a notification shows raw at:// URI text instead of navigating to the thread") which is already filed and `open`. The screenshot here makes the symptom concrete: the user sees the AT-URI verbatim because the placeholder's body literally renders the URI as text.

## Steps to reproduce

1. Open Notifications.
2. Tap any row (a like, reply, mention, quote, or subscribed-post).
3. Observe a screen titled "Thread" whose body is the AT-URI rendered as a centered string.

## Expected behavior

Tapping a notification navigates to the **real** `ThreadView` from `BlueskyFeed`, with the focal post (the one the notification refers to), parent chain above, and reply tree below — same surface the Home feed → post tap path produces.

## Actual behavior

The placeholder ThreadView renders only the AT-URI string. No post, no replies, no actions.

## Root cause

`NotificationsScreen` owned its own `navigationDestination(isPresented:)` that mounted a literal `Text("Thread: \(uri.rawValue)")` placeholder rather than the real `ThreadView`. The placeholder was deliberate scaffolding — `BlueskyNotifications` is a Layer-3 feature module and cannot import `BlueskyFeed` (also Layer-3) without forming a cross-feature edge that the architecture forbids (the same constraint that produced the `ThreadPlaceholder` workaround inside `BlueskyProfile`). Because the destination was internal to the screen, `MainTabView`'s `case .notifications:` had no opportunity to override it, so every notification tap rendered the AT-URI verbatim instead of the thread the user wanted to read.

## Fix

Hoisted the destination wiring up to the app shell, mirroring the pattern landed in #0146 for `LikedByScreen` / `RepostedByScreen` / `QuotesOfScreen` and the per-screen `onPostTap` pattern from #0108:

- **`NotificationsScreen` (BlueskyKit)** — added an optional `onPostTap: ((ATURI) -> Void)?` init parameter. Row taps now route through a small `handlePostTap(_:)` dispatcher: when the parent supplies `onPostTap`, the dispatcher forwards the URI; when it's `nil`, it falls back to the existing internal `threadURI` `@State`. The placeholder `navigationDestination` now activates only in the `onPostTap == nil` case (kept alive so previews and any unwired callsite still degrade gracefully rather than swallowing the tap).
- **`MainTabView` (Bluesky-SwiftUI)** — `case .notifications:` now passes `onPostTap: { uri in threadURI = uri }` into `NotificationsScreen`, then attaches `navigationDestination(item: $threadURI)` mounting the real `ThreadView(uri:network:accountStore:bookmarks:onAuthorTap:onPostTap:onLikedByTap:onRepostedByTap:onQuotesTap:)`. The same Liked-by / Reposted-by / Quotes-of `navigationDestination(item:)` blocks from the Home and Saved tabs are duplicated under the notifications case so the focal-post stat row (#0146) and in-thread reply / quote taps keep working when the user reached the thread via notifications.

The `threadURI` / `likedByPostURI` / `repostedByPostURI` / `quotesPostURI` `@State`s are shared with the Home and Saved tabs, but `MainTabView` already keys its `NavigationStack` on `selectedTab` (`.id(activeTab)` / `.id(selectedTab ?? .home)`), and `resetTransientNavState()` clears these on every tab switch — so reuse is safe and matches the same idiom Home/Saved already use.

## Files changed

- `BlueskyKit/Sources/BlueskyNotifications/NotificationsScreen.swift` — new `onPostTap` init parameter, new `handlePostTap(_:)` dispatcher, internal placeholder `navigationDestination` gated on `onPostTap == nil`.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` — `case .notifications:` now passes `onPostTap` through, and adds five `navigationDestination(item:)` blocks (`threadURI`, `likedByPostURI`, `repostedByPostURI`, `quotesPostURI` — the existing `feedProfileDID` block was already in place from #0080).

## Gotchas

- This is the same root cause as **#0062** — that issue has been open since 2026-05-05 and the screenshot in #0160 is fresh evidence the placeholder was still shipping. The fix in this commit pair resolves both. **Per the issue skill's resolution rule, #0062 is left as `open` pending explicit user confirmation**; once the user confirms the notification-tap path now works, #0062 can be flipped to `resolved` against the same commit hashes (`f5cc464d` / `b78828af`). #0062 has been cross-linked with a `## Notes` line so the link survives.
- The placeholder destination inside `NotificationsScreen` is intentionally retained for previews and any future caller that doesn't wire `onPostTap`. It behaves exactly as before in that mode (renders the AT-URI as text), so this change is strictly additive.
- The shared `threadURI` / liked-by / etc. `@State` reuse across tabs relies on the existing `resetTransientNavState()` call on tab change. If a future change moves those states into a tab-scoped container (e.g. an `@Observable` per-tab navigation model), this wiring should follow.
- #0030 (push notification routing) already drives `threadURI` and `selectedTab = .home` from `handlePushPostThread(_:)`. With this fix, push routing into the notifications tab specifically would now also work because the destination exists there too — no follow-up needed for that path.
