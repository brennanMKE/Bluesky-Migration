## Description

Manual sign-off pass for authentication and session management against
the React Native reference. Confirm the SwiftUI app reaches feature
parity with the RN client's sign-in, sign-out, and multi-account flows.

This is a test plan, not a bug report — fill in the **Actual behavior**
column as you walk through it. File one issue per problem found rather
than expanding this one.

## Test sequence

1. **Fresh launch — sign-in screen.** Delete the app's keychain entry
   (or run a fresh build with no saved sessions). Launch the app.
2. **Sign in with handle + app password.** Enter a Bluesky handle
   (e.g. `alice.bsky.social`) and a valid app password. Tap Sign In.
3. **Sign in with handle + main password + 2FA.** Repeat with an
   account that has 2FA enabled. Confirm the email-code prompt
   appears and accepts the code.
4. **Failed sign-in.** Enter a wrong password. Confirm the error
   message surfaces inline (not a silent failure).
5. **Background → relaunch.** Background the app, kill it from the
   app switcher (iOS) or quit (macOS), relaunch. Confirm the same
   account is signed in without re-entering credentials.
6. **Sign out.** Sign out from Settings → Account. Confirm the
   sign-in screen reappears and the keychain entry for that account
   is cleared.
7. **Multi-account add.** From the sign-in screen (or the account
   picker), add a second account. Confirm both accounts appear in
   the picker.
8. **Multi-account switch.** Switch between accounts via the picker.
   Confirm the feed, notifications, and profile all reflect the new
   active account, not stale data from the previous one.
9. **Token refresh.** Leave the app open long enough for the access
   token to expire (≈2 hours), or manually invalidate the token in
   the keychain. Trigger any networked action. Confirm the session
   silently refreshes; no sign-in prompt should appear unless the
   refresh token itself is invalid.
10. **Forced re-auth.** Manually invalidate both access and refresh
    tokens. Trigger a networked action. Confirm the user is bounced
    to the sign-in screen with a clear message ("Your session has
    expired").

## Expected behavior (per step)

| # | RN parity expectation |
|---|---|
| 1 | Identical landing page to RN: handle/email + password fields, "Sign in" CTA, "Create account" link below |
| 2 | After successful auth, user lands on Home feed with their selected feed loaded |
| 3 | 2FA code field appears in same modal/sheet pattern as RN; success path same as #2 |
| 4 | Inline error text, focus stays on password field, no toast or alert popup |
| 5 | App opens directly to last-viewed tab with no flicker of sign-in screen |
| 6 | Sign-out is instant, no confirmation needed (matches RN; settings has the destructive-style row) |
| 7 | Account picker shows both with avatars/handles |
| 8 | Cross-contamination check — feed should not flash content from the previous account |
| 9 | Network request retries transparently after refresh |
| 10 | Sign-in screen appears with a single inline message, not a generic error toast |
