## Description

The React Native reference shows an inline composer prompt at the very top of the Home feed, between the tab strip and the first post. It contains the user's avatar on the left, the prompt text "What's up?" centered, and a camera icon plus a photo-library icon on the right. Tapping anywhere in the row opens the full `ComposerSheet`. The SwiftUI iOS app has no equivalent — the only composer entry is the top-right pencil icon (replaced by a FAB in #0071).

Adding this prompt closes a discoverability gap and matches the long-standing RN pattern.

## Scope

- **Add a new view** at the top of the Home feed list, below the tab strip and above the first post.
- **Layout**:
  - Leading: circular avatar (24pt) for the signed-in user
  - Center: secondary-color text "What's up?" left-aligned with a small leading gap from the avatar
  - Trailing: two outlined icon buttons — camera (`camera`) and photos (`photo`) — separated by a small gap
- **Tap anywhere on the row → open `ComposerSheet`** with no pre-set body, no reply target, no quoted post.
- **Tap camera button → open `ComposerSheet`** with the camera-capture flow primed (iOS only — uses `UIImagePickerController` with `.camera` source).
- **Tap photos button → open `ComposerSheet`** with the photo-picker flow primed.
- **Visible only on Home**, not on other feeds. Hide when scrolled past the top? Optional polish — match RN behavior (RN keeps it pinned at top of the feed list, scrolls away normally).

## Implementation notes

- New `InlineComposerPrompt` view in `BlueskyFeed` (or `BlueskyComposer`).
- Inject as the first row of the `LazyVStack` in `FeedView`, gated `#if os(iOS)`.
- Pre-priming the composer with camera/photos: extend `ComposerSheet` to take an optional initial-attachment-source enum (`.none`, `.camera`, `.photoLibrary`).
- On macOS, do not render this prompt — the macOS toolbar already provides a compose action and the layout doesn't fit.

## Acceptance

- iOS Home: an inline composer row with avatar, "What's up?", camera, and photo icons appears at the top of the feed.
- Tapping the row, the camera button, or the photo button opens `ComposerSheet` with the appropriate primed state.
- macOS Home: unchanged.
- iOS Simulator and macOS builds pass.

## Root cause

Parity gap, not a bug. The SwiftUI iOS Home feed jumps straight from the
tab strip into the post list with no inline composer entry — the only
compose entry points are the FAB (#0071) and (on iPad) the toolbar
pencil. The React Native reference has shipped a pinned "What's up?" row
between the tab strip and the first post for years; new users find that
row first when scanning the feed for "where do I write a post". The
SwiftUI app inherited a stripped-down list and never grew the row.

## Fix

Adds a new `InlineComposerPrompt` view in `BlueskyFeed`, gated to iOS,
and wires it as the first row of every Home tab's `LazyVStack` (just
below the existing scroll-to-top anchor). The row is a single `Button`
with a 24pt `AvatarView`, a `"What's up?"` `Text` in
`theme.colors.textSecondary`, and two outlined trailing icon buttons
(`camera` and `photo`). Tapping the row opens the existing
`ComposerSheet` with no primed picker; tapping either icon button opens
the same sheet but pre-primes the corresponding attachment flow.

`ComposerSheet` is extended with a public
`ComposerInitialAttachmentSource` enum (`.none`, `.camera`,
`.photoLibrary`, default `.none`) and an optional
`initialAttachmentSource:` init parameter — every existing call site is
unchanged. On iOS, the sheet's `onAppear` checks the source and, after a
small `asyncAfter` tick to let the sheet's own presentation animation
finish, raises a `fullScreenCover` (camera) or `sheet` (photo library)
on top of itself. A guard flag (`didTriggerInitialAttachment`) ensures
this fires exactly once per sheet instance.

The camera flow uses a new `CameraImagePicker`
`UIViewControllerRepresentable` wrapping `UIImagePickerController` with
`.camera` source — the system `PhotosPicker`/`PHPicker` paths don't
expose camera capture and `UIImagePickerController` is still the
supported way. The photo flow reuses the existing
`PHPickerRepresentable`. Both feed picked image data straight into
`viewModel.addImage(data:mimeType:)`, so the rest of the composer flow
is untouched.

`FeedView` fetches the viewer's avatar lazily on first appearance via
`accountStore.loadCurrentDID()` + `accountStore.load(did:)` for the
handle, then `app.bsky.actor.getProfile` for the avatar URL. This
mirrors the workaround added in #0071 because `Account.avatarURL` is
currently always `nil` after login. A new
`showsInlineComposerPrompt: Bool = true` init flag preserves an opt-out
if the view is ever reused on a non-Home surface; today it's only used
from `MainTabView` for the Home tab so the default is fine.

The Bluesky-SwiftUI app target adds `INFOPLIST_KEY_NSCameraUsageDescription`
and `INFOPLIST_KEY_NSPhotoLibraryUsageDescription` to both Debug and
Release build settings; without them the camera picker would crash
immediately. The macOS path is untouched — the inline prompt is gated
behind `#if os(iOS)` and `FeedView` keeps its existing toolbar compose
button on the Mac.

## Gotchas

- **Camera picker is `UIImagePickerController`, not `PhotosPicker`.**
  SwiftUI's `PhotosPicker` and the underlying `PHPickerViewController`
  do not support a `.camera` source — they're photo-library-only
  pickers, by design. The supported way to capture from the camera in
  SwiftUI / UIKit is still `UIImagePickerController` with
  `sourceType = .camera`. `CameraImagePicker` is the iOS-26 equivalent
  of every SwiftUI camera-capture demo from 2020. Compresses the picked
  image as JPEG at 0.85 quality before handing it to the composer.
- **Initial picker raises after a `0.25s asyncAfter` tick.** The first
  sheet (`ComposerSheet`) needs a moment to finish its presentation
  animation before we layer a second picker on top, otherwise iOS logs
  the noisy "Attempt to present … which is already presenting …"
  warning and the picker never appears. The tick is short enough that
  the user perceives it as a single composite gesture.
- **Avatar fetch is lazy on first appearance.** `Account.avatarURL` is
  always `nil` after login, same as in #0071. The first feed load
  briefly shows the initials placeholder while
  `app.bsky.actor.getProfile` resolves; this is a one-shot per
  `FeedView` lifetime and the result isn't yet cached across view
  recreations. A future refactor that populates `Account.avatarURL`
  during session creation would let both this prompt and the iOS tab
  bar's Profile-tab avatar drop the workaround.
- **`NSCameraUsageDescription` and `NSPhotoLibraryUsageDescription` are
  load-bearing.** Without them, the camera picker crashes the app on
  first present (privacy compliance) and PHPicker logs a runtime
  warning. They're scoped to iOS via the INFOPLIST_KEY mechanism so
  the macOS sandbox profile is unchanged.
- **The prompt appears at the top of every Home tab.** Following,
  Mentions, Discover, Popular-With-Friends, and any pinned feed all
  show it as the first row of their `LazyVStack`. This matches the RN
  reference where the prompt is part of the home pager, not just the
  Following list. It scrolls away with the feed (per the issue's
  "scrolls away normally" note) — it's not pinned to the top.
- **Bluesky-SwiftUI app code did not need to change.** The existing FAB
  sheet plumbing (#0071) and the Home-tab `FeedView` invocation handle
  the wiring; the inline prompt's sheet lives entirely inside
  `FeedView`. Only the build settings needed an update for the iOS
  privacy strings.
