## Description

`AccountSettingsScreen` is currently a private stub inside `SettingsScreen.swift` (lines 120–133) that displays a note directing users to bsky.app for email / password / 2FA changes. The React Native client implements all of these inline:

- **Email**: change email + verification status with badge color (verified / unverified) + resend verification.
- **Password**: change password (current + new + confirm) inline.
- **Handle**: change handle (with `.bsky.social` domain or custom domain — `com.atproto.identity.updateHandle`).
- **Birthday**: edit birthdate.
- **Two-factor (Email 2FA)**: toggle.
- **Switch hosting provider / migrate PDS**: advanced.
- **Export my data**: download CAR file.
- **Deactivate account**: with warning + final-confirm.
- **Delete account**: end-of-life flow.

The stub directs users out of the app, which is a usability regression — these are core account operations that RN handles natively.

## Steps to reproduce

1. Settings → Account.
2. Compare to RN's Account screen.
3. Observe SwiftUI shows only an info note; no inline forms or actions.

## Expected behavior

A full `AccountSettingsScreen.swift` with rows for: Email (with verified badge + change), Password (open change-password sheet), Handle (open handle-change flow), Birthday, Email 2FA toggle, Export my data (download .car), Deactivate account, Delete account. Each row navigates to a focused sub-sheet that calls the appropriate AT Proto endpoint.

## Actual behavior

A single info note: "Manage your account on bsky.app". No inline functionality.

## Root cause

`AccountSettingsScreen` was a placeholder that landed during the initial `BlueskySettings` skeleton — a private struct inside `SettingsScreen.swift` whose body was a single secondary-text label directing users to bsky.app. None of the RN account operations (email verify/update, handle, birthday, deactivate, etc.) had been ported because the SessionManager surface didn't expose them and there was no spec for which subset to build inline vs. punt to the web app.

## Fix

Replaced the stub with a dedicated public screen at `BlueskyKit/Sources/BlueskySettings/AccountSettingsScreen.swift` that mirrors RN's row order from `Bluesky-ReactNative/src/screens/Settings/AccountSettings.tsx`:

- **Email** — current address + verified badge (`checkmark.shield.fill`); when `emailConfirmed != true`, an inline "Verify your email" row that calls `com.atproto.server.requestEmailConfirmation` and surfaces a one-shot success alert.
- **Update email** — sheet that runs `requestEmailUpdate` (capturing `tokenRequired`), then either auto-submits via `updateEmail` or asks for the confirmation token first. Refreshes the local `Account` snapshot from `getSession` on success.
- **Password** — opens a referral sheet that explains the Forgot Password flow on the sign-in screen and offers an "Open bsky.app" link. Deferred per the issue spec.
- **Handle** — current handle row + a sheet referring users to bsky.app for now.
- **Birthday** — pulls the `personalDetailsPref.birthDate` from `app.bsky.actor.getPreferences`; an inline `DatePicker` writes a new `personalDetailsPref` via `putPreferences` with optimistic local update + rollback on failure.
- **Export my data / Delete account** — referral sheets pointing at bsky.app, mirroring the row shape RN ships.
- **Deactivate account** — destructive sheet with the same copy as RN's `DeactivateAccountDialog`. On Submit it calls `com.atproto.server.deactivateAccount` and invokes the host-supplied `onDeactivated` callback. `MainTabView` wires that to `session.logout`, so the next sign-in routes to `DeactivatedView` (RN parity: `logoutCurrentAccount('Deactivated')`).

State is held in a small `@MainActor @Observable AccountSettingsViewModel` keyed off the passed-in `Account?`. `refreshFromServer()` runs on `task`-attach and re-pulls email/verification status from `getSession` plus the birthday from `getPreferences`.

To support the new endpoints, BlueskyCore now defines `EmptyBody`, `RequestEmailUpdateResponse`, `UpdateEmailRequest`, `UpdateHandleRequest`, and `DeactivateAccountRequest` (with ISO-8601 `deleteAfter` encoding). `PutPreferencesRequest` gained a `birthDate:` initializer that emits an `app.bsky.actor.defs#personalDetailsPref`, and `GetPreferencesResponse` parses the matching `birthDate` so the screen has read access. `SettingsScreen.init` accepts `currentAccount: Account?` and an optional `onAccountDeactivated` callback so the host can forward state cleanly.

## Gotchas

- **Deferrals are explicit.** Change password, change handle, export, and delete all open a referral sheet ("Open bsky.app") rather than a real form. The screen still ships those rows in RN order so the layout is right; the inline flows will be filed as follow-ups.
- **Change-password punt rationale.** RN's `ChangePasswordDialog` reuses `requestPasswordReset` + `resetPassword`, which are already implemented for the forgot-password flow (#0093). Wiring a second entry point is straightforward but felt out of scope for this hub-shaping pass — the referral sheet copy explicitly points users at the existing forgot-password flow.
- **Email update token UX.** RN typically requires the user to copy a token out of the confirmation email. Our sheet handles both `tokenRequired = true` (two-step) and `tokenRequired = false` (single-step) paths but does not auto-detect when the user receives the email; they paste the code manually.
- **Deactivate logout.** Because `deactivateAccount` may invalidate the access token immediately on some PDS deployments, we don't try to `getSession` afterwards. Instead the host fires `onDeactivated` which calls `session.logout`, matching RN. The user re-authenticates and the gate in `ContentView` routes them to `DeactivatedView`.
- **Birthday DatePicker firing on every change.** SwiftUI's `DatePicker` calls the binding setter for each adjustment. The view model debounces only by no-op-ing if the picker re-emits the same value; rapid spinning will still issue multiple `putPreferences` calls. Acceptable for now — RN's modal commits once on Done — but worth revisiting if it shows up as a hotspot.
- **Entitlements drift not bundled.** A local `Bluesky-SwiftUI.entitlements` change (likely auto-touched by Xcode signing) showed up in the working tree but was deliberately left out of the resolution commits since it's unrelated to this issue.
