## Description

The React Native composer has a `DraftsButton` in the top toolbar that opens a list of saved drafts: each draft shows a preview, timestamp, and tap-to-resume / swipe-to-delete affordances. A separate "Save Draft" action lets the user save the in-progress composer state without posting. The SwiftUI composer has only auto-save-on-dismiss persistence (per #0034), which means the user has no way to see, manage, or load past drafts.

This is filed separately from #0034 because that issue covers draft *persistence* mechanics; this one covers the user-facing **drafts UI** (button + list + manage).

## Steps to reproduce

1. Open the SwiftUI composer, type a partial post.
2. Look for a way to save the draft explicitly, or to view past drafts.
3. Observe neither affordance exists.

## Expected behavior

- **Drafts button** in the composer top toolbar (RN places it to the left of the Post button).
- Tapping it opens a `DraftsSheet`:
  - Empty state if no drafts.
  - List of drafts: avatar + first 2 lines of text + relative timestamp + image-count badge if any.
  - Tap a draft to load it into the composer.
  - Swipe to delete a draft.
- **Save Draft** action: when the composer is dismissed via Cancel and the user has typed text, prompt "Save Draft / Discard / Cancel". The Save path stores the state in the existing persistence mechanism.
- **Auto-save** continues to write a single "in-progress" draft so a force-quit doesn't lose work.

## Actual behavior

No drafts button, no list, no explicit save action. The composer either submits or discards.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/com/composer/drafts/`.
- Storage: drafts can live in `UserDefaultsPreferencesStore` (small) or `SwiftDataCacheStore` (better — supports per-draft id + media references).
- Files to edit: `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` (toolbar + dismiss-with-text prompt), new `DraftsSheet.swift`, plus a `DraftsStore` in `BlueskyComposer` that wraps the cache.

## Root cause

The SwiftUI composer only persisted a single "current text" auto-save under a fixed `UserDefaults` key (`composer.draft.text` / `composer.draft.reply.<uri>`). There was no list, no per-draft entry, no UI surface to view, load, or delete past drafts, and no explicit "Save Draft" affordance. RN's parallel surface in `view/com/composer/drafts/` (DraftsButton + DraftsListDialog + DraftItem + the `state/queries.ts` mutation hooks) had no SwiftUI equivalent.

## Fix

Built a small drafts subsystem inside `BlueskyComposer` (Layer 3) without taking a dependency on `BlueskyDataStore`:

- **Draft model** (`Drafts.swift`): `Draft: Codable, Identifiable, Sendable` carrying text, timestamps, reply / quote context URIs, language, self-labels, threadgate selection, quote-toggle, GIF, and reserved slots for `imagePreviews` / `additionalPosts`. The threadgate enum is flattened into `ThreadgateAllowSelectionSnapshot` (an explicit `kind` discriminator + flat fields) so a future enum-case addition doesn't invalidate persisted JSON.
- **DraftsStore** (`DraftsStore.swift`): `DraftsStoring` protocol with `loadAll` / `save` / `delete` / `deleteAll`. Default implementation `UserDefaultsDraftsStore` JSON-encodes the array under `bluesky.composer.drafts`. `MockDraftsStore` for previews and tests.
- **DraftsSheet** (`DraftsSheet.swift`): NavigationStack list, sorted newest-first, with relative-timestamp formatter (`now`/`Nm`/`Nh`/`Yesterday`/`MMM d`), image-count + GIF badges, reply / quote chips, swipe-to-delete (iOS) and context-menu Delete (macOS). Empty-state copy "No drafts yet" mirrors RN. Light + Dark previews.
- **Composer integration** (`ComposerSheet.swift` / `ComposerViewModel.swift`): added a `tray.full` Drafts button to the top toolbar to the left of the Post button (RN places its DraftsButton in the same position). Tapping presents `DraftsSheet`. Tapping Cancel with non-empty composer state opens a `confirmationDialog` with **Save Draft** / **Discard** / **Keep Editing** actions. Save Draft writes via `viewModel.saveCurrentAsDraft()`, Discard clears state, Keep Editing closes the dialog. Loading a draft from the list calls `viewModel.loadDraft(_:)` which replaces the visible composer state and remembers the draft id so a subsequent Save overwrites instead of appending. After a successful post originating from a draft, the draft is deleted from the list (mirrors RN's `deleteDraftAfterPost`).
- **TenorGif** flipped from `Decodable` to `Codable` (with its nested `TenorMediaFormats` / `TenorMediaObject`) so a selected GIF survives the round trip into a saved draft.

The legacy single-slot auto-save (`saveDraft()` writing to `UserDefaults` on every text change and on disappear) is retained — it's the safety net for force-quits while the explicit drafts list is the user-facing affordance.

## Gotchas

- **Image attachments are not persisted.** RN keeps a stable file path for each picked image and re-loads from the local media library when a draft is opened. The SwiftUI composer holds raw `Data` blobs from `PHPickerResult` / `NSOpenPanel` that have no stable round-trip handle, and storing the bytes themselves in `UserDefaults` is a non-starter (`UserDefaults` is sized for kilobytes, not megabytes). The `DraftImage` type in `Drafts.swift` is reserved for a future thumbnail-persistence pass; the composer never writes it today. **User impact:** loading a draft restores text + GIF + threadgate + labels, but image attachments are dropped. The drafts list shows an image count of zero for those drafts — there is no "Media stored on this device" / "Missing media" badge yet (RN displays one).
- **Thread-draft persistence is deferred.** The composer's `additionalPosts: [String]` is not persisted. `Draft.additionalPosts: [DraftPost]?` is a reserved slot. Drafting a thread, hitting Save Draft, then loading it back will restore only the root post.
- **Full background auto-save is deferred.** The original single-slot auto-save (`saveDraft()` on text change / on disappear) is unchanged. The explicit Save Draft action is the entry point into the drafts list. A debounced periodic auto-save into the drafts list (so a force-quit produces a list entry, not just a single restore slot) is a follow-up that would require careful UX work to avoid spawning many near-duplicate drafts.
- **macOS picker frame.** The drafts sheet sets `minWidth: 480, minHeight: 560` on macOS. Without it, the sheet renders narrow enough to truncate text aggressively. iOS uses default sheet sizing.
- **`UserDefaults` is single-writer in practice.** `UserDefaultsDraftsStore` does a read-modify-write of the full drafts array on every save. A second composer window writing concurrently would race and lose one of the writes; the iOS / macOS app is single-window-composer in the current shell, so this isn't exercised. If multi-window composing lands, swap to a serial actor or a SwiftData-backed store.
