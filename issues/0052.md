## Description

Opening a `bluesky://` URL (e.g. `open "bluesky://profile/brennan.sstools.co"`) causes macOS to open a new app window rather than routing within the existing window. The `onOpenURL` handler in `MainTabView` is called but the SwiftUI scene machinery creates a second window instead of reusing the first.

## Steps to reproduce

1. Build and run the app on macOS.
2. In Terminal, run: `open "bluesky://profile/brennan.sstools.co"`
3. Observe that a second app window appears instead of the existing window navigating to the profile.

## Expected behavior

The existing window handles the URL and navigates to the appropriate screen (profile, thread, etc.) without opening a new window.

## Actual behavior

A new app window is created.

## Notes

This is a macOS SwiftUI scene management issue. By default, `DocumentGroup` or multiple `WindowGroup` scenes will open a new window per URL. Fixes to investigate:

1. In `Bluesky_SwiftUIApp.swift`, ensure only a single `WindowGroup` is declared and it is not configured to open new windows for URLs.
2. Add `.handlesExternalEvents(matching: Set(arrayLiteral: "*"))` to the `WindowGroup` so the OS routes URLs to the existing window rather than spawning a new one.
3. On the `MainTabView` or its scene, add `.handlesExternalEvents(preferring: Set(arrayLiteral: "*"), allowing: Set(arrayLiteral: "*"))` to opt the existing window in as the URL handler.

Universal links would exhibit the same behavior since the underlying scene routing is the same.

## Root cause

macOS SwiftUI's default behaviour for an incoming `bluesky://` URL (or any external event) is to open a new window for the WindowGroup unless the scene explicitly claims the URL pattern. The original fix attempt (commit `f3ace76`) tried to apply `.handlesExternalEvents(preferring:allowing:)` as a Scene modifier on the `WindowGroup`, but that signature is a View modifier, not a Scene modifier — it only compiled because Swift inferred a different overload, and the scene-level routing never saw the matcher. A follow-up (`ade936f`) moved the View modifier inside the `WindowGroup`'s content, which is syntactically correct but still missing the Scene-level `.handlesExternalEvents(matching:)` declaration that the WindowGroup needs in order to claim the URL pattern in the first place.

Without the Scene-level matcher, macOS treats every incoming URL as needing a fresh window because no existing scene has registered for the URL pattern.

## Fix

Apply both modifiers in `Bluesky_SwiftUIApp.swift`:

- **Scene-level** `.handlesExternalEvents(matching: Set(arrayLiteral: "*"))` on the `WindowGroup` — declares that this scene type accepts all external URL/activity events.
- **View-level** `.handlesExternalEvents(preferring: Set(arrayLiteral: "*"), allowing: Set(arrayLiteral: "*"))` on the root view inside the WindowGroup — marks the existing window instance as the preferred handler so macOS routes URLs to it instead of spawning a new window.

The existing `onOpenURL { handleDeepLink($0) }` in `MainTabView` then receives the URL and updates `selectedTab` for paths like `profile/...`, `notifications`, `messages`/`chat`.

## Files changed

- `Bluesky-SwiftUI/Bluesky-SwiftUI/Bluesky_SwiftUIApp.swift` — added Scene-level `handlesExternalEvents(matching:)` alongside the existing View-level `handlesExternalEvents(preferring:allowing:)`.

## Gotchas

- `Scene.handlesExternalEvents(matching:)` and `View.handlesExternalEvents(preferring:allowing:)` are two different APIs with different signatures. They are commonly mistaken for one another and both are required for the deep-link-into-existing-window pattern on macOS.
- The empty-string set trick (`Set([""])`) is sometimes suggested to suppress new-window creation; here we use `"*"` to match all URLs because the app declares only one `WindowGroup` and should always reuse it.
- Verification of the actual routing behaviour requires running the app and triggering `open "bluesky://profile/<handle>"` from Terminal — the build itself only confirms the API shapes are correct.
