## Description

React Native exposes a dedicated **Following Feed Preferences** screen (Settings → Content & Media → Following feed preferences) where the user controls how the Following timeline composes:

- Show replies (toggle) + "From people you follow more than" threshold slider.
- Show reposts (toggle).
- Show quote posts (toggle).
- Show samples from your other feeds (Merge feed lab toggle).

These preferences live at `app.bsky.actor.defs#feedViewPref` keyed by `home`. The SwiftUI client has #0017 (filter toggles) marked resolved, but the implementation is per-tab transient state, not preference-backed and not exposed via Settings.

## Steps to reproduce

1. Look in Settings for "Following feed preferences" or similar.
2. Observe there's no settings entry — the filter toggles are buried in feed UI.

## Expected behavior

A new `FollowingFeedPreferencesScreen.swift` accessible from `ContentSettingsScreen`. Same fields as RN. Persists via `putPreferences` and is consumed by `FeedViewModel` so settings are global (not per-session).

## Actual behavior

Screen does not exist; filter toggles per #0017 are only in-feed UI and don't persist.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Settings/FollowingFeedPreferences.tsx` (single file, not a folder index).
- Lexicon: `app.bsky.actor.defs#feedViewPref` (`feed: 'home'`, `hideReplies: bool`, `hideRepliesByUnfollowed: bool`, `hideRepliesByLikeCount: int`, `hideReposts: bool`, `hideQuotePosts: bool`).
- File to create: `BlueskyKit/Sources/BlueskySettings/FollowingFeedPreferencesScreen.swift`.

## Root cause

`ContentSettingsScreen` only exposed a Threads section (added in #0116). There was no Settings entry point for the per-Following-feed `feedViewPref` flags, and `BlueskyCore` did not model the lexicon's `app.bsky.actor.defs#feedViewPref` value type, so `getPreferences` simply ignored those entries.

## Fix

Modeled the lexicon and added a parity screen.

1. **`BlueskyCore/Moderation.swift`**:
   - New `FeedViewPref` value type with all six lexicon fields (`feed`, `hideReplies`, `hideRepliesByUnfollowed`, `hideRepliesByLikeCount`, `hideReposts`, `hideQuotePosts`) plus a `defaultHome` matching RN's `DEFAULT_HOME_FEED_PREFS`.
   - Extended `GetPreferencesResponse` with a `feedViews: [String: FeedViewPref]` dictionary keyed by `feed`, plus a `homeFeedView` convenience accessor.
   - Added `PutPreferencesRequest(feedView:)` initializer that writes a single `feedViewPref` payload (mirrors the project's existing one-pref-per-PUT convention used by `threadView`, `birthDate`, `interests`, etc.).
2. **`BlueskySettings/FollowingFeedPreferencesScreen.swift`** (new): three toggles — Show replies / Show reposts / Show quote posts — with optimistic update + rollback on failure, matching RN's `FollowingFeedPreferences.tsx`. The view model loads via `app.bsky.actor.getPreferences` and writes via `app.bsky.actor.putPreferences`.
3. **`BlueskySettings/ContentSettingsScreen.swift`**: added a "Feeds" section with a NavigationLink labeled "Following feed preferences" pointing to the new screen, sitting above the existing Threads section.

## Files changed

- `BlueskyKit/Sources/BlueskyCore/Moderation.swift`
- `BlueskyKit/Sources/BlueskySettings/FollowingFeedPreferencesScreen.swift` (new)
- `BlueskyKit/Sources/BlueskySettings/ContentSettingsScreen.swift`

## Gotchas

- **RN's screen has fewer fields than the lexicon advertises.** `hideRepliesByUnfollowed` and `hideRepliesByLikeCount` are commented as **"Legacy, ignored"** in `DEFAULT_HOME_FEED_PREFS` and are *not* surfaced by `FollowingFeedPreferences.tsx`. The dispatch brief asked for an "Engagement threshold" picker and a "Show replies from people you don't follow" toggle, but RN does not ship either — strict-RN-parity won. Both fields still round-trip on `FeedViewPref` (read on load, written back unchanged) so the server's stored values are preserved.
- **Labs flag deferred.** RN exposes a `lab_mergeFeedEnabled` toggle in an "Experimental" group. Following the precedent set by `lab_treeViewEnabled` in #0116, labs flags are not modeled in the SwiftUI `FeedViewPref` value type yet. If/when we ship the merged-feed experiment, add `labMergeFeedEnabled` to `FeedViewPref` and a matching toggle here.
- **Cross-cuts #0017.** That issue resolved by giving the home feed per-tab transient filter chips. With these prefs now server-backed and global, `FeedViewModel` should consume `homeFeedView.hideReplies` / `hideReposts` / `hideQuotePosts` so the Following timeline reflects the user's persisted choice across launches and devices. That wiring is intentionally out of scope here — opening a follow-up if it's not already tracked.
- **`putPreferences` semantics.** The project-wide convention is to PUT a single-element `preferences` array carrying only the pref being edited (see `threadView`, `birthDate`, `interests`). The AT Proto server merges by `$type` and (for `feedViewPref`) by `feed`, so other prefs survive. Confirmed by examining the existing `PutPreferencesRequest(threadView:)` path that has been in production since #0116.
