# 0067 — Search field does not match against issue numbers

| | |
|---|---|
| **Status** | open |
| **Module** | Issues.app |
| **Platform** | macOS |
| **First seen** | 2026-05-05 |

## Description

The search field in the Issues Mac app does not search the issue number. Typing `31` does not narrow the list to issue 0031, and typing `003` does not narrow the list to all issues whose number starts with `003` (0030, 0031, 0032, …). Today the search appears to match against title (and possibly module/platform) only.

The issue number is the most stable identifier — it's how issues are referenced in commit messages, conversations, and cross-issue links — so it should be the most searchable field.

## Steps to reproduce

1. Open Issues.app on the Bluesky-Migration project.
2. Type `31` into the search field.
3. Observe that issue 0031 is **not** in the filtered list (or at best is matched coincidentally because "31" appears in its title or body).

## Expected behavior

Search matches against the issue number as a substring, in addition to whatever it already matches:

- Typing `31` filters to issues whose number contains `31` (matches 0031, 0131, 0231, …).
- Typing `003` filters to issues whose number contains `003` (matches 0030 through 0039, plus any `*003*`).
- Typing `0031` matches exactly 0031.
- Typing the title text continues to match as it does today.

Substring match (not "starts with") is preferable so the user can type the meaningful digits and ignore leading zeros — `31` should reach 0031 just as `0031` does. Pad the user's query with implicit leading zeros up to 4 digits if it is purely numeric, then run substring match against the 4-digit ID; that handles both "31" and "003" naturally.

## Actual behavior

Search ignores the issue number. The user's example: typing `31` does not surface issue 0031.

## Notes

- The Issues.app source lives outside this Bluesky-Migration repo (likely a sibling Issues/ or Issues.app/ project — confirm location before starting).
- Search probably runs over a `searchText` predicate composed of `title + module + platform`. Adding `+ id` (or `+ String(format: "%04d", number)`) to that predicate is the minimum fix.
- Pure-numeric queries should also match unpadded — implementation suggestion:
  ```swift
  let needle = query.trimmingCharacters(in: .whitespaces).lowercased()
  let isNumeric = !needle.isEmpty && needle.allSatisfy(\.isNumber)
  let numericMatch = isNumeric && (
      issue.idString.contains(needle) ||              // "0031" contains "31"
      issue.idString.contains(String(format: "%04d", Int(needle)!))  // "31" → "0031"
  )
  let textMatch = issue.title.localizedStandardContains(needle)
              || issue.module.localizedStandardContains(needle)
              || issue.platform.localizedStandardContains(needle)
  return numericMatch || textMatch
  ```
- Consider also exposing a "Filter by number" hint in the search-field placeholder, so users discover the behavior.

## Related

- This is an Issues.app feature gap; not blocking any Bluesky-Migration work, but it directly impacts the workflow this project uses.
