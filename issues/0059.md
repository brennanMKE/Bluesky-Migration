# 0059 — Network reachability gating using Network framework

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskyNetworking / BlueskyKit |
| **Platform** | All |
| **First seen** | 2026-04-30 |

## Description

The app currently issues network requests without first checking whether a viable network path exists. When the device is offline (no Wi-Fi, no cellular, airplane mode, captive portal blocking), requests fail late at the URLSession layer, producing slow timeouts, noisy errors, and wasted retries. We should use Apple's `Network` framework (`NWPathMonitor`) to track reachability continuously, gate outbound requests on a viable path, and automatically refresh the feed and other relevant data the moment connectivity is restored.

## Expected behavior

- A single shared reachability monitor (built on `NWPathMonitor`) publishes the current network status (`viable` / `not viable`, plus interface type and constrained/expensive flags) as observable state.
- While the path is not viable:
  - `NetworkClient` short-circuits outbound XRPC calls with a typed `noNetwork` error instead of attempting them.
  - The UI surfaces an offline indicator (banner or status pill) and disables actions that require the network (post, like, repost, refresh).
  - Cached content (per issue #0048) continues to render so the app stays usable offline.
- The instant the path becomes viable again:
  - The monitor emits a `.viable` event.
  - Subscribers refresh: home feed, current thread, current profile, notifications badge — whichever screens are visible — and any pending writes are retried.
  - The offline indicator is dismissed.
- The monitor lives in a low layer (Layer 1 `BlueskyKit` protocol + Layer 2 implementation in `BlueskyNetworking`) so any module can observe it via DI without importing `Network` directly.

## Actual behavior

- No reachability check exists. Requests are attempted regardless of path state.
- Offline failures appear as URLSession timeouts after a long delay.
- When connectivity returns, nothing refreshes automatically — the user has to pull-to-refresh or navigate away and back.

## Steps to reproduce

1. Launch the app with a working connection and load the home feed.
2. Enable Airplane Mode (or disable Wi-Fi on macOS).
3. Try to refresh the feed, like a post, or open a profile — observe long timeouts and generic errors with no offline indication.
4. Re-enable connectivity — observe that the feed does not refresh on its own.

## Notes

Implementation sketch:

1. **Protocol in `BlueskyKit`** — define `NetworkPathMonitoring` exposing:
   - `var isViable: Bool { get }` (current snapshot)
   - an `AsyncStream<NetworkPathStatus>` (or Combine publisher) for changes
   - `nonisolated` so non-MainActor callers can read it
2. **Implementation in `BlueskyNetworking`** — wrap `NWPathMonitor` on a dedicated dispatch queue; map `NWPath.Status` plus `isExpensive` / `isConstrained` into a `NetworkPathStatus` value type in `BlueskyCore`.
3. **Gate `NetworkClient`** — before issuing a request, check `isViable`; if false, throw `BlueskyError.noNetwork` immediately. Add the new error case to the existing error enum and map it to a user-friendly message in the UI.
4. **Refresh on reconnect** — feature view models (FeedViewModel, ThreadViewModel, ProfileViewModel, NotificationsViewModel) subscribe to the path stream; on a transition from `not viable → viable`, kick off their refresh routine. Coalesce so multiple simultaneous viewmodels don't stampede the API.
5. **UI affordance** — add a thin offline banner component in `BlueskyUI` that observes the monitor and shows when offline; place it under the navigation bar across all tabs.
6. **Pending writes** — likes, reposts, posts created while offline should queue (or at minimum show an explicit "can't send while offline" error). Initial scope can be: block the action with a clear message; queueing can be a follow-up.

Related issues:
- #0048 — feed cache (offline content)
- #0053 — like state persistence (writes that should survive a reconnect)

Test approach:
- Unit-test the path-status mapping with a mock `NWPathMonitor`-shaped seam.
- Manual test matrix: airplane mode toggle, Wi-Fi off, captive portal, slow network, expensive (cellular) path. Verify gating, offline banner, and auto-refresh on each.

### 2. New file: `BlueskyKit/NetworkPathMonitoring.swift`

**Path:** `/Users/brennan/Developer/ReactNative/BlueskyKit/Sources/BlueskyKit/NetworkPathMonitoring.swift`

**Purpose:** Layer 1 protocol. `BlueskyKit` depends on `BlueskyCore`, so it can reference `NetworkPathStatus`.

**Protocol shape:**

```swift
import BlueskyCore

/// Observes the device's network path and vends a live status stream.
///
/// Implementations live in `BlueskyNetworking` (Layer 2). All requirements are
/// `nonisolated` so the protocol is usable from background actors and non-`@MainActor`
/// contexts without extra hops.
public protocol NetworkPathMonitoring: AnyObject, Sendable {
    /// Current path viability — a synchronous snapshot safe to read anywhere.
    nonisolated var isViable: Bool { get }

    /// Current status value — a synchronous snapshot.
    nonisolated var status: NetworkPathStatus { get }

    /// Infinite async stream of status changes. Each element is emitted whenever
    /// the underlying `NWPath` transitions (viable ↔ not viable, interface change, etc.).
    /// The stream never completes normally; cancel the consuming `Task` to stop it.
    nonisolated var statusStream: AsyncStream<NetworkPathStatus> { get }
}
```

**Imports:** `BlueskyCore` only (no `Foundation`, no `Network`).

**Gotchas:**
- `BlueskyKit` has `swiftSettings: swiftSettings` (`defaultIsolation(.MainActor)`) — adding `nonisolated` on each requirement is required so callers outside `@MainActor` can use the protocol without hopping to the main actor.
- Do not mark the protocol `@MainActor`. The monitor must be queryable from `ATProtoClient` (a custom actor) and from `FeedViewModel` background Tasks.

### 3. New file: `BlueskyNetworking/NWPathMonitorAdapter.swift`

**Path:** `/Users/brennan/Developer/ReactNative/BlueskyKit/Sources/BlueskyNetworking/NWPathMonitorAdapter.swift`

**Purpose:** Layer 2 concrete implementation. This is the only file in the package that imports `Network`.

**Type shape:**

```swift
import Foundation
import Network
import BlueskyCore
import BlueskyKit

/// Production implementation of `NetworkPathMonitoring` backed by `NWPathMonitor`.
///
/// Create one shared instance at app boot and pass it through `BlueskyEnvironment`.
/// Call `start()` immediately after creation.
public final class NWPathMonitorAdapter: NetworkPathMonitoring, @unchecked Sendable {

    private let monitor = NWPathMonitor()
    private let queue = DispatchQueue(label: "co.sstools.bluesky.network-path-monitor")

    // Continuation for the AsyncStream. Stored so we can yield to it from the
    // NWPathMonitor handler running on the dedicated queue.
    private var continuation: AsyncStream<NetworkPathStatus>.Continuation?

    // nonisolated storage backed by a lock so reads are safe from any context.
    private let _statusLock = NSLock()
    private var _status: NetworkPathStatus = .viable  // optimistic default until first callback

    public nonisolated var isViable: Bool { status == .viable }

    public nonisolated var status: NetworkPathStatus {
        _statusLock.withLock { _status }
    }

    public nonisolated let statusStream: AsyncStream<NetworkPathStatus>

    public init() {
        var captured: AsyncStream<NetworkPathStatus>.Continuation?
        statusStream = AsyncStream { continuation in
            captured = continuation
        }
        continuation = captured
    }

    /// Starts the underlying `NWPathMonitor`. Call once at app boot.
    public func start() {
        monitor.pathUpdateHandler = { [weak self] path in
            guard let self else { return }
            let newStatus = NetworkPathStatus(path)
            self._statusLock.withLock { self._status = newStatus }
            self.continuation?.yield(newStatus)
        }
        monitor.start(queue: queue)
    }

    /// Cancels the monitor. Call when the app is terminating (optional in practice).
    public func stop() {
        monitor.cancel()
        continuation?.finish()
    }
}

// MARK: - NWPath → NetworkPathStatus mapping

private extension NetworkPathStatus {
    init(_ path: NWPath) {
        if path.status == .satisfied {
            self = .viable
        } else {
            let reason: String?
            switch path.status {
            case .unsatisfied:  reason = "no network interface available"
            case .requiresConnection: reason = "connection required (captive portal?)"
            default:            reason = nil
            }
            self = .notViable(reason: reason)
        }
    }
}
```

**Imports:** `Foundation`, `Network`, `BlueskyCore`, `BlueskyKit`.

**Gotchas:**
- `BlueskyNetworking` has no `swiftSettings` (no `defaultIsolation`) in `Package.swift`. The class must be marked `@unchecked Sendable` because `NSLock` is not `Sendable`, and the manual locking makes it safe.
- `NWPathMonitor` must be cancelled on the same queue it was started on — do not call `monitor.cancel()` from a different thread without care. Here `stop()` is called from the main thread at app termination, which is fine because `NWPathMonitor.cancel()` is thread-safe per Apple's documentation.
- The `AsyncStream` is constructed once; the continuation is stored. Yields happen from the `NWPathMonitor` handler on the dedicated `DispatchQueue`. `AsyncStream` is safe for cross-actor consumption.

### What NOT to do

- **Do not import `Network` in `BlueskyCore` or `BlueskyKit`.** The `Network` framework is a Layer 2 concern and must stay in `BlueskyNetworking` only. This is why `NetworkPathStatus` is a plain Swift enum rather than a wrapper around `NWPath`.
- **Do not use Combine.** The existing codebase uses `async/await` and `AsyncStream` throughout. Adding a `Publisher` would introduce an inconsistent pattern and force callers to manage cancellables.
- **Do not make `NetworkPathMonitoring` `@MainActor`.** The protocol is used inside `ATProtoClient` (a custom actor) and from background `Task`s. A `@MainActor` constraint would require every caller to hop to the main actor just to read `isViable`.
- **Do not use `NWPathMonitor` inside `BlueskyCore` or `BlueskyKit`.** It requires `import Network`, which belongs in Layer 2 only.
- **Do not add `pathMonitor` as an `@EnvironmentObject` or `@Environment` key** separate from `BlueskyEnvironment`. It should be accessed as `env.pathMonitor` to keep DI consistent with the rest of the app.
