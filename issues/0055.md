## Description

When the Home tab is showing a thread view (pushed via NavigationStack), clicking a different tab in the sidebar (e.g. Profile) does not switch to that tab. The thread view remains visible. The sidebar highlights the new tab but the detail pane does not change.

## Steps to reproduce

1. Open the app and tap any post to open the thread view.
2. Click "Profile" in the sidebar.
3. Observe that the thread view is still displayed — Profile does not appear.
4. Press the back button to return to the feed — only then does the Profile screen appear.

## Expected behavior

Each tab has its own independent navigation stack. Clicking a sidebar tab immediately switches the detail pane to that tab's root screen, regardless of what is pushed on any other tab's stack.

## Actual behavior

The detail pane stays on the thread view until the user manually pops back to the feed. The tabs share a single NavigationStack instead of having independent stacks per tab.

## Notes

On macOS, `MainTabView` uses a `NavigationSplitView` with a single `NavigationStack` wrapping `tabContent(selectedTab)`. When `selectedTab` changes, the stack's root changes but the existing pushed views may not be cleared. The fix is to ensure each tab's navigation stack is isolated — either by keying the `NavigationStack` on the selected tab (`.id(selectedTab)`) so it resets when the tab changes, or by maintaining a separate `NavigationPath` per tab.

## Root cause

`MainTabView` shares a single `NavigationStack` across all tabs in the `NavigationSplitView` detail column (used on macOS and on iPadOS in regular size class). Switching `selectedTab` only swaps the stack's root content but leaves any previously-pushed destinations (e.g. `ThreadView`, `ProfileScreen`, `SettingsScreen`) on the stack, so the detail pane keeps rendering the pushed view until the user manually pops back. There is also no per-tab `NavigationPath`, so navigation state from one tab leaks into another when the user revisits.

## Fix

Two coordinated changes in `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift`:

1. Apply `.id(selectedTab ?? AppTab.home)` to the detail-column `NavigationStack` (both the macOS branch and the iPadOS regular-width branch). Re-keying the view forces SwiftUI to rebuild a fresh `NavigationStack` whenever the tab changes, which discards any pushed destinations and shows the new tab's root immediately.
2. In `.onChange(of: selectedTab)` invoke a new `resetTransientNavState()` helper that clears every navigation-state binding owned by `MainTabView`: `threadURI`, `feedProfileDID`, `showSavedFeeds`, `showModeration`, `showSettings`, `showBookmarks`, and `showLists`. Without this, an `isPresented` binding left at `true` would re-push its destination onto the freshly-rebuilt stack the next time the user returns to that tab.

The first commit (`19c72ed`) added `.id` plus the partial reset (only `threadURI` and `feedProfileDID`); the follow-up (`287d836`) extends the reset to all the Profile-tab `isPresented` flags and consolidates the cleanup into the shared helper.

## Files changed

- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift`

## Gotchas

- `.id(selectedTab)` is unconditionally destructive: it tears down the entire stack on every switch, which means in-flight loads in pushed views are cancelled and view state is lost. That is acceptable here because the bug is exactly that pushed views were sticking around — but if a future requirement wants per-tab navigation history, switch to a `[AppTab: NavigationPath]` dictionary instead and bind each tab's `NavigationStack(path:)` to its own path.
- The iPhone (compact) branch uses `TabView` with one `NavigationStack` per tab and is unaffected — `TabView` already isolates stacks per tab.
- Any `@State` flag added to `MainTabView` that drives a `navigationDestination(isPresented:)` should also be cleared in `resetTransientNavState()`, otherwise the same re-push bug will resurface.
