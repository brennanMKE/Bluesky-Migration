## Description

A black gap sits between the iOS status bar and the top of the app's chrome on every screen — Home, Notifications, Search, and Profile. The status bar is rendered against a black backing rather than the screen's background, so the slim top bars (#0072, #0077) and the Profile banner (#0083) appear to "float" with a dead band above them. The React Native reference has no such gap — the app fills behind the status bar.

The user's diagnosis is on the nose: the screens are respecting the top safe-area inset where they should be ignoring it (banner / top bar should extend behind the status bar, with the *content* respecting the safe area).

## Steps to reproduce

1. Run the SwiftUI iOS app on an iPhone with a notch (any post-iPhone X).
2. Open any of Home, Notifications, Search, or Profile.
3. Observe the dark band between the status-bar elements (time, signal, battery) and the app's top bar / banner.

## Expected behavior

The app's surface fills the entire screen vertically. The status bar's content (time / battery / signal) sits over the same color as the rest of the top bar (Home: dark chrome; Profile: the banner image). RN parity.

## Actual behavior

A solid black band runs across the top above every screen's chrome.

## Steps to fix

The fix is almost certainly in the app shell, not in each screen:

- Apply `.ignoresSafeArea(edges: .top)` to the top-bar background **only** (`BlueskyTopBar`'s background fill, the banner image in `ProfileHeaderView`), keeping the bar's *content* (hamburger, title, gear) inside the safe area via padding.
- For `MainTabView`'s compact iOS layout, audit the outer container — currently a `VStack` likely respects safe area at the top; the chrome row needs to render behind the status bar while the scrollable feed below stays inside the safe area.
- Verify with a fresh iPhone 17 Simulator screenshot on each of Home / Notifications / Search / Profile. The dead band should disappear and the status-bar glyphs should sit over the bar's color.

## Implementation notes

- This may be a single-line fix at the right level of the view hierarchy — apply `.background(theme.colors.surface, ignoresSafeArea(.top))` on the top-bar wrapper, or use `.safeAreaInset(.top)` to mount the bar with its background extending to the device edge.
- For Profile (#0083), the banner already has `.ignoresSafeArea(edges: .top)`, but the gap is still visible — suggests the parent `NavigationStack` or `ScrollView` is constraining the banner rather than letting it bleed up. Check the order of modifiers and whether `.toolbar(.hidden)` is actually hiding the bar's container.
- Search screen has a giant "Search" headline (separate concern, tracked in #0090) but the safe-area gap above it is the same root cause.

## Root cause

The iPhone-compact app shell wrapped each tab's content in a
`VStack(spacing: 0)` inside a `NavigationStack`. SwiftUI's default is to
respect the top safe-area inset on a `VStack`, so every child — the
`BlueskyTopBar` on Home/Notifications, the giant `.large` system title on
Search, and the `ProfileScreen` `ScrollView` whose first child is the
banner — started at safe-area-top. The status-bar zone above the chrome
was the system's default backing (black on dark mode), which read as a
dead band above the chrome on every screen.

`BlueskyTopBar` itself owned a 44pt-tall HStack with a `theme.colors.background` fill that didn't extend past its bounds, so even when mounted flush with the safe-area edge, the chrome color stopped at the top of the bar instead of bleeding behind the status bar. `ProfileScreen` already had `.ignoresSafeArea(edges: .top)` on the banner image, but a constraining ancestor (the `VStack` plus the `NavigationStack` plus the `ScrollView`'s own safe-area handling) blocked it from reaching the screen edge.

## Fix

Three coordinated, minimal changes — the principle is *background ignores the top safe area; content stays inside it*:

1. **`BlueskyTopBar.swift`** — apply `.ignoresSafeArea(edges: .top)` to the bar's background fill (theme color + hairline border), not to the bar's HStack content. The 44pt content height is unchanged; only the fill bleeds up. The modifier is a no-op when the bar isn't at the screen's top edge, so it's safe to leave on regardless of where the bar is mounted.
2. **`MainTabView.swift`** — in `iosCompactLayout`, replace `NavigationStack { VStack { topBar; tabContent } }` with `NavigationStack { tabContent.safeAreaInset(edge: .top) { topBar } }`. The `safeAreaInset` modifier mounts the bar inside the safe area while letting its (now bleeding) background extend behind the status bar.
3. **`ProfileScreen.swift`** — apply `.ignoresSafeArea(edges: .top)` to the outer `ScrollView` on iOS so the LazyVStack's first child (the banner) actually reaches the screen edge. macOS keeps its `.navigationTitle` path untouched.

Search still uses the system large title, which already bleeds behind the status bar — there was no extra change needed for the safe-area gap; the giant headline itself is a separate concern tracked in #0090.

## Files changed

- `BlueskyKit/Sources/BlueskyUI/BlueskyTopBar.swift` — background `.ignoresSafeArea(edges: .top)` so the chrome color extends behind the status bar.
- `BlueskyKit/Sources/BlueskyProfile/ProfileScreen.swift` — outer ScrollView `.ignoresSafeArea(edges: .top)` (iOS only) so the banner reaches the screen edge.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` — `iosCompactLayout` now mounts the slim top bar via `.safeAreaInset(edge: .top)` instead of the previous `VStack`-as-first-child layout.

## Gotchas

- **`.ignoresSafeArea` belongs on the *background*, not the bar's content.** Putting it on the HStack would push the icons up under the status bar; putting it on the bar's `.background(...)` lets the fill extend without touching layout. Same principle for the banner: image bleeds, the avatar / action row stays put.
- **No app-wide `.ignoresSafeArea`.** The fix is pinned to the chrome row's background and to ProfileScreen's outer ScrollView. Going broader would push nested ScrollView content under the status bar.
- **Search's giant headline (#0090) is separate.** This fix doesn't change Search's chrome — when Search picks up a slim `BlueskyTopBar` later, it'll inherit the same status-bar bleed for free because the bar's background already ignores top safe area.
- **`.safeAreaInset(.top)` with `EmptyView()`.** SwiftUI special-cases `EmptyView` as a 0-size inset, so Profile's case (no chrome row) is effectively a no-op — Profile's own `.ignoresSafeArea(.top)` on the ScrollView is what makes the banner reach the status bar.
- **macOS / iPad regular paths untouched.** The change in `MainTabView` is inside `iosCompactLayout` (iPhone only); the `BlueskyTopBar` background change is harmless on other platforms because the bar isn't mounted at the screen's top edge there. `ProfileScreen` gates its new modifier with `#if os(iOS)`.
