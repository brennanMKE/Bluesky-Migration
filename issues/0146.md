## Description

In the React Native thread view, the **focal** post (the one the user navigated to) is visually distinct from its parents and replies. RN's `ThreadItemAnchor`:

- Uses a **larger avatar** (~42pt vs the ~32pt used elsewhere).
- Uses **larger text** (`text_lg` body).
- Shows a **full timestamp** ("Mar 4, 2026 · 10:42 AM"), not just the relative "2h" used in feed.
- Renders an **expanded stats section**: tappable "X likes · Y reposts · Z quotes" rows that open the screens in #0139.
- Includes a **"Who can reply"** badge if the thread has a threadgate.
- The avatar/handle/body block has more vertical breathing room.

The SwiftUI `ThreadView` reuses the same `PostCard` for every node with no per-position styling. The user can't immediately tell which post they tapped into.

## Steps to reproduce

1. Open any thread.
2. Identify which post is the one you navigated to.
3. Observe nothing visually distinguishes it from the surrounding posts.

## Expected behavior

`ThreadView` detects the focal post (the one matching the requested `uri`) and renders it via a different layout:

- Larger avatar (42pt).
- Larger body text (use `theme.typography.bodyLarge`).
- Full date subtitle below display name + handle.
- Tappable stat row above the action bar (gates into the screens from #0139).
- "Who can reply" badge if threadgate exists.
- Slightly larger vertical padding.

All other rows stay on the existing `PostCard` (or a tree-renderer per #0141).

## Actual behavior

Every post node uses the same `PostCard`; focal post is indistinguishable.

- RN reference: `Bluesky-ReactNative/src/view/com/post-thread/ThreadItemAnchor.tsx`.
- File to edit: `BlueskyKit/Sources/BlueskyFeed/ThreadView.swift`. Either extract a new `FocalPostCard` view or pass an `isFocal` parameter to `PostCard` and switch the styling internally.

## Root cause

`ThreadView` rendered every node — focal post, ancestors, and replies — through the same `PostCard`. There was no per-position styling, so the post the user navigated to was visually indistinguishable from the surrounding rows. RN's `screens/PostThread/components/ThreadItemAnchor.tsx` is a dedicated component that swaps in a larger avatar, larger typography, a full-precision timestamp, a tappable stats strip, and a "Who can reply" badge — none of those treatments existed on the SwiftUI side.

The plumbing for the stat row's destinations had landed in #0139 (the three optional callbacks `onLikedByTap` / `onRepostedByTap` / `onQuotesTap` on `ThreadView`) and the unclamped body had landed in #0142 (`PostBodyView(isFocal: true)`), so this issue is the UI that fires those callbacks.

## Fix

Added a new `FocalPostCard` view in `BlueskyUI` as a sibling of `PostCard` (Option A from the spec — keeps `PostCard` simple). `ThreadView` swaps it in for the row whose URI matches `focalPostURI(...)`; all other rows in `linearRows` continue to render `PostCard`. RN parity items, all driven from `ThreadItemAnchor`:

- **Avatar 42pt** — RN's `LINEAR_AVI_WIDTH = 42`. (`PostCard` uses 44pt.)
- **Display name uses `Typography.lg, semibold`** and handle uses `Typography.body` — matches RN's `text_lg` / `text_md`.
- **Body uses `PostBodyView(isFocal: true)`** so the long-post clamp from #0142 is bypassed, and it's set to `Typography.bodyLarge` to match RN's `a.text_lg`.
- **Full-precision timestamp** rendered as `time · date` (e.g. `10:42 AM · Mar 4, 2026`) using two `DateFormatter` instances — matches RN's `niceDate(i18n, indexedAt, 'dot separated')`.
- **"Who can reply" badge** rendered when the focal post's `threadgate.record.allow` is non-nil. Empty array → "Replies disabled"; non-empty → "Only mentioned users / followed users / your followers / a custom list can reply" (with English-list join — RN uses `lingui` for the localized version, which is a follow-up). Hidden when no threadgate is attached, matching RN's "default → omit the badge" behavior.
- **Tappable stat row** rendered above the action bar inside two thin border lines (mirrors RN's `border_t + border_b + py_md`). Order is `reposts · quotes · likes`, each with the count formatted by `CompactNumberFormatter` and the singular/plural label following it. Each tap fires the matching callback on `FocalPostCard`, which `ThreadView` forwards from its existing `onLikedByTap` / `onRepostedByTap` / `onQuotesTap` (added by #0139). The strip is suppressed when all three counts are zero, matching RN.
- **Larger vertical padding** (`Spacing.lg` top/bottom on the card) and a slightly larger action-bar icon size (18pt vs 16pt) to mirror RN's `<PostControls big />`.

Wired `MainTabView` (Bluesky-SwiftUI) for the new destinations: three new `@State` AT-URIs (`likedByPostURI`, `repostedByPostURI`, `quotesPostURI`) plus `navigationDestination(item:)` blocks attached to both ThreadView call sites (Home and Saved tabs) that push `LikedByScreen`, `RepostedByScreen`, and `QuotesOfScreen`. Author taps inside those screens reuse the existing `feedProfileDID` so back-navigation behaves consistently.

## Files changed

- `BlueskyKit/Sources/BlueskyUI/FocalPostCard.swift` — new file. Larger-avatar focal-post layout with timestamp subtitle, threadgate badge, stat row, action bar, plus light/dark previews.
- `BlueskyKit/Sources/BlueskyFeed/ThreadView.swift` — swap `FocalPostCard` in for the focal row in `linearRows`; new helper `focalThreadgateAllow(_:)` that pulls `threadgate.record.allow` off the loaded thread.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` — three new `@State` ATURIs; `onLikedByTap` / `onRepostedByTap` / `onQuotesTap` callbacks attached to both `ThreadView` constructions; `navigationDestination(item:)` blocks for `LikedByScreen` / `RepostedByScreen` / `QuotesOfScreen`.

## Gotchas

- RN's `WhoCanReply` localizes through `lingui`; this implementation uses an inline English list-formatter (`a, b, and c`) to avoid pulling a localization dependency. The label copy ("Only X can reply" / "Replies disabled") matches RN; the localized variants will need to come back through a future shared string-table.
- The `Bluesky-SwiftUI.entitlements` file showed an unrelated diff at commit time (Xcode added `aps-environment` for development push during a previous build session). That diff was deliberately left out of the resolution commit.
- `PostBodyView(isFocal: true)` already existed from #0142; the `.font(Typography.bodyLarge)` modifier on the wrapping view sets the displayed font without changing the `RichTextView` internals. If we later want true variable-font sizing inside `RichTextView`, this is the spot to thread a `font:` parameter through.
