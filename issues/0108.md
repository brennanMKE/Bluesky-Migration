## Description

In React Native, a message bubble can contain rich embeds the same way a post can: link cards (`app.bsky.embed.external`), quoted posts (`app.bsky.embed.record`), and record-with-media compounds. RN's `MessageItemEmbed` renders all three. SwiftUI's `MessageBubble` only handles raw image embeds — link previews and shared posts inside DMs render as plain text URLs or worse, are silently dropped.

A practical consequence: when a user shares a Bluesky post into a DM, the recipient sees the at:// URI as text instead of an inline preview of the post.

## Steps to reproduce

1. Send (or receive) a DM that shares a Bluesky post (use the iOS share-sheet "Send via DM" flow, or the recipient sends from RN).
2. Open the thread in the SwiftUI app.
3. Observe the post is shown as raw text, not as an inline post preview.

## Expected behavior

`MessageBubble` consumes the `MessageView.embed` field and renders:

- `app.bsky.embed.external` — link card with thumbnail, title, host, description (existing `PostEmbedView` handles this).
- `app.bsky.embed.record` — inline quoted post (avatar + name + truncated body + first image if any). Tap navigates to the thread for that post.
- `app.bsky.embed.recordWithMedia` — both, side by side.

For best parity, `PostEmbedView` from `BlueskyUI` is reusable here — the message bubble just sizes it down.

## Actual behavior

Only `app.bsky.embed.images` is rendered. All other embed types are ignored.

## Notes

- RN reference: `Bluesky-ReactNative/src/components/dms/MessageItemEmbed.tsx` (the path in the original spec — `screens/Messages/components/MessageItemEmbed.tsx` — does not exist; RN actually wires the chat embed through `components/dms/MessageItemEmbed.tsx`, which simply renders `<Embed>` from `components/Post/Embed`).
- Lexicon: `chat.bsky.convo.defs#messageView.embed` carries the same union as a post's embed.
- File to edit: `BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift`. Inside `MessageBubble`, switch on `message.embed` and dispatch to `PostEmbedView` for non-image cases.

## Root cause

`MessageBubble` only consumed the `images` and `recordWithMedia(_, .images)` cases of `MessageView.embed`. `external` (link cards) and `record` (quoted posts) were silently dropped, and the quote half of `recordWithMedia` was discarded. Users saw at:// URIs as plain text and no link previews at all.

## Fix

`MessageBubble` now also routes link cards and quoted posts through `PostEmbedView` from `BlueskyUI`, the same renderer used by feed and thread post cards.

A new `richEmbedForRender` helper picks out the embed branch that should flow to `PostEmbedView`:

- `external` → renders as-is (link card with thumb / title / host).
- `record` → renders as-is (inline quote post — avatar + name + truncated body).
- `recordWithMedia` → strips the media half (already drawn by `imageStack` above) and hands `PostEmbedView` only the quote half, so images do not appear twice.
- `images`, `video`, `unknown` → returns `nil`; the bubble's own image renderer handles images, the others are deferred.

`PostEmbedView`'s callbacks are wired to platform-native handlers:

- `onLinkTap` → `@Environment(\.openURL)` — launches in the system / in-app browser, mirroring RN's link handoff.
- `onRecordTap` → a new `onPostTap: ((ATURI) -> Void)?` closure piped up through `MessageThreadScreen`'s initializer, so the host app (`MainTabView`) can route into the existing `threadURI` navigation destination just like feed/profile post taps.

The rendered embed is constrained to `maxWidth: 280` so it sits comfortably inside the message column.

## Gotchas

- The original issue spec cited `Bluesky-ReactNative/src/screens/Messages/components/MessageItemEmbed.tsx`. That path does not exist — the actual file lives at `src/components/dms/MessageItemEmbed.tsx`. The RN component is a thin wrapper around the standard `<Embed/>` from `components/Post/Embed`, which is exactly the pattern we follow by reusing `PostEmbedView`.
- Quoted-post tap navigation is opt-in via the new `onPostTap:` parameter on `MessageThreadScreen`. The current `MainTabView` call site (`case .messages:`) does not yet pass it, so taps on a quoted post in DMs are no-ops in the app shell. Wiring this through the messages-tab navigation stack (which currently has no `threadURI` destination on the messages tab) is a small follow-up — the BlueskyKit-side plumbing is in place.
- Video embeds in DMs are not rendered; RN's chat lexicon does not currently allow them. If that changes upstream we can add a `.video` branch to `richEmbedForRender`.
