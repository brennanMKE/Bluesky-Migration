## Description

Once a list is created in the SwiftUI app, it cannot be edited or deleted. The React Native client lets the list creator:

- **Edit** the list — change name, description, avatar, purpose. Reuses the create form in edit mode.
- **Delete** the list — destructive confirmation, then `com.atproto.repo.deleteRecord` on `app.bsky.graph.list`.
- **Add / remove members** — managed inline from the Members tab.

SwiftUI has `ListCreateSheet` for creation but no edit path and no delete affordance.

## Steps to reproduce

1. Create a list.
2. Open it.
3. Look for an Edit button or a delete option in any menu.
4. Observe none exist.

## Expected behavior

In the list-detail header's secondary-actions menu (added by #0125):

- **Edit list** — opens an edit sheet that mirrors `ListCreateSheet` but pre-populated and writes via `com.atproto.repo.putRecord`.
- **Delete list** — destructive confirm dialog. On confirm, deletes the list record and pops to `ListsScreen`.

The Members tab gains Add / Remove member actions for the list creator.

## Actual behavior

No edit or delete UI; lists are append-only after creation.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/ProfileList/Header.tsx` (menu) and `src/screens/Lists/components/EditListDialog.tsx`.
- Endpoints: `com.atproto.repo.putRecord` (edit), `com.atproto.repo.deleteRecord` (delete), `com.atproto.repo.applyWrites` (member add/remove batches).
- Files to edit: `BlueskyKit/Sources/BlueskyLists/ListDetailScreen.swift` (menu wiring), `ListsViewModel.swift` / `ListDetailViewModel.swift` (mutations), and a new `ListEditSheet.swift`.

## Root cause

`ListDetailScreen` shipped (with #0125) without owner-only edit / delete
affordances on the More-options menu, and the underlying store / view
model exposed no `editList` or `deleteList` mutation. RN's
`MoreOptionsMenu.tsx` puts both items in a creator-only `Menu.Group`
just below Share, but the SwiftUI port stopped at Share / Report. The
list-record lexicon (`ListRecord` in `BlueskyCore/Graph.swift`) was
also write-only (`Encodable`-only) and could not round-trip a
`getRecord` → mutate → `putRecord` flow without dropping the avatar
blob, which is the path RN's `useListMetadataMutation` takes.

## Fix

- **Lexicon**: promoted `ListRecord` to `Codable` and added an optional
  `avatar: BlobRef?` field. The struct now decodes from the result of
  `com.atproto.repo.getRecord` (so the avatar blob and `createdAt`
  survive the round-trip) and re-encodes for `com.atproto.repo.putRecord`.
- **Store mutations**: extended `ListDetailStoring` with
  `editList(name:description:)` and `deleteList()`. The implementation
  resolves the signed-in viewer DID via the existing `AccountStore`
  protocol, fetches the existing record, mutates name/description (RN's
  read-modify-write pattern in `useListMetadataMutation`), then
  `putRecord`s it back. Edit optimistically rebuilds the local
  `ListView` so the header / About tab update without a refetch.
  Delete just calls `com.atproto.repo.deleteRecord` and returns
  `true` so the screen can pop.
- **View model**: `ListDetailViewModel` gained matching `editList` /
  `deleteList` passthrough methods and now accepts an optional
  `accountStore` so the store can resolve the viewer DID.
- **Sheet**: new `ListEditSheet.swift` mirrors `ListCreateSheet` but in
  edit mode — pre-populates name / description from the loaded
  `ListView`, shows the purpose read-only (the lexicon does not allow
  changing purpose after creation; RN's edit dialog also treats it as
  immutable), and renders title `Edit user list` / `Edit moderation
  list` per RN's `CreateOrEditListDialog`. Save fires an
  `(name, description?)` callback the screen forwards to the store.
- **More-options menu**: added two new items in `ListDetailScreen` —
  *Edit list details* (pencil icon) and *Delete list* (trash, role
  `.destructive`) — gated on `viewerDID == list.creator.did` to match
  RN's `isOwner` check. For non-owners the menu still shows *Report
  list* in the same slot. The unmute group moved below the divider so
  it always renders last. Delete fires a destructive
  `confirmationDialog` ("Delete this list? — If you delete this list,
  you won't be able to recover it.") matching RN's prompt copy. On
  confirm the store mutation runs and on success the screen calls
  `dismiss()` to pop back to the lists hub.
- **Wiring**: `ListsScreen` resolves `viewerDID` once from
  `accountStore.loadCurrentDID()` and forwards it (along with the
  account store) to `ListDetailScreen`. The new init parameters are
  optional so the existing previews and call sites continue to compile.

## Files changed

- `BlueskyKit/Sources/BlueskyCore/Graph.swift` — `ListRecord` is now

- `BlueskyKit/Sources/BlueskyLists/ListsStore.swift` —

- `BlueskyKit/Sources/BlueskyLists/ListDetailViewModel.swift` —

  optional `accountStore` parameter on the initializer.
- `BlueskyKit/Sources/BlueskyLists/ListDetailScreen.swift` — owner
  detection (`isOwner`), Edit / Delete menu items, edit sheet

- `BlueskyKit/Sources/BlueskyLists/ListsScreen.swift` — resolves the

## Gotchas

- **Inline member add / remove is deferred.** RN's Members tab lets
  the creator add / remove members inline, and the issue spec lists
  this as part of the original scope. The dispatch instructions
  flagged it as a bail-rule deferral; this change implements only
  edit + delete for the list itself. The underlying mutations are
  already available on `ListsStore` (`addMember` / `removeMember`),
  so a follow-up issue can add the inline UI on top without store
  changes. File the follow-up in the issues hub when ready.
- **Avatar editing is not supported in the sheet.** The sheet edits
  name and description only. Avatar editing requires a blob upload
  (`com.atproto.repo.uploadBlob` → `BlobRef`) and an image picker —
  matching the precedent set by `EditProfileSheet` (which also defers
  avatar / banner editing). The lexicon side is ready: `ListRecord`
  now carries `avatar: BlobRef?` and the `getRecord` → `putRecord`
  flow preserves it across edits, so the existing avatar is **not**
  wiped when only name / description change. Adding the picker is a
  follow-up.
- **Purpose is rendered immutable.** The `app.bsky.graph.list`
  lexicon does not allow changing `purpose` after creation, and RN's
  `CreateOrEditListDialog` reads `list.purpose` once and never lets
  the user change it. The SwiftUI sheet shows it as a read-only row
  with a "Cannot be changed" hint to match.
- **Delete only removes the list record.** RN's
  `useListDeleteMutation` also bulk-deletes the list's `listitem`
  records via `applyWrites`. The appview cleans up orphaned listitem
  records server-side, and the SwiftUI client does not currently
  iterate the items, so we ship the minimal `deleteRecord` call.
  If the appview ever stops cleaning up, switch the call to an
  `ApplyWritesRequest` that batches the listitem deletes.
- **Owner detection threads `viewerDID` through `ListsScreen`.**
  `ListDetailScreen` cannot import an authenticated session manager
  (the lower layers don't ship one), so `ListsScreen` resolves the
  current DID via `AccountStore.loadCurrentDID()` and passes it down.
  When viewing another user's lists hub, `viewerDID` is still the
  signed-in user's DID, and the comparison against
  `list.creator.did` in `ListDetailScreen` correctly hides the
  edit / delete menu for other people's lists.
