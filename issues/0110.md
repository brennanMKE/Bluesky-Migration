## Description

The React Native chat client supports per-message emoji reactions: a user picks an emoji from a picker, the reaction attaches to the message, and a small reaction strip renders below the bubble showing each emoji and its count. Tapping the strip opens a dialog listing who reacted with what. RN exposes the picker via the per-message context menu (covered by #0106) and via a "+" affordance on long-press.

The SwiftUI client has no reactions UI: no picker, no rendered reaction strip, no dialog.

## Steps to reproduce

1. Open a thread where the other party (in RN) has reacted to one of your messages.
2. Observe the reaction is not shown in the SwiftUI bubble.
3. Try to add a reaction — there's no affordance.

## Expected behavior

- **Reaction strip**: under any bubble that has reactions, render a horizontal pill row with each unique emoji and its count (e.g. "👍 3 ❤️ 1").
- **Tap a pill**: opens a "Reacted by" sheet listing each user grouped under each emoji.
- **Long-press on bubble** (or context menu — see #0106): opens an emoji picker. Selecting one writes the reaction via `chat.bsky.convo.addReaction`.
- **Tap your own emoji in the strip**: removes the reaction (`chat.bsky.convo.removeReaction`).

## Actual behavior

No reactions UI at all; reactions present on incoming messages are dropped silently.

## Notes

- RN reference: `Bluesky-ReactNative/src/components/dms/MessageItem.tsx` (reaction strip / `groupedReactions`), `EmojiReactionPicker.tsx` (quick picker), `ReactionsDialog.tsx` (per-user "reacted by" sheet — deferred), and `state/messages/convo/agent.ts` (`addReaction` / `removeReaction`).
- Lexicon: `chat.bsky.convo.defs#reactionView`, plus the `addReaction` / `removeReaction` procs. Added the new types to `BlueskyCore/Chat.swift`.
- File to edit: `BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift` and `MessageThreadViewModel.swift`.

## Root cause

The chat surface had no representation of `chat.bsky.convo.defs#reactionView`. `MessageView` did not decode `reactions`, the store had no `addReaction` / `removeReaction` mutations, and `MessageBubble` had nowhere to render a reaction strip. Reactions arriving on inbound messages were dropped during JSON decode and the user had no affordance to add or remove a reaction.

## Fix

- `BlueskyCore/Chat.swift`: added `ReactionView` (`value`, `sender: ReactionSender`, `createdAt`) and `ReactionSender` (`did`). Added optional `reactions: [ReactionView]?` to `MessageView`. Added `AddReactionRequest`, `RemoveReactionRequest`, and `ReactionResponse { message: MessageView }`.
- `MessageThreadStore`: added `addReaction(messageId:emoji:viewerDID:convoId:)` and `removeReaction(...)` with optimistic local mutation, server confirmation, and rollback on failure. Both mutators replace the local `MessageView` with the server-of-record copy returned by the endpoint.
- `MessageThreadViewModel`: pass-throughs `addReaction(messageID:emoji:)` and `removeReaction(messageID:emoji:)` that thread the caller's `viewerDID` through.
- `MessageBubble`: added a `reactionStrip` rendered immediately below the bubble — pills are grouped by emoji with a count, the viewer's own reactions get an accent-tinted background and tapping one removes the reaction; tapping any other pill adds the viewer's reaction with that emoji. Added a "React" item to the per-message context menu that opens a `popover` with the same five-emoji quick row RN uses (`👍 😆 ❤️ 👀 😢`); selecting one calls `addReaction` (or `removeReaction` if the viewer already reacted with that emoji).

## Gotchas

- The full system emoji picker (RN's `EmojiPopup` "more" affordance behind the quick row) is intentionally deferred — SwiftUI has no native emoji-only picker and a focused `UITextField` workaround is out of scope for this pass. The five-emoji quick row covers RN's most common reactions.
- The "Reacted by …" detail sheet (`ReactionsDialog`) is deferred. It requires resolving each reaction's sender DID to a profile, and `MessageThreadViewModel` does not have a profile resolver wired into it yet. Pills are tappable for add/remove, but tapping does not currently open a per-user sheet.
- SwiftUI's `.contextMenu` cannot host a custom popover inline, so the "React" menu item flips a `@State` flag that drives a separate `.popover` on the bubble. This lands the picker in roughly the right place, with macOS using a true popover and iOS adapting to a compact popover via `presentationCompactAdaptation(.popover)`.
- `MessageView.reactions` is decoded as an optional array because the server omits the field entirely when no one has reacted; synthesized `Codable` handles that via `decodeIfPresent`.
