# 0048 — Feed cache not used: no stale content shown when offline

| | |
|---|---|
| **Status** | in-progress |
| **Module** | BlueskyFeed / BlueskyDataStore |
| **Platform** | All |
| **First seen** | 2026-04-29 |

## Description

Launching the app without a network connection shows an error state with zero posts. The `SwiftDataCacheStore` is passed into `FeedView` and `FeedViewModel` but the feed store never reads from it before attempting the network fetch, so there is no stale-while-revalidate behavior — offline or slow-network users see nothing.

## Steps to reproduce

1. Launch the app and let the feed load at least once.
2. Turn off Wi-Fi / network.
3. Force-quit the app.
4. Relaunch.
5. Observe the feed shows an error banner and zero posts.

## Expected behavior

Previously fetched posts are served from the local `SwiftDataCacheStore` immediately on launch. The feed shows cached content while a background network refresh is attempted. On failure the cached content remains visible.

## Actual behavior

```
fetch error: BlueskyCore.ATError.network(... "The Internet connection appears to be offline." ...)
loadInitial returned, posts=0, error=...
```

Feed is blank with an error message. No cached posts are shown.

## Notes

`FeedStore` receives a `CacheStore` dependency but likely only writes to it after a successful fetch and never reads from it before the first network call. The fix is to populate `posts` from cache in `loadInitial()` before firing the network request, then replace/merge with fresh results on success.

## Investigation findings (2026-04-30)

The cache read in `loadInitial()` was already implemented — it fetched from `SwiftDataCacheStore` and populated `posts` before the network call. However, two bugs were found and fixed:

**Bug 1 — Duplicate posts on successful online fetch (cache hit path):**
`loadInitial` passed `reset: posts.isEmpty` to `fetch()`. When cache was populated, `posts` was not empty, so `reset` was `false`. On network success with `reset=false`, the fetch method called `posts.append(contentsOf: response.feed)` — appending ~50 fresh posts after ~50 already-cached posts, resulting in duplicates. Fixed by always passing `reset: true` in `loadInitial`.

**Bug 2 — Blank feed when offline after cache hit (the reported bug):**
With the `reset=true` fix above, a new edge case needed handling: if the network call fails (offline), the pre-existing cached posts would be lost because `posts` could be overwritten by an empty set. Fixed in `fetch()` by snapshotting `postsBeforeReset` before the network call and restoring it in the catch block when `reset` is true and the snapshot is non-empty.

The fix is in `BlueskyKit/Sources/BlueskyFeed/FeedStore.swift`, committed as `fix(#0048): serve cached feed posts when offline` (commit `521bbab`). Needs live testing on device with network disabled to confirm the stale-while-revalidate behavior works end-to-end.
