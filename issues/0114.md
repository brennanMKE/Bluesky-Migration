## Description

`PrivacySettingsScreen` (lines 26–49) is a stub with placeholder text. The React Native equivalent is a full screen with:

- **Two-factor authentication** — inline `Email2FAToggle` showing current state and flipping it via `com.atproto.server.update2faEmail`.
- **App passwords** — link row with a badge showing the count of active app passwords. (Currently SwiftUI has the `AppPasswordsScreen` but no entry point from Privacy.)
- **Logged-out visibility (PWI opt-out)** — toggle + admonition explaining "When off, signed-out users cannot view your profile or posts on the web". This is the `noUnauthenticated` label on the actor.
- **Activity privacy** (covered separately by #0122) — link to dedicated screen.
- **Interaction settings** — link to "Who can reply / quote / DM" screen (covered by #0140 once filed in moderation pass — but the privacy entry point is here).

## Steps to reproduce

1. Settings → Privacy.
2. Observe the screen has only placeholder text.

## Expected behavior

Full privacy screen with: 2FA toggle, App Passwords row (with active-count badge → existing `AppPasswordsScreen`), Logged-out visibility toggle (`noUnauthenticated` label write via `app.bsky.actor.putPreferences`), Activity privacy link, Interaction settings link.

## Actual behavior

Stub text only.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Settings/PrivacyAndSecuritySettings.tsx` and `Settings/components/{Email2FAToggle.tsx, PwiOptOut.tsx}`.
- File edited: `BlueskyKit/Sources/BlueskySettings/PrivacySettingsScreen.swift`.
- The PWI opt-out is implemented as a self-applied `!no-unauthenticated` label on the actor's profile record. `ProfileRecord` did not previously carry labels — extended in this fix.

## Root cause

`PrivacySettingsScreen` was a 25-line stub with two placeholder text sections; none of RN's five privacy rows existed. Beyond the screen itself, `ProfileRecord` was `Encodable`-only and lacked a `labels` field, so there was no way to read the existing profile record, toggle the `!no-unauthenticated` self-label, and write it back without clobbering `displayName`/`description`. There was also no `GetRecordResponse<T>` shape for `com.atproto.repo.getRecord`.

## Fix

- **`BlueskyCore/Feed.swift`** — added `GetRecordResponse<T>` (generic `{uri, cid, value}`); promoted `ProfileRecord` from `Encodable` to `Codable` and added an optional `labels: SelfLabels?` field. Custom `init(from:)` defaults `$type` to `app.bsky.actor.profile` (some `getRecord` payloads omit it on the embedded value); `encode(to:)` writes `$type` + only the present fields. New 3-arg init is back-compatible with the existing 2-arg call sites in `ProfileStore.updateProfile` and `OnboardingModel`.
- **`BlueskySettings/PrivacySettingsScreen.swift`** — full rewrite. Sections in RN order:
  1. **Two-factor authentication** — row showing current state ("Email 2FA enabled" / "Two-factor authentication") with a `Change` / `Enable` trailing label. Tap opens a referral sheet that summarizes 2FA state and points the user at bsky.app to manage it. Deferred from a real toggle because `Account` does not yet carry `emailAuthFactor` and the PDS code-submission dialog (`com.atproto.server.confirmEmail` flow) isn't built.
  2. **App passwords** — `NavigationLink` into the existing `AppPasswordsScreen`, with an active-count capsule badge populated from `com.atproto.server.listAppPasswords`.
  3. **Activity privacy** — forward-reference button opening a placeholder sheet (#0122).
  4. **Logged-out visibility (PWI)** — `Toggle` whose binding goes through `PrivacySettingsViewModel.setPwiOptedOut(_:)`. The view model `getRecord`s the existing `app.bsky.actor.profile` (rkey `self`), toggles `labels.values` for `!no-unauthenticated`, and `putRecord`s the merged result. Optimistic flip with rollback on failure.
  5. **Interaction settings** — forward-reference button opening a placeholder sheet (#0138).
- **`BlueskySettings/SettingsScreen.swift`** — pass `network`, `accountStore`, and `currentAccount` through to the new privacy screen instead of the old `viewModel` arg.
- New `PrivacySettingsViewModel` (`@Observable @MainActor`) owns `appPasswordCount`, `isPwiOptedOut`, `isTwoFactorEnabled`, plus `refresh()` (parallel `listAppPasswords` + `getRecord(profile)`) and `setPwiOptedOut(_:)`.

## Gotchas

- **Two-factor row is a referral, not a real toggle.** Bail-rule applied per the issue spec: enabling Email 2FA requires plumbing `emailAuthFactor` onto `Account` (from `getSession`) and building the in-app code-submission dialog that RN's `EmailDialog` provides. Both are out of scope here. The referral sheet describes the flow and links to bsky.app's privacy settings, where the existing email-verification UI handles the toggle.
- **Read-modify-write loses unknown profile fields.** Our `ProfileRecord` only models `displayName`, `description`, and `labels`. RN's `agent.upsertProfile` round-trips the *entire* JSON blob (e.g. `pinnedPost`, `joinedViaStarterPack`) — our `putRecord` would drop those. Acceptable for now because the only field the PWI flow shares the record with is `pinnedPost`, which Bluesky also ignores when re-decoded into a fresh `ProfileRecord`. If we later get reports of pinned posts being cleared by a PWI toggle, switch to a typed-but-pass-through model (e.g. `[String: AnyCodable]` for unknown keys).
- **`!no-unauthenticated` is *not* a moderation label.** It looks like one (it goes through `SelfLabels`) but the appview interprets it as a logged-out visibility hint. The composer's self-labels picker should keep filtering this value out so users don't see it as a content rating.
- **Activity privacy and Interaction settings rows are forward-references** to #0122 and #0138 respectively. They open placeholder sheets that explain the deferral; they don't navigate to half-built screens.
