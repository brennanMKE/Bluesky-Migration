# 0030 — Module 10 feature: push notification receipt does not open the correct thread

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskyNotifications / Bluesky-SwiftUI |
| **Platform** | iOS |
| **First seen** | 2026-04-29 |

## Description

When a push notification is received (like, reply, follow, mention), tapping it should navigate directly to the relevant post thread or profile. This deep-link handling from APNs is not implemented.

## Steps to reproduce

1. Register for push notifications (requires `app.bsky.notification.registerPush`).
2. Have another account like or reply to one of your posts.
3. Tap the push notification banner.

## Expected behavior

The app opens and navigates directly to the relevant thread or profile.

## Actual behavior

Push notification registration is not implemented, and no APNs payload handler exists in the app delegate / `SceneDelegate`. Tapping a notification likely does nothing or just opens the app to its current state.

## Notes

Requires: (1) APNs entitlement in the Xcode project, (2) `registerPush` call in `SessionManager` after login, (3) `UNUserNotificationCenterDelegate` in the app target to parse the payload and call the deep-link router.

### 2026-05-05 — bail (code-only changes insufficient)

Tried to scope a fix and concluded this can't be resolved in a code-only pass. End-to-end push delivery and deep-link routing requires infrastructure that is outside this repo:

- **APNs entitlement + provisioning**: needs `aps-environment` entitlement on the app target plus a provisioning profile with Push Notifications capability. Requires Apple Developer account configuration (App ID with Push enabled, APNs key/cert).
- **Bluesky push server**: `app.bsky.notification.registerPush` is implemented against the Bluesky notification gateway, which requires a real APNs device token from a registered build. Without a registered build + valid token the registration call will fail or be a no-op.
- **Live device testing**: the simulator cannot receive real APNs payloads from Bluesky's service. Verifying that tapping a notification opens the right thread requires a real iOS device signed in with a real account that another account interacts with.
- **Payload schema**: would also want to confirm the exact APNs payload shape Bluesky's gateway sends (reason + subject URI fields) against current production behavior — best done by capturing one on-device.

What I'd ask the user before resuming:
1. Is there an Apple Developer account + provisioning profile already configured for the `Bluesky-SwiftUI` target with Push entitlement?
2. Is there a registered iOS device available for live testing, and which account should be used as the receiver?
3. Should the deep-link router (Module 11) be in place first, so push handler can hand off to a stable router API rather than navigating ad hoc?

What I'd try next once unblocked:
1. Add `aps-environment` to the entitlements file and a `Push Notifications` capability to the app target.
2. Implement `UNUserNotificationCenterDelegate` on the app delegate / `BlueskyApp` (via `UIApplicationDelegateAdaptor`) to capture device token and forward `didReceive` payloads.
3. Add `registerPush(token:)` to `SessionManager` (or a new `PushRegistrationService` in `BlueskyNotifications`) that calls `app.bsky.notification.registerPush` after login and on token refresh.
4. Parse APNs payload `reason` + `uri`/`subject` fields and call into the deep-link router to push the right destination (post thread, profile, or notifications tab).
5. Verify with a real device: like → tap banner → opens thread; reply → tap → opens thread; follow → tap → opens profile.
