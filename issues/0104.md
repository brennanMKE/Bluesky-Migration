## Description

The SwiftUI `ConvoRow` (in `ConversationListScreen.swift` ~lines 168–179) shows the avatar, display name, and a single-line of `message.text`. The React Native reference shows several more details:

- **Relative timestamp** right-aligned next to the unread badge (e.g. "2h").
- **System message handling**: when the last "message" is actually a system event ("Conversation deleted", "@alice left the group", "@bob added @carol"), RN renders an italicized system text instead of `message.text`.
- **Two-line preview** with truncation, not single-line. Long messages or multi-line content are properly handled.
- **Sender prefix on group chats**: e.g. "@alice: hello" so the row identifies who sent the last message in a group thread.

## Steps to reproduce

1. Open the SwiftUI Messages tab.
2. Compare a row to the RN reference: timestamp, multi-line preview, sender prefix on groups.

## Expected behavior

`ConvoRow` renders:

- Avatar (existing).
- Display name + handle (existing).
- Right-aligned relative timestamp ("now", "2h", "Yesterday", "Mar 4") matching RN's helper.
- 2-line preview with `lineLimit(2)` and truncation tail.
- For group convos: "@<sender>: <text>" prefix on the preview.
- For system events: italicized system text in place of the preview.
- Unread badge stays at trailing edge below the timestamp.

## Actual behavior

Single-line preview, no timestamp, no system handling, no sender prefix.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Messages/components/ChatListItem.tsx`.
- File to edit: `BlueskyKit/Sources/BlueskyMessages/ConversationListScreen.swift`.
- The system-message detection lives on the lexicon: `chat.bsky.convo.defs#deletedMessageView` and similar variants. `ConvoView.lastMessage` is a discriminated union — the SwiftUI side may need to handle the system-variant case in `ConversationStore.parseLastMessage`.

## Root cause

Two related shortcomings:

1. The lexicon model `BlueskyCore/Chat.swift` declared `ConvoView.lastMessage` as a flat `MessageView?`, with no representation of the `chat.bsky.convo.defs#deletedMessageView` variant. There was no way for the UI to tell the difference between "no message yet" and "the last message was deleted."
2. The SwiftUI `ConvoRow` only rendered the display name and a single line of `message.text`. It had no timestamp, no system-message handling, no group sender prefix, and used `.lineLimit(1)` instead of two lines with tail truncation.

## Fix

- **`BlueskyCore/Chat.swift`** — introduced `ConvoLastMessage`, a `Codable` enum that decodes `lastMessage` based on its `$type` discriminator into either `.message(MessageView)` or `.deleted(DeletedMessageView)`. Added `DeletedMessageView` (`id`, `rev`, `sender`, `sentAt`) and convenience accessors (`messageView`, `deletedView`, `sentAt`, `sender`). Default decode falls back to `.message` when the server omits `$type`.
- **`BlueskyMessages/ConversationListScreen.swift`** — rebuilt `ConvoRow`:
  - Top-aligned `HStack` with avatar, content column, and trailing unread badge.
  - Header row: display name + spacer + relative timestamp.
  - `previewLine` returns a `Text` covering all four cases: no message ("No messages yet" italic), deleted ("Message deleted" italic), regular message, and group-chat message with bolded `@handle:` prefix (or `You: ` when the viewer sent it).
  - Body uses `.lineLimit(2)` with `.truncationMode(.tail)`.
  - Added a pure `shortTimeAgo(from:now:)` helper mirroring RN's `useTimeAgo` short form: `now`, `Nm`, `Nh`, `Yesterday`, `Nd`, `MMM d`, `MMM d, yyyy`.
  - Group detection: `convo.members.count > 2` (matches RN's `parseConvoView` rule of self + 2+ others).
- **`BlueskyMessages/MessageRequestsScreen.swift`** — adapted to the new union (`.messageView` accessor) and added a deleted-variant fallback line.
- **`BlueskyMessages/MessageThreadScreen.swift`** — preview's `lastMessage:` now wraps in `.message(...)`.

## Gotchas

- The dispatch referenced an existing `ConvoWithDetails` discriminated union "from the 2026-04-26 RN drift work," but no such type exists in BlueskyKit yet. Group detection is therefore inferred from `members.count > 2` instead of a parsed group/direct kind. If `ConvoWithDetails` is added later, the row should switch to using its `kind == .group` predicate.
- System-event variants beyond `deletedMessageView` (e.g. group "@alice left" / "@bob added @carol") are not modelled in `chat.bsky.convo.defs` here yet — only the deleted case is handled. RN's `getSystemMessageInfo` reads from `lastMessage.data` of a `systemMessageView`, which can be added to `ConvoLastMessage` later as a third case without breaking call sites.
- `previewLine` is a plain computed `Text` (not `@ViewBuilder`) so the `Text + Text` concatenation for the bolded sender prefix compiles.
