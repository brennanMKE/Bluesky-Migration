## Description

The iOS bottom chrome (tab bar + compose entry) needs to match the React Native reference. Today the iOS app uses a labeled five-tab bar with a generic "person" icon for the Profile tab and no floating compose button — the compose action lives in the top-right of every screen. RN ships an icon-only tab bar, uses the user's avatar for the Profile tab, indicates selection by swapping outline icons for filled ones, and surfaces a primary blue floating compose button (FAB) at bottom-right that overlaps the feed.

This is part of the larger iOS parity audit in #0068.

## Scope

### Tab bar

- **Drop the text labels.** Tabs render icon-only on iOS. (macOS keeps labels — this is iOS-only.)
- **Replace the Profile tab icon with the user's avatar.** A circular AsyncImage backed by the signed-in account's avatar URL, falling back to a placeholder if not loaded. Should respect dark/light backgrounds.
- **Swap selected-state.** Today the selected tab gets a pill-shaped capsule background and the icon stays the same. Match RN: the icon itself swaps from outline to filled when selected (e.g. `house` ↔ `house.fill`, `magnifyingglass` ↔ `magnifyingglass.circle.fill`, `bell` ↔ `bell.fill`). For Profile, ring the avatar with the brand color when selected.
- **Tab order stays Home · Search · Messages · Notifications · Profile** — that already matches.

### Floating compose button (FAB)

- **Add a circular blue button at bottom-right** with the pencil/compose SF Symbol, placed above the bottom tab bar with a 16pt margin from the right edge and ~16pt above the tab bar.
- **Tap opens the existing composer sheet** (`ComposerSheet`).
- **Visible on Home, Search, Notifications, Profile** (matches RN). Hide on Messages.
- **Removes the top-right compose action from those screens on iOS** — the FAB replaces it. (macOS keeps the top-right compose entry; this is iOS-only.)

## Implementation notes

- Code: `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` is the natural place for the tab bar redesign and FAB overlay. Use `#if os(iOS)` to gate the icon-only / avatar / FAB changes; keep macOS labeled and FAB-less.
- For the Profile-tab avatar, the existing `SessionManager` (or the active `AccountStore`) already has the viewer's `ProfileBasic` with `avatar` URL — read from there. Cache via `AsyncImage`'s default LRU is fine.
- The FAB should use `.zIndex` and `.safeAreaInset` (or a `ZStack` overlay on the tab content) so it floats above the feed without breaking pull-to-refresh or scroll-to-top behavior on the underlying view.
- Verify scroll-to-top still works on the Home tab when the user taps the Home icon while already on Home — RN behavior — once the icon-fill swap is in.

## Root cause

Not a bug per se — a parity gap. The iOS compact layout in `MainTabView` was a stock SwiftUI `TabView` with `.tabItem { Label(title, systemImage:) }`. SwiftUI's `tabItem` only accepts `Label`/`Image`/`Text`, so it can't render an `AsyncImage` for the Profile tab, and there's no API to suppress labels per-tab. RN renders its own tab bar from scratch, which is why it gets icon-only tabs, an avatar Profile tab, and a floating compose FAB. The SwiftUI app inherited the system tab bar's labeled-icon look and pushed compose into the navigation toolbar instead.

## Fix

On iOS compact widths, replaced the system `TabView` with a custom SwiftUI tab bar inserted via `.safeAreaInset(edge: .bottom)`. The bar is a `HStack` of `Button`s — one per `AppTab.compactTabs` — that swap SF Symbols between outline and filled variants when active and render the viewer's avatar (with a brand-blue stroked ring when active) for the Profile slot. A circular brand-blue FAB is overlaid at `.bottomTrailing` *before* the safe-area inset so its alignment bounds end at the top of the bar (16pt margins). The FAB drives a top-level `.sheet(isPresented: $showComposer) { ComposerSheet(...) }` and is hidden when `selectedTab == .messages`. The Home tab's toolbar compose button is now gated on `horizontalSizeClass == .regular` so iPad regular still gets a toolbar shortcut while iPhone uses the FAB exclusively.

The viewer's avatar URL is fetched lazily on first appearance (and on account switch) by calling `app.bsky.actor.getProfile` directly via the injected `NetworkClient` and decoding into `ProfileDetailed`. `Account.avatarURL` is currently always nil after login, so the per-session fetch is required.

To preserve RN's "tap active Home tab → scroll to top" behavior — which the system `TabView` provided implicitly — the custom bar broadcasts a `scrollFeedToTop` `NotificationCenter` notification, and `FeedView` wraps its post list in a `ScrollViewReader` with a stable top anchor that `scrollTo`s on receipt.

macOS sidebar and iPad regular `NavigationSplitView` paths are untouched.

## Files changed

- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` — replaced the iOS compact `TabView` with a custom `iosCompactLayout` that uses `.safeAreaInset` for the tab bar and a `.bottomTrailing` overlay for the FAB; added `iosCustomTabBar`, `iosTabBarLabel`, `iosTabIcon`, `composeFAB`, `refreshViewerAvatar`; added `viewerAvatarURL` state; gated the Home toolbar compose button behind `horizontalSizeClass == .regular` on iOS.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/PushRouter.swift` — declared `Notification.Name.scrollFeedToTop`.
- `BlueskyKit/Sources/BlueskyFeed/FeedView.swift` — wrapped the post list in `ScrollViewReader` with a stable top anchor and an `onReceive` that scrolls to it when the `co.sstools.bluesky.scrollFeedToTop` notification fires.

## Gotchas

- The custom iOS tab bar uses `.id(activeTab)` on the `NavigationStack`, so switching tabs recreates the stack from scratch — pushed destinations within a tab don't survive a tab switch (the same behavior as the iPad regular `NavigationSplitView` and macOS sidebar). The system `TabView` previously preserved per-tab navigation history; that's now gone on iPhone. Acceptable per existing behavior elsewhere, but worth knowing.
- `co.sstools.bluesky.scrollFeedToTop` is referenced as a literal string inside `FeedView` (BlueskyKit) because feature modules can't import the app target where `Notification.Name.scrollFeedToTop` is canonically declared. The two strings must stay in sync; both reference the same opaque identifier.
- The viewer avatar is fetched ad-hoc in `MainTabView` rather than wired through `SessionManager` / `Account.avatarURL`, because `Account` currently stores `avatarURL: nil` after login. A future refactor could populate `Account.avatarURL` during session creation and remove the extra `getProfile` call. Until then the FAB's first appearance shows the initials placeholder briefly while the network call resolves.
- `.safeAreaInset` for the tab bar is layered *after* the `.overlay` for the FAB. SwiftUI evaluates modifiers bottom-up, so the FAB overlay's coordinate space is the inner content view (which `safeAreaInset` then shrinks to make room for the bar). Reversing the order would push the FAB behind the bar.
