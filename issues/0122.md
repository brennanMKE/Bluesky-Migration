## Description

React Native has a dedicated **Activity Privacy Settings** screen (Settings → Privacy → Activity privacy) controlling who is allowed to receive notifications when you interact with their content (likes, reposts):

- "Allow others to be notified of your likes / reposts" — radio: **Anyone** / **People you follow** / **Mutuals only** / **Nobody**.

This maps to `app.bsky.actor.defs#postInteractionSettingsPref` (or the activity-privacy variant). SwiftUI's `PrivacySettingsScreen` is a stub (see #0114) and does not link out to a per-activity privacy screen.

## Steps to reproduce

1. Settings → Privacy.
2. Look for "Activity privacy".
3. Observe the screen does not exist.

## Expected behavior

A new `ActivityPrivacySettingsScreen.swift` accessible from `PrivacySettingsScreen`. Single radio control with the four visibility options. Persist via `app.bsky.actor.putPreferences`.

## Actual behavior

Screen does not exist.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Settings/ActivityPrivacySettings.tsx` (the file at `ActivityPrivacy/index.tsx` named in the original spec lives at `ActivityPrivacySettings.tsx` in the current RN tree).
- File to create: `BlueskyKit/Sources/BlueskySettings/ActivityPrivacySettingsScreen.swift`.

## Root cause

The Activity Privacy screen had never been ported. The privacy hub (#0114) shipped a forward-reference button that opened a placeholder sheet; tapping the row in Settings → Privacy was a dead end. There was also no model in `BlueskyCore` for the underlying record (`app.bsky.notification.declaration`).

## Fix

Confirmed RN's data shape against `Bluesky-ReactNative/src/state/queries/activity-subscriptions.ts` and `src/screens/Settings/ActivityPrivacySettings.tsx` before coding. Important parity correction vs. the original spec: this is **not** an `app.bsky.actor.putPreferences` write, and there are **three** values (not four). The setting lives in a record on the user's repo:

- Endpoints: `com.atproto.repo.getRecord` / `com.atproto.repo.putRecord`
- `repo=<did>`, `collection=app.bsky.notification.declaration`, `rkey=self`
- Field: `allowSubscriptions: 'followers' | 'mutuals' | 'none'`
- "Record not found" → fall back to `'followers'` (matches RN's `useNotificationDeclarationQuery`).

Changes:

- **`BlueskyCore/Feed.swift`** — added `AllowSubscriptions` (Codable enum: `followers | mutuals | none`) and `NotificationDeclarationRecord` (Codable, embeds `$type=app.bsky.notification.declaration`). Decode tolerates a missing `$type` (some `getRecord` payloads omit it on the embedded value) and an unknown enum raw value (falls back to `.followers`, matching RN's missing-record fallback).
- **`BlueskySettings/ActivityPrivacySettingsScreen.swift`** (new) — `Form` with one section. Header copy mirrors RN exactly (`"Allow others to be notified of your posts"`); footer mirrors the RN subtitle (`"This feature allows users to receive notifications for your new posts and replies. Who do you want to enable this for?"`). Three radio rows match RN's `<Toggle.Item name=...>` labels exactly: **Anyone who follows me** / **Only followers who I follow** / **No one**. The radio is rendered with `largecircle.fill.circle` / `circle` SF Symbols and tinted with the accent color when selected.
- **`ActivityPrivacySettingsViewModel`** (`@Observable @MainActor`) — owns `allowSubscriptions`, `isLoading`, `isUpdating`, `loadFailed`, and `errorMessage`. `refresh()` calls `getRecord` and treats "Could not locate record" / "RecordNotFound" / "not found" errors as the default `.followers` value rather than a load failure (parity with RN). `setAllowSubscriptions(_:)` does an optimistic flip + rollback-on-error around `putRecord`.
- **`BlueskySettings/PrivacySettingsScreen.swift`** — replaced the placeholder button + sheet branch with a real `NavigationLink` into `ActivityPrivacySettingsScreen`, threading the existing `network` / `accountStore` / `currentAccount`. Removed the now-unused `.activityPrivacyPlaceholder` `SheetKind` case and updated the screen's docstring.

## Gotchas

- **Issue spec listed four options; RN has three.** The original spec mentioned `Anyone | Following | Mutuals | Nobody`. RN's actual radio group is `followers | mutuals | none` — there is no separate "anyone (including non-followers)" option, because the underlying capability is "who can subscribe to my activity feed", and only followers can subscribe in the first place. We shipped the three RN options.
- **Lexicon is a record write, not a `putPreferences` write.** The spec also suggested `app.bsky.actor.putPreferences` / `postInteractionSettingsPref`. RN actually writes a per-account record at `app.bsky.notification.declaration#self`, not a preference. Using `putPreferences` would be a no-op as far as the appview's notification-subscription gating is concerned. `GetPreferencesResponse` / `PutPreferencesRequest` were not extended.
- **Read-modify-write isn't required.** The declaration record only has `allowSubscriptions` today, so `putRecord` writes a fresh struct rather than round-tripping a fetched value (unlike the PWI flow which has to merge into `ProfileRecord`). If the lexicon ever gains additional fields we'd need to switch to read-modify-write to avoid clobbering them.
- **Activity *subscriptions* list (RN's `listActivitySubscriptions`) is a separate feature** and not in scope here. Also not yet implemented in SwiftUI; that's the per-user "who do *I* subscribe to" inverse, surfaced from `useActivitySubscriptionsQuery`. File a follow-up if/when needed.
