## Description

`ComposerSheet` is missing four features present in the React Native composer:

1. **Video picker + upload** — select a video from the library, upload as a blob, and embed it.
2. **Link card preview** — paste a URL in the text and a card preview (title, description, thumbnail) appears.
3. **Thread / multi-post** — add additional posts to compose a thread in one shot.
4. **Draft persistence** — closing the composer mid-draft saves it and restores it on next open.

## Steps to reproduce

1. Open `ComposerSheet`.
2. Look for a video picker button — absent.
3. Paste a URL in the text body — no link card preview appears.
4. Look for a "+" button to add another post to the thread — absent.
5. Close the composer with text entered, reopen it — draft is gone.

## Expected behavior

All four features work as described.

## Actual behavior

None of the four features are implemented.

## Notes

Each sub-feature is independent and can be tackled separately. Draft persistence is likely the lowest-effort (store draft text in `UserDefaults` keyed by composer context).

## Root cause

The four features were scaffolded into `ComposerSheet`, `ComposerViewModel`, and
`ComposerStore` but the link card was rendering only the URL itself (no
title/description/thumbnail) and the macOS build had no video picker (the
existing `PhotosPicker` was wrapped in `#if os(iOS)`). Thread composing,
draft persistence, and the basic video flow on iOS were already in place;
this resolution focuses on the gaps and polish.

## Fix

- Added `LinkMetadataFetcher` (actor) which downloads the first 256 KB of a
  URL, parses the `<head>` for `og:title`, `og:description`, `og:image`
  (with `twitter:*`, `<title>`, and `meta name="description"` fallbacks),
  and returns a `LinkMetadata` value.
- The composer view-model debounces detection, kicks off a fetch when the
  detected URL changes, exposes `linkMetadata` and `isFetchingLinkMetadata`,
  and clears state if the URL changes mid-flight.
- `ComposerSheet` renders the richer card: 60×60 thumbnail (via
  `AsyncImage`), title, two-line description, host. Shows a
  `ProgressView` placeholder while the fetch is in flight.
- `ComposerStore.post(...)` now accepts `linkMetadata`. When the user posts
  with no images/video and a metadata thumbnail is available, the store
  downloads the thumbnail (capped at 1 MB), uploads it as a blob, and
  embeds it as `EmbedExternal.thumb`.
- macOS now has its own video picker via `NSOpenPanel` (`UTType.movie`,
  `.video`, `.mpeg4Movie`, `.quickTimeMovie`).
- After a successful post, transient composer state (`additionalPosts`,
  `attachedVideo`, `linkMetadata`, `detectedURL`) is cleared so the next
  open starts fresh — only the saved-draft text is restored, and that's
  cleared via `clearDraft()` on success too.

## Gotchas

- Xcode builds the package with `-default-isolation MainActor` (Swift 6),
  so file-level `let` globals are MainActor-isolated. The fetcher's
  static parsing helpers run inside an actor, so they need to be
  `nonisolated`. The OSLog `Logger` is wrapped in a `nonisolated static`
  property of a private enum to keep the access outside of the
  parsing path actor-friendly across both `swift build` and Xcode.
- Link metadata fetching uses no proxy — RN uses a `LINK_META_PROXY`
  service, which we don't have. We fetch the page directly with a short
  timeout. Sites that gate scraping (e.g. anti-bot) may not return a
  preview; that's a pre-existing limitation of the AT Proto external
  embed format.
- The image thumbnail is downloaded on post, not on detection, so a
  failure to fetch the thumb degrades to a text-only external embed
  rather than blocking the post.
