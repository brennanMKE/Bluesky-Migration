## Description

Two small visual polish items on the Notifications screen, bundled because they each touch one tiny piece of the row layout. Splitting would make two near-identical sub-PRs.

This is a follow-up to the iOS Notifications parity audit in #0069.

## Scope

### 1. Concise time-ago format

- **Today**: SwiftUI renders relative timestamps as "9 hr, 49 min", "12 hr, 17 min", "4 days, 5 hr". Verbose, multi-unit, and inconsistent with the feed's format.
- **RN**: renders single-unit, abbreviated form — `9h`, `2d`, `4d`, `12h`, `5d`, `15s`, `3m`, `1w`, `1y`.
- **Fix**: replace the formatter with a single-unit shortform helper. The feed module already has the `niceDate`-equivalent — extract it to `BlueskyUI` if not already shared, and use it on every time-ago label across the app (not just notifications).

### 2. Filled bell when unread

- **Today**: every notification row uses an outlined grey bell icon regardless of read state.
- **RN**: filled (`bell.fill`) bell in brand blue when the group is unread; outlined (`bell`) grey bell when seen.
- **Fix**: bind the icon style to `group.anyUnread` (already on `GroupedNotification`). Use `.foregroundStyle` to color the bell with the brand accent when unread.

## Implementation notes

- The shared time formatter belongs in `BlueskyUI` — name it `RelativeTimeFormatter` or expose a `String(relativeTo: Date)` helper.
- The unread-bell change is one ternary in `GroupedNotificationRow`.
- Both changes have no data dependencies — they're purely view-layer.

## Acceptance

- Notification timestamps show single-unit short form (`9h`, `2d`).
- Unread groups have a filled blue bell; seen groups have an outlined grey bell.
- Same time format is used in the feed (already consistent there) — confirm no regression.
- iOS Simulator and macOS builds pass.

## Root cause

Two unrelated regressions colliding on the notification row:

- **Verbose timestamps.** `GroupedNotificationRow` used `Text(date, style: .relative)`, which is SwiftUI's system relative-date renderer — multi-unit by default ("9 hr, 49 min", "4 days, 5 hr"). The feed's `PostCard` had its own private `relativeTimestamp(_:)` that produced the desired short form for under-24h durations, but it fell through to a literal `MMM d` date for anything older, and notifications didn't use it at all. There was no shared helper, so the two surfaces drifted.
- **Always-grey bell.** The catch-all `iconName` was a hardcoded `"bell.fill"` and `iconColor` was `.secondary` regardless of read state. RN flips between filled/brand-color (unread) and outlined/grey (seen); the SwiftUI row had no signal in either direction.

## Fix

- **`RelativeTimeFormatter` in `BlueskyUI`** — new `public enum` with one entry point (`string(from:reference:)`) that mirrors the short-form branch of RN's `useTimeAgo` (`src/lib/hooks/useTimeAgo.ts`). Returns single-unit output: `"now"` for <5s, then `"15s"`, `"3m"`, `"9h"`, `"2d"`, `"1w"`, `"5mo"`, `"1y"`. Includes a `reference` parameter (default `Date.now`) so future tests can pin the comparison instant. Lives next to `CompactNumberFormatter` in `BlueskyUI` so all formatting helpers stay co-located.
- **`PostCard` adopts the shared helper.** The private `relativeTimestamp(_:)` function is deleted; the post header timestamp now calls `RelativeTimeFormatter.string(from:)`. This keeps feed cards on the same renderer notifications use, and silently extends day → week → month → year coverage that the old fall-through `MMM d` path used to handle.
- **Notifications adopt the shared helper.** Both `defaultGroupedRow` (timestamp on the right edge) and `replyRow` (timestamp inline next to the handle) replace `Text(group.indexedAt, style: .relative)` with `Text(RelativeTimeFormatter.string(from: group.indexedAt))`.
- **Filled-bell unread state.** `GroupedNotificationRow` reads `@Environment(\.blueskyTheme)` and updates the catch-all branches in `iconName` / `iconColor`: `bell.fill` + `theme.colors.link` when `!group.isRead`, `bell` + `.secondary` otherwise. Reason-specific icons (heart/pink, repost/green, follow/blue, mention, reply, quote) are deliberately untouched — their colors already convey meaning, and the issue scope restricts the change to "rows that still show the bell".

## Files changed

- `BlueskyKit/Sources/BlueskyUI/RelativeTimeFormatter.swift` — new. Public `enum RelativeTimeFormatter` with `string(from:reference:)`. Single-unit short-form output, brackets matching RN's `useTimeAgo`.
- `BlueskyKit/Sources/BlueskyUI/PostCard.swift` — header timestamp now calls `RelativeTimeFormatter.string(from:)`; private `relativeTimestamp(_:)` helper removed.
- `BlueskyKit/Sources/BlueskyNotifications/NotificationsScreen.swift` — both row variants render the timestamp via `RelativeTimeFormatter`. `GroupedNotificationRow` gains an `@Environment(\.blueskyTheme)` and flips the catch-all bell icon to `bell.fill` / brand-link color when unread, outlined `bell` / secondary grey when seen.

## Gotchas

- **Shared formatter lives in `BlueskyUI` (Layer 2)**, not `BlueskyCore`, because it's a presentation concern (display string for the UI) and follows the same placement pattern as `CompactNumberFormatter`. Profile timestamps (#0070) and message-thread timestamps will naturally pick this up when they land — both of those modules already depend on `BlueskyUI`. No follow-up issue needed yet because Profile / Messages don't currently render any relative-time labels; once they do, they should call `RelativeTimeFormatter.string(from:)` directly.
- **Unread-bell change is scoped to the catch-all icon only.** Reason-specific icons (heart for `like`, arrow for `repost`, person.fill.badge.plus for `follow`, etc.) keep their fixed colors. Re-coloring them based on read state would muddle the per-reason semantics RN preserves on those rows; only the catch-all bell flips. Reply rows from #0081 already drop the icon entirely and are not touched.
- **`PostCard` no longer falls through to `"MMM d"` for old posts.** The previous helper rendered "Apr 25" for anything older than 24h; the shared formatter renders `"3d"` / `"2w"` / `"5mo"` / `"1y"` instead. This matches RN behavior more closely and keeps the format consistent across recent and old posts. If a downstream surface specifically wants a date string for very old posts, it can add a separate fallback path — but no current call site needs that.
