# 0079 — Notifications: render inline post-content preview on each row

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskyNotifications / BlueskyFeed |

## Description

Each notification row in the React Native client renders an inline preview of the relevant post text — multi-line, with bullet points and link cards — so the user can read the content without tapping through to the thread. The SwiftUI client currently shows only the actor name and the reason label; there is no post body, making notifications useless for at-a-glance reading. This is the single biggest UX gap on the Notifications screen.

This is a follow-up to the iOS Notifications parity audit in #0069.

## Scope

- **Render an inline post-text excerpt** on each notification row when the reason is post-related (`reply`, `mention`, `quote`, `like`, `repost`, `subscribed-post`, `like-via-repost`, `repost-via-repost`).
- **Source of the text**: the post the notification refers to. For `reply`/`mention`/`quote`, the post is the notification's own `uri`. For `like`/`repost`, it's the `subjectUri` (the post that was liked/reposted).
- **Layout**: text excerpt sits below the group label, indented to align with the avatar stack (or with the reason text). Use the same typographic styling as the feed body — same font, same secondary color.
- **Multi-line**: allow up to 4 lines before truncating with an ellipsis. Match RN behavior.
- **Inline link cards**: if the post has an external link with an OG card, render a compact version of the card. Skip image embeds at full size — keep the row dense.
- **Skip rendering** if the reason is non-post (`follow`, `verified`, `contact-match`).

## Data fetching

The `app.bsky.notification.listNotifications` response gives notifications but **does not include the full post body**. To render the preview, we need to fetch the underlying posts:

- Collect the unique `subjectUri` values across the notification list.
- Batch-fetch via `app.bsky.feed.getPosts(uris: [...])` (limit 25 per call).
- Cache per-URI in a `NotificationPostCache` keyed by AT-URI.
- Render the row with a placeholder while loading; populate when the post arrives.

`NotificationsStore.loadInitial` should kick off the post-fetch after the notifications come back. Append-loading should batch the additional URIs.

## Implementation notes

- New `NotificationPostCache` actor in `BlueskyNotifications` (or extend `NotificationsStore` with a cache field — actor-isolated).
- Reuse the post-text-rendering view from `PostCardView` (extract a `PostBodyView` if it isn't already).
- Limit text height to ~4 lines via `.lineLimit(4)`.

## Acceptance

- Notification rows for post-related reasons show the relevant post text inline, capped at ~4 lines.
- Long-press / hover (macOS) reveals the full text — optional polish.
- Posts are batch-fetched; UI shows a placeholder during the fetch.
- Non-post reasons (follow, verified) render without a preview.
- iOS Simulator and macOS builds pass.
