## Description

Each notification row in the React Native client renders an inline preview of the relevant post text — multi-line, with bullet points and link cards — so the user can read the content without tapping through to the thread. The SwiftUI client currently shows only the actor name and the reason label; there is no post body, making notifications useless for at-a-glance reading. This is the single biggest UX gap on the Notifications screen.

This is a follow-up to the iOS Notifications parity audit in #0069.

## Scope

- **Render an inline post-text excerpt** on each notification row when the reason is post-related (`reply`, `mention`, `quote`, `like`, `repost`, `subscribed-post`, `like-via-repost`, `repost-via-repost`).
- **Source of the text**: the post the notification refers to. For `reply`/`mention`/`quote`, the post is the notification's own `uri`. For `like`/`repost`, it's the `subjectUri` (the post that was liked/reposted).
- **Layout**: text excerpt sits below the group label, indented to align with the avatar stack (or with the reason text). Use the same typographic styling as the feed body — same font, same secondary color.
- **Multi-line**: allow up to 4 lines before truncating with an ellipsis. Match RN behavior.
- **Inline link cards**: if the post has an external link with an OG card, render a compact version of the card. Skip image embeds at full size — keep the row dense.
- **Skip rendering** if the reason is non-post (`follow`, `verified`, `contact-match`).

## Data fetching

The `app.bsky.notification.listNotifications` response gives notifications but **does not include the full post body**. To render the preview, we need to fetch the underlying posts:

- Collect the unique `subjectUri` values across the notification list.
- Batch-fetch via `app.bsky.feed.getPosts(uris: [...])` (limit 25 per call).
- Cache per-URI in a `NotificationPostCache` keyed by AT-URI.
- Render the row with a placeholder while loading; populate when the post arrives.

`NotificationsStore.loadInitial` should kick off the post-fetch after the notifications come back. Append-loading should batch the additional URIs.

## Implementation notes

- New `NotificationPostCache` actor in `BlueskyNotifications` (or extend `NotificationsStore` with a cache field — actor-isolated).
- Reuse the post-text-rendering view from `PostCardView` (extract a `PostBodyView` if it isn't already).
- Limit text height to ~4 lines via `.lineLimit(4)`.

## Acceptance

- Notification rows for post-related reasons show the relevant post text inline, capped at ~4 lines.
- Long-press / hover (macOS) reveals the full text — optional polish.
- Posts are batch-fetched; UI shows a placeholder during the fetch.
- Non-post reasons (follow, verified) render without a preview.
- iOS Simulator and macOS builds pass.

## Root cause

`app.bsky.notification.listNotifications` returns the actor and reason
but no post body. The previous `GroupedNotificationRow` rendered only
the actor stack, the actor summary, and a one-line reason label — there
was no inline post text and no fetch step to hydrate one. To read
"Alice replied: …" the user had to tap through to the thread.

## Fix

Hydrate posts after every notification fetch and render an inline
preview on each row:

- **`NotificationPostCache`** — new `@MainActor @Observable` cache in
  `BlueskyNotifications` keyed by AT-URI (raw value) with three
  observable buckets: `posts` (resolved), `inFlight` (fetching), and
  `missing` (server returned no entry, don't retry). Exposes
  `post(for:)`, `isLoading(uri:)`, and `isMissing(uri:)`.
- **Hydration** — `NotificationsViewModel.loadInitial`/`loadMore`/
  `refresh` now `await` `hydratePreviewPosts()` after the store fetch
  completes. The view model's `postPreviewURIs` derives the URIs to
  hydrate by walking `store.allNotifications` (unfiltered, so the
  cache is shared across the All / Mentions tabs from #0078) and
  picking the right URI per reason: notification's own `uri` for
  `reply`/`mention`/`quote`, `reasonSubject` for `like`/`repost`/
  `subscribed-post`/`like-via-repost`/`repost-via-repost`. Other
  reasons (`follow`, `verified`, etc.) contribute no URI.
- **Batch fetch via per-URI fan-out** — see Gotchas. The cache fans
  out one `app.bsky.feed.getPosts` call per URI through `withTaskGroup`
  rather than calling the endpoint with a 25-URI batch, because the
  shared `NetworkClient.get(_:params:)` signature is `[String: String]`
  and can't carry repeated `uris=` query items. The fan-out matches the
  pattern `SuggestedFeedsStore.resolve(uris:)` already uses for
  `getFeedGenerator`. `inFlight` is set up-front so concurrent
  hydration calls (e.g. `loadInitial` then `loadMore`) don't restampede
  the same URIs.
- **`PostBodyView`** — extracted from `PostCard.postBody` into
  `BlueskyUI/PostBodyView.swift`. Wraps `RichTextView` with the same
  font, foreground/link colors, and facet-link routing
  (`bluesky://hashtag/<tag>` → `onHashtagTap`, `bluesky://profile/<did>`
  → `onMentionTap`). Adds an optional `lineLimit:` parameter — `nil`
  matches the feed's uncapped behavior; the notification row passes
  `4`. `PostCard` now calls `PostBodyView`, so any future tweak to body
  rendering applies to feed and notifications uniformly. Lives in
  `BlueskyUI` (Layer 2) — no feed-specific types involved, and
  `BlueskyNotifications` already depends on `BlueskyUI`.
- **Row layout** — `GroupedNotificationRow` adds a `postPreview`
  ViewBuilder block below the existing `reasonText` line. When the
  preview URI is resolved it renders `PostBodyView(lineLimit: 4)` plus
  an optional compact link card. While the fetch is in flight it shows
  a two-line skeleton (rounded gray rectangles, ~10pt tall).
  Post-reason rows for which the server returned no entry render
  nothing. Non-post reasons (`follow`, `verified`) render no preview at
  all because `previewPostURI` is `nil`.
- **Compact link card** — for `external` embeds (and the external half
  of `recordWithMedia`), the row renders a single-line title plus
  hostname (e.g. `bsky.app`) inside a 0.5pt rounded outline. Image,
  video, and quote-record embeds are intentionally skipped to keep
  rows dense; quote text is already implied by the body's facet
  rendering and the issue's "keep rows dense" rule.
- **Grouping change** — reply/mention/quote notifications are now
  keyed by their notification `uri` instead of `(reason,
  reasonSubject)` in `NotificationsViewModel.groupedNotifications`.
  Multiple replies to the same parent post used to merge into a single
  row, which was wrong once we surface body text — each reply has
  unique text. Likes and reposts continue to group on
  `(reason, reasonSubject)` so "Alice, Bob, and 3 others liked your
  post" still collapses correctly.
- **Tap target** — the row now navigates to `previewPostURI ??
  reasonSubject` rather than `reasonSubject` alone. For replies/
  mentions/quotes, this routes to the actor's new post (which is what
  the preview is showing); for likes/reposts, it falls back to the
  viewer's own post as before.

  routes the row's tap target to `previewPostURI ?? reasonSubject`.

## Gotchas

- **Per-URI fan-out, not a 25-URI batch.** The issue suggested
  `getPosts(uris: [...])` in chunks of 25, which is the AT Proto
  endpoint's actual shape. Our shared `NetworkClient.get(_:params:)`
  takes `[String: String]`, so it can't emit the repeated
  `uris=at://…&uris=at://…` query items the lexicon requires for
  `array` parameters. (Comma-joining is not part of the AT Proto
  spec and is unreliable across PDS implementations — same constraint
  documented in #0078's "filter is client-side" gotcha.) Until the
  `NetworkClient` protocol grows array-param support — a
  cross-module refactor (`NetworkClient` in `BlueskyKit`,
  `ATProtoClient` in `BlueskyNetworking`, plus test doubles) — the
  cache fans out one `getPosts` call per URI in parallel via
  `withTaskGroup`. This matches `SuggestedFeedsStore.resolve(uris:)`'s
  approach. The `keepReasons`-style URI-collection logic is centralized
  in `postPreviewURIs` so swapping to a real batched call later is a
  one-line change inside `NotificationPostCache.hydrate`.
- **Cache lives on the view model, not the store.** Storing and
  fetching unrelated to listNotifications fits the view-model boundary
  better, and it sidesteps changing the existing `NotificationsStoring`
  protocol surface (which would force every test double to implement
  no-op `postCache` accessors). The store still owns the notification
  array; the view model orchestrates the listNotifications + getPosts
  hydration sequence.
- **Reply/mention/quote ungrouping is intentional.** Multiple replies
  to the same parent post used to share one row because they shared
  `reasonSubject`. Now each reply is its own row keyed by the
  notification's own `uri`, since each reply has unique body text we
  can't sensibly collapse. Likes and reposts continue to group; the
  "Alice and 3 others liked your post" UX is preserved. This makes
  #0081 (reply-row affordance) cleaner — `if group.reason == "reply"
  { ReplyRow(...) }` operates on a single-actor group every time.
- **Loading skeleton is intentionally light.** The skeleton is two
  rounded gray bars (~10pt tall, secondary 15% opacity) rather than a
  shimmer or `ProgressView` — rows already have an avatar stack and
  reason text visible, so showing a chunky spinner per row would feel
  busy. The skeleton occupies roughly the same vertical space the
  resolved body will take, so the row doesn't visibly resize when the
  fetch completes.
- **`isMissing` short-circuits perpetual placeholders.** If
  `getPosts` returns no entry for a URI (deleted, blocked, takedown),
  the cache marks the URI in `missing` and the row drops the preview
  block entirely. Without this, the skeleton would show forever after
  a single transient empty response.
- **Compact link card vs. full embed.** Image, video, and quote
  embeds are intentionally skipped on notification rows — the issue's
  "keep rows dense" rule wins over richer previews. External link
  cards render in a tighter form (no thumbnail, single-line title +
  hostname) than the full `PostEmbedView` link card. If a future issue
  wants thumbnails on link cards in notifications, swap
  `compactLinkCard` for a configurable `PostEmbedView` mode rather
  than a second renderer.
