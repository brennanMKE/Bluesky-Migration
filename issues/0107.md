## Description

Two related gaps in the SwiftUI message thread:

1. **No per-message timestamps.** RN shows a relative timestamp under each bubble (or under the last bubble in a clustered run from one sender). SwiftUI bubbles have no timestamp at all.
2. **No date dividers.** RN renders a centered date pill ("Today", "Yesterday", "Mar 4") between groups of messages from different days. SwiftUI's `LazyVStack` is one continuous list with no day separators.

Together these make it hard to anchor a conversation in time.

## Steps to reproduce

1. Open a message thread that spans multiple days.
2. Scroll through.
3. Observe: no timestamps under bubbles, no date dividers between days.

## Expected behavior

- **Per-message timestamp**: under each bubble (or the last bubble in a run from the same sender, RN's clustering rule). Format: relative for today ("2h ago"), time-only for older same-day, day+time for older days. Match RN's `niceDate` helper for consistency.
- **Date dividers**: a small centered pill ("Today" / "Yesterday" / "Mar 4") above the first message of a new day. The pill uses the secondary text color and a subtle background.

## Actual behavior

No timestamps, no dividers.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Messages/components/DateDivider.tsx` and the timestamp rendering in `MessageItem.tsx`.
- File to edit: `BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift`. The `LazyVStack` ForEach should be re-grouped so dividers can be inserted between adjacent messages whose `sentAt` falls on different calendar days.
- Helper: a small `Calendar.current.isDate(_:inSameDayAs:)` check works.

## Root cause

The `MessageThreadScreen` `LazyVStack` rendered bubbles end-to-end with
no day boundary detection and no per-bubble timestamp. The recently added
run-clustering helper from #0105 (`isFirstInRun`) identified the first
bubble in a same-sender run but had no symmetric "last in run" helper to
anchor a timestamp under. RN's `MessageItem.tsx` does both: emits a
`<DateDivider date={message.sentAt} />` when the previous message is
from a different day, and emits `<MessageItemMetadata>` (containing the
relative timestamp) on the *last* bubble of each cluster.

## Fix

In `MessageThreadScreen.swift`:

- Added `isLastInRun(at:)` mirroring `isFirstInRun(at:)`.
- Added `shouldShowDateDivider(at:)` using
  `Calendar.current.isDate(_:inSameDayAs:)` against the previous message,
  always true for index 0.
- Added a pure `bubbleTimestamp(for:now:)` helper producing RN-style
  short labels: time-only for today, "Yesterday HH:MM" for yesterday,
  "Mon d, HH:MM" (year added when out of current year) otherwise.
- The ForEach now emits a `DateDivider` row before each bubble that
  starts a new calendar day, and passes `isLastInRun` +
  `timestampLabel` into `MessageBubble`.
- `MessageBubble` renders the timestamp as a `caption2` secondary-color
  label inside the bubble's own `VStack`, so the alignment follows the
  bubble (trailing for own, leading otherwise) rather than the row
  edges.
- Added a private `DateDivider` view that renders a centered label with
  thin horizontal rules on either side ("Today at 3:42 PM" / "Yesterday
  at 3:42 PM" / "Monday at 3:42 PM" within the past week / "Mon, March
  4 at 3:42 PM" otherwise), matching RN's `DateDivider.tsx` formatting
  and ordering.

## Files changed

- `BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift`

## Gotchas

- The timestamp string is computed once at render time using `Date()`;
  if a thread stays open across midnight the labels won't auto-rebucket
  until the view is reloaded. Acceptable for now — RN exhibits the same
  behavior because it captures `today` in `DateDivider`'s render.
- The bubble timestamp label is built with `Locale.current` and the
  `"jm"` template so 12/24-hour rendering follows the user's locale,
  consistent with RN's `Intl.DateTimeFormat({timeStyle: 'short'})`.
- Run clustering ignores the 5-minute time gap that RN applies in
  `isWithinClusterBoundary` — same caveat called out by #0105. A future
  pass can introduce a `MESSAGE_GAP_THRESHOLD_MS` analogue if we ever
  surface "isPending" or want sub-day cluster splits.
