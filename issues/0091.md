## Description

The React Native client supports new-account creation directly inside the app via a multi-step signup flow (`Bluesky-ReactNative/src/screens/Signup/`): **StepInfo** (email + password + birthdate), **StepHandle** (handle picker with availability check) and **StepCaptcha** (device attestation). The SwiftUI port has no signup UI at all — a brand-new user has nowhere to go from `LoginView`. The only way to create an account is to leave the app and use bsky.app, then return to log in.

For parity (and so the SwiftUI app is usable as a primary client), signup needs to be implemented as a SwiftUI flow that mirrors the RN steps and posts to the same `com.atproto.server.createAccount` endpoint.

## Steps to reproduce

1. Launch the SwiftUI app fresh (no saved account).
2. On `LoginView`, look for a way to create a new account.
3. Observe there is no "Create account" / "Sign up" affordance.

## Expected behavior

A "Create account" button on `LoginView` that opens a multi-step signup flow:

- **Step 1 — Info**: hosting provider picker (defaults to `bsky.social`), invite code (if required), email, password, birthdate.
- **Step 2 — Handle**: handle text field with live availability check, `.bsky.social` (or PDS) suffix preview, suggestions.
- **Step 3 — Captcha**: device attestation / hCaptcha equivalent. iOS already has access to `DCAppAttestService`, which is the natural fit; macOS may need a fallback.
- **Step 4 (deferred to onboarding, see #0092)**: profile setup → interests → suggested follows → completion.

Each step has back/next navigation, validates inputs locally, and displays server errors inline. On success the new session is saved via `SessionManager` exactly like login.

## Actual behavior

No signup affordance anywhere. `LoginView` is the only auth screen and only handles existing-account login.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Signup/index.tsx` (orchestrator) and `state/StepInfo.tsx`, `state/StepHandle.tsx`, `state/StepCaptcha.tsx`.
- AT Proto endpoint: `com.atproto.server.createAccount`. Lexicon types may need to be added to `BlueskyCore/Auth.swift` (CreateAccountRequest/Response, hosting-provider URL list, invite-code requirements).
- This issue is the first half of a two-part missing flow; #0092 covers the post-signup onboarding (interests, suggested follows, etc.) that RN runs immediately after a successful create-account.
- Files to create: new `SignupFlowView.swift` (container) + step views, in `BlueskyKit/Sources/BlueskyAuth/`. Wire into `LoginView` via a NavigationLink / sheet.

## Root cause

This was a missing-feature issue, not a regression. The SwiftUI port shipped with `LoginView` only and no path for new-account creation, so the app could not be used as a primary client. The fix is a deliberate, parity-driven implementation choice for the signup flow's three steps and how each step talks to the AT Proto API.

Design decisions:

* **Captcha:** RN native uses `DCAppAttestService` (iOS) / Play Integrity (Android) and stitches a token onto a `WKWebView`-loaded `/gate/signup/attempt-attest` URL. Wiring App Attest cleanly is involved (key generation, attestation challenge round-trip, JS bridge for token injection). For the first pass we ship the canonical web gate at `/gate/signup` — the same hCaptcha used by every Bluesky web user — and intercept the redirect query string for `verification_code`. App Attest can be layered in later without changing the call-sites.
* **Onboarding:** Issue #0092 covers the post-signup interests / suggested-follows / completion onboarding. This issue stops at `createAccount` succeeding and `SessionManager.currentAccount` being set, which moves `RootView` onto `MainTabView` exactly like a successful login.
* **Validation:** A `SignupValidation` namespace in `BlueskyCore` holds `createFullHandle`, `validateServiceHandle`, `isOverAge`, `isLikelyEmail`, and the constants RN uses (`maxFrontHandleLength = 18`, `minPasswordLength = 8`, `minSignupAge = 13`).

## Fix

* New `SignupFlowView` (3-step container, `@Observable SignupModel`) presented as a sheet from `LoginView` via a "Create an account" call-to-action.
* Three step views that mirror RN one-for-one:
  * `SignupStepInfoView` — hosting provider, optional invite code (only when the PDS reports `inviteCodeRequired`), email, password, birthdate. Drives `com.atproto.server.describeServer` to gate which fields show.
  * `SignupStepHandleView` — handle entry with debounced availability via `com.atproto.identity.resolveHandle`, suffix preview, character / length / hyphen requirement messages.
  * `SignupStepCaptchaView` — `WKWebView` pointed at `<service>/gate/signup`, intercepts the redirect for `verification_code`, then submits.
* Lexicon types added in `BlueskyCore/Auth.swift`: `DescribeServerResponse`, `CreateAccountRequest`, `CreateAccountResponse`.
* `SessionManager` learns `describeServer(_:)` and `createAccount(request:serviceURL:birthDate:email:)`. The latter posts to `com.atproto.server.createAccount`, saves the resulting JWTs through `AccountStore` exactly like login, and writes a best-effort `app.bsky.actor.putPreferences` for the birth date.
* `LoginView` gains a "Create an account" button that presents the signup sheet; on success the same `onSuccess` closure that login uses fires.

## Gotchas

* Swift 6 strictness made `@MainActor @Sendable` on `WKNavigationDelegate.decidePolicyFor:` necessary; the closure is split into a `@MainActor`-isolated extension on the coordinator to satisfy the protocol witness.
* `availableUserDomains` from `describeServer` includes the leading dot (`".bsky.social"`); `createFullHandle` strips it before joining so the displayed full handle is `name.bsky.social`, not `name..bsky.social`.
* The captcha redirect URL is intercepted by inspecting `URLComponents.queryItems` for `verification_code` and `code` (both have appeared in practice). The state nonce is checked to defend against replay.
* `SignupModel` is `@Observable` and `@MainActor`; `Bindable($model.handle)` is used in the step views so SwiftUI bindings survive the model living outside `@State` storage of the step itself.
