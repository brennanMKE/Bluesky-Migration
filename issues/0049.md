# 0049 — FeedView task fires twice on launch, creating duplicate FeedViewModels

| | |
|---|---|
| **Status** | resolved |
| **Module** | BlueskyFeed |
| **Platform** | All |
| **First seen** | 2026-04-29 |

## Description

On every launch, `FeedView`'s `.task(id: selection)` fires twice with the same `selection` value. Both firings find `viewModels[selection] == nil` and create separate `FeedViewModel` instances, triggering two concurrent network fetches. The first fetch is cancelled (error -999) when the original view is torn down; the second succeeds or fails independently.

## Steps to reproduce

1. Launch the app.
2. Observe the console output.

## Expected behavior

```
task fired, selection=timeline
creating FeedViewModel for timeline
calling loadInitial
```
(fires once, creates one VM, one fetch)

## Actual behavior

```
task fired, selection=timeline
creating FeedViewModel for timeline   ← first instance
calling loadInitial
...
task fired, selection=timeline
creating FeedViewModel for timeline   ← second instance, viewModels[] reset
calling loadInitial
fetch error: ... Code=-999 "cancelled"   ← first fetch cancelled
```

## Notes

Root cause: `FeedView` is being constructed twice during launch — likely because `MainTabView` renders its detail content before the environment is fully settled (e.g. auth state changes trigger a re-render), creating a second `FeedView` instance with a fresh `@State`. The `@State private var viewModels` is per-instance, so the second view cannot see the first's populated dict. Fix options: (1) lift `viewModels` state up into a parent `@State` or `@StateObject`; (2) key `FeedView` with a stable `.id()` to prevent recreation; (3) investigate why `MainTabView` is rebuilding the detail content pane during launch.

## Resolution (2026-04-30)

Fixed by wrapping the `[FeedSelection: FeedViewModel]` dictionary in `FeedViewModelCache`, a reference-type class stored as `@State`. SwiftUI preserves the same reference-type object across view re-inits, so the second `.task` fire finds the existing VM and logs `reusing existing FeedViewModel for timeline` instead of creating a duplicate. The `.timeline` case is additionally protected by the boot-time `FeedStore` injected via environment — `loadInitial` is started as a plain `Task` in `boot()` and cannot be cancelled by view lifecycle.

Confirmed in launch logs:
```
task fired, selection=timeline
attaching pre-built timeline store from boot()
task fired, selection=timeline
reusing existing FeedViewModel for timeline
```
