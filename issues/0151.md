## Description

Tapping the compose button (FAB / pencil icon / inline "What's up?" prompt) opens the composer with text **already in the editor** — leftover from a previous session that the user never saved as a draft. There was no draft-resume action; the user expected a blank editor.

This is the legacy single-slot auto-save (force-quit safety net) from #0034 firing on every fresh open. After #0100 shipped an explicit Drafts list with Save Draft / Discard / Keep Editing on cancel, the safety-net auto-restore is no longer the right default — it's surfacing pre-cancelled / pre-discarded state without consent.

## Steps to reproduce

1. Open the composer.
2. Type some text.
3. Cancel out (and decline to save as a draft, or just dismiss).
4. Later, tap the compose button again.
5. Observe: the editor shows the text from step 2.

## Expected behavior

A fresh tap on the compose button always opens an empty editor.

The only paths that should pre-fill the composer:

- Explicit **Resume from Drafts** (Drafts sheet → tap a draft).
- Explicit **Reply** to a specific post (the reply context is the pre-fill, not residual text).
- Explicit **Quote** of a specific post.
- A force-quit recovery offer — if it's kept at all, it should be **opt-in**: a small banner at the top of the empty editor saying "Restore previous draft?" with Restore / Discard buttons, NOT an automatic re-fill of the body.

## Actual behavior

Body text from a prior cancelled / dismissed compose session pre-populates the editor with no UX cue, no banner, no explicit action.

## Notes

- Legacy auto-save lives in `ComposerViewModel` / `ComposerSheet.swift` — see #0034 (filed) and #0100 (drafts UI). #0100's Gotchas explicitly retained the single-slot auto-save as a "safety net for force-quits while the explicit drafts list is the user-facing affordance" — but the auto-restore on every fresh open was not what the user wanted.
- Two acceptable resolutions:
  1. **Drop the auto-restore entirely.** Cancel always wipes (or defers to the Save Draft prompt); the explicit drafts list is the only resume path. Simplest fix.
  2. **Keep the safety-net but make it opt-in.** On open, if there's residual auto-saved text, render the editor empty and show a dismissible banner: "Restore unsaved draft?" — only fills the body if the user taps Restore.
  
  Recommendation: **option 1.** The Save Draft / Discard / Keep Editing prompt added by #0100 already covers the user-intent case for keeping work in progress. The auto-save slot is now redundant.
- Cross-cuts: image attachments, GIF, threadgate selection, self-labels, and quotes-enabled may also be auto-restoring incorrectly. Audit the full composer-state reset path on open.
- File to edit: `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` and `ComposerViewModel.swift`. Look for a `loadDraft()` / `saveDraft()` pair tied to `UserDefaults` keys like `composer.draft.text` / `composer.draft.reply.<uri>`.

## Root cause

`ComposerViewModel.init` ended with `self.text = UserDefaults.standard.string(forKey: draftKey) ?? ""`, where `draftKey` resolved to `composer.draft.text` for the top-level composer (and `composer.draft.reply.<uri>` for a reply context). The matching `saveDraft()` was wired into `ComposerSheet` two ways: `.onChange(of: viewModel.text) { viewModel.saveDraft() }` and `.onDisappear { viewModel.saveDraft() }`, plus `onTextChange()` itself called `saveDraft()`. The net effect was that *every* keystroke and *every* dismiss persisted the in-progress text under a single `UserDefaults` key, and *every* fresh open re-read it — completely silent, with no banner, no Restore prompt, no opt-in. After #0100 added the explicit Drafts list with Save Draft / Discard / Keep Editing on cancel, the auto-save slot was redundant: the user-intent path for keeping work in progress was now covered by an explicit affordance, but the implicit auto-restore kept firing and surfacing pre-cancelled / pre-discarded body text on the next open.

## Fix

Picked option 1 from the issue Notes — **removed the auto-save entirely** rather than keeping it as a defensive crash-recovery trace. Rationale: less state, fewer footguns. The Save Draft / Discard / Keep Editing prompt added by #0100 is the user-intent persistence path; a parallel implicit slot would either drift out of sync with the drafts list or re-introduce the same silent re-fill that this issue is about. The drafts list is the only place a draft can come back from now.

Concretely:

- Deleted `saveDraft()` and `clearDraft()` from `ComposerViewModel`.
- Deleted the `draftKey` computed property.
- Removed the `self.text = UserDefaults.standard.string(forKey: draftKey) ?? ""` line from `init`. A fresh composer open now starts with an empty `text` (the property's default `""`), which combined with the rest of the property defaults (`images = []`, `attachedVideo = nil`, `selectedGIF = nil`, `selectedLabels = []`, `threadgateAllow = .everyone`, `quotesEnabled = true`, `additionalPosts = []`, `editingDraftID = nil`, etc.) gives a guaranteed-empty editor unless the caller passes `replyTo:` / `quotedPost:` or a draft is loaded via `loadDraft(_:)`.
- Removed the `saveDraft()` call from `onTextChange()` so keystrokes no longer persist to `UserDefaults`.
- Removed `.onChange(of: viewModel.text) { viewModel.saveDraft() }` and `.onDisappear { viewModel.saveDraft() }` from `ComposerSheet.body`.
- Removed `clearDraft()` from `post()` (no key to clear) and from `discardCurrentContent()` (same).

**One-shot legacy-key migration.** Added `purgeLegacyAutoSaveKeysIfNeeded()` in `ComposerViewModel.init`. The first time a build containing this code constructs a composer, it removes `composer.draft.text` and every `composer.draft.reply.*` key from `UserDefaults.standard`, then sets `composer.draft.legacyAutoSave.purged.v1 = true` so the sweep doesn't run again. Existing users who already had stale text in those keys get a clean state on the next composer open instead of carrying the leftover blob around forever.

**Reset-path audit.** On `post()` success the view-model now nils `text`, `images`, `additionalPosts`, `attachedVideo`, `selectedGIF`, `linkMetadata`, `detectedURL`, `mentionPrefix`, `mentionDIDs`, `selectedLabels`, `threadgateAllow` (back to `.everyone`), `quotesEnabled` (back to `true`), and `editingDraftID`. `replyTo` / `replyToView` / `quotedPost` / `quotedPostView` are intentionally not cleared because they are caller-supplied context for the lifetime of *this* sheet and the sheet is dismissed on success — the next compose construction builds a brand-new view-model from `init`, so those start at their `nil` defaults. `discardCurrentContent()` performs the same reset (minus the touched-on-post-only fields) for the cancel-with-content "Discard" branch.

## Files changed

- `BlueskyKit/Sources/BlueskyComposer/ComposerViewModel.swift` — removed `draftKey`, `saveDraft()`, `clearDraft()`, the auto-restore line in `init`, and the `saveDraft()` call in `onTextChange()`. Added `purgeLegacyAutoSaveKeysIfNeeded()` migration. Removed `clearDraft()` calls from `post()` and `discardCurrentContent()`.
- `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` — removed the `.onChange(of: viewModel.text) { viewModel.saveDraft() }` and `.onDisappear { viewModel.saveDraft() }` modifiers on the `NavigationStack`.

## Gotchas

- **Force-quit recovery is now a non-feature.** If the app is killed mid-compose (force-quit, OOM, crash) the user loses the in-progress text. The previous auto-save would have restored it (silently, which was the bug). The trade-off was deliberate: option 2 in the issue Notes — "keep auto-save but never auto-restore, then surface a Restore banner" — would have been less destructive but adds new UX surface that wasn't asked for in this fix. If a force-quit recovery banner is wanted later, it can be re-added as a separate issue with explicit opt-in UX; the migration sentinel `composer.draft.legacyAutoSave.purged.v1` would not need to change.
- **Drafts list is the only persistence path.** Save Draft on cancel and the Drafts sheet for resume are both explicit; the implicit slot is gone. This matches RN's user-facing model (Drafts button opens a list, Save Draft on cancel is an explicit action).
- **Migration runs lazily.** `purgeLegacyAutoSaveKeysIfNeeded()` runs the first time a `ComposerViewModel` is constructed in this build, not at app launch. Users who never open the composer never pay the sweep; users who do pay it once. The legacy keys are small strings, so the sweep is cheap even when present.
- **`onTextChange()` is now URL detection only.** Don't reintroduce a `saveDraft()` call there or the bug returns. If a future periodic background save into the *drafts list* is needed (the deferred follow-up called out in #0100's Gotchas), it should be a separate debounced action with explicit deduping rather than a call back into a single-slot key.
