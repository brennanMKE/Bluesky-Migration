# 0151 — Composer opens with stale text pre-filled when no draft was resumed

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskyComposer |
| **Platform** | All |
| **First seen** | 2026-05-07 |

## Description

Tapping the compose button (FAB / pencil icon / inline "What's up?" prompt) opens the composer with text **already in the editor** — leftover from a previous session that the user never saved as a draft. There was no draft-resume action; the user expected a blank editor.

This is the legacy single-slot auto-save (force-quit safety net) from #0034 firing on every fresh open. After #0100 shipped an explicit Drafts list with Save Draft / Discard / Keep Editing on cancel, the safety-net auto-restore is no longer the right default — it's surfacing pre-cancelled / pre-discarded state without consent.

## Steps to reproduce

1. Open the composer.
2. Type some text.
3. Cancel out (and decline to save as a draft, or just dismiss).
4. Later, tap the compose button again.
5. Observe: the editor shows the text from step 2.

## Expected behavior

A fresh tap on the compose button always opens an empty editor.

The only paths that should pre-fill the composer:

- Explicit **Resume from Drafts** (Drafts sheet → tap a draft).
- Explicit **Reply** to a specific post (the reply context is the pre-fill, not residual text).
- Explicit **Quote** of a specific post.
- A force-quit recovery offer — if it's kept at all, it should be **opt-in**: a small banner at the top of the empty editor saying "Restore previous draft?" with Restore / Discard buttons, NOT an automatic re-fill of the body.

## Actual behavior

Body text from a prior cancelled / dismissed compose session pre-populates the editor with no UX cue, no banner, no explicit action.

## Notes

- Legacy auto-save lives in `ComposerViewModel` / `ComposerSheet.swift` — see #0034 (filed) and #0100 (drafts UI). #0100's Gotchas explicitly retained the single-slot auto-save as a "safety net for force-quits while the explicit drafts list is the user-facing affordance" — but the auto-restore on every fresh open was not what the user wanted.
- Two acceptable resolutions:
  1. **Drop the auto-restore entirely.** Cancel always wipes (or defers to the Save Draft prompt); the explicit drafts list is the only resume path. Simplest fix.
  2. **Keep the safety-net but make it opt-in.** On open, if there's residual auto-saved text, render the editor empty and show a dismissible banner: "Restore unsaved draft?" — only fills the body if the user taps Restore.
  
  Recommendation: **option 1.** The Save Draft / Discard / Keep Editing prompt added by #0100 already covers the user-intent case for keeping work in progress. The auto-save slot is now redundant.
- Cross-cuts: image attachments, GIF, threadgate selection, self-labels, and quotes-enabled may also be auto-restoring incorrectly. Audit the full composer-state reset path on open.
- File to edit: `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` and `ComposerViewModel.swift`. Look for a `loadDraft()` / `saveDraft()` pair tied to `UserDefaults` keys like `composer.draft.text` / `composer.draft.reply.<uri>`.
