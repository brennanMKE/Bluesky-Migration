## Description

When a user signs in with an account whose status is `deactivated`, the React Native client shows a dedicated screen at `Bluesky-ReactNative/src/screens/Deactivated.tsx` with: an explanatory message, a "Reactivate your account" button (calls `com.atproto.server.activateAccount`), a "Sign out" button, and an account picker so the user can switch to a non-deactivated account. The SwiftUI client has no equivalent — at best the user lands on a broken/empty MainTabView, since `SessionManager` doesn't branch on the deactivated status.

## Steps to reproduce

1. Have a Bluesky account in deactivated state.
2. Sign in to the SwiftUI app with that account.
3. Observe what the app does (likely: blank/error main tab view, no clear path to reactivate).

## Expected behavior

When `SessionManager` detects the active session is on a deactivated account, the app routes (above MainTabView) to a `DeactivatedView` that:

- Displays a status banner ("Your account is deactivated").
- Shows a "Reactivate account" primary button → calls `com.atproto.server.activateAccount`, then re-bootstraps to MainTabView on success.
- Shows a "Sign out" secondary action.
- Optionally exposes the existing `AccountPickerView` so the user can switch to another account.

## Actual behavior

No deactivated handling. The session is treated as a normal logged-in session and the user lands on MainTabView with API calls failing (deactivated accounts can't read/write).

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Deactivated.tsx`.
- Detection: account status comes from `com.atproto.server.getSession`'s `status` field (`active` / `takendown` / `suspended` / `deactivated`). May need to be added to `Account` / `StoredAccount` in `BlueskyCore`.
- AT Proto endpoint: `com.atproto.server.activateAccount`.
- File to create: `BlueskyKit/Sources/BlueskyAuth/DeactivatedView.swift`; gate from `RootView` in the app target.
- Pairs with #0095 (takendown screen) — both branch off the same status field.

## Root cause

`SessionManager` neither captured the `status` field returned by `createSession` / `refreshSession` / `getSession` nor exposed `com.atproto.server.activateAccount`, so a deactivated account flowed straight into `MainTabView`. RN consumes `currentAccount?.status === 'deactivated'` in `view/shell/index.tsx` to gate above the main shell — there was no equivalent in `RootView`.

## Fix

Plumbed the status from the wire all the way to the gate, then added the holding screen.

1. `BlueskyCore`
   - New `AccountStatus` enum with `.active / .takendown / .suspended / .deactivated`, raw values matching the AT Proto strings.
   - `Account.status: AccountStatus?` (nil treated as active) plus a `with(status:)` copy helper. `Account.isActive` convenience.
   - `GetSessionResponse` lexicon shape (`did / handle / email / emailConfirmed / active / status`).
2. `BlueskyAuth/SessionManager`
   - Captures `active` + `status` on `createSession` / `refreshSession` and merges them via `normalizeStatus(active:status:)`.
   - `resumeSession` now follows up with `getSession` to fetch the authoritative status (best-effort; non-fatal on failure) so the gate sees the right value on launch.
   - New `activateAccount()` (POST `com.atproto.server.activateAccount` with the access JWT) and `refreshCurrentSession()` for the post-reactivate refresh.
3. `BlueskyAuth/DeactivatedView`
   - Mirrors RN's `screens/Deactivated.tsx`: "Welcome back" copy + `@handle`, primary "Yes, reactivate my account" button, secondary "Cancel" sign-out, inline error banner (with the App Password / "Bad token scope" callout from RN), and the existing `AccountPickerView` when other stored accounts exist.
   - Lives in BlueskyAuth (Layer 2) — does not import BlueskyUI; styling uses `.quaternary.opacity(0.5)` matching `LoginView` / `SignupFlowView`.
4. `Bluesky-SwiftUI/ContentView`
   - `RootView` adds a fourth branch above the existing logged-out / onboarding / main split: when `currentAccount?.status == .deactivated`, render `DeactivatedView`. The `task(id: …)` already re-evaluates on account change, so reactivate / sign-out flow naturally back through the gate.

## Gotchas

- The status enum string raw values are case-sensitive and must match the AT Proto wire format (`takendown`, not `takenDown`). RN's lexicon uses lowercase.
- `getSession` after `resumeSession` is wrapped in a `do/catch` and only logged on failure: a flaky network on launch must not block sign-in. The status is also persisted into the keychain so the gate works offline on the next launch.
- `activateAccount` requires the **access** JWT (not the refresh JWT); using the refresh JWT will return `Bad token scope`. RN surfaces that exact message to nudge App Password users back to a real password — `DeactivatedView` does the same.
- Plumbing reused for #0095: the `AccountStatus` enum, the `Account.status` field, the `getSession` lexicon, and the `RootView` gate already cover `.takendown` / `.suspended` — #0095 only needs to add a parallel `TakendownView` and one more `else if` branch in `RootView`.
