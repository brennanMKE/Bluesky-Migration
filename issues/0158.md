## Description

Notification rows for `subscribed-post` reasons (and possibly other variants) render only the actor's name and a raw label ŌĆö `Matt Massicotte / subscribed-post`, `Point-Free / subscribed-post`, `Paul Hudson / subscribed-post` ŌĆö with no preview of the post itself. The whole point of a "new post from someone you subscribe to" notification is to surface the post; rendering only the reason name makes the row useless.

The reply and like rows further down in the same screenshot **do** render the post text correctly ("You get to decide how to *spell* thingsŌĆ”", "Last year I watched a few conference videosŌĆ”"), so the post-preview rendering path works for some reasons but not others.

This regresses #0079 ("Notifications: render inline post-content preview on each row"), which was resolved 2026-05-05. Either the `subscribed-post` reason was never wired into that path, or a subsequent change broke the routing. Pairs with #0063 which already noted that the **label** `subscribed-post` renders verbatim instead of friendlier copy ("Subscribed: ŌĆ”" / "New post from ŌĆ”").

## Steps to reproduce

1. Subscribe to an account's posts (use the bell on a profile in RN if needed).
2. Wait for that account to post.
3. Open Notifications.
4. Observe the row shows only `<actor> / subscribed-post` with no post preview.

## Expected behavior

The row layout for `subscribed-post` matches the reply / like / quote rows: actor avatar + actor name, the friendly reason copy ("New post from @actor"), and a 2-line preview of the post text below ŌĆö same `PostBodyView` (or compact equivalent) used by the reply/like rows. Tapping the row navigates to the thread for that post (covered by #0062 / #0160).

## Actual behavior

Bare `<actor> / subscribed-post` two-line cell. No post text, no embed thumbnail, no link card.

## Root cause

`NotificationsViewModel.actorPostReasons` ŌĆö the set that decides which
reasons preview the actor's *own* post (notification `uri`) versus the
viewer's post (`reasonSubject`) ŌĆö only listed `reply`/`mention`/`quote`.
`subscribed-post` was in `postPreviewReasons` but NOT in
`actorPostReasons`, so it fell through to the `reasonSubject` branch.
The AT Proto wire shape for `subscribed-post` puts the new post at
`notification.uri`; `reasonSubject` is typically nil/wrong. With a nil
`previewURI`, no post was hydrated and the row rendered just the actor
name + raw `subscribed-post` label. RN does the same routing ŌĆö
`Bluesky-ReactNative/src/state/queries/notifications/util.ts:288`
(`getSubjectUri`) returns `notif.uri` for `reply | quote | mention |
subscribed-post`. We were one entry short of parity.

## Fix

- **`NotificationsViewModel.actorPostReasons`** ŌĆö add
  `"subscribed-post"`. Two effects flow from this single change:
  1. `postPreviewURIs` now collects `n.uri` for `subscribed-post` rows,
     so `NotificationPostCache.hydrate` fetches the new post via
     `getPosts`.
  2. `groupedNotifications` keys subscribed-post rows by
     `"subscribed-post:<n.uri>"` instead of
     `"subscribed-post:<reasonSubject ?? authorDID>"`, so each new post
     is its own row even when the same author posts multiple times.
     This matches RN (`util.ts:167` carves out `subscribed-post` as a
     never-merge-same-author case).
- **`NotificationsScreen.reasonText`** ŌĆö added a `"subscribed-post"`
  case that returns `"posted"`. The actor name is already on the line
  above, so "Matt Massicotte / posted" reads naturally without needing
  to interpolate the actor into the label string. The full RN-matching
  copy ("New posts from {actor}", with stacked-actor pluralization)
  for every remaining reason still belongs to #0063 ŌĆö applying it here
  would require restructuring the row to interpolate actor names into
  the reason label, which is exactly the scope #0063 owns.

## Gotchas

- **Label fix here is intentionally minimal.** Per the bail rule, the
  full reason-label overhaul (icons, colors, RN-matching copy with
  actor interpolation) stays in #0063 ŌĆö that ticket also covers
  `like-via-repost`, `repost-via-repost`, `feedgen-like`,
  `starterpack-joined`, `contact-match`, `verified`, `unverified`. The
  one-liner "posted" label added here only ensures `subscribed-post`
  doesn't leak the raw machine string while #0063 is pending.
- **Grouping behavior changes for `subscribed-post`.** Previously all
  subscribed-post notifications from the same actor merged under one
  row (because `reasonSubject` was nil so the bucket key collapsed to
  `subscribed-post:<authorDID>`). Now each post is its own row. This
  matches RN's explicit carve-out and is the right behavior for a row
  that previews unique post text ŌĆö same reasoning as the
  reply/mention/quote ungrouping in #0079's gotchas.
- **Tap target works automatically.** `previewPostURI` is now non-nil
  for `subscribed-post`, so `onTap(group.previewPostURI ??
  group.reasonSubject)` routes to the actor's new post. The thread
  destination is still the placeholder `Text("Thread: ...")` ŌĆö proper
  thread navigation is #0062/#0160's territory.
- **Icon stays bell.** The `iconName` switch wasn't extended; the
  default branch shows a filled/outlined bell depending on read state.
  RN uses a distinct icon for subscribed-post; that visual lift is in
  #0063's scope.
