## Description

The Notifications screen needs a two-tab segmented selector at the top — **All** and **Mentions** — matching the React Native reference. Today the screen is a flat list with no filter. The Mentions filter is critical UX because it lets users jump straight to direct interactions without scrolling past likes/follows/reposts/etc.

This is a follow-up to the iOS Notifications parity audit in #0069.

## Scope

- **Add a segmented control** below the top bar (#0077) with two segments: **All** and **Mentions**.
- **All** — current behavior: every notification reason from the server.
- **Mentions** — client-side filter or, if the API supports it, a server-side filter that returns only `reply`, `mention`, `quote` reasons (i.e. things directed at you specifically).
- **Selected-tab indicator**: brand-color underline (matches RN; do not use the iOS-default segmented control look).
- **Persist the active tab** within the session; reset to All on app cold-start unless the user explicitly chose Mentions.
- **Underlying store** (`NotificationsStore`/`NotificationsViewModel`) should expose a filter parameter; UI just toggles it.

## Implementation notes

- If the AT Proto API exposes a server-side filter for mentions-only, prefer that to avoid over-fetching. Otherwise filter `groupedNotifications` in the view model based on `reason in ["reply", "mention", "quote"]`.
- Reuse `notificationList` rendering — only the source data array changes.
- The unread-badge computation should reflect the All count regardless of which segment is shown.

## Acceptance

- Two-segment selector appears at the top of Notifications.
- Switching to Mentions filters the list to reply/mention/quote reasons only.
- Switching back to All restores the full list.
- iOS Simulator and macOS builds pass.

## Root cause

`NotificationsScreen` rendered a single `notificationList` bound straight to
`viewModel.groupedNotifications` with no concept of a filter. The store
fetched every reason from `app.bsky.notification.listNotifications` and
dumped them into one array. Users who only wanted the directed-at-them
subset (replies, mentions, quotes) had to scroll past every like, repost,
and follow.

## Fix

Added a `NotificationFilter` enum (`.all` / `.mentions`) to
`NotificationsStore` and a `setFilter(_:)` entry point. The store keeps a
single underlying `allNotifications` array (one fetch, one cursor) and
exposes a derived `notifications` computed property that filters by the
active filter's `keepReasons` set. `.all` returns the full array;
`.mentions` keeps only entries with `reason ∈ {"mention", "reply", "quote"}`,
matching the React Native reference (`feed.ts` line 99-106). Switching
filters is a synchronous flip — both segments are backed by the same
fetched data so there's no refetch and no per-segment cursor to manage.

`NotificationsViewModel` exposes the filter via a get/set property and a
`setFilter(_:)` method; `groupedNotifications` automatically reflects the
active filter because it reads `store.notifications`. `NotificationsScreen`
adds a `NotificationsFilterStrip` (private to the file) above the list:
two text buttons in an `HStack`, each with a 2pt `Color.accentColor`
underline when selected — same visual pattern as the Home feed tab strip
from #0074, deliberately not the iOS-default segmented control look. A
local `@State activeFilter` cold-starts at `.all` and is propagated to
the view model via `.onChange(of: activeFilter)`. There's no
`@AppStorage` — the selection lives only for the screen's lifetime, as
the issue specified.

The unread badge stays bound to the All count: `markSeen()` and
`fetchUnreadCount()` both run against the unfiltered server endpoints
(`app.bsky.notification.updateSeen` / `getUnreadCount`), and
`onUnreadCountChange` fires off `viewModel.unreadCount` which is the
total — flipping the segment never zeros or filters the badge.

Added an empty-state view for the Mentions tab so it doesn't render a
blank rectangle when there are no mentions to show.

## Gotchas

- **Filter is client-side, not server-side — by design.** AT Proto's
  `app.bsky.notification.listNotifications` lexicon does expose a
  `reasons` array parameter (`['mention', 'reply', 'quote']` for
  Mentions in the RN reference at `feed.ts:99-106`), and we'd normally
  prefer server-side filtering. But our `NetworkClient.get(_:params:)`
  signature is a flat `[String: String]` dictionary — it can't carry
  repeated query items (`reasons=mention&reasons=reply&...`), and the
  AT Proto server requires repeated keys for `array` lexicon
  parameters (comma-joining is not part of the spec and is unreliable
  across pds implementations). Widening the protocol to support array
  params is a multi-module refactor (`NetworkClient` protocol in
  `BlueskyKit`, `ATProtoClient` impl in `BlueskyNetworking`, and any
  test doubles) with marginal payoff: the typical first page is 50
  items, the Mentions reasons are a small subset, and switching tabs
  is instantaneous because both segments are backed by the same array.
  If a future issue tackles array query params, the `keepReasons`
  set already aligns with the server `reasons` parameter so swapping
  to server-side filtering would be a one-line change in the store's
  `loadInitial`/`loadMore`/`refresh` paths.
- **Pagination is shared across both filters.** Because both segments
  read from `allNotifications`, scrolling to the bottom of Mentions
  triggers `loadMore` for the All cursor — which is fine, but means
  the Mentions tab pulls in additional likes/reposts/follows under the
  hood as it paginates. The user only sees the filtered subset; the
  network cost is identical to scrolling All. If a sparse-mentions
  user runs into an "everything got fetched but nothing showed up"
  pause, switching to server-side filtering (per the gotcha above)
  would help — until then, the existing on-appear `loadMore()` trigger
  is good enough.
- **Filter is per-screen, not per-store-instance-share.** The screen
  owns its `NotificationsViewModel`, which owns its `NotificationsStore`.
  Push-driven jumps into the Notifications tab from #0030 will land
  on whichever filter the user last chose during the same screen
  lifetime; if the screen is torn down and rebuilt, it cold-starts at
  `.all`. This matches the issue's "session-only" persistence rule.
- **Unread badge is intentionally filter-blind.** `unreadCount` is
  fetched and zeroed via the unfiltered `getUnreadCount` /
  `updateSeen` endpoints — flipping to Mentions never changes the
  badge, even if every unread notification on the server happens to
  be a like. Documented inline so the next reader doesn't try to
  "fix" it.
