## Description

The Bluesky website has a dedicated "Saved" section in the sidebar that lists all bookmarked posts as a scrollable feed. In the SwiftUI app, bookmarks are accessible only via the `•••` menu on the Profile tab → "Bookmarks", which is not discoverable. The `BookmarksScreen` also needs visual parity with the website's Saved Posts experience.

## Expected behavior

- A "Saved" entry is accessible from the app's primary navigation (sidebar on macOS/iPad, or via Profile tab menu on iPhone) at a consistent, discoverable location.
- `BookmarksScreen` shows the full post card for each saved post, with correct author info, text, embeds, and action bar.
- Pull-to-refresh and infinite scroll (load more) work correctly.
- Unbookmarking a post from the Saved screen removes it immediately (optimistic removal).

## Actual behavior

- Bookmarks are buried under Profile → `•••` → Bookmarks.
- The `BookmarksScreen` was rewritten to use the network `BookmarksStore` but visual and interaction parity with the website's "Saved Posts" view has not been validated.

## Notes

- `BookmarksStore` (`Sources/BlueskyFeed/BookmarksStore.swift`) already implements `getBookmarks`, `loadMore`, and `delete`.
- `BookmarksScreen` (`Sources/BlueskyFeed/BookmarksScreen.swift`) was rewritten in the #0057 fix to use the network store — it needs live testing and likely refinement.
- Consider whether "Saved" deserves a top-level `AppTab` entry rather than a nested menu item, to match the website's navigation hierarchy.

## Root cause

The Saved Posts surface had two gaps from the bsky.app reference:

1. **Discoverability.** Bookmarks were only reachable via Profile → `•••` → "Bookmarks". The web `LeftNav` (and the RN drawer) treat "Saved" as a top-level destination next to Home/Search/Notifications/Lists/Profile.
2. **Screen parity.** `BookmarksScreen` rendered post cards inside `List` rows with no pull-to-refresh and no tap-handling — taps on a saved post or its author were dead-ends. The empty state used `ContentUnavailableView` instead of a copy-matched "Nothing saved yet" message.

## Fix

- Added `.saved` to `AppTab` and split the cases into `compactTabs` (iPhone TabView keeps the existing five) and `sidebarTabs` (macOS / iPad regular include Saved). This mirrors the RN web LeftNav while leaving the iPhone tab bar at five tabs and keeping the Profile menu entry as a fallback.
- Wired the `.saved` tab content in `MainTabView.tabContent(_:)` to `BookmarksScreen` with `navigationDestination` for thread and profile navigation, matching the home tab pattern.
- Introduced a single shared `BookmarksStore` via `savedStoreOrCreate()` so both the sidebar tab and the Profile menu's "Saved" entry reuse the same store and avoid duplicate fetches.
- Reworked `BookmarksScreen`:
  - `List` → `ScrollView` + `LazyVStack` for visual parity with `FeedView`.
  - Added `.refreshable` for pull-to-refresh.
  - Added `onPostTap` and `onAuthorTap` callbacks so a saved post navigates to its thread / author.
  - Replaced the empty state with a "Nothing saved yet" message that matches the RN reference.
  - Renamed the navigation title to "Saved" to match the website.
- Renamed the Profile menu entry "Bookmarks" → "Saved" for consistency.

## Files changed

- `BlueskyKit/Sources/BlueskyFeed/BookmarksScreen.swift`
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift`

## Gotchas

- iPhone compact `TabView` is intentionally limited to five tabs (`AppTab.compactTabs`) — adding `.saved` there would push the system into "More…" overflow. Saved on iPhone stays in the Profile menu, matching the bsky.app mobile drawer.
- The shared `BookmarksStore` is created lazily on first access so unauthenticated launches don't pay for it.
