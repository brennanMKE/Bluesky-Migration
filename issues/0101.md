# 0101 — Composer doesn't warn when alt-text-required is on but images have no alt text

## Description

The React Native composer respects an "alt text required" preference (set in Accessibility settings). When that preference is on and the user has attached images without alt text, RN displays an `Admonition` error banner above the action bar reading approximately "All images must have alt text" and disables the Post button until alt text is provided.

The SwiftUI composer has the preference stored in `SettingsViewModel` (Accessibility) but doesn't consult it. A user with the preference enabled can still post without alt text, defeating the accessibility setting.

## Steps to reproduce

1. In Settings → Accessibility, enable "Require alt text".
2. Open the composer and attach an image.
3. Without typing alt text, tap Post.
4. Observe the post submits.

## Expected behavior

When `requireAltText == true` AND any attached image lacks alt text:

- A red admonition banner appears above the toolbar: "All images need alt text" (or RN's exact copy).
- The Post button is disabled until every image has non-empty alt text.
- Once all images have alt text, the banner clears and Post re-enables.

When the preference is off (default), behavior is unchanged.

## Actual behavior

Preference is ignored by the composer. Posts submit regardless of missing alt text.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/com/composer/Composer.tsx` — search for `requireAltTextEnabled` / `Admonition`.
- Preference key: `requireAltText` in `UserDefaultsPreferencesStore`. Already wired in `AccessibilitySettingsScreen` but not consumed by the composer.
- Files to edit: `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` (read preference + render admonition), `ComposerViewModel.swift` (computed property: `canPost` returns false when missing alt text).

## Root cause

`SettingsViewModel.altTextRequired` was wired to the
`AccessibilitySettingsScreen` toggle and persisted under the JSON-encoded
`settings.altTextRequired` key in `UserDefaultsPreferencesStore`, but
`ComposerViewModel` never read it. The composer's `canPost` only gated on
text emptiness, character limit, and `isPosting`; missing alt text on
attached media was not consulted. RN's
`view/com/composer/Composer.tsx` derives a `missingAltError` via
`useRequireAltTextEnabled()` and renders it through
`AltTextReminder` (`<Admonition type="error">`) above the toolbar, while
gating its `canPost` on `!missingAltError`. Our composer had no
equivalent path.

## Fix

**`ComposerViewModel`** gained a `requireAltText: Bool` flag, hydrated
once at construction time. The constructor now accepts an optional
`(any PreferencesStore)?` parameter; when provided it's queried directly
(this is the production path used by `MainTabView`'s `env.preferences`).
When omitted, the view model falls back to reading the same JSON-encoded
`settings.altTextRequired` key from `UserDefaults.standard` —
`UserDefaultsPreferencesStore` writes through `JSONEncoder().encode(value)`
to `.standard` defaults, so a `JSONDecoder().decode(Bool.self, from:)`
recovers the same value. This avoids a wide plumbing change to the
existing `ComposerSheet` call sites in `FeedView`, `ThreadView`,
`CustomFeedTimelineView`, etc., which take explicit `network` /
`accountStore` parameters but not a `BlueskyEnvironment`.

A new `altTextWarning: String?` computed property mirrors RN's
`missingAltError` `useMemo`. It returns `nil` when `requireAltText` is
off; otherwise it walks the attached media in the same order RN does
and returns the first matching warning string verbatim:

- `"One or more images is missing alt text."` — any image with empty
  `altText`.
- `"One or more GIFs is missing alt text."` — selected GIF whose
  Tenor-derived `altText` is empty.
- `"One or more videos is missing alt text."` — attached video whose
  `altText` is empty.

`canPost` was extended to require `altTextWarning == nil`, which is
what disables the Post button.

**`VideoAttachment`** gained a `var altText: String = ""` field. The
composer doesn't yet ship a UI for editing video alt text (see #0103
for the captions / alt-editing surface), but the field is here so the
require-alt-text gate can apply uniformly across images, GIFs, and
video — matching RN, which checks all three media types in
`missingAltError`.

**`ComposerSheet`** renders an `altTextReminder` view above the
`mediaToolbar`. The view shows a red exclamation icon + the warning
text on a `Color.red.opacity(0.12)` rounded background — the closest
SwiftUI analog to RN's `<Admonition type="error">` styling without
introducing a new shared component. The view is `@ViewBuilder` and
collapses to nothing when `viewModel.altTextWarning == nil`, so the
banner appears and disappears as the user adds / removes alt text on
attached images. SwiftUI's reactivity handles the live update because
`canPost` and `altTextWarning` both observe the same `images`
collection on the `@Observable` view model.

The `ComposerSheet.init` now also takes an optional
`preferences:` parameter (defaulted to `nil`) and threads it down to
the view model. `MainTabView`'s three `ComposerSheet(...)` call sites
in `Bluesky-SwiftUI` were updated to pass `env.preferences` — the rest
of the in-package call sites (FeedView, ThreadView, etc.) take the
default `nil` and fall back to the `UserDefaults.standard` read.

## Files changed

- `BlueskyKit/Sources/BlueskyComposer/ComposerViewModel.swift` — new
  `requireAltText` stored property, `altTextWarning` computed property,
  `canPost` extended, `init(...)` gained the optional `preferences:`
  parameter, plus the `loadRequireAltText(from:)` helper that picks
  between the injected store and the `UserDefaults.standard` JSON
  fallback.
- `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` —
  `altTextReminder` view inserted above `mediaToolbar`; the public
  `init(...)` gained the optional `preferences:` parameter and threads
  it to the view model.
- `BlueskyKit/Sources/BlueskyComposer/ComposerStore.swift` —
  `VideoAttachment.altText: String` added (default `""`), so the
  warning logic can apply uniformly to videos.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` — three
  `ComposerSheet(...)` call sites pass `preferences: env.preferences`
  so the production composer reads the preference via the injected
  `BlueskyEnvironment` rather than the `UserDefaults.standard` fallback.

## Gotchas

- **Two read paths for `requireAltText`.** When
  `BlueskyEnvironment.preferences` is in scope we go through the
  injected `PreferencesStore` (the production path from
  `MainTabView`). When it's not — every `ComposerSheet(...)` call site
  inside `BlueskyFeed` (`FeedView`, `ThreadView`,
  `CustomFeedTimelineView`) takes explicit `network` / `accountStore`
  parameters but no environment, so threading a preference store would
  be a wide refactor — we fall back to a direct
  `UserDefaults.standard.data(forKey:)` decode of the same key. Both
  paths read the same JSON-encoded blob, so the result is identical;
  the duplication is intentional to keep the fix scoped.
- **`requireAltText` is read once at construction.** This matches RN's
  `useRequireAltTextEnabled()` which is read at the top of `Composer`
  and held for the compose session — toggling the preference in
  Settings while the composer is open won't re-evaluate. Dismissing
  and re-opening the composer picks up the new value.
- **Video alt text has no editing UI yet.** The composer's
  `VideoAttachment` now carries an `altText` field but there's no
  composer surface to edit it — see #0103, which scopes the
  alt-text + captions UI for video. Consequence: with
  "Require Alt Text" on, attaching a video unconditionally blocks Post
  until the user either disables the preference or removes the video.
  This is the parity-correct enforcement of the user's accessibility
  opt-in (RN behaves the same way when video alt is missing) — the
  cliff is the missing video-alt UI, not the warning logic. Once #0103
  ships, this resolves naturally.
- **GIF alt text is Tenor-derived, not user-editable.** Our composer
  uses `gif.altText` from `TenorGif` (`content_description ?? title`)
  as the alt text. RN's `selectedGIF` lets the user override the alt
  inline; we don't yet. In practice Tenor results almost always carry
  a `content_description`, so the GIF check is a near-no-op in
  practice — but if Tenor returns a result with no description and no
  title, the warning fires correctly. A future GIF alt-text editor
  would slot into the same field.
- **Banner copy is an exact RN match.** The three strings come from
  `Composer.tsx` lines 784 / 787 / 794 — verbatim. Don't rewrite them
  for "naturalness"; they're the user-facing strings RN ships with
  i18n already pinned.
- **No new shared `Admonition` component yet.** RN uses
  `#/components/Admonition` across the codebase; we render the
  composer banner inline because it's the only consumer today. If a
  second consumer appears, hoist the inline `HStack` into a reusable
  `BlueskyUI.Admonition` view.
