## Description

The React Native client has an **External Media Preferences** screen (Settings → Content & Media → External media) with per-source toggles for which embedded players to inline-render: YouTube, YouTube Music, Spotify, Apple Music, Twitch, SoundCloud, Vimeo, Giphy, Tenor, Flickr, etc. Sources that are off render as plain link cards instead of inline players.

The SwiftUI `ContentSettingsScreen` has only a single boolean "Load external embeds" toggle. The per-source granularity is lost; users either get all third-party iframes or none.

## Steps to reproduce

1. Settings → Content & Media.
2. Look for per-source toggles.
3. Observe a single coarse toggle.

## Expected behavior

A new `ExternalMediaPreferencesScreen.swift` accessible from `ContentSettingsScreen` (or replace the single toggle with a navigation row). Each source row has its own toggle. Defaults match RN. Embed renderers in `BlueskyUI` (specifically `PostEmbedView`) consult the per-source flags before rendering inline players.

## Actual behavior

Single toggle; no per-source granularity.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Settings/ExternalMediaPreferences.tsx` (single file, not `index.tsx` directory). Source list lives at `src/lib/strings/embed-player.ts` (`embedPlayerSources` + `externalEmbedLabels`).
- Storage: RN keeps these in local preferences (not in AT Proto preferences) — verify the same is appropriate here. They probably belong in `UserDefaultsPreferencesStore`.
- Files to create: `BlueskyKit/Sources/BlueskySettings/ExternalMediaPreferencesScreen.swift`, plus a small struct in `BlueskyKit` for the per-source flags so feeds can read them.

## Root cause

The SwiftUI Content & Media screen had a single `Load External Embeds` boolean. RN exposes a 12-source list (YouTube, YouTube Shorts, Vimeo, Twitch, GIPHY, Tenor, KLIPY, Spotify, Apple Music, SoundCloud, Flickr, Bandcamp) where each source is independently toggleable, persisted locally in `persisted.Schema.externalEmbeds` as `'show' | 'hide' | undefined` (default `{}`, i.e. all off until the user opts in).

## Fix

- New value types in `BlueskyCore`:
  - `ExternalMediaSource` enum — 12 cases mirroring `embedPlayerSources` order, each with a display name (matching `externalEmbedLabels`) and an SF Symbol fallback glyph (`play.rectangle` / `music.note` / `sparkles` / `photo.on.rectangle`).
  - `ExternalEmbedsPreferences` struct — per-source `Bool` flags with subscript access by `ExternalMediaSource`. Defaults all-off to match RN's `{}` semantics (`true` ↔ `'show'`, `false` ↔ `'hide'`/undefined).
  - `ExternalEmbedsPreferencesKey.storage` — canonical `PreferencesStore` key (`"settings.externalEmbeds"`) so the future `PostEmbedView` consumer reads exactly what the screen writes.
  - Lives in `BlueskyCore` (Layer 0) rather than `BlueskyKit` (Layer 1) because `BlueskySettings` has `defaultIsolation(MainActor.self)` set; synthesized `Codable` conformance there gets MainActor-isolated and fails to satisfy the `Sendable`-bounded `PreferencesStore.get` generic. `BlueskyCore` has no isolation default, so the conformance stays nonisolated.
- New `ExternalMediaPreferencesScreen.swift` in `BlueskySettings`:
  - `Form` with an info section ("External media may allow websites to collect information…") followed by a per-source `Toggle` list.
  - Each row: SF Symbol + source name + "Enable inline players" caption.
  - `ExternalMediaPreferencesViewModel` (`@Observable @MainActor`) loads on init and persists every change via the injected `PreferencesStore`.
  - Light/Dark `#Preview` blocks using a small in-memory `PreferencesStore` stub.
- Wired into `ContentSettingsScreen`: kept the existing `Load External Embeds` master toggle for backward compat, added a `NavigationLink` row "External media" that opens the new screen, and updated the section footer to call out the master vs. per-source split. The screen receives `viewModel.preferences` (the same `PreferencesStore` instance as the rest of the settings hub).

## Files changed

- `BlueskyKit/Sources/BlueskyCore/ExternalEmbedsPreferences.swift` (new)
- `BlueskyKit/Sources/BlueskySettings/ExternalMediaPreferencesScreen.swift` (new)
- `BlueskyKit/Sources/BlueskySettings/ContentSettingsScreen.swift` (NavigationLink + footer copy)

## Gotchas

- `PostEmbedView` consumption is **out of scope** — this issue only persists the per-source flags and adds the UI. The renderer in `BlueskyUI/PostEmbedView.swift` still gates inline players on the master `viewModel.externalEmbeds` Bool. A follow-up should inject a `PreferencesStore` (or a snapshot type) into the embed renderer and read `ExternalEmbedsPreferences` keyed at `ExternalEmbedsPreferencesKey.storage`.
- Tenor is included in the SwiftUI list even though RN currently filters it (`// TODO: Remove special case when we disable the old integration.`); when RN re-enables it the list is already correct here.
- RN screen does not ship a master "Disable all" toggle — the existing SwiftUI master `Load External Embeds` boolean is a SwiftUI-side affordance kept for backward compat and as a kill-switch for the renderer.
- Defaults: RN's `defaults.externalEmbeds = {}` means an unconfigured user sees no inline players. We mirror that with `ExternalEmbedsPreferences()` having every flag default to `false`.
