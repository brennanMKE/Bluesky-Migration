## Description

`ContentFilterSettingsScreen` exposes a fixed set of global Bluesky labels (porn, sexual, nudity, graphic-media, hate, spam) with hide/warn/show pickers. The React Native reference builds the screen dynamically from each subscribed labeler's `labelValueDefinitions`:

- Each labeler gets its own section.
- Each section lists every label the labeler defines (which can be custom — e.g., a labeler might define `politics`, `news-outlet`, etc.).
- Per-label visibility picker writes to `ContentLabelPref` keyed by `(labeler DID, label name)`.
- Some labels are "self-applied only" or "system" — those rows are read-only.

The SwiftUI version cannot configure custom labeler labels; only the built-in global set works.

## Steps to reproduce

1. Subscribe to a custom labeler (one that defines its own labels).
2. Open Content Filters.
3. Observe the labeler's custom labels are absent.

## Expected behavior

`ContentFilterSettingsScreen` becomes per-labeler:

- One section per subscribed labeler (loaded from `subscribedLabelers` per #0133).
- Each row in a section is a label defined by that labeler, with hide/warn/show picker.
- Built-in Bluesky labels render under a "Bluesky Moderation" labeler section.
- Per-row visibility writes to `ContentLabelPref(label, visibility, labelerDid)`.

## Actual behavior

Six global labels, hardcoded.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Moderation/ContentFilterSettings.tsx` and `state/queries/preferences/labelers.ts`.
- Lexicon: `app.bsky.labeler.defs#labelerView.policies` carries `labelValueDefinitions` and `labelValues`.
- File to edit: `BlueskyKit/Sources/BlueskyModeration/ContentFilterSettingsScreen.swift` and `ModerationViewModel.swift` (load labeler policies).

## Root cause

`ContentFilterSettingsScreen` rendered a hardcoded set of six global Bluesky labels and never read `LabelerView.policies`. The lexicon pieces (`labelValueDefinitions`, `labelValues`) weren't even modeled in `BlueskyCore.LabelerView`, so the screen had no way to discover what labels a custom labeler exposed. Even though `ModerationStore.loadSubscribedLabelers` already passed `detailed=true` (per #0135), the policies bundle was being silently dropped in decoding.

## Fix

- Added `LabelerPolicies`, `LabelValueDefinition`, and `LabelValueDefinitionStrings` value types to `BlueskyCore/Moderation.swift`, plus a `policies: LabelerPolicies?` field on `LabelerView`. The detailed `getServices` response now round-trips per-labeler policies.
- Rewrote `ContentFilterSettingsScreen` to render dynamically:
  - One `Section` per subscribed labeler. Bluesky's official labeler (`did:plc:ar7c4by46qjdydhdevvrndac`) is pinned to the top under the heading "Bluesky Moderation"; everything else follows in the order returned by the appview.
  - Rows are built by combining `policies.labelValues` (deduped while preserving order) with `policies.labelValueDefinitions` for custom-label metadata. System labels prefixed with `!` are skipped — they match RN's `def.configurable === false` filter.
  - Built-in labels (`porn`, `sexual`, `nudity`, `graphic-media`) get their display title, severity, and `adultOnly` flag from a small constants table that mirrors RN's `LABELS` import.
  - Per-row picker reads from `ContentLabelPref` keyed by `(label, labelerDid)`. Built-in Bluesky labels are written with `labelerDid == nil` so other RN clients see the same global value (matches RN's `usePreferencesSetContentLabelMutation` behavior); custom labels carry the owning DID.
  - Adult-gated rows (built-in or custom with `adultOnly: true`) dim and lock to "Hide" when the adult-content toggle is off, mirroring RN's `prefAdjusted` calculation in `LabelerLabelPreference`.
- The screen now calls both `loadPreferences` and `loadSubscribedLabelers` on appear and on pull-to-refresh, so the policies are guaranteed to be available before the rows render.

## Files changed

- `BlueskyKit/Sources/BlueskyCore/Moderation.swift`
- `BlueskyKit/Sources/BlueskyModeration/ContentFilterSettingsScreen.swift`

## Gotchas

- The picker writes `"ignore"` (the lexicon's canonical value) for the "Show" tag; legacy `"show"` is normalized to `"ignore"` at read time so the UI stays stable across older preference payloads.
- The "Warn" option is hidden when `severity == "none"` (RN's `canWarn` rule). Custom labelers that mark severity as `"none"` will only see Hide/Show, matching RN.
- `ContentLabelPref` is keyed by `(label, labelerDid)` already (no schema change). Built-in Bluesky labels are intentionally stored with `labelerDid == nil`, not the Bluesky labeler DID, so they round-trip with RN's existing prefs payloads.
