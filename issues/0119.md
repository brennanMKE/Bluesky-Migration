## Description

`AppearanceSettingsScreen` currently shows only a font-size slider (12–24pt) and a static text note. The React Native equivalent has:

- **Color mode** (radio): **System** / **Light** / **Dark**.
- **Dark theme variant** (radio when Dark or System+dark): **Dark** / **Dim** (the lower-contrast OLED-friendly variant).
- **Font** (radio): **System** / **Theme** (RN's branded font, Inter).
- **Font size** (radio with 5 fixed levels) — RN uses 5 named tiers (XS / S / M / L / XL) rather than a continuous slider.

The SwiftUI `Theme` enum (`light` / `dark` / `dim`) already exists in `BlueskyUI`, but no UI lets the user pick. The app currently follows the system trait collection always, which means Dim is unreachable and Light/Dark can't be forced.

## Steps to reproduce

1. Settings → Appearance.
2. Compare to RN — SwiftUI lacks color-mode and theme-variant pickers.

## Expected behavior

`AppearanceSettingsScreen` shows:

- **Color mode** picker (System / Light / Dark) — applies via `.preferredColorScheme(_:)` on the root.
- **Dark theme variant** picker (Dark / Dim) — visible when color mode is Dark or System (and system is dark). Applies the existing `Theme.dim` palette.
- **Font** picker (System / Theme) — Theme uses `BlueskyUI`'s branded font when available.
- **Font size** discrete picker (5 tiers) replacing the slider for parity.

Persist all four choices via `UserDefaultsPreferencesStore` and apply at the app root.

## Actual behavior

Only a font-size slider. Dim theme is unreachable; color mode is forced to system.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Settings/AppearanceSettings.tsx` (the planning doc cited an `index.tsx` path; the actual RN file is the flat one).
- File to edit: `BlueskyKit/Sources/BlueskySettings/AppearanceSettingsScreen.swift`. App-root wiring lives in `Bluesky-SwiftUI/Bluesky-SwiftUI/Bluesky_SwiftUIApp.swift` / `RootView.swift`.
- Pairs with #0040 (color palette refinement) — material Bluesky-blue and the Dim theme are both color-system work.

## Root cause

`AppearanceSettingsScreen` only exposed a continuous font-size slider (12–24
pt) and a static text note. None of the four RN appearance preferences had a
control or persisted value:

1. **Color mode** (System / Light / Dark) — the SwiftUI shell hardcoded
   `adaptiveBlueskyTheme()`, which always tracked the host trait collection.
   The user could not force Light or Dark independent of the OS.
2. **Dark theme variant** (Dim / Dark) — the `BlueskyTheme.dim` palette
   already existed in `BlueskyUI` but was unreachable: `adaptiveBlueskyTheme`
   only ever resolved to `.light` or `.dark`.
3. **Font family** (System / Theme) — no preference, no picker.
4. **Font size tiers** — RN ships a discrete five-tier picker (XS / S / M /
   L / XL); SwiftUI shipped a 13-step slider that doesn't match RN parity.

The four prefs also had no shared source of truth: the Settings screen and
the app shell needed to read the same persisted values, but only the Settings
screen had a `SettingsViewModel`. Plumbing `viewModel` up to
`Bluesky_SwiftUIApp` would have created a layering inversion, so a dedicated
appearance store was needed.

## Fix

Added an `@Observable` `AppearanceStore` in `BlueskyKit` that owns the four
preferences, persists each on `didSet` via the injected `PreferencesStore`,
and is created once in `boot()` so the same instance reaches both the SwiftUI
shell (for `.preferredColorScheme(_:)` + `.blueskyTheme(_:)`) and the
Settings appearance screen (for the four pickers). Persistence keys are all
namespaced under `appearance.*` to keep them distinct from
`SettingsViewModel`'s `settings.*` keys.

In the SwiftUI shell, replaced `.adaptiveBlueskyTheme()` with a new
`AppearanceShellModifier` that reads the store and applies:

- `colorMode` → `.preferredColorScheme(_:)`. `.system` returns `nil` so the
  modifier is a no-op and the host trait collection wins; `.light` / `.dark`
  force the SwiftUI color scheme.
- The resolved `colorScheme` + `darkVariant` → which `BlueskyTheme` palette
  is injected. Light mode always uses `.light`; dark mode picks between
  `.dim` and `.dark` based on the user's variant choice.

In `AppearanceSettingsScreen`, replaced the slider + note with four
`Picker(.segmented)` rows that bind directly to `appearance.*` via
`@Environment(AppearanceStore.self)`. The dark-theme picker is gated on
`shouldShowDarkVariant` — visible when `colorMode == .dark`, or when
`colorMode == .system` and the resolved `\.colorScheme` is `.dark`. This
matches RN's "show when not in pure light" logic while honouring System on
both light and dark hosts.

`SettingsViewModel` lost its `fontSize: Double` field and the
`settings.fontSize` persistence key — both replaced by
`AppearanceStore.fontSize` (the discrete tier). No production callers read
`viewModel.fontSize` outside the now-rewritten Appearance screen.

## Files changed

- `BlueskyKit/Sources/BlueskyKit/AppearanceStore.swift` — new file. Defines
  `AppearanceColorMode`, `DarkThemeVariant`, `AppearanceFontFamily`, and
  `AppearanceFontSize` (with `bodyPointSize` mapping XS=14 → XL=18 pt and a
  `shortLabel` for the picker). Holds an `@Observable` store that loads from
  / persists to `PreferencesStore` under namespaced `appearance.*` keys.
- `BlueskyKit/Sources/BlueskySettings/AppearanceSettingsScreen.swift` —
  full rewrite. Four segmented pickers (Color Mode / Dark Theme / Font /
  Font Size), gated dark-variant section, live font-size preview using the
  selected tier's `bodyPointSize`. Pulls `AppearanceStore` from the SwiftUI
  environment.
- `BlueskyKit/Sources/BlueskySettings/SettingsViewModel.swift` — removed
  `fontSize: Double`, the `settings.fontSize` persistence key, and the
  matching load / save lines. Added a comment pointing to `AppearanceStore`.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/Bluesky_SwiftUIApp.swift` — created the
  `AppearanceStore` in `boot()`, injected via `.environment(appearance)` on
  `RootView`. Replaced `.adaptiveBlueskyTheme()` with a new
  `AppearanceShellModifier` that resolves `preferredColorScheme` and the
  active `BlueskyTheme` from the store.

## Gotchas

- **`Theme` font family is currently a no-op.** RN's "Theme" font is the
  bundled "Inter" face; no branded font is bundled in the SwiftUI app yet,
  so the Font picker persists the user's choice but both `.system` and
  `.theme` render with the system font today. The picker is shipped now so
  the preference round-trips and so the rendering layer can switch over
  without a data migration once Inter is added to the asset bundle. Track
  as a follow-up if this needs to land sooner.
- **Tier count diverges from RN.** RN's font-size picker has three options
  (Smaller / Default / Larger). The issue spec asked for five tiers
  (XS / S / M / L / XL); the implementation follows the spec rather than
  RN here. Body sizes 14 / 15 / 16 / 17 / 18 pt straddle RN's three steps,
  so a future migration to three tiers is a label change plus collapsing
  XS+S → Smaller and L+XL → Larger; persisted values would map cleanly.
- **`adaptiveBlueskyTheme()` is no longer used at the shell.** The view
  modifier still lives in `BlueskyUI` for any standalone preview / test
  context that wants automatic light/dark resolution without an
  `AppearanceStore`. The shell explicitly does not call it because the
  store-driven modifier supersedes it (otherwise the dim variant would
  never apply).
- **Dark-variant section uses resolved `\.colorScheme`, not the raw
  `colorMode` value.** The picker hides itself when the system trait is
  light even if the user picked System, mirroring RN's `colorMode !==
  'light'` guard with the additional refinement that System+light hides
  the variant control (it has no visible effect).
- **`SettingsViewModel` doesn't gain new properties.** The task suggested
  "(or similar)" — the appearance store is the source of truth and the
  Settings screen reads it via the SwiftUI environment, so adding mirror
  properties to the view model would create two writers for the same
  persistence keys with no win. Existing setting screens that don't touch
  appearance (Content / Notification / etc.) are unaffected.
