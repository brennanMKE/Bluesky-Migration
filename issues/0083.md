## Description

The Profile screen's banner area diverges from the React Native reference in three coordinated ways: a giant handle ("brennan.sstools.co") is rendered as a navigation title above the banner, the banner image sits in the middle third of the header instead of running edge-to-edge under the status bar, and the "Edit Profile" pill plus the blue-dotted ellipsis are not arranged in their own row below the banner. Together these change how the top of the profile feels — and they should ship as one redesign.

This is a follow-up to the iOS Profile parity audit in #0070.

## Scope

### Drop the handle navigation title

- **Today**: `brennan.sstools.co` renders as a giant `.large` navigation title above the banner.
- **Fix**: switch to `.toolbar(.hidden)` for the navigation bar on the Profile screen, or use `.inline` mode and only show the handle when scrolled past the banner. RN shows nothing above the banner.

### Banner edge-to-edge

- **Today**: the banner image sits inset, with the giant title above and a black background visible above the banner.
- **Fix**: banner runs flush under the status bar to the full width of the screen. Honor the safe-area inset by extending the banner image into it (with `.ignoresSafeArea(edges: .top)` on the banner only, not the rest of the content).
- **Avatar position**: stays at the bottom-left of the banner, overlapping by half its diameter. **Drop the white ring** around the avatar (RN does not render a ring).

### Edit Profile + blue-dotted ellipsis row

- **Today**: the "Edit Profile" pill sits inline with the avatar at the right edge, with the ellipsis on its own at the top-right of the screen.
- **Fix**: put both controls in their own row below the banner, right-aligned. The ellipsis carries a small blue dot indicator (matches RN — signals new options/menu items).
- **Order**: "Edit Profile" pill on the left of the row, ellipsis-with-dot on the right.

## Implementation notes

- Code: `BlueskyKit/Sources/BlueskyProfile/ProfileScreen.swift` (or `ProfileHeader.swift` if the header is its own view).
- `.toolbar(.hidden, for: .navigationBar)` on iOS hides the giant title cleanly.
- For the inline-on-scroll variant of the title (optional polish), use `.toolbar { ToolbarItem(placement: .principal) { Text(handle).opacity(scrollOffset > threshold ? 1 : 0) } }`.
- The blue-dot indicator is a small overlay on the ellipsis — a 6pt blue circle at top-right of the icon, conditionally shown when the user has unviewed menu items (or always shown if the indicator state isn't yet wired).

## Acceptance

- iOS profile: no giant handle headline; banner runs edge-to-edge under the status bar.
- Avatar overlaps banner without a white ring.
- "Edit Profile" pill + blue-dotted ellipsis sit in their own right-aligned row below the banner.
- macOS: layout unchanged.
- iOS Simulator and macOS builds pass.

## Root cause

Three independent layout choices in the SwiftUI Profile screen had drifted from the RN reference. (1) `ProfileScreen` applied `.navigationTitle(handle)` unconditionally, which on iOS produces a `.large` headline floating above the banner. (2) The banner inside the LazyVStack section header respected the top safe-area inset and used a fixed 130pt height, so it never reached the status bar. (3) The own-profile actions were split between two surfaces — the toolbar carried an `ellipsis.circle` Menu (Settings/Saved/My Lists/Moderation) while the in-screen avatar row held a lone Edit Profile pill on the right; RN unifies these as a single right-aligned row of [Edit Profile, ellipsis-with-blue-dot] at the avatar overlap band.

## Fix

All changes are gated `#if os(iOS)` so macOS layout is untouched.

1. `ProfileScreen.swift` now applies `.toolbar(.hidden, for: .navigationBar)` on iOS in place of the giant `.navigationTitle`, and keeps `.navigationTitle(handle)` only on macOS.
2. `ProfileHeaderView.swift` extends the banner with `.ignoresSafeArea(edges: .top)` on iOS and bumps the iOS banner height to 150pt so it absorbs the status-bar inset; the avatar's white ring is dropped on iOS.
3. A new iOS-only `iosAvatarAndActions` row lays out the avatar (size 84, half-overlap, no ring) on the left and a right-aligned `iosActionRow` containing Edit Profile pill + ellipsis Menu. The ellipsis carries a 6pt accent-color dot overlay at top-right. For own profile the ellipsis Menu hosts Settings/Saved/My Lists/Moderation via new optional `onSettings`/`onSaved`/`onMyLists`/`onModeration` callbacks threaded from `ProfileScreen` into `ProfileHeaderView`. `MainTabView`'s iOS profile case now passes those callbacks and drops the toolbar Menu (the toolbar Menu still ships on macOS via `#if os(macOS)`).

## Files changed

- `BlueskyKit/Sources/BlueskyProfile/ProfileHeaderView.swift` — banner edge-to-edge, drop white ring, new iOS avatar+actions row, blue-dot ellipsis Menu, new own-profile menu callbacks.
- `BlueskyKit/Sources/BlueskyProfile/ProfileScreen.swift` — hide nav bar on iOS, plumb new callbacks through.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` — pass `onSettings`/`onSaved`/`onMyLists`/`onModeration` to `ProfileScreen`; toolbar Menu now `#if os(macOS)`-only.

## Gotchas

- **Blue-dot indicator is unwired.** It's rendered unconditionally as a placeholder. There is no "unviewed menu items" signal in `BlueskyKit` yet. Follow-up: tie this to a real indicator state once the source is decided (RN appears to flag new menu options).
- **Inline-on-scroll handle title polish was skipped.** With the nav bar hidden the banner stays unannounced even when scrolled past — adding a scroll-position-driven principal toolbar item would have required a `.scrollPosition` listener that was out of scope for this issue.
- **Banner safe-area bleed only triggers when the nav bar is hidden.** If a future change re-shows the iOS nav bar, the banner will get clipped by the bar's frame — keep the two coordinated.
- **iOS banner height.** Bumped to 150pt so the visible banner pixels above the status-bar zone match the previous 130pt feel. Adjust if a future device model changes the typical inset materially.
