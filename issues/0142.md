## Description

The React Native client clamps long post bodies to ~25 lines (`MAX_POST_LINES`) and shows a **"Show more"** button below; tapping expands the full text and replaces the button with **"Show less"**. The SwiftUI `PostCard` / `PostBodyView` either renders the full body uncapped (in feed) or applies a fixed `lineLimit` set by the caller (in compact contexts) — but there's no expansion control either way. Long-form posts break the feed visually with massive cards.

## Steps to reproduce

1. Find a post with a body longer than ~25 lines.
2. Observe in feed it renders the entire wall of text, with no "Show more" affordance.

## Expected behavior

- `PostBodyView` clamps to a configurable max (default 25 lines for feed, unlimited for thread highlighted post).
- When clamped, a "Show more" button appears below the text.
- Tapping expands; the button becomes "Show less".
- Auto-expanded when post is the focal post in a thread.

Use `Text.lineLimit(_:)` plus a measure pass (e.g., `GeometryReader` + `PreferenceKey`) or SwiftUI's `truncationMode` + line-count helper.

## Actual behavior

No expansion control. Posts render at their full length in feed.

## Notes

- RN reference: `Bluesky-ReactNative/src/components/Post/ShowMoreTextButton.tsx` and `src/lib/strings/helpers.ts` (`countLines`).
- File to edit: `BlueskyKit/Sources/BlueskyUI/PostBodyView.swift` (or wherever the post text view lives — `RichTextView`?).
- Match RN's threshold (~25 lines) and copy ("Show more" / "Show less").

## Root cause

`PostBodyView` rendered the post body as `RichTextView` with at most a caller-supplied `lineLimit` (e.g. `4` for notification rows) and no expansion affordance. Long bodies in the feed therefore rendered uncapped and stretched the cards vertically. RN clamps bodies whose `countLines(text) >= MAX_POST_LINES` (= 25) to `MAX_POST_LINES` and renders a `<ShowMoreTextButton>` below them.

## Fix

- Added `MAX_POST_LINES = 25` and a `countPostLines(_:)` helper to `PostBodyView.swift` (RN parity with `MAX_POST_LINES` in `src/lib/constants.ts` and `countLines` in `src/lib/strings/helpers.ts`; both count `"\n"` occurrences, not visual wraps).
- Extended `PostBodyView` with a local `@State private var isExpanded = false` and a new `isFocal: Bool = false` initializer parameter for #0146's focal-post path.
- When `lineLimit == nil`, `isFocal == false`, and `countPostLines(text) >= MAX_POST_LINES`, the body is clamped to `MAX_POST_LINES` lines and a "Show More" button renders below it. Tapping the button sets `isExpanded = true` (with a 0.2 s ease-in-out animation), drops the line limit, and hides the button — mirroring RN exactly. RN's `ShowMoreTextButton` only renders the "Show More" label; once expanded there is no "Show Less" return, so this implementation deliberately drops the issue's "Show less" suggestion in favor of live RN behavior (per `CLAUDE.md` "trust the live RN code").
- Notifications still pass an explicit `lineLimit: 4` and bypass the new path entirely (no expansion in compact previews — matches RN's notification-feed clamp behavior).
- Focal-post callers (#0146) can pass `isFocal: true` to render the full body without a clamp regardless of length.

## Files changed

- `BlueskyKit/Sources/BlueskyUI/PostBodyView.swift` — `MAX_POST_LINES`, `countPostLines`, expansion state, `isFocal` parameter, "Show More" button, updated previews.

## Gotchas

- RN's `countLines` counts `\n` characters only, so a single-paragraph 1,000-character post does **not** trip the threshold — only posts with at least 25 hard line breaks. This matches RN behavior; if we later want a character-count fallback we can add it without changing the API surface.
- The `isExpanded` state is per-`PostBodyView` instance. SwiftUI may recycle the view when the surrounding `ForEach` reorders, but because we key feed rows by post URI, a recycled cell will collapse a previously-expanded post when scrolled off-screen and back on. RN behaves the same way (`useState` is local to the rendered post component).
