## Description

After a successful create-account in the React Native client, the user is dropped into a multi-step onboarding flow at `Bluesky-ReactNative/src/screens/Onboarding/` that shapes the empty home feed into something useful. It includes (roughly): profile setup (display name + avatar pick), **interests** picker (multi-select of topic chips that seeds Discover), **suggested accounts** to follow (per-interest), **suggested feeds**, **suggested starter packs**, and a completion screen. The SwiftUI app has none of this — even if signup landed (#0091), a brand-new user would arrive at an empty Home with no recommendations.

This is also relevant for *existing* RN users who haven't been through onboarding yet, which RN handles via the `hasOnboarded` preference flag in `app.bsky.actor.preferences`.

## Steps to reproduce

1. (Once #0091 lands, or with a fresh test account) sign in as a new account that has not completed onboarding in RN.
2. Observe that the app goes straight to MainTabView with a blank/empty feed.

## Expected behavior

After successful signup (or after sign-in when `hasOnboarded` is false) the app presents an onboarding `NavigationStack` with:

1. Welcome / value-prop screen.
2. Profile setup: display name, avatar pick (PHPicker), short bio.
3. Interests picker: grid of topic chips (Tech, Sports, News, Art, etc.); multi-select; seeds Discover algorithm.
4. Suggested accounts: per-interest list of recommended profiles with Follow buttons.
5. Suggested feeds: per-interest list of feed generators with Follow / Pin buttons.
6. Suggested starter packs (optional — RN shows them).
7. Find friends (links to existing `FindContactsScreen`).
8. Completion: writes `hasOnboarded = true` to preferences, dismisses to `MainTabView`.

Each step has skip / back / next navigation. Pinned interests + selected follows are written via `app.bsky.actor.putPreferences` and `app.bsky.graph.follow`.

## Actual behavior

No onboarding UI at all. New accounts land on MainTabView with an empty feed and no recommendations.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Onboarding/index.tsx` and `Steps/` siblings.
- Onboarding completion flag lives in preferences as `app.bsky.actor.defs#interestsPref` and a `hasOnboarded`-style flag — see RN's `state/preferences/onboarding.ts`.
- This is a Module 1 / Module 14 cross-cutting feature: needs `BlueskyAuth` for the gate, `BlueskySettings` for the preferences write, and (ideally) a new `BlueskyOnboarding` module to keep the screens isolated.
- Pairs with #0091 (signup) — together they form the full new-user journey.

## Root cause

`BlueskyAuth.SignupFlowView` previously dropped users straight into
`MainTabView` after `createAccount`. There was no onboarding flow,
no interests picker, and no `hasOnboarded` gate in `RootView`.

## Fix

1. Added a new Layer-3 `BlueskyOnboarding` Swift package target
   under `BlueskyKit/Sources/BlueskyOnboarding/`. It depends on
   Layer 0–2 (`BlueskyCore`, `BlueskyKit`, `BlueskyAuth`,
   `BlueskyDataStore`, `BlueskyUI`, `BlueskyNetworking`) and on no
   other Layer-3 modules — onboarding stands alone.
2. Implemented an `@Observable @MainActor` `OnboardingModel` that
   holds the per-step state and drives the four side-effects RN
   does in `StepFinished`: `app.bsky.actor.putPreferences` for the
   interests pref, a best-effort `com.atproto.repo.putRecord` for
   the profile (display name + bio), per-actor follows via
   `app.bsky.graph.follow`, and a local `hasOnboarded` flag.
3. Built five SwiftUI step views — `OnboardingWelcomeStepView`,
   `OnboardingProfileStepView`, `OnboardingInterestsStepView`
   (with a hand-rolled `FlowLayout` for the chip grid),
   `OnboardingSuggestedAccountsStepView` (per-interest tab strip
   pulling from `app.bsky.unspecced.getSuggestedOnboardingUsers`,
   falling back to `app.bsky.actor.getSuggestions` when the
   unspecced lexicon is unavailable), and
   `OnboardingFinishedStepView`.
4. Added `GetSuggestedOnboardingUsersResponse` to `BlueskyCore`
   and a new `PutPreferencesRequest(interests:)` constructor that
   encodes `app.bsky.actor.defs#interestsPref`.
5. Plumbed a separate `onSignupSuccess` callback through
   `LoginView` so the host app can distinguish a fresh
   create-account from a sign-in. The shipped `RootView` now has
   three states: logged-out → `LoginView`, logged-in but
   onboarding-pending → `OnboardingFlowView`, logged-in and
   onboarded → `MainTabView`. The local pref defaults to `true`
   so existing users upgrading into the build are not re-routed
   through the flow; signup explicitly flips it to `false`.

## Gotchas

- **Avatar upload via PHPicker is deferred.** The Profile step
  renders the user's existing avatar (or initials) but does not
  yet open `PHPickerViewController` to let them swap it. RN's
  `StepProfile/index.tsx` does this via `expo-image-picker` plus
  blob upload; the SwiftUI parity port wraps that work in a
  follow-up issue.
- **Suggested feeds, suggested starter packs, and find-contacts
  steps are deferred.** RN includes these between
  `suggestedAccounts` and `finished`; this pass ships only the
  core flow. Find-contacts already exists at
  `BlueskySettings.FindContactsScreen` and could be linked from
  the Finished step in the next pass.
- **Bulk follow / interests-topic header is simplified.** RN sends
  the selected interests as a custom `bsky-topics` header on the
  suggested-onboarding-users call to bias the response, and bulk-
  writes follows via `bulkWriteFollows`. Custom headers aren't yet
  threaded through `NetworkClient`, so we send the `category`
  param only and follow profiles one-at-a-time. Both are
  follow-up wins.
- **The local `hasOnboarded` flag is per-device only.** RN reads
  the equivalent from a server-synced preference. Round-tripping
  this through `app.bsky.actor.preferences` is the correct long-
  term fix; for now, completing the flow once on each device is
  acceptable parity.
