## Description

The React Native post controls include a **Translate** option (visible inline near the post body when language differs from app language, and also in the ellipsis menu). When tapped, it routes through the platform translate API or the web Translate service. The SwiftUI `PostCard` ellipsis menu (added in #0047) has Copy text / Mute thread / Report — no Translate option.

## Steps to reproduce

1. Open a post in a foreign language.
2. Open the ellipsis menu.
3. Look for Translate.
4. Observe it's missing.

## Expected behavior

- **Inline affordance**: when `post.langs[0]` differs from the app/content language and the post body is long enough to warrant translation, render a small "Translate" link below the post body.
- **Menu item**: also include "Translate post" in the ellipsis menu always.
- Tapping invokes `TranslationKit` (iOS 17.4+ / macOS 14.4+) via `.translationPresentation(...)`. On older systems, fall back to opening the system Translate app or the bsky.app web translate page.

## Actual behavior

No translate option anywhere on a post.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/com/post/Post.tsx` (translate row) and the ShareMenu translate item.
- iOS API: `TranslationSession` / `.translationPresentation(isPresented:text:)`.
- File to edit: `BlueskyKit/Sources/BlueskyUI/PostCard.swift`.

## Root cause

`PostCard` was added in #0047 with a Copy / Mute / Report ellipsis menu but no Translate affordance. The RN post (`src/components/Post/Translated/index.tsx`) renders a `TranslationLink` directly under the body when `!isPostInLanguage(post, [primaryLanguage])` and the share menu always offers a translate item. The SwiftUI port had neither inline link nor menu entry.

## Fix

Reused the same SwiftUI Translation pattern already shipped for message bubbles in #0106 (`MessageThreadScreen.swift` line 757). In `PostCard.swift`:

1. `import Translation` and added `@State private var isTranslating: Bool = false`.
2. Anchored `.translationPresentation(isPresented: $isTranslating, text: item.post.record.text)` on the outer card so both entry points present the same system sheet.
3. Inserted a "Translate" link below `postBody` (above any embed) gated by `shouldShowInlineTranslate`, which checks the BCP-47 primary subtag of `post.record.langs[0]` against `Locale.current.language.languageCode` and requires `text.count > 20` — matches the spec's pragmatic heuristic.
4. Added a "Translate post" item to the ellipsis Menu (always visible per RN's ShareMenu, disabled only when the body is empty for image-only posts), placed between Copy and Mute thread.

Copy: inline label is "Translate" (RN parity with `<Trans>Translate</Trans>`); menu label is "Translate post" (matching the spec wording in the issue). Icon `character.bubble` matches the message-thread translate item.

## Files changed

- `BlueskyKit/Sources/BlueskyUI/PostCard.swift` — added Translation import, `isTranslating` state, `translateLink` subview, `shouldShowInlineTranslate` helper, ellipsis-menu "Translate post" button, `.translationPresentation` modifier.

## Gotchas

- `.translationPresentation` requires iOS 17.4 / macOS 14.4. The project's deployment baseline is well above this so no availability-gating was needed (matches the precedent set by #0106).
- The inline-visibility heuristic intentionally uses `Locale.current` rather than `LanguageSettingsViewModel.contentLanguages` per the spec ("pragmatic: just always show inline when `post.langs[0] != Locale.current.languageCode` for now"). When the language settings model is integrated more broadly, swap the helper to read user content-language preferences for full RN parity.
- Posts without declared `langs` never show the inline link (RN's `isPostInLanguage` also short-circuits when no language metadata exists). The ellipsis menu still offers Translate so the user can force a translation.
