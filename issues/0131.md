## Description

`ListsScreen` shows a single flat list of every list the user can see (created + subscribed). The React Native reference splits this into segmented sections — **My Lists** (ones the viewer created) and **Subscribed Lists** (ones the viewer has subscribed to without owning) — typically as either tabs or labeled sections in a single scroll. This separation matters because the actions per row differ: edit/delete on owned lists, unsubscribe on subscribed.

## Steps to reproduce

1. Open the Lists hub when you both own lists and subscribe to others.
2. Observe everything is one flat list with no ownership indicator.

## Expected behavior

Either:

- **Tabs**: "My Lists" / "Subscribed" / "Moderation" (matching RN's segmentation).
- Or **sections**: a single scroll with two `Section` headers.

Each row carries the appropriate trailing affordance (edit menu vs unsubscribe). Use `creatorDID == viewerDID` to classify each list.

## Actual behavior

One flat list; no ownership distinction.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Lists/MyLists.tsx` (the `filter` prop discriminates the two cases).
- File to edit: `BlueskyKit/Sources/BlueskyLists/ListsScreen.swift`.
- Mod lists currently surface separately in `ModerationListsScreen` (`BlueskyModeration`) — leave them there; this issue is about curate-list ownership in the Lists hub specifically.

## Root cause

`ListsScreen` rendered `viewModel.lists` as a single flat `List` with no
ownership classification, even though RN's `MyLists` component surfaces
ownership through its `filter` prop and segments the curate vs mod views.
Per-row affordances were also undifferentiated — `.onDelete` ran on every
row regardless of whether the viewer authored the list.

## Fix

Split the hub into two `Section`s — **My Lists** and **Subscribed** — keyed
on `creator.did == viewerDID`:

- `ListsViewModel` now exposes `viewerDID`, `ownedLists`, and
  `subscribedLists` computed arrays plus a `resolveViewerDID()` helper. When
  the viewer is logged out, `ownedLists` returns the full set so the screen
  never strands rows.
- `ListsScreen` renders two `Section` headers. The "My Lists" section keeps
  the `.onDelete` swipe; the "Subscribed" section never offers delete
  (matches RN — subscribed-list management lives behind unsubscribe, not
  delete). Pagination still triggers off the underlying flat-list tail so
  it fires regardless of which section the last row landed in.
- Empty states are per-section: "Lists you create will appear here." for
  owned, "Subscribed lists will appear here." for subscribed.

Per RN parity: the curate hub (`MyLists filter="curate"`) only fetches
`getLists({actor: currentAccount.did})` and does NOT pull `getListMutes` /
`getListBlocks` (those are gated to the `mod` filter, which has its own
screen via `ModerationListsScreen`). So the "Subscribed" section is
expected to be empty in practice — the structure exists for parity with
mod-list expectations and to land the per-row affordance differentiation.

## Files changed

- `BlueskyKit/Sources/BlueskyLists/ListsScreen.swift` — two sections + per-section row builders.
- `BlueskyKit/Sources/BlueskyLists/ListsViewModel.swift` — viewer DID resolution + `ownedLists`/`subscribedLists` computed arrays.

## Gotchas

- The pagination trigger (`onAppear` + tail check) intentionally compares
  against `viewModel.lists.last?.uri` (the flat array), not
  `ownedLists.last`. If we keyed off the section array, the trigger could
  miss when the last row lives in the "Subscribed" section once that fetch
  is wired up (#0137 territory).
- `ownedLists` returns the full set when `viewerDID == nil` (logged-out
  preview / unauthenticated path) so previews still render rows.
- Mod lists are surfaced by `ModerationListsScreen` (BlueskyModeration);
  this hub remains curate-only per RN's `Lists.tsx` screen.
