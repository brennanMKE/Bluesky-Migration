## Description

The React Native client has a dedicated **Post Interaction Settings** screen (`Bluesky-ReactNative/src/screens/ModerationInteractionSettings/`) where the user sets *defaults* for who can interact with their posts and DMs:

- **Allow replies from**: Everyone / People you follow / Mentioned / Lists / Nobody.
- **Allow quotes from**: Everyone / Nobody (or list combo).
- **Allow incoming DMs from**: Everyone / People you follow / Nobody.

These defaults are written via `app.bsky.actor.putPreferences` (specifically `app.bsky.actor.defs#postInteractionSettingsPref` and the DM-allowance pref) and applied as the default `threadgate` / `postgate` on new posts (see composer issue #0098) and as the DM allow-list.

SwiftUI has no Interaction Settings surface at all â€” neither in Moderation nor in Privacy.

## Steps to reproduce

1. Look for "Interaction settings" in the SwiftUI Moderation hub or Privacy settings.
2. Observe there is none.

## Expected behavior

A new `PostInteractionSettingsScreen.swift` (or split into `ReplyDefaultsScreen` / `QuoteDefaultsScreen` / `DMDefaultsScreen`) accessible from both the Moderation hub (per #0133) and the Privacy hub (per #0114). Each section is a radio picker. Persists via `putPreferences` and is consulted by:

- Composer (`#0098` threadgate/postgate UI uses the saved defaults as initial values).
- DM allow logic (incoming DMs honor the allow rule on the server side; client-side just displays the setting).

## Actual behavior

Screen does not exist.

## Root cause

Neither the Moderation hub (#0133) nor the Privacy & Security hub (#0114) exposed an editable surface for the user's default reply/quote rules. Both rows were forward-references that opened bsky.app placeholder sheets. The `app.bsky.actor.defs#postInteractionSettingsPref` lexicon shape was also unmodeled in `GetPreferencesResponse` / `PutPreferencesRequest`, so even if the UI existed there was no path to read or write the pref.

## Fix

- **`BlueskyCore/Moderation.swift`**: added `PostInteractionSettingsPref` (round-tripped through `GetPreferencesResponse.postInteractionSettings`, omitted-key vs. empty-array vs. populated semantics preserved) and a `PutPreferencesRequest.init(postInteractionSettings:)` initializer that emits a single-pref payload. Reused the existing `ThreadgateAllowRule` / `PostgateEmbeddingRule` enums from `Post.swift` rather than declaring duplicates â€” the lexicon shares wire types between per-post records and the prefs blob.
- **`BlueskyModeration/PostInteractionSettingsScreen.swift`**: new `Form` screen with two server-backed sections plus an admonition. Top section is a three-way radio (Anyone / Combined / Nobody); when Combined is selected the screen reveals Followers / Following / Mention toggles plus a read-only summary row for any list rules pulled in from the server (so writes preserve list selections even though list creation isn't built yet). Quote-posts is a single toggle backed by `#disableRule` presence. `@Observable` `@MainActor` view model: optimistic updates with snapshot-based rollback on `putPreferences` failure.
- **Hub linkage**: replaced the placeholder `Button â†’ sheet` pattern in both `ModerationScreen.swift` and `BlueskySettings/PrivacySettingsScreen.swift` with a real `NavigationLink` into `PostInteractionSettingsScreen`. Added `BlueskyModeration` as a dependency of `BlueskySettings` (no cycle â€” both already depend on `BlueskyKit/Core/UI`).
- **DM allowance: deliberately deferred.** RN's RN-side dialog also covers DM allowance, but DMs ride on `chat.bsky.actor.declaration` (a separate record on a different collection), not on `actor.defs` prefs. Per the issue spec, ship the reply + quote defaults aligned to `postInteractionSettingsPref` and track DM allowance as a follow-up. The Privacy hub label was updated from "Who can reply, quote, or DM you" to "Who can reply or quote your posts" so the row description matches the screen content.

## Gotchas

- `ThreadgateAllowRule` and `PostgateEmbeddingRule` already exist in `BlueskyCore/Post.swift` (used by per-post threadgate/postgate records). The pref reuses those wire types â€” do not add parallel enums under different names. The first cut of this fix duplicated them and caused redeclaration errors.
- The `nil` vs. empty-array distinction on `threadgateAllowRules` is load-bearing: nil means "anyone replies" and `[]` means "nobody replies". The decoder uses `KeyedDecodingContainer.contains(_:)` to disambiguate "key absent" from "key present with empty array", and the `PutPreferencesRequest` encoder omits the field for the nil branch and emits `[]` for the empty branch. Both branches must round-trip without leaking the wrong meaning.
- DM allowance (`chat.bsky.actor.declaration`) is deliberately not on this screen. A follow-up issue should add a dedicated DM defaults row on the Privacy hub once the chat declaration record is plumbed; the current copy says "reply or quote" so the scope is honest.
- The composer side (#0098) can read the new `GetPreferencesResponse.postInteractionSettings` to seed its initial threadgate/postgate UI. Wiring that read is a small follow-up â€” out of scope for this issue per the spec.
- The "Combined" reply mode behaves like RN's lower toggle group: when zero sub-toggles are checked, the screen falls back to writing `nil` (anyone) rather than `[]` (nobody), so the user can't accidentally lock themselves out by exiting the screen mid-edit.
