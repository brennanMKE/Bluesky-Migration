# 0001 — Account session not persisted across app launches

| | |
|---|---|
| **Status** | closed |
| **Module** | BlueskyDataStore |
| **Platform** | macOS |
| **First seen** | 2026-04-29 |
| **Closed** | 2026-04-29 |

## Description

After signing in successfully, the session is not persisted across app launches. On every relaunch the user is presented with the sign-in screen and must re-authenticate.

## Steps to reproduce

1. Launch the app.
2. Sign in with a valid Bluesky account.
3. Confirm the home feed is active.
4. Quit the app completely.
5. Re-launch the app.

## Expected behavior

The app resumes the existing session and goes directly to the home feed.

## Actual behavior

The app shows the sign-in screen on every launch, as if no session was previously stored.

## Root cause

`KeychainAccountStore._loadAll()` used `kSecMatchLimitAll` combined with `kSecReturnData: true` in the same `SecItemCopyMatching` query. This combination is not supported on macOS and returns `errSecParam` (-50). The error propagated up through `SessionManager.restoreLastSession()`, which caught it silently and fell through to the login screen.

The write path (`SecItemAdd`) worked correctly — tokens were saved to the Keychain after login — but the read path on relaunch always failed.

## Fix

Changed `_loadAll()` to issue two queries instead of one:

1. Fetch attributes only (`kSecReturnAttributes: true`, no `kSecReturnData`) with `kSecMatchLimitAll` to enumerate stored account keys — this query succeeds on macOS.
2. Call the existing `_load(did:)` for each key, which uses `kSecMatchLimitOne` and is reliable on all platforms.

Added OSLog structured logging to `KeychainAccountStore` and `SessionManager` so the exact Keychain status code and restore path are visible in Console.app.

## Files changed

- `Sources/BlueskyDataStore/KeychainAccountStore.swift` — fix `_loadAll()`; add OSLog
- `Sources/BlueskyAuth/SessionManager.swift` — add OSLog to `restoreLastSession`
