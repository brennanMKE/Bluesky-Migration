## Description

The iOS top bar (#0072) introduces a `#` button that, in the React Native reference, navigates to a "My Feeds" screen. That screen does not yet exist in the SwiftUI app and needs to be built. It is the home for managing custom feeds — the user's pinned feeds, saved feeds, suggested feeds, and a search/discovery entry point. It is one of the most-used screens in the RN client because it's the only way to switch the active feed beyond the small set in the Home tab strip.

## Reference

`Bluesky-ReactNative/src/view/screens/Feeds.tsx` (or `MyFeeds.tsx` depending on version) is the canonical implementation. Cross-reference `src/state/queries/preferences/savedFeeds.ts` for the data model.

## Scope

### Sections

The screen is a single scrollable view with these sections, in order:

1. **Search bar** — type-ahead search against the feed-discovery API (`app.bsky.unspecced.getSuggestedFeeds` plus `app.bsky.feed.searchPosts`-style fallback). Tapping a result opens that feed.
2. **My Feeds** — the user's saved feeds (pinned + saved-but-unpinned). Pinned feeds appear first with a pin indicator; user can drag to reorder, swipe to unpin, or tap to open. Reorder persists via `app.bsky.actor.putPreferences` saved-feed entries.
3. **Discover new feeds** — a curated list of suggested feeds from `app.bsky.unspecced.getSuggestedFeeds`. Each card shows feed name, description, author, like count, and a "+ Add" button.

### Interactions

- **Tap a feed card → open the feed timeline** in a new navigation destination using the existing `FeedView` (or the equivalent timeline view).
- **Pin / unpin toggle** on each saved feed.
- **Drag-to-reorder pinned feeds** with haptic feedback on iOS.
- **Swipe-to-delete** unpinned saved feeds.
- **"+ Add"** on a discover card adds the feed to saved feeds (unpinned by default).

### State

- `SavedFeedsViewModel` already exists in `BlueskyFeed/SavedFeedsScreen.swift` plus stores. Reuse rather than duplicate.
- Suggested-feeds list lives in a new `SuggestedFeedsStore` (or as a method on the existing `SavedFeedsStore`).
- Search results list is its own ephemeral state in the view model.

## Implementation notes

- Place this screen in `BlueskyKit/Sources/BlueskyFeed/MyFeedsScreen.swift` since it's primarily a feed-management surface.
- Wire it as the `navigationDestination` for the top-bar `#` button added in #0072.
- iOS-only consideration: use `.draggable` / `.dropDestination` for reordering and `.swipeActions` for unpin. macOS may need a different reorder affordance — confirm before shipping.
- Use the existing component patterns from `SavedFeedsScreen` and `BookmarksScreen` for visual consistency.

## Root cause

The iOS Home top bar shipped in #0072 with a `#` button that routed
to a temporary `MyFeedsPlaceholderView`. The full feed-management
surface — search-by-feed, pinned/saved list with reorder, and a
Discover section — did not yet exist; this issue fills that gap.

## Fix

Added three new files in `BlueskyKit/Sources/BlueskyFeed/`:

- **`MyFeedsScreen.swift`** — the screen itself. Three sections in
  the order the issue specifies: a search bar, a `My Feeds` list,
  and a `Discover new feeds` list. The screen hosts a
  `SavedFeedsViewModel` (reused unchanged from `SavedFeedsScreen`)
  and a `SuggestedFeedsStore`. Tapping a feed routes via two
  closure props (`onFeedTap`, `onSuggestedTap`) so the screen stays
  decoupled from the app target's navigation. Reorder uses `↑↓`
  buttons next to each pinned row instead of drag-to-reorder — see
  Gotchas. Swipe-to-unpin / swipe-to-remove via `.swipeActions` on
  iOS, since macOS lacks that affordance.
- **`SuggestedFeedsStore.swift`** — `@Observable` store with three
  arrays: `suggested` (one-shot load from
  `app.bsky.feed.getSuggestedFeeds`), `searchResults` (live
  type-ahead from `app.bsky.unspecced.getPopularFeedGenerators`,
  monotonic request-id guard so a slow earlier response can't
  overwrite a faster later one), and `resolvedFeeds` (a
  `[String: GeneratorView]` dictionary populated via
  `app.bsky.feed.getFeedGenerator` so saved-feed AT-URIs render with
  real names/avatars instead of the rkey fallback).
- **`CustomFeedTimelineView.swift`** — single-feed counterpart to
  the existing `FeedView`. `FeedView` is wedded to its inline
  Following / Discover switcher, so opening an arbitrary feed
  needed a parameterised view. This one wraps `FeedStore` /
  `FeedViewModel` directly with a fixed `FeedSelection.feed(uri:)`
  and reuses the same like / repost / bookmark / reply
  ergonomics so behavior matches the rest of the app.

In `BlueskyCore/FeedGenerator.swift` added `GetFeedGeneratorResponse`
(the singular `view`/`isOnline`/`isValid` response shape) so the new
store has a Codable target without going around the layered design.

In `Bluesky-SwiftUI/MainTabView.swift` swapped the placeholder
destination for the new `MyFeedsScreen`, added an
`openCustomFeedURI: ATURI?` navigation flag that pushes a
`CustomFeedTimelineView`, hooked the `following` saved-feed entry up
to dismiss the My Feeds screen rather than push a no-op timeline, and
removed the no-longer-referenced `MyFeedsPlaceholderView` declaration.

## Gotchas

- **Reorder ergonomics — `↑↓` buttons over drag-to-reorder.** The
  issue's open question called this out. SwiftUI's `.draggable` /
  `.dropDestination` are flaky on macOS (the drag preview snaps
  oddly outside `List`, and re-ordering in a hand-rolled `LazyVStack`
  needs preference-key plumbing that gets fragile fast), so each
  pinned row carries an inline `arrow.up` / `arrow.down` pair next
  to the pin indicator. The buttons disable themselves at the ends
  of the list. The same affordance works identically on iOS and
  macOS. If a future pass wants drag-to-reorder on iOS, layer it on
  top — the `↑↓` pair stays as the macOS fallback.
- **Saved-feed URI resolution is one call per URI.** The lexicon's
  bulk endpoint (`app.bsky.feed.getFeedGenerators`) takes repeated
  `feeds=` query params, but the shared `NetworkClient.get` API
  passes `[String: String]` — one value per key — so the bulk form
  isn't reachable yet. `SuggestedFeedsStore.resolve(uris:)` issues
  one parallel `getFeedGenerator` (singular) per unresolved URI via
  `withTaskGroup` instead. The typical saved-feeds list is small
  (<10), so this is fine in practice; if a multi-value `get` helper
  ever lands on `NetworkClient`, swap the loop for the bulk lexicon.
- **Suggested-feed `+ Add` saves immediately.** Tapping `+ Add`
  appends the feed to `savedFeedsViewModel.feeds` and then triggers
  `savedFeedsViewModel.save()` right away rather than waiting for
  the section's `Save` button. Pin-state and reorder edits stage
  changes locally and only persist on `Save` — RN parity.
- **Search debounce is per-keystroke 250ms.** Implemented inside
  `.onChange` with a `Task.sleep` plus a same-text re-check. Matches
  the existing search debounce pattern in `SearchScreen` without
  pulling in the extra dependency on a Combine debouncer.
