# 0135 — Report dialog uses flat reasons; needs categorized picker + labeler selection

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskyModeration |
| **Platform** | All |
| **First seen** | 2026-05-06 |

## Description

`ReportDialog` exposes a small flat list of report reasons (six entries). The React Native reference uses a richer model:

1. **Categorized reason tree** — top-level categories (Spam, Harassment, Sexual content, Violence/threats, Illegal, Misleading, Rude, Other) with sub-reasons under each. RN maps these to Ozone-namespaced types (`REASONVIOLENCE`, `REASONSEXUAL`, `REASONHARASSMENT`, etc.) and falls back to base `com.atproto.moderation.defs#reason*` codes for older labelers.
2. **Labeler picker** — when multiple labelers can handle a category, RN shows a labeler list so the user can choose which moderation service to send the report to. The available labelers are filtered by both the subject type and the chosen reason.

The SwiftUI version skips both: reasons are flat, and the report always goes to a single hard-coded labeler (or no labeler at all).

## Steps to reproduce

1. Open the report dialog from any post / profile.
2. Compare to RN's dialog.

## Expected behavior

Refactor `ReportDialog`:

- **Step 1 — Reason category**: list of top-level categories.
- **Step 2 — Sub-reason** (where applicable): list of sub-reasons under the chosen category. Maps to Ozone reason type.
- **Step 3 — Labeler**: list of labelers that handle the chosen reason for the chosen subject type, each with "Send to <labeler>" button. Single-labeler case skips this step.
- **Step 4 — Details**: optional textarea + Submit.

Loaded labelers come from `ModerationViewModel.subscribedLabelers` (introduced for #0133). Reason filtering uses each labeler's declared `policies.labelValueDefinitions` and `reasonTypes` from the labeler service record.

## Actual behavior

Flat reason picker, hardcoded labeler.
