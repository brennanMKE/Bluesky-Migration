## Description

`ReportDialog` exposes a small flat list of report reasons (six entries). The React Native reference uses a richer model:

1. **Categorized reason tree** — top-level categories (Spam, Harassment, Sexual content, Violence/threats, Illegal, Misleading, Rude, Other) with sub-reasons under each. RN maps these to Ozone-namespaced types (`REASONVIOLENCE`, `REASONSEXUAL`, `REASONHARASSMENT`, etc.) and falls back to base `com.atproto.moderation.defs#reason*` codes for older labelers.
2. **Labeler picker** — when multiple labelers can handle a category, RN shows a labeler list so the user can choose which moderation service to send the report to. The available labelers are filtered by both the subject type and the chosen reason.

The SwiftUI version skips both: reasons are flat, and the report always goes to a single hard-coded labeler (or no labeler at all).

## Steps to reproduce

1. Open the report dialog from any post / profile.
2. Compare to RN's dialog.

## Expected behavior

Refactor `ReportDialog`:

- **Step 1 — Reason category**: list of top-level categories.
- **Step 2 — Sub-reason** (where applicable): list of sub-reasons under the chosen category. Maps to Ozone reason type.
- **Step 3 — Labeler**: list of labelers that handle the chosen reason for the chosen subject type, each with "Send to <labeler>" button. Single-labeler case skips this step.
- **Step 4 — Details**: optional textarea + Submit.

Loaded labelers come from `ModerationViewModel.subscribedLabelers` (introduced for #0133). Reason filtering uses each labeler's declared `policies.labelValueDefinitions` and `reasonTypes` from the labeler service record.

## Actual behavior

Flat reason picker, hardcoded labeler.

## Notes

- RN reference: `Bluesky-ReactNative/src/components/ReportDialog/`.
- Lexicon: `tools.ozone.moderation.defs#reasonType` plus the older `com.atproto.moderation.defs` reason codes.

## Root cause

`ReportDialog` shipped the very first cut from Phase 0: a six-row inline `Picker`
mapping directly to legacy `com.atproto.moderation.defs#reason*` codes. RN had
since moved to the Ozone `tools.ozone.report.defs#reason*` taxonomy with eight
top-level categories and per-category sub-reasons, and added a labeler picker so
the user can choose which moderation service receives the report. The SwiftUI
client also had no way to attach the per-call `atproto-proxy` header that the
labeler picker requires.

## Fix

- **Categorized reason picker.** Added `ReportCategoryKey`, `ReportCategoryConfig`,
  `ReportOption`, and a static `ReportReasons.categories` table mirroring RN's
  `useReportOptions.ts` (8 categories, 35 sub-reasons). Auto-skips the sub-reason
  step for single-option categories ("Other" collapses to a one-row picker).
- **Backwards-compat reason mapping.** Ported `NEW_TO_OLD_REASONS_MAP` so reports
  fall back to the legacy `com.atproto.moderation.defs#reason*` value when the
  chosen labeler only declares the older types.
- **Labeler picker.** Filters the caller-supplied labelers by `subjectTypes`,
  `subjectCollections`, and `reasonTypes` (matching RN's `supportedLabelers`
  memo). Reasons in `bskyOnlyReasons` (CSAM / extremist content) collapse to
  Bluesky's official labeler. Single-labeler case auto-selects and skips the
  step. Empty-set case shows an inline notice and submits without proxy.
- **`atproto-proxy` per-call header.** Added an overload of `NetworkClient.post`
  that takes an optional `proxyDID`. `ATProtoClient` attaches
  `<did>#atproto_labeler` when set; the protocol extension provides a default
  that ignores `proxyDID` so existing preview / mock conformers don't have to
  change. `ModerationStore.report` now accepts `labelerDID`.
- **Detailed labeler view.** `loadSubscribedLabelers` now requests
  `detailed=true` so the response includes `subjectTypes`, `subjectCollections`,
  and `reasonTypes`. `LabelerView` gained matching optional fields.
- **Optional details textarea.** Auto-opens for `*Other` reason types
  (mirroring RN's `OTHER_REPORT_REASONS`); user can also expand it manually.

## Gotchas

- **`ListDetailScreen` call site not yet wired to labelers.** The existing list
  report sheet keeps the legacy two-arg `onSubmit` initializer (still supported
  for back-compat) and passes an empty `availableLabelers` array, so the
  labeler picker is hidden there until the screen is plumbed with the
  user's subscribed labelers (`ModerationViewModel.subscribedLabelers`). The
  dialog still benefits from the categorized reason taxonomy.
- **`reasonTypes` is read from the labeler service record's policies.** RN
  reads it from the same field; not all third-party labelers populate it, in
  which case the dialog treats them as accepting all reason types
  (matching RN's "`undefined` = accept all" rule).
- **No `convoMessage` subject.** Chat / DM reports aren't yet exposed on the
  SwiftUI side; `ReportSubjectKind` only models account + record. Adding chat
  later means a new case + a `chat.bsky.convo.defs#messageRef` encoder.
