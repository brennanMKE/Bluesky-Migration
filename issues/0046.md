# 0046 — Tapping a post author avatar does not navigate to their profile

| | |
|---|---|
| **Status** | resolved |
| **Module** | BlueskyFeed / BlueskyProfile |
| **Platform** | All |
| **First seen** | 2026-04-29 |
| **Closed** | 2026-05-05 |
| **Commit** | 7c195f7 |

## Description

Tapping the avatar or author name on a post card in the feed does nothing. Navigation to the tapped author's profile is not wired up, so there is no way to view another user's profile, follow or unfollow them, or browse their posts from the feed.

## Steps to reproduce

1. Open the Home feed.
2. Tap the avatar or author name on any post.
3. Observe that nothing happens — no navigation occurs.

## Expected behavior

Tapping an author opens their `ProfileScreen` with their posts, followers, and follow/unfollow controls.

## Actual behavior

No navigation. The tap is silently ignored.

## Notes

`PostCard.Actions.onAuthorTap` exists and is plumbed through `FeedView.actions(for:vm:)`, but `FeedView` sets `onAuthorTap` from an optional closure passed in at the call site. `MainTabView` constructs `FeedView` without supplying `onAuthorTap`, so the closure is nil and taps are dropped. The fix needs to wire `onAuthorTap` in `MainTabView` (and `ThreadView`) to push `ProfileScreen` via a `navigationDestination`.

## Root cause

The fix had two layers:

1. **App-shell wiring** (already landed in `e020951` on 2026-04-30): `MainTabView` did not pass an `onAuthorTap` closure to `FeedView` or `ThreadView`, and there was no `navigationDestination(item: $feedProfileDID)` registered. That commit added a `feedProfileDID: DID?` `@State`, supplied `onAuthorTap` to both views, and registered the `ProfileScreen` destination.
2. **Post-card tap region** (this commit): the avatar already had its own `onTapGesture` calling `actions?.onAuthorTap`, but the author display name and handle lived inside the post-content `VStack` whose tap gesture fires `actions?.onTap` (open thread). Tapping the name therefore opened the thread instead of the profile, contradicting the issue's expected behavior of "tapping the avatar or author name".

## Fix

In `PostCard.authorHeader`, wrap the display name + handle in their own `HStack` with `.contentShape(Rectangle())` and `.onTapGesture { actions?.onAuthorTap?(item.post.author) }`. The relative timestamp and trailing spacer remain inside the surrounding content tap region so tapping to the right of the name still opens the thread, preserving the existing post-tap UX.

## Files changed

- `BlueskyKit/Sources/BlueskyUI/PostCard.swift` — author name/handle now have their own author-tap region (commit `7c195f7`).
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` — `feedProfileDID` state, `onAuthorTap` wiring, and `navigationDestination` for `ProfileScreen` (already in commit `e020951`).

## Gotchas

- `navigationDestination(item:)` requires the bound type to be `Hashable` (not `Identifiable`); `DID` already conforms.
- The avatar's tap gesture must remain attached directly to `avatarColumn` rather than letting it inherit the surrounding content tap, since the avatar is in the sibling `HStack` outside the content `VStack`.
- The author tap region uses `contentShape(Rectangle())` so the gap between display name and handle is also tappable.
