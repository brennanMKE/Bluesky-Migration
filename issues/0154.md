## Description

After #0152 fixed the outer-item shape (`subject.uri` instead of flat `uri`) the Bookmarks screen still fails to load with a different decoding error:

> Response decoding failed: DecodingError.keyNotFound: Key 'cid' not found in keyed decoding container. Path: `bookmarks[38].item`. Debug description: No value associated with key CodingKeys(stringValue: "cid", intValue: nil) ("cid").

This is *not* a re-occurrence of #0152. The path is now `bookmarks[38].item` — the embedded `item` field on the 38th bookmark is missing `cid`. #0152 modeled `item` as `PostView?`, but in the AT Proto lexicon `app.bsky.bookmark.defs#bookmarkView.item` is a **discriminated union** keyed on `$type`:

- `app.bsky.feed.defs#postView` — a normal post (has `uri` + `cid`).
- `app.bsky.feed.defs#notFoundPost` — `{ uri, notFound: true }` — **no `cid`**.
- `app.bsky.feed.defs#blockedPost` — `{ uri, blocked: true, author }` — **no `cid`**.

Bookmark 38's `item` is one of the latter two variants (deleted or blocked post), and the current `PostView` decoder rejects it.

## Steps to reproduce

1. Sign in.
2. Open the Saved / Bookmarks screen.
3. Observe the alert when the page that contains a bookmark for a now-deleted or blocked post is decoded.

## Expected behavior

`BookmarkView.item` decodes as a discriminated union, with each variant rendering appropriately:

- `postView` → existing `PostCard` rendering.
- `notFoundPost` → a tombstone row ("This post no longer exists" or similar).
- `blockedPost` → a tombstone row ("This post is from a blocked account").

A bookmark whose post is deleted or blocked must not break the entire list — RN gracefully renders a tombstone for those cases.

## Actual behavior

`item: PostView?` decoder hits the first non-`postView` variant in the page and aborts the entire response decode. No bookmarks render past the alert.

## Notes

- Cross-reference the `ThreadViewPost` discriminated enum in `BlueskyCore/Feed.swift` — that's the existing pattern for `app.bsky.feed.defs#threadViewPost` and its sibling `notFoundPost` / `blockedPost` cases. Apply the same shape to a new `BookmarkItemView` enum in `BlueskyCore/Bookmark.swift`.
- The Codable should switch on `$type` and decode to one of:
  - `.post(PostView)` — the happy path.
  - `.notFound(NotFoundPost)` — `{ uri, notFound }`.
  - `.blocked(BlockedPost)` — `{ uri, blocked, author }`.
  - `.unknown` (or `.unsupported`) — defensive fallback for new variants the server adds.
- `BookmarksScreen` needs to switch on the variant when rendering: `PostCard` for `.post`, a small tombstone view for the others. RN's `Bookmarks/index.tsx` does the same.
- Add a Codable round-trip test for each variant (mirrors the test added for #0152). Make sure a page containing a mix of post / notFound / blocked still decodes end-to-end.
- Defensive idea: even after fixing the union, wrap `getBookmarks` decoding in a per-item fallback so a single malformed item doesn't break the entire page. RN does this client-side. **Pragmatic**: ship the union fix first; per-item defensive decode is a follow-up.
- Cross-cuts #0057 / #0060 (Saved Posts UX work).

## Root cause

`BookmarkView.item` was modeled as `PostView?` after #0152. The AT Proto
lexicon defines `app.bsky.bookmark.defs#bookmarkView.item` as an open
union keyed on `$type`:

- `app.bsky.feed.defs#postView`     — full hydrated post (has `cid`).
- `app.bsky.feed.defs#notFoundPost` — `{uri, notFound}` — **no `cid`**.
- `app.bsky.feed.defs#blockedPost`  — `{uri, blocked, author}` — **no `cid`**.

Bookmark 38 in the failing page was a `notFoundPost` (post deleted by
its author). The plain `PostView` decoder rejected the missing `cid`
and the entire response decode aborted — no bookmarks rendered.

Confirmed against the React Native client in
`Bluesky-ReactNative/src/screens/Bookmarks/index.tsx`, which switches
on `AppBskyFeedDefs.isNotFoundPost` / `AppBskyFeedDefs.isPostView`
when assembling its list items and renders a `BookmarkNotFound`
tombstone for the not-found case ("This post was deleted by its
author").

## Fix

In `BlueskyKit/Sources/BlueskyCore/Bookmark.swift`:

- New `BookmarkItemView` open-union enum with `.post(PostView)`,
  `.notFound(NotFoundPost)`, `.blocked(BlockedPost)`, and
  `.unknown(String)`. The Codable reads `$type` and dispatches to the
  matching inner decoder, mirroring the `ThreadViewPost` pattern in
  `BlueskyCore/Feed.swift`.
- New top-level `NotFoundPost` / `BlockedPost` types — they were
  previously inline within `ThreadViewPost`'s decoder (`NF` / `BL`
  private structs). Lifting them out lets `BookmarkItemView` and
  future open-union sites reuse them.
- `BookmarkView.item` becomes `BookmarkItemView?` (still optional —
  the lexicon allows the field to be omitted entirely).
- `GetBookmarksResponse` gained a custom `init(from:)` that decodes
  the `bookmarks` array element-by-element and skips any element whose
  outer envelope fails to decode. This is in addition to the inner
  open-union; together they guarantee a single bad row can never break
  the whole page. Mirrors RN's tolerant client-side handling.

In `BlueskyKit/Sources/BlueskyFeed/BookmarksScreen.swift`:

- `bookmarkList` now calls a `row(for:)` builder that switches on
  `bookmark.item`. The `.post` case renders the existing `PostCard`;
  `.notFound` / `.blocked` / `.unknown` render a compact tombstone row
  with the matching copy and a Remove button. `nil` skips the row.
- Tombstone copy mirrors RN: "This post was deleted by its author"
  for `.notFound`; for `.blocked` and `.unknown` we use generic
  Bluesky-side strings ("This post is from a blocked account" /
  "Unsupported bookmark") since RN doesn't render those variants in
  the bookmarks list at all.

Tests added in `LexiconCodableTests.BookmarkCodableTests`:

- `bookmarkView.item` decodes the `postView` variant.
- `bookmarkView.item` decodes the `notFoundPost` variant (no `cid`).
- `bookmarkView.item` decodes the `blockedPost` variant (no `cid`).
- Unknown `$type` lands as `.unknown` rather than throwing.
- A mixed page (post + notFound + blocked + unknown) decodes
  end-to-end and each item lands in the right case.
- A malformed bookmark envelope (missing `subject`) is skipped, while
  surrounding well-formed entries still decode.

## Gotchas

- `Feed.swift`'s `ThreadViewPost` already handled the
  `notFoundPost` / `blockedPost` variants but with **inline** anonymous
  structs (`NF` / `BL`) inside its decoder — there were no top-level
  `NotFoundPost` / `BlockedPost` types to reuse. Lifting them out is a
  small enhancement; `ThreadViewPost` continues to work with its
  `case notFound(uri:)` / `case blocked(uri:)` shape and was not
  changed in this commit.
- Per-item defensive decode of the `bookmarks` array is included in
  the same change (rather than deferred). It's small and pairs
  naturally with the open-union fix — a future fifth variant on the
  outer bookmark envelope would otherwise re-break the entire page.
- Pre-existing test compile failure noted in past issues' Gotchas
  (`BlueskyKitTests.swift:90`) blocks `swift test`. The new
  `BookmarkCodableTests` cases follow the same pattern as the
  surrounding `@Suite` cases and were verified by inspection plus a
  full `swift build`.
