# 0154 — Bookmarks decoding still fails: `bookmarks[38].item` is a discriminated union, not a plain PostView

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskyCore / BlueskyFeed |
| **Platform** | All |
| **First seen** | 2026-05-07 |

## Description

After #0152 fixed the outer-item shape (`subject.uri` instead of flat `uri`) the Bookmarks screen still fails to load with a different decoding error:

> Response decoding failed: DecodingError.keyNotFound: Key 'cid' not found in keyed decoding container. Path: `bookmarks[38].item`. Debug description: No value associated with key CodingKeys(stringValue: "cid", intValue: nil) ("cid").

This is *not* a re-occurrence of #0152. The path is now `bookmarks[38].item` — the embedded `item` field on the 38th bookmark is missing `cid`. #0152 modeled `item` as `PostView?`, but in the AT Proto lexicon `app.bsky.bookmark.defs#bookmarkView.item` is a **discriminated union** keyed on `$type`:

- `app.bsky.feed.defs#postView` — a normal post (has `uri` + `cid`).
- `app.bsky.feed.defs#notFoundPost` — `{ uri, notFound: true }` — **no `cid`**.
- `app.bsky.feed.defs#blockedPost` — `{ uri, blocked: true, author }` — **no `cid`**.

Bookmark 38's `item` is one of the latter two variants (deleted or blocked post), and the current `PostView` decoder rejects it.

![Bookmarks fatal decoding error: cid not found at bookmarks[38].item](error.png)

## Steps to reproduce

1. Sign in.
2. Open the Saved / Bookmarks screen.
3. Observe the alert when the page that contains a bookmark for a now-deleted or blocked post is decoded.

## Expected behavior

`BookmarkView.item` decodes as a discriminated union, with each variant rendering appropriately:

- `postView` → existing `PostCard` rendering.
- `notFoundPost` → a tombstone row ("This post no longer exists" or similar).
- `blockedPost` → a tombstone row ("This post is from a blocked account").

A bookmark whose post is deleted or blocked must not break the entire list — RN gracefully renders a tombstone for those cases.

## Actual behavior

`item: PostView?` decoder hits the first non-`postView` variant in the page and aborts the entire response decode. No bookmarks render past the alert.

## Notes

- Cross-reference the `ThreadViewPost` discriminated enum in `BlueskyCore/Feed.swift` — that's the existing pattern for `app.bsky.feed.defs#threadViewPost` and its sibling `notFoundPost` / `blockedPost` cases. Apply the same shape to a new `BookmarkItemView` enum in `BlueskyCore/Bookmark.swift`.
- The Codable should switch on `$type` and decode to one of:
  - `.post(PostView)` — the happy path.
  - `.notFound(NotFoundPost)` — `{ uri, notFound }`.
  - `.blocked(BlockedPost)` — `{ uri, blocked, author }`.
  - `.unknown` (or `.unsupported`) — defensive fallback for new variants the server adds.
- `BookmarksScreen` needs to switch on the variant when rendering: `PostCard` for `.post`, a small tombstone view for the others. RN's `Bookmarks/index.tsx` does the same.
- Add a Codable round-trip test for each variant (mirrors the test added for #0152). Make sure a page containing a mix of post / notFound / blocked still decodes end-to-end.
- Defensive idea: even after fixing the union, wrap `getBookmarks` decoding in a per-item fallback so a single malformed item doesn't break the entire page. RN does this client-side. **Pragmatic**: ship the union fix first; per-item defensive decode is a follow-up.
- Cross-cuts #0057 / #0060 (Saved Posts UX work).
