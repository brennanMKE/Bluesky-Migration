# 0062 ‚Äî Tapping a notification shows raw at:// URI text instead of navigating to the thread

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskyNotifications |
| **Platform** | All |
| **First seen** | 2026-05-05 |

## Description

Tapping a notification row in the Notifications tab should navigate to the relevant destination ‚Äî the thread for likes/replies/quotes/mentions/reposts, the actor's profile for follows, the feed for `feedgen-like` / `starterpack-joined`, and the activity view for `subscribed-post`. Today the destination is a placeholder that just renders the raw AT-URI as text (`Thread: at://did:plc:‚Ä¶/app.bsky.feed.post/‚Ä¶`), so the user has no working navigation from notifications.

## Steps to reproduce

1. Open the Notifications tab.
2. Tap any notification row (like, reply, repost, follow, etc.).
3. Observe the destination view.

## Expected behavior

For each notification reason, navigation goes to the right destination:

| Reason | Destination |
|---|---|
| `like`, `repost`, `like-via-repost`, `repost-via-repost` | Thread for `subjectUri` |
| `reply`, `mention`, `quote` | Thread for the notification's own `uri` |
| `follow`, `contact-match`, `verified`, `unverified` | Profile of the notification's author |
| `feedgen-like`, `starterpack-joined` | Feed/starter pack at `subjectUri` |
| `subscribed-post` | Activity view listing the matching posts |

(Mapping mirrored from React Native: `Bluesky-ReactNative/src/view/com/notifications/NotificationFeedItem.tsx`, around line 129.)

## Actual behavior

`NotificationsScreen.swift:48` renders the destination as:

```swift
Text("Thread: \(uri.rawValue)").navigationTitle("Thread")
```

This is a placeholder. The real `ThreadView` is never pushed, and follows / feed-likes / activity destinations have no routing at all.

## Steps to fix

1. Replace the placeholder `Text("Thread: ‚Ä¶")` in `NotificationsScreen.navigationDestination` with the real `ThreadView` (push the post by `subjectUri` or notification `uri` depending on reason).
2. For `follow` / `contact-match` / `verified` / `unverified`, push `ProfileScreen` for the notification's author DID.
3. For `feedgen-like` / `starterpack-joined`, push the feed view for `subjectUri`.
4. For `subscribed-post`, push the activity-style screen (lists multiple subscribed posts at once). RN passes a query string with comma-separated post URIs ‚Äî match that grouping.
5. Update `GroupedNotificationRow.onTap` to surface the full notification (not just `subjectUri`) so the screen can pick the right destination by reason.

### 2026-05-12 ‚Äî actual code landed (`#0160`'s referenced commits were never in either repo's git history)

The `f5cc464d` / `b78828af` SHAs in the prior note do not exist in
the `BlueskyKit` or `Bluesky-SwiftUI` repos respectively
(`git log --oneline --all` returns nothing for them). At pickup time
on 2026-05-12, `NotificationsScreen.body` still rendered the
`Text("Thread: \(uri.rawValue)")` placeholder verbatim and
`MainTabView`'s `case .notifications:` was still a bare
`NotificationsScreen(network:onUnreadCountChange:)` call with no
`onPostTap` wiring. So the work attributed to #0160 had never
actually shipped.

This session implemented the fix directly:

- **BlueskyKit `ff0cfd7`** (`Sources/BlueskyNotifications/NotificationsScreen.swift`)
  ‚Äî added `onPostTap: ((ATURI) -> Void)?` and `onAuthorTap: ((DID) -> Void)?`
  init parameters. New `handleRowTap(_:)` switch dispatches:
  - `follow` / `contact-match` / `verified` / `unverified` ‚Üí
    `onAuthorTap(group.actors.first?.did)` (previously these rows
    were untappable because `reasonSubject` is nil).
  - All other reasons (thread-style) ‚Üí `onPostTap(group.reasonSubject)`.
  - Internal `navigationDestination` placeholder is kept but only
    activates when `onPostTap == nil`, so the previews and any
    unwired call site degrade gracefully.
- **Bluesky-SwiftUI `f870f23`** (`Bluesky-SwiftUI/MainTabView.swift`)
  ‚Äî `case .notifications:` passes both callbacks
  (`onPostTap: { uri in threadURI = uri }`,
  `onAuthorTap: { did in feedProfileDID = did }`) into
  `NotificationsScreen` and attaches `navigationDestination(item: $threadURI)`
  mounting the real `ThreadView` plus
  `navigationDestination(item: $feedProfileDID)` mounting
  `ProfileScreen`. Reuses the same `@State` slots the Home tab uses;
  `resetTransientNavState()` already clears them on tab switch.

Reasons still left without a dedicated destination (deferred ‚Äî
the fallback below is acceptable but not ideal):

- `feedgen-like` / `starterpack-joined` ‚Äî `reasonSubject` is a feed
  or starter pack URI, not a post URI. Routed through `onPostTap`
  which currently mounts `ThreadView` for that URI ‚Äî `ThreadView`
  will fail to render a feed URI, so the user will see an empty
  thread / error. A dedicated `FeedScreen` / `StarterPackScreen`
  destination is the right fix; tracked here for follow-up.
- `subscribed-post` ‚Äî `reasonSubject` is the post URI in the
  current `GroupedNotification` shape, so it routes via `onPostTap`
  to `ThreadView` ‚Äî that's the *right* destination for a single
  subscribed post. The RN client renders a multi-post "activity
  view" when multiple subscribed posts arrive in the same window;
  that grouping shape isn't in `GroupedNotification` today, so a
  full activity-view fix needs a viewmodel change first.

Status stays `open` per the prior note's explicit "awaiting user
confirmation" pending the user verifying that tapping notifications
now opens real threads / profiles on a running build.
