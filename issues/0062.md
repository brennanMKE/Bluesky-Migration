# 0062 — Tapping a notification shows raw at:// URI text instead of navigating to the thread

| | |
|---|---|
| **Status** | open |
| **Module** | BlueskyNotifications |
| **Platform** | All |
| **First seen** | 2026-05-05 |

## Description

Tapping a notification row in the Notifications tab should navigate to the relevant destination — the thread for likes/replies/quotes/mentions/reposts, the actor's profile for follows, the feed for `feedgen-like` / `starterpack-joined`, and the activity view for `subscribed-post`. Today the destination is a placeholder that just renders the raw AT-URI as text (`Thread: at://did:plc:…/app.bsky.feed.post/…`), so the user has no working navigation from notifications.

## Steps to reproduce

1. Open the Notifications tab.
2. Tap any notification row (like, reply, repost, follow, etc.).
3. Observe the destination view.

## Expected behavior

For each notification reason, navigation goes to the right destination:

| Reason | Destination |
|---|---|
| `like`, `repost`, `like-via-repost`, `repost-via-repost` | Thread for `subjectUri` |
| `reply`, `mention`, `quote` | Thread for the notification's own `uri` |
| `follow`, `contact-match`, `verified`, `unverified` | Profile of the notification's author |
| `feedgen-like`, `starterpack-joined` | Feed/starter pack at `subjectUri` |
| `subscribed-post` | Activity view listing the matching posts |

(Mapping mirrored from React Native: `Bluesky-ReactNative/src/view/com/notifications/NotificationFeedItem.tsx`, around line 129.)

## Actual behavior

`NotificationsScreen.swift:48` renders the destination as:

```swift
Text("Thread: \(uri.rawValue)").navigationTitle("Thread")
```

This is a placeholder. The real `ThreadView` is never pushed, and follows / feed-likes / activity destinations have no routing at all.

## Steps to fix

1. Replace the placeholder `Text("Thread: …")` in `NotificationsScreen.navigationDestination` with the real `ThreadView` (push the post by `subjectUri` or notification `uri` depending on reason).
2. For `follow` / `contact-match` / `verified` / `unverified`, push `ProfileScreen` for the notification's author DID.
3. For `feedgen-like` / `starterpack-joined`, push the feed view for `subjectUri`.
4. For `subscribed-post`, push the activity-style screen (lists multiple subscribed posts at once). RN passes a query string with comma-separated post URIs — match that grouping.
5. Update `GroupedNotificationRow.onTap` to surface the full notification (not just `subjectUri`) so the screen can pick the right destination by reason.

## Notes

- Code locations in BlueskyKit:
  - `Sources/BlueskyNotifications/NotificationsScreen.swift:43-50` — placeholder navigation destination.
  - `Sources/BlueskyNotifications/NotificationsScreen.swift:59-61` — `onTap` handler captures `subjectUri`.
  - `Sources/BlueskyNotifications/NotificationsScreen.swift:106-108` — row-level tap dispatch.
  - `Sources/BlueskyNotifications/NotificationsViewModel.swift` — `GroupedNotification` aggregates by `(reason, subject)`; needs to expose enough to drive the destination switch.
- Consider extracting the destination decision into a small helper (`func destination(for: GroupedNotification) -> Route`) to keep `body` clean.
- Blocks #0031 (Module 10 gate) — list rendering already works, but a clean gate close needs working tap navigation.
