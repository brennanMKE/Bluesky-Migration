## Description

When multiple actors trigger the same notification group (e.g. five different people liked the same post), the React Native client displays up to 5 stacked actor avatars in the row, with a chevron (`v`) to expand the rest. The SwiftUI client currently shows only the first actor's avatar ā€” losing the social context that "Point-Free **and 3 others** subscribed to this post" is meant to convey.

This is a follow-up to the iOS Notifications parity audit in #0069.

## Scope

### Avatar stack

- **Render up to 5 actor avatars** stacked horizontally in a row to the right of the bell icon.
- **Avatars overlap** by ~30% ā€” leading edge of each subsequent avatar covers the trailing edge of the previous one (matches RN visual style).
- **Each avatar is tappable** ā€” taps open that actor's profile via the standard navigation path.
- **Avatar sizes**: ~28pt diameter; adjust to feel right against the bell icon.
- **Order**: most-recent-first (matches the order of `actors` in the `GroupedNotification`).

### Expand chevron

- **Chevron button** (`chevron.down`) appears at the end of the avatar stack when the group has more than 5 actors.
- **Tap expands the row** to show all actors as a vertically-stacked list with full names.
- **Tapping the chevron again collapses** back to the 5-avatar stack.
- **Animate the expand/collapse** with a smooth height transition.

### Group label

- Update the group label to "**{first actor}** and **N others**" when the group has more than one actor ā€” matches RN. The current SwiftUI label only shows the first actor.

## Implementation notes

- Code lives in `BlueskyKit/Sources/BlueskyNotifications/NotificationsScreen.swift` ā€” `GroupedNotificationRow`.
- The `GroupedNotification` view-model already aggregates `actors: [ProfileBasic]`. The store doesn't need changes ā€” only the row view.
- For overlap, use `ZStack` + `.offset` per avatar (or `HStack` with negative spacing).
- The chevron's visibility binds to `actors.count > 5`.

## Acceptance

- Notification rows with multiple actors show overlapping stacked avatars (up to 5).
- Tapping any avatar opens that actor's profile.
- A chevron button appears when there are more than 5 actors; tapping it expands to a full list.
- The group label uses the "X and N others" form when applicable.
- iOS Simulator and macOS builds pass.

## Root cause

`GroupedNotificationRow` already received the full `actors` array on the
view-model (each grouped notification aggregates `[ProfileBasic]` ordered
most-recent-first), but the row only ever rendered the first three
avatars and there was no tap handler on any of them. Rows with five or
more actors had no affordance to see the rest, and even the first three
avatars couldn't navigate to those actors' profiles ā€” the social
context "Point-Free **and 3 others** subscribed to this post" the RN
client conveys at a glance was lost.

## Fix

Replaced the fixed three-avatar stack with up to five overlapping
avatar buttons, an optional chevron-down expand button at the trailing
edge of the stack, and an inline expanded actor list:

- **Avatar stack** ā€” `actorAvatarRow` renders `min(actors.count, 5)`
  avatars in an `HStack(spacing: -8)` (each avatar is 28pt, so -8pt
  spacing = ~28% overlap, matching the RN reference). Each avatar is
  wrapped in a `Button` whose action fires the row's new
  `onAuthorTap(ProfileBasic)` closure. The avatars retain their
  existing 1.5pt background-colored ring so the overlap reads as
  discrete circles in both light and dark mode.

- **Expand chevron** ā€” when `actors.count > 5`, a 28pt circular button
  with a `chevron.down` glyph (rotating 180Ā° when open) sits at the
  end of the stack. Tapping it toggles a row-local
  `@State isExpanded`. The overall row is animated with
  `.animation(.smooth, value: isExpanded)` so the height change reads
  as a smooth slide-down, and the expanded list itself uses an
  `opacity + move(.top)` transition for a subtle fade-in.

- **Expanded actor list** ā€” `expandedActorList` renders every actor as
  a vertically-stacked entry (28pt avatar + display name + handle),
  each entry a `Button` that also routes through `onAuthorTap`. Indented
  36pt to align under the avatar stack so it reads as a continuation of
  the row, not a sibling.

- **Group label** ā€” `actorSummary` now reads "{first} and N others" for
  any multi-actor group (with singular `other` when N == 1) instead of
  enumerating two names plus an overflow count. Single-actor groups
  keep the existing display-name-or-`@handle` label.

Wired the new callback through `NotificationsScreen.init`, threaded it
into the row via the same closure-based pattern `FeedView` and
`ThreadView` already use, and updated `MainTabView.tabContent(.notifications)`
to bind it to `feedProfileDID = profile.did` and attach a
`.navigationDestination(item: $feedProfileDID)` that pushes
`ProfileScreen` ā€” same shared state Home / Saved / Search use, so
back-navigation behaves identically across tabs.

## Files changed

- `BlueskyKit/Sources/BlueskyNotifications/NotificationsScreen.swift` ā€”
  added `onAuthorTap: ((ProfileBasic) -> Void)?` to `NotificationsScreen`
  with a default-`nil` initializer parameter; threaded it into
  `GroupedNotificationRow`. Replaced the row's `Button`-wrapped HStack
  with a plain VStack + `contentShape` + `onTapGesture` because the row
  now contains multiple independent tap targets (post body, each
  avatar, chevron, each expanded actor) ā€” wrapping everything in a
  single Button would consume the inner taps. Replaced
  `actorAvatarStack` with `actorAvatarRow` (up to 5 avatars + chevron),
  added `expandedActorList` with `@State isExpanded`, and rewrote
  `actorSummary` to use the "{first} and N others" form. Added
  `.animation(.smooth, value: isExpanded)` for the height transition.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` ā€” passed
  `onAuthorTap: { profile in feedProfileDID = profile.did }` to
  `NotificationsScreen` and attached a
  `.navigationDestination(item: $feedProfileDID)` to the
  `.notifications` case that pushes `ProfileScreen`.

## Gotchas

- **Row container had to drop the outer `Button`.** The row used to be
  one big `Button { onTap(...) }` wrapping the entire HStack. With the
  new tap targets (avatars, chevron, expanded actor rows) nested
  inside, an outer `Button` would have intercepted every inner tap and
  broken everything. Switched to `VStack` + `.contentShape(Rectangle())`
  + `.onTapGesture` for the row's own tap-to-thread behavior ā€” the post
  body / icon / reason text region still navigates, while inner buttons
  consume their own taps. This is the same pattern `PostCard` uses for
  its overlapping author / body / action regions.
- **Profile navigation is wired via callback, not module-internal
  navigation.** `NotificationsScreen` lives in `BlueskyNotifications`
  (Layer 3), `ProfileScreen` lives in `BlueskyProfile` (Layer 3) ā€” peers,
  not parent/child. Following the established pattern, the screen
  exposes an `onAuthorTap` closure and the app shell (`MainTabView`)
  decides what to push. This keeps `BlueskyNotifications` from gaining a
  cross-feature dependency on `BlueskyProfile`.
- **Animation lives at the row level, not on each child.** Putting
  `.animation(.smooth, value: isExpanded)` on the outer container
  animates both the height change and the `transition` on the
  expanded list cleanly. Earlier attempts that scoped the animation to
  the chevron rotation only ā€” leaving the list to use SwiftUI's
  default ā€” produced a visibly-jumpy expand because the row resized
  before the list faded in.
- **Chevron stack-position deliberately overlaps the last avatar.** It
  uses the same negative-spacing HStack with a 4pt leading pad so it
  reads as an extension of the avatar row rather than a separate
  control. The 28pt frame and background-colored ring match the avatar
  visual weight; the rotation animation makes the open/closed state
  obvious without changing the icon.
