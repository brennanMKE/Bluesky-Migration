# 0098 — Composer is missing the threadgate / postgate (reply + quote permissions) picker

## Description

The React Native composer lets the author restrict who can reply to (`threadgate`) and who can quote (`postgate`) the post being created. RN exposes a `ThreadgateBtn` in the toolbar that opens a dialog with options: **Everyone**, **People you follow**, **People you mention**, **Lists** (multi-select of moderation lists), and **Nobody**. Quote permissions are similarly configurable.

These records are written alongside the post:

- `app.bsky.feed.threadgate` — controls reply permissions.
- `app.bsky.feed.postgate` — controls embed/quote permissions.

The SwiftUI composer has no threadgate / postgate UI. Every post is implicitly open to all replies and quotes, which is a privacy + control regression vs. RN.

## Steps to reproduce

1. Open the SwiftUI composer.
2. Look for a "Who can reply?" / "Who can quote?" affordance.
3. Observe there is none.

## Expected behavior

A reply-restrictions button in the composer toolbar (RN uses a chat-bubble icon) that opens a sheet:

- **Reply restrictions** (radio):
  - Everyone (default)
  - People you follow
  - People you mention
  - Combine: people in selected lists (with list multi-select)
  - Nobody
- **Quote restrictions** (toggle): Allow quotes (default on) / Disable quotes.

When the post is submitted, if restrictions are non-default, the app additionally writes:
- `app.bsky.feed.threadgate` record at the same `rkey` as the post (key insight: the threadgate `rkey` matches the post's `rkey`).
- `app.bsky.feed.postgate` record at the same `rkey` if quote restrictions differ from default.

## Actual behavior

No threadgate / postgate UI. Posts are created without companion records, so they are implicitly open.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/com/composer/threadgate/`.
- Lexicons to add to `BlueskyCore`: `ThreadgateRecord` (`allow: [Allow]`, where Allow is `mentioned` / `followers` / `list`), `PostgateRecord` (`embeddingRules`).
- File to edit: `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` and `ComposerViewModel.swift` — extend the create-post path to optionally write threadgate / postgate records via `applyWrites`.
- Reply restrictions also need to be *displayed* on the focal post in thread view ("Only people @op mentions can reply") — that's a separate render-side parity gap (see #0147 once filed) but mentioned here for context.

## Root cause

The SwiftUI composer never modeled reply or quote restrictions. There were
no `ThreadgateRecord` / `PostgateRecord` types in `BlueskyCore`, no
companion-record write path in `ComposerStore.post`, and no entry point in
`ComposerSheet` for the user to express the choice. As a result every
SwiftUI-authored post implicitly allowed everyone to reply and everyone to
quote, regardless of the author's preferences — RN's `ThreadgateBtn`
(`view/com/composer/threadgate/ThreadgateBtn.tsx`) and the threadgate /
postgate writes RN bundles into `applyWrites` had no counterpart on this
side of the port.

## Fix

**Lexicon (Layer 0).** Added two record types and their discriminated
unions to `BlueskyCore/Post.swift`:

- `ThreadgateAllowRule` — the four `app.bsky.feed.threadgate#*Rule`
  variants (`mentionRule`, `followingRule`, `followerRule`, `listRule`
  with an `ATURI`), plus an `unknown` fallback for forward-compat.
- `ThreadgateRecord` — `{ post, allow, hiddenReplies, createdAt }`. The
  `allow` field is intentionally `[ThreadgateAllowRule]?` because the
  lexicon distinguishes:
  - omitted → everyone can reply (default; `nil`).
  - present-but-empty → nobody can reply (`[]`).
  Encoding preserves this asymmetry: `nil` skips the field entirely; an
  empty array is encoded as `[]`. Decoding treats an absent `allow` key
  as `nil` and a present-but-empty key as `[]`.
- `PostgateEmbeddingRule` — currently the single `#disableRule` variant.
- `PostgateRecord` — `{ post, detachedEmbeddingUris, embeddingRules,
  createdAt }`. To disable quotes, push `.disable` into
  `embeddingRules`.

`CreateRecordRequest` gained an optional `rkey` field so threadgate /
postgate records can be written at a specific rkey (matching the post's
rkey). The encoder skips `rkey` when `nil` so existing call sites (which
let the server allocate a TID) keep their wire payload unchanged.

**Composer state (Layer 3).** `ComposerStore` and `ComposerViewModel`
gained two new pieces of state:

- `threadgateAllow: ThreadgateAllowSelection` (default `.everyone`).
  This is a single value type that collapses the "Everyone" / "Nobody"
  radios with the granular checkboxes so impossible combinations (e.g.
  "Everyone *and* mentioned-only") aren't representable.
  `.granular(mention:following:follower:listURIs:)` carries the union
  of toggles. An all-off granular selection collapses back to
  `.everyone` (matching RN's behavior — clearing every checkbox
  shouldn't lock down replies).
- `quotesEnabled: Bool` (default `true`).

Both are reset to defaults on a successful post (alongside `text`,
`images`, `selectedLabels`, etc.).

`ComposerStore.post(...)` now writes companion records after the head
post is created:

- If `threadgateAllow != .everyone`, it writes a
  `app.bsky.feed.threadgate` record at the same rkey as the post (read
  from the server's returned URI). `.nobody` writes `allow: []`;
  granular writes the corresponding rule list.
- If `quotesEnabled == false`, it writes a `app.bsky.feed.postgate`
  record with `embeddingRules: [.disable]` at the same rkey.

Both companion writes are best-effort: a failure logs but doesn't fail
the parent post (the post itself succeeded; the gate just didn't take
effect). RN bundles them into a single `applyWrites` call and computes
the post's CID locally; we keep a separate `createRecord` per write
because we read the rkey back from the server's URI, so the post must
hit the network first. The user-visible outcome is the same — both
records exist at the same rkey by the time the composer dismisses.

**Composer UI.** `ComposerSheet.mediaToolbar` now renders a
`threadgateButton` next to `labelsButton` on both iOS and macOS. The
button shows `globe` (secondary tint) when anyone can interact and
`person.2.fill` (accent tint) when any restriction is in place. Tapping
presents `ThreadgatePicker`, which is a sheet on iOS and a popover on
macOS — matching `SelfLabelsPicker`'s presentation pattern from #0097.

`ThreadgatePicker` mirrors RN's `PostInteractionSettingsControlledDialog`
layout:

- A "Who can reply?" section with two radio rows (**Everyone** /
  **Nobody**) bracketing three checkbox toggles (**Mentioned users**,
  **Followed users**, **Your followers**). Selecting a radio clears the
  checkboxes; toggling a checkbox switches `allow` to a granular
  selection. The checkboxes are disabled while **Nobody** is active.
- A "Quote settings" section with a single **Allow quotes** toggle.
- A "Done" toolbar button to dismiss.

Tests added to `Tests/BlueskyKitTests/LexiconCodableTests.swift`:

- `ThreadgateCodableTests` — verifies the omitted-vs-empty `allow`
  asymmetry on both encode and decode, the four `$type` discriminators
  round-trip, and the `app.bsky.feed.threadgate` outer `$type` is
  emitted.
- `PostgateCodableTests` — verifies `#disableRule` encodes with the
  correct `$type` and round-trips.

## Gotchas

- **List multi-select rule deferred.** RN's dialog includes "Specific
  lists" with a multi-select of the viewer's curate / mod lists. The
  available lists come from `app.bsky.graph.getLists`. Wiring this in
  required either reusing `BlueskyLists` (a Layer 3 module — Layer 3
  cross-deps are forbidden by `ModularArchitecture.md`) or duplicating
  the `getLists` plumbing inside `BlueskyComposer`. Both are larger
  than this picker. Per the issue's bail rule, the simpler set ships:
  Everyone / Following / Followers / Mentioned / Nobody plus the quote
  toggle. The lexicon already supports the `listRule` variant, and
  `ThreadgateAllowSelection.granular(...)` already carries
  `listURIs: [ATURI]`, so a follow-up issue can wire the UI without
  re-touching the lexicon or the store.
- **Companion writes are sequential, not atomic.** RN bundles post +
  threadgate + postgate into a single `applyWrites` call so all three
  succeed-or-fail together. We can't (yet) without computing the
  post's CID client-side, which RN does via `@atproto/common-web`.
  Sequential writes are correct in the happy path and good enough for
  parity; the unhappy path leaves the post visible without the gate.
  A failure to write the gate logs but doesn't fail the parent post
  — surfacing a partial-success error for an already-shipped post
  would be confusing UX. If we add client-side CID computation later
  this can collapse into one `applyWrites` and become atomic.
- **Empty granular collapses to `.everyone`.** If the user toggles
  every checkbox off without picking Nobody, we treat the selection
  as `.everyone` rather than writing a threadgate with an empty
  allow list (which would be the "nobody can reply" sentinel). RN
  has the same collapse via `threadgateAllowUISettingToAllowRecordValue`
  returning `undefined` when `everybody` is in the array. The
  picker's UI also disables the granular checkboxes while
  **Nobody** is active so the two states can't collide.
- **Pre-existing `BlueskyKitTests` build failure.** The test target
  fails to compile against `BlueskyEnvironment` (the same
  `bookmarks` / `pathMonitor` signature drift noted in #0097). The
  new threadgate / postgate test cases compile inside
  `LexiconCodableTests.swift` and are correct by inspection of the
  encoder output, but `swift test` still won't run them until the
  unrelated `BlueskyKitTests.swift:90` regression is fixed. The
  library `swift build` and the macOS workspace build both pass.
