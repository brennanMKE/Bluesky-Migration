## Description

Both 1:1 and group threads can include system events that look like messages but aren't authored by anyone: "@alice left the group", "@bob added @carol", "Conversation locked", "Group name changed to <new>", message-deleted tombstones, etc. React Native renders these as italicized centered text (no bubble, no avatar). The SwiftUI thread loop only handles `MessageView` and skips other variants of the discriminated union — system events don't render at all, breaking the narrative of the conversation.

## Steps to reproduce

1. Open a group thread where members have been added/removed since you last opened it.
2. Compare to the same thread in RN.
3. Observe SwiftUI silently drops the system events while RN shows an italic line for each.

## Expected behavior

- The thread iterator handles all variants of `chat.bsky.convo.defs#message*` — `messageView`, `deletedMessageView`, and any group-events variants.
- For system events / tombstones: render as a centered italic small-text row in the secondary text color, no bubble, no avatar. Examples:
  - "Message deleted"
  - "@alice left the group"
  - "@bob added @carol"
  - "@op renamed the group to <new>"
- For unknown variants, render a generic "Unsupported message" placeholder rather than dropping silently.

## Actual behavior

Anything that isn't a `MessageView` is dropped. Group threads look incomplete.

## Notes

- RN reference: `Bluesky-ReactNative/src/components/dms/SystemMessageItem.tsx` and `getSystemMessageInfo.ts` for the variant dispatch and copy.
- Lexicon variants: actually shipped under `chat.bsky.convo.defs#systemMessageView` (the 2026-04-26 RN drift turned out to keep group events inside the existing `convo.defs` namespace rather than splitting them out under `chat.bsky.group.*`). The discriminated `data` union covers add/remove/join/leave member, lock/unlock/lockPermanently, editGroup (rename), and four invite-link variants.
- File to edit: `BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift` (the messages loop ~lines 74–77) — switch on the discriminated message union, dispatch to `MessageBubble` or `SystemMessageRow`.

## Root cause

The chat surface only modeled two variants of `chat.bsky.convo.defs#getMessages` output (`messageView`, `deletedMessageView`) and the in-thread store's `messages` array was typed as `[MessageView]`. Anything that wasn't a real authored message — tombstones plus the entire system-event family (members added/left, group rename, lock changes, invite-link mutations) — failed type discrimination during JSON decode and was silently dropped. There was no rendering path for tombstones or system events either: the `LazyVStack` loop hard-coded `MessageBubble` for every row.

## Fix

- `BlueskyCore/Chat.swift`: added `SystemMessageView { id, rev, sender, sentAt, data: SystemMessageData }`, the `SystemMessageReferredUser` actor reference, and the `SystemMessageData` discriminated union over the 12 variants RN ships in `getSystemMessageInfo.ts` (add/remove/join/leave member, lock/unlock/lockPermanently, editGroup, four invite-link mutations) plus an `.unknown` catch-all so future server additions render a placeholder rather than fail decode. Introduced `ConvoMessage` — the in-thread enum mirroring `ConvoLastMessage` with `.message`/`.deleted`/`.system` cases — and retyped `GetMessagesResponse.messages` to `[ConvoMessage]`. Extended `ConvoLastMessage` with the same `.system(_)` case and a `systemView` accessor so the conv-list row can preview system events as the most-recent activity.
- `MessageThreadStore` and `MessageThreadViewModel`: retyped `messages` to `[ConvoMessage]`. The store's mutators (`deleteMessage`, `addReaction`, `removeReaction`) drill into `.message(_)` cases via a shared `messageIndex(of:)` helper and no-op (return `false`) when the targeted ID belongs to a system or deleted row. `sendMessage` and `sendImageAttachment` wrap their server response in `.message(_)` before appending. `isOwn(_:)` now takes a `ConvoMessage` and only returns `true` for real-message rows the viewer authored — system events are never the viewer's "own" bubble even if they performed the action.
- `MessageThreadScreen`: replaced the bubble-only `LazyVStack` row with a `switch` over `ConvoMessage` — `.message` renders `MessageBubble`, `.deleted` renders a `SystemMessageRow(text: "Message deleted")`, and `.system` renders a `SystemMessageRow` with the localized line and matching SF Symbol. Added `SystemMessageRow` (centered, italic, secondary, small caption) modeled on RN's `SystemMessageItem.tsx`. Run-clustering helpers `isFirstInRun(at:)` / `isLastInRun(at:)` now treat any non-message neighbor as a run break, so the next bubble after a system event re-shows the sender header / avatar and the bubble before it carries its timestamp. Date-divider logic continues to fire across system rows because every variant carries `sentAt`. Added `systemMessageText(_:)` for line generation (mirroring RN's wording) and `systemMessageIcon(for:)` for the leading glyph (Join/Leave/Lock/Pencil/Link).
- `ConversationListScreen` and `MessageRequestsScreen`: the inbox-row preview line now handles `.system(_)` with the same wording the thread uses, so the row caption is informative when the most recent activity isn't an authored message.

## Gotchas

- RN's `getSystemMessageInfo.ts` (line 36+) names the lock event "Chat locked" / "Chat unlocked" with no actor, while the SwiftUI line reads "@actor locked the conversation". The actor-prefixed phrasing is more useful in the thread context (you almost always want to know who hit the button) and the issue spec called for "@op locked the conversation" wording, so we deliberately diverge from RN's no-actor copy here. Easy to swap to the actor-less form later if RN parity becomes important.
- `SystemMessageView.sender` carries the actor's DID; resolving to a handle goes through the convo member list. If the actor has since left the group their DID will not resolve and the line falls back to a truncated DID. RN avoids this by carrying a `relatedProfiles: Map<DID, ProfileViewBasic>` on each `ConvoItem` (snapshotted at the moment the event fired) — adopting that map is deferred until the firehose-driven `relatedProfiles` work lands.
- The full localization story (Lingui's `msg` macro in RN) is intentionally out of scope. Strings are English-only for now; the line is built as `AttributedString` so future localized formatting (e.g. handle styling, click targets) can drop in without changing the row layout.
- The `.unknown` `SystemMessageData` variant renders as "Unsupported event" so any future server-side variant the SwiftUI client doesn't know about still draws a row instead of failing decode and dropping the message — a strict improvement over the previous behavior.
