## Description

When the original poster has set a threadgate (composer issue #0098 covers writing one) restricting who can reply, the thread response from `getPostThread` flags certain replies as "hidden by threadgate". React Native renders these as either a tombstone ("This reply is hidden") or in a collapsed "Hidden replies" section below the visible replies, with a divider. The `ThreadView` in SwiftUI does neither — hidden replies either don't show or render alongside visible ones with no distinction.

There's also an OP-controlled mechanism to hide individual replies even without a threadgate; same UX applies.

## Steps to reproduce

1. Find a thread where the OP has hidden a specific reply (or has a restrictive threadgate).
2. Compare to RN.
3. Observe SwiftUI either shows the hidden reply mixed in or omits it silently.

## Expected behavior

- A reply marked `hidden` in the thread response is grouped under a "Hidden replies" divider at the bottom of the visible reply list.
- The divider is collapsed by default; tapping expands it to show the hidden replies as muted/dimmed `PostCard`s with a small badge ("Hidden by author").
- Replies hidden by a threadgate (e.g., user not in the allow rule) render as tombstones inline at their position: small text "This reply is hidden by the author".

The thread response provides the metadata (`threadContext.viewer.threadgateHidden` or similar). Confirm shape against the lexicon.

## Actual behavior

Hidden replies render mixed in or are dropped without any indicator.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/com/post-thread/ThreadItemPostTombstone.tsx` and `useMergedThreadgateHiddenReplies.ts`.
- Lexicon: `app.bsky.feed.defs#threadgateView` plus `#hiddenReplies` field on the threadgate record.
- File to edit: `BlueskyKit/Sources/BlueskyFeed/ThreadView.swift` and `ThreadViewModel.swift`. Possibly extend `ThreadPost` in `BlueskyCore` with a `hiddenByThreadgate: Bool` flag.

## Root cause

Two gaps. (1) `BlueskyCore.ThreadPost` did not decode the optional `threadgate: threadgateView` field that `app.bsky.feed.getPostThread` attaches to the focal `threadViewPost`. The threadgate landed on the wire but was discarded on decode, so the SwiftUI side had no visibility into the OP's `hiddenReplies` URI list. (2) `ThreadView`'s recursive walker (added in #0141) iterated `viewModel.sortedReplies` blind — visible and hidden replies rendered side-by-side with no divider, no dimming, and no badge.

## Fix

- `BlueskyCore/Feed.swift`:
  - Added `ThreadgateView` (mirrors `app.bsky.feed.defs#threadgateView`) carrying `uri`, `cid`, and an embedded `ThreadgateRecord?`. Tolerates a missing record via `try?` so older posts that ship a threadgate envelope without a usable record decode cleanly.
  - Extended `ThreadPost` with `threadgate: ThreadgateView?` and a custom `init(from:)` so existing `post / parent / replies` decoding stays unchanged.
- `BlueskyFeed/ThreadViewModel.swift`:
  - Added `hiddenReplyURIs: Set<ATURI>` derived from the focal post's `threadgate.record.hiddenReplies`. Drives the divider/dimming logic in the view layer. Mirrors RN's `useMergedThreadgateHiddenReplies` minus the optimistic-overlay piece (that lives in the threadgate-edit flow we don't ship yet).
- `BlueskyFeed/ThreadView.swift`:
  - New `@State private var showHiddenReplies: Bool = false`.
  - New `partitionReplies(_:hidden:)` splits `viewModel.sortedReplies` into `visible` and `hidden` lists by URI membership in `hiddenReplyURIs`. Non-`.post` sentinels stay visible (they paint their own tombstone).
  - The recursive `replyTree(of:focalURI:)` now renders the visible list as before, then — when there are hidden replies — emits a `hiddenRepliesDivider(count:)` (chevron + "Hidden replies" / "Hidden reply", centered, secondary). Tapping toggles `showHiddenReplies`; expanding renders the hidden replies via the existing `replyRow(_:depth:focalURI:dimmed:)` path.
  - `replyRow(...)` gained a `dimmed: Bool = false` parameter. When true the row paints a `hiddenByAuthorBadge(depth:)` ("eye.slash" + "Hidden by author") above the row and renders the `TreeReplyRow` at `opacity 0.55`. Nested children inherit the dimming so a hidden subtree reads as a single muted block.

The rules-based detection path (replies a viewer can't see because they violate the threadgate's `allow` rules) is documented as a follow-up — V1 `getPostThread` does not surface a per-reply "hidden by threadgate rules" flag, so the explicit `hiddenReplies` URI list is the only authoritative signal we ship today.

## Gotchas

- `ThreadgateRecord` requires `createdAt` on decode. The embedded record from a `threadgateView.record` payload normally includes it, but the wrapping `try?` in `ThreadgateView.init(from:)` keeps malformed envelopes from poisoning the entire thread response.
- The divider only appears when the partitioned `hidden` list is non-empty, so threads without any explicit `hiddenReplies` render exactly as before — no extra trailing whitespace or empty divider rows.
- `dimmed` is propagated through recursion so children of a hidden direct reply also paint dimmed; the badge however only shows on the depth-1 row to avoid stacking badges down a hidden subtree.
