## Description

`StarterPackCreateSheet` collects name, description, and a list selection in a single sheet. The React Native client uses a multi-step wizard at `Bluesky-ReactNative/src/screens/StarterPack/Wizard/`:

1. **Profiles** — actor picker; the user searches and selects accounts to include directly (without needing to first build a list).
2. **Feeds** — feed picker; up to N feeds to include.
3. **Details** — name, description, optional banner.
4. **Preview** — preview how the starter pack will appear; final Create button.

The single-sheet version misses the feeds slot entirely and forces the user to pre-build a list before creating the pack.

## Steps to reproduce

1. Open the starter-pack create flow.
2. Compare to RN's wizard.

## Expected behavior

Refactor `StarterPackCreateSheet` into a `NavigationStack`-driven wizard with the four steps above. Each step has Next / Back. Profiles step uses an actor-search field with debounced lookup; Feeds step uses a feed-search field. Final preview step calls `com.atproto.repo.applyWrites` to create both the list (containing selected actors) and the starter pack record in one transaction.

## Actual behavior

Single sheet; feeds slot missing; requires a pre-existing list.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/StarterPack/Wizard/index.tsx` and per-step files.
- Lexicons: `app.bsky.graph.starterpack` (record), `app.bsky.graph.list`, `app.bsky.graph.listitem` — all already in `BlueskyCore`.
- File to edit: `BlueskyKit/Sources/BlueskyLists/StarterPackCreateSheet.swift` (refactor), and likely add per-step view files.

## Root cause

`StarterPackCreateSheet` shipped as a single `Form` that collected only
`name`, `description`, and a list selection (via a sub-sheet
`ListPickerView`). The flow assumed the user had already built a curated
list — the create path could not select profiles directly, and the
feeds slot was missing entirely. RN's
`screens/StarterPack/Wizard/index.tsx` is a four-step wizard backed by
a `useWizardState` reducer (`State.tsx`) that owns selected profiles,
selected feeds, and details, with each step (`StepProfiles`,
`StepFeeds`, `StepDetails`) binding to the same store.

The lexicon side was also incomplete: `StarterPackRecord` only encoded
`{name, description, list, createdAt}` and did not include the optional
`feeds` array that RN's `useCreateStarterPackMutation` writes
(`feeds?.map(f => ({uri: f.uri}))`). And `ListsStore.createStarterPack`
took only a `listURI`, so callers had to pre-build the backing list.

## Fix

- **Lexicon**: `StarterPackRecord` now encodes an optional
  `feeds: [StarterPackFeedItem]?` array per the lexicon. New
  `StarterPackFeedItem` value type wraps each `{uri}` entry. Defaults
  to `nil` so existing call sites continue to compile.
- **Wizard model**: new
  `BlueskyLists/StarterPackWizardModel.swift` — a `@MainActor
  @Observable` container holding `currentStep`, `selectedProfiles`
  (`[ProfileBasic]`), `selectedFeeds` (`[GeneratorView]`), `name`,
  `description`, `processing`, `errorMessage`, plus per-step gating
  (`canAdvance(from:)`). Mirrors the RN reducer's shape, including
  the 8-profile minimum to advance off the Profiles step and the
  3-feed cap on Feeds.
- **Search store**: new `StarterPackWizardSearchStore.swift` exposes
  four result sets — `topActors`, `actorResults`, `popularFeeds`,
  `feedResults` — plus matching loading flags. Profile typeahead
  uses `app.bsky.actor.searchActorsTypeahead` with a 200 ms debounce
  (matches `ComposerStore.searchMentions`); feed search uses
  `app.bsky.unspecced.getPopularFeedGenerators` with 300 ms debounce.
  Empty queries fall back to `app.bsky.actor.searchActors?q=*` (top
  actors) and `getPopularFeedGenerators` with no query
  (popular feeds), matching RN's empty-state behavior.
- **Sheet refactor**: `StarterPackCreateSheet` is now a
  `NavigationStack`-driven wizard with a 4-segment progress bar,
  shared model bindings, Back / Next / Skip / Create footer, and an
  alert for store errors. Steps in RN order:
  - **Profiles** (`StarterPackWizardProfilesStep`): horizontal chip
    strip of selected profiles, search field with debounced typeahead,
    multi-select rows with checkmark trailing, fallback to
    "top actors" before the user types. Footer disables Next until
    the 8-profile minimum is met (matches RN copy *"Add N more to
    continue"*).
  - **Feeds** (`StarterPackWizardFeedsStep`): same pattern, popular
    feeds default, debounced query against `getPopularFeedGenerators`,
    cap of 3 selections enforced by the model. Footer button label
    flips between "Skip" (when zero feeds) and "Next" (one or more)
    to match RN's `wizardUiStrings.Feeds.nextBtn` logic.
  - **Details** (`StarterPackWizardDetailsStep`): a `Form` with the
    starter-pack hero panel ("Invites, but personal" — RN copy), the
    name field with a 50-char counter (matches RN's
    `state.name.length`/50), and a multi-line description editor.
  - **Preview** (`StarterPackWizardPreviewStep`): name + description
    recap, included-people count with overlapping avatar strip
    (mirrors RN's `Footer` avatar stack), included-feeds count and
    list of feed rows. Final Create button.
- **Submission**: new
  `ListsStore.createStarterPackWithProfiles(name:description:profileDIDs:feedURIs:)`
  ports RN's `useCreateStarterPackMutation` /
  `createStarterPackList` two-step:
  1. `createRecord` for the backing list (purpose
     `app.bsky.graph.defs#referencelist`).
  2. `applyWrites` chunks of 50 to add `app.bsky.graph.listitem`
     rows for every selected profile (matches RN's
     `chunk(profiles, 50)`).
  3. `createRecord` for the `app.bsky.graph.starterpack` record
     pointing at the new list URI plus the optional feeds.
  Returns `Bool` so the sheet can flip processing state and show the
  error alert. `ListsViewModel` exposes a passthrough.

## Files changed

- `BlueskyKit/Sources/BlueskyCore/StarterPack.swift` — adds
  `StarterPackFeedItem` and an optional `feeds` field on
  `StarterPackRecord`.
- `BlueskyKit/Sources/BlueskyLists/ListsStore.swift` — new
  `createStarterPackWithProfiles(...)` protocol method and impl
  (list create → applyWrites → starterpack create).
- `BlueskyKit/Sources/BlueskyLists/ListsViewModel.swift` —
  passthrough wrapper for the new store method.
- `BlueskyKit/Sources/BlueskyLists/StarterPackCreateSheet.swift` —
  full rewrite into a `NavigationStack`-driven wizard with progress
  bar, shared model + search store, four step views, and a unified
  Next/Create footer.
- `BlueskyKit/Sources/BlueskyLists/StarterPackWizardModel.swift` —
  new `@MainActor @Observable` container holding wizard state, with
  per-step gating, profile/feed selection helpers, and effective
  name/description resolvers (50-char clamp, blank fallback to
  suggested name).
- `BlueskyKit/Sources/BlueskyLists/StarterPackWizardSearchStore.swift`
  — new debounced search store (`searchActorsTypeahead` for actors,
  `getPopularFeedGenerators` for feeds), with empty-query defaults
  and request-id guards so a slow earlier response cannot overwrite
  a faster later one.

## Gotchas

- **Banner image picker is deferred.** The dispatch spec listed the
  banner as part of the Details step. Per the bail-rule on heavy
  image-picker wiring, this change ships the four-step wizard with a
  text-only Details step. The blob upload helper
  (`network.upload(lexicon: "com.atproto.repo.uploadBlob", ...)`)
  already exists, so adding banner upload is a contained follow-up
  on the existing wizard scaffold.
- **List edit-mode is not implemented.** RN's wizard doubles as the
  edit screen via the `rkey` route param (`useEditStarterPackMutation`
  diff-applies the new vs. existing members and `putRecord`s the
  starterpack record). The SwiftUI port covers create only; an edit
  flow can reuse `StarterPackWizardModel` and add a sibling
  `editStarterPack(...)` method on `ListsStore`.
- **Suggested-name seed is a placeholder.** RN seeds the Details
  field with `"{currentProfile.displayName}'s Starter Pack"`. The
  sheet does not currently accept a profile, so the suggested name
  defaults to `"My Starter Pack"`. When the host wires the wizard
  into `StarterPackScreen`, threading the current profile down lets
  the suggested name match RN exactly.
- **The 8-profile minimum is hard-coded.** RN uses
  `STARTER_PACK_MAX_SIZE` for the cap and a separate `8` for the
  minimum-to-advance. Both numbers live as constants on
  `StarterPackWizardModel` (`maxProfiles = 50`, hard-coded `8` in
  `canAdvance`). If RN changes the minimum, update both call sites.
- **`searchActorsTypeahead` returns `ProfileBasic`, not
  `ProfileView`.** The model stores `[ProfileBasic]` so the empty-
  query default (`searchActors`) is mapped down via a small adapter.
  This works because the wizard rows only need handle / displayName /
  avatar; if a future step needs follow counts, switch the storage
  type to `ProfileView` and adapt the typeahead path the other way.
