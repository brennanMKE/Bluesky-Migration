## Description

Tapping the like button on a post appears to fire a network request (multiple `_loadCurrentDID` calls visible in the log) but the like state does not stick — the heart returns to unfilled and the like count does not increment. This is a regression; liking was previously confirmed working.

## Steps to reproduce

1. Open the Home feed.
2. Tap the like (heart) button on any post.
3. Observe the heart briefly fills, then reverts to unfilled.
4. Check the Bluesky website — the post shows no new like.

## Expected behavior

The heart fills and stays filled. The like count increments. The like is visible on bsky.app.

## Actual behavior

The like state reverts. The only log output is repeated `_loadCurrentDID` calls — no success or error message from the like/unlike API call is logged.

## Notes

The log shows `_loadCurrentDID` called 4 times in rapid succession when the like button is tapped, suggesting multiple concurrent requests may be racing or the optimistic update is being overwritten by a background refresh. Possible causes:

1. The optimistic post update in `FeedViewModel.like()` is being wiped out by a concurrent `loadMore()` or `refresh()` call that replaces the `posts` array.
2. The `createRecord` XRPC call for `app.bsky.feed.like` is silently failing and the error is not being surfaced.
3. Issue #0049 (duplicate FeedViewModel) may be causing the second VM's state to overwrite the first's optimistic update.

Check `FeedStore.like()` and `FeedViewModel.like()` for error handling and confirm the optimistic update is not being clobbered.

## Root cause

`FeedStore.performLike` cleared its `pendingLikeURIs[uri]` entry synchronously at the end of the function, immediately after the `createRecord` XRPC call returned. Any subsequent feed fetch (`getTimeline`/`getFeed`) or `freshenedPost` (`getPosts`) that ran *after* `performLike` returned but *before* the AT Protocol indexer had reflected the new like would receive `viewer.like == nil` for that post and overwrite the local optimistic state, reverting the heart. The same race existed for repost.

In addition, `freshenedPost` always called `updatePost(uri:) { _ in fresh }` with the raw server response, so even within a single toggle flow it could clobber a pending optimistic state if the indexer was slightly behind.

## Fix

In `Sources/BlueskyFeed/FeedStore.swift`:

- Hold `pendingLikeURIs` / `pendingRepostURIs` entries for a 30s safety window after a successful create-record (new `pendingURICleanupWindow` constant). A scheduled Task clears each entry only if it still matches the URI we wrote (so a quick unlike/re-like isn't undone).
- Clear those entries early when a fetch or `freshenedPost` observes the server returning the like/repost in `viewer` state (indexer has caught up).
- Drop the entry on explicit `unlike` / `unrepost` so a deferred cleanup or late-arriving fetch can't re-apply a stale like.
- Add `overlayPendingState(on:)` and call it from `freshenedPost` so the post returned to callers (and written into `posts`) preserves any in-flight or recently-confirmed optimistic state when the server is still lagging.
- Tighten the `fetch(reset:true)` overlay so it only injects the optimistic placeholder when the server response itself is missing the like/repost (the indexer could legitimately confirm during the same fetch).

## Files changed

- `BlueskyKit/Sources/BlueskyFeed/FeedStore.swift`

## Gotchas

- The pending-URI window (30s) is a safety net; the entries are normally cleared earlier when a refresh confirms the indexer caught up. If a test or simulator scenario relies on these entries being purged immediately, it will need updating.
- `pendingLikeURIs` lookups use AT-URI value equality. If the AT Protocol ever returns a different normalisation of the post URI in `getPosts` versus `getTimeline`, the overlay would miss; investigate that path before adding handle-based or DID-based fallbacks.
