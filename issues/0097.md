## Description

Before posting, the React Native composer lets a user attach self-labels — content warnings that downstream clients use to blur or hide the post. The labels include `nudity`, `sexual`, `porn`, and `graphic-media` (this set follows the AT Proto self-label vocabulary). RN exposes them via a `LabelsBtn` in the composer toolbar that opens a dialog with toggleable rows (`Bluesky-ReactNative/src/view/com/composer/labels/`).

The SwiftUI composer has no UI for self-labels. A user cannot mark their own post as NSFW, which means SwiftUI-posted content is mis-categorized when viewed by users who hide adult content.

## Steps to reproduce

1. Open the SwiftUI composer.
2. Look for an icon/button to mark the post NSFW or as containing graphic media.
3. Observe there is none.

## Expected behavior

A label button (e.g., warning-triangle icon) in the composer toolbar that opens a sheet:

- Toggle: **Sexually explicit** (`porn`)
- Toggle: **Sexually suggestive** (`sexual`)
- Toggle: **Nudity** (`nudity`)
- Toggle: **Graphic media** (`graphic-media`)

Selected labels are written to `PostRecord.labels` as an `app.bsky.feed.post#selfLabels` object (see RN's post-create code path). The button shows a filled / colored state when any label is selected.

## Actual behavior

No labels button. `PostRecord` is created with no `labels` field regardless of content.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/com/composer/labels/LabelsBtn.tsx` and `LabelsDialog.tsx`.
- Lexicon: `app.bsky.feed.post#selfLabels` (a `com.atproto.label.defs#selfLabels` value).
- File to edit: `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift`; possibly add a `selfLabels` field to `PostRecord` in `BlueskyCore/Feed.swift` if not present.
- The four options above are the only valid self-label values for posts. Don't expose other values.

## Root cause

The SwiftUI composer never modeled self-labels. `PostRecord` (in
`BlueskyCore/Post.swift`) had no `labels` field, the composer toolbar
exposed only the image and video pickers, and `ComposerStore.post` built
the AT Proto record with no path to attach a
`com.atproto.label.defs#selfLabels` payload. The end result was that
SwiftUI-authored posts were always indexed as unlabeled, regardless of
their content — RN's `LabelsBtn` (`Bluesky-ReactNative/src/view/com/composer/labels/LabelsBtn.tsx`)
and the `if (draft.labels.length)` branch in `lib/api/index.ts` had no
counterparts on the SwiftUI side.

## Fix

**Lexicon (Layer 0).** Added two new value types to
`BlueskyCore/Post.swift`:

- `SelfLabelValue` — `{ val: String }` corresponding to
  `com.atproto.label.defs#selfLabel`.
- `SelfLabels` — `{ $type, values: [SelfLabelValue] }` corresponding to
  `com.atproto.label.defs#selfLabels`. Encodes its own `$type`
  discriminator so the appview accepts it as the tagged-union variant
  for a post's `labels` field.

`PostRecord` gained an optional `labels: SelfLabels?` field. When
`nil`, the JSON encoder omits the key entirely (matching RN's
`if (draft.labels.length)` guard, so unlabeled posts don't carry an
empty container).

**Composer state (Layer 3).** `ComposerViewModel` gained
`selectedLabels: Set<String>`, reset on a successful post alongside the
other transient draft state. `ComposerStoring.post(...)` and
`ComposerStore.post(...)` accept a `selfLabels: Set<String>` parameter
and translate it into a `SelfLabels` value on the head post record.
The same labels are carried onto every additional thread post, since
the SwiftUI thread editor doesn't expose per-post label UI yet
(RN supports per-draft labels because each thread row has its own
`LabelsBtn`; we mirror only the simpler "one set of labels for the
thread" model for now).

**Composer UI.** `ComposerSheet.mediaToolbar` now renders a
`labelsButton` next to the image and video pickers on both iOS and
macOS. The button shows `shield` (secondary tint) when no labels are
selected and `exclamationmark.triangle.fill` (orange tint) when at
least one label is attached, signalling that a content warning will
be applied. Tapping presents `SelfLabelsPicker`, which is a sheet on
iOS and a popover on macOS.

`SelfLabelsPicker` mirrors RN's `LabelsDialog` layout:

- An "Adult Content" section with three rows — *Suggestive* (`sexual`),
  *Nudity* (`nudity`), *Adult* (`porn`). Tapping a row applies the
  same `updateAdultLabels` semantics RN uses: only one of the three
  may be active at a time (tapping the active row clears the adult
  selection entirely). Per-selection helper text appears below the
  group with the same copy RN ships ("Pictures meant for adults.",
  "Artistic or non-erotic nudity.", "Sexual activity or erotic
  nudity.").
- An "Other" section with a single *Graphic Media* (`graphic-media`)
  row that toggles independently. Selecting it surfaces RN's
  "Media that may be disturbing or inappropriate for some audiences."
  helper text.

A "Done" toolbar button dismisses the picker. The composer's existing
post submission path (`ComposerStore.post`) wires the resulting
`Set<String>` through to `PostRecord.labels`, omitting the field
entirely when no labels are selected.

A `PostCodableTests` suite was added to
`Tests/BlueskyKitTests/LexiconCodableTests.swift` covering the
`SelfLabels` `$type` discriminator, the round-trip of a `PostRecord`
with labels, and the omission of the field on an unlabeled post.

## Files changed

- `BlueskyKit/Sources/BlueskyCore/Post.swift` — added `SelfLabelValue`,
  `SelfLabels`, and the optional `labels` field on `PostRecord`.
- `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` — added
  `labelsButton` to the media toolbar, the `SelfLabelsPicker` view,
  and the iOS/macOS sheet/popover wiring (plus an iOS
  `dismissKeyboard()` helper to mirror RN's `Keyboard.dismiss()`).
- `BlueskyKit/Sources/BlueskyComposer/ComposerViewModel.swift` —
  added `selectedLabels: Set<String>`, threaded it through `post()`,
  cleared it on successful submission.
- `BlueskyKit/Sources/BlueskyComposer/ComposerStore.swift` — added
  `selfLabels: Set<String>` to the protocol and implementation, built
  a `SelfLabels` value when non-empty, and attached it to both the
  head and any additional thread posts.
- `BlueskyKit/Tests/BlueskyKitTests/LexiconCodableTests.swift` —
  added a `PostCodableTests` suite verifying the encoder behavior
  described above.

## Gotchas

- **RN label copy is *not* the issue spec's copy.** The issue lists
  the toggles as "Sexually explicit", "Sexually suggestive", "Nudity",
  "Graphic media". The live RN component
  (`view/com/composer/labels/LabelsBtn.tsx`) ships **"Suggestive"**,
  **"Nudity"**, **"Adult"**, **"Graphic Media"** with section headers
  **"Adult Content"** and **"Other"**, and groups the first three as
  single-selection. We mirrored RN exactly per the strict-parity
  rule; the issue's list of "expected" labels is doc drift, not a
  spec.
- **No separate `LabelsDialog.tsx` file.** Despite the issue mentioning
  one, the entire `LabelsBtn` + `DialogInner` lives in a single
  `LabelsBtn.tsx`. Read that file for the dialog layout.
- **Adult labels are radio, not checkbox.** RN's `updateAdultLabels`
  filters out the other two before adding the new one. If we modeled
  all four toggles as independent checkboxes, the appview would still
  accept it, but the post would carry contradictory labels. The
  picker enforces single-selection within the adult group to match
  RN.
- **No `warning` color token in `BlueskyTheme`.** Following the same
  fallback used in #0096, the active labels button uses
  `Color.orange` rather than introducing a new theme token. Issue
  #0040 (color palette refinement) is the right place to mint a
  proper warning token if the design system grows one.
- **Thread posts inherit the head's labels.** RN gives each draft its
  own `LabelsBtn`, so different rows in a thread can carry different
  labels. The SwiftUI thread editor only exposes a single
  `selectedLabels` for the entire compose session today; the store
  applies that one set to every post in the thread. If the thread UI
  ever grows per-row labels, the store already takes a
  `Set<String>` per call, so the change would be confined to the
  view model and the picker plumbing.
- **Labels field is omitted, not encoded as `null` or empty.** The
  appview tolerates `{}` but RN's encoder skips the field entirely
  for unlabeled posts. We match by making `labels` optional and
  building the `SelfLabels` value only when `!selfLabels.isEmpty`.
  Verified by `PostCodableTests.encodePostRecordWithoutLabels`.
- **Pre-existing test target build failure.** The `BlueskyKitTests`
  target fails to compile against `BlueskyEnvironment` due to an
  earlier signature drift unrelated to this issue (`bookmarks` /
  `pathMonitor` parameters in `BlueskyKitTests.swift:90`). The new
  `PostCodableTests` cases are correct by inspection of the encoder
  output; the pre-existing failure prevents `swift test` from
  running them. The library `swift build` succeeds cleanly, as does
  the macOS Bluesky-SwiftUI workspace build.
