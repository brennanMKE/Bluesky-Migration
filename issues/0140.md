## Description

The React Native thread view exposes a header dropdown (RN's `HeaderDropdown`) where the user picks how replies are sorted: **Hot** (engagement-weighted), **Top** (likes), **Newest**, **Oldest**, **Random**. The default comes from the user's Thread Preferences (see #0116). The SwiftUI thread renders replies in API order with no UI to change sort.

## Steps to reproduce

1. Open any thread.
2. Look for a sort selector / dropdown in the toolbar.
3. Observe none.

## Expected behavior

A small Menu in the thread toolbar with sort options. Selection updates the loaded thread in place (re-sort client-side from the loaded `thread.replies` tree, or refetch with a different `?sort=` parameter on `getPostThread` if supported). The default value reads from `app.bsky.actor.defs#threadViewPref.sort`.

## Actual behavior

No sort UI; replies are rendered in API order regardless of the user's preference.

## Notes

- RN reference: `Bluesky-ReactNative/src/view/screens/PostThread.tsx` (`HeaderDropdown`).
- File to edit: `BlueskyKit/Sources/BlueskyFeed/ThreadView.swift` and `ThreadViewModel.swift`.
- Pairs with #0116 (Thread Preferences screen). The preference there sets the default; this dropdown is the per-view override.

## Root cause

`ThreadView` rendered direct replies via `tp.replies` straight from the `app.bsky.feed.getPostThread` payload, in whatever order the appview shipped them. There was no toolbar control to override the order, no model state for the active sort, and no path to seed an initial value from the user's `app.bsky.actor.defs#threadViewPref.sort` preference (already exposed on `GetPreferencesResponse.threadView` since #0116). RN's `HeaderDropdown` had no SwiftUI counterpart.

## Fix

- **`BlueskyKit/Sources/BlueskyFeed/ThreadViewModel.swift`** — added a public `ThreadSort` enum mirroring the lexicon string set (`hotness`, `most-likes`, `newest`, `oldest`, `random`) with `displayOrder` matching the issue spec (Hot, Top, Newest, Oldest, Random) and per-case localized labels. Added `var sort: ThreadSort` on the view model (`.hotness` default), `setSort(_:)` to flip it (reshuffling the random seed when `.random` is tapped or re-tapped), and `sortedReplies` computed property that re-orders the focal post's `replies` client-side based on the active selection. Sort comparators: `.mostLikes` by `likeCount` desc with newest-first then server-position tie-breakers; `.newest`/`.oldest` by `indexedAt`; `.hotness` by an engagement-weighted, age-decayed score (`(likes + 2·reposts + 4·replies + 1) / (hours + 2)^1.8`) — best-effort client approximation since the V1 `getPostThread` lexicon doesn't accept a server-side `sort`; `.random` via a seeded splitmix64 RNG so the order stays stable across re-renders within a session. Non-`.post` reply nodes (notFound / blocked / unknown) keep their server position at the end. `load()` now hydrates the user's default `sort` from `app.bsky.actor.getPreferences` exactly once (first call), then preserves the per-session override on subsequent refreshes; failures fall back silently to `.hotness`.
- **`BlueskyKit/Sources/BlueskyFeed/ThreadView.swift`** — added a `sortToolbar` `ToolbarContent` with a `Menu` containing an inline `Picker` over `ThreadSort.displayOrder`. The picker is bound to `viewModel.sort` via `setSort(_:)` and disabled until the thread loads. The toolbar item uses `.primaryAction` placement and a `slider.horizontal.3` SF Symbol (closest match to RN's `SettingsSliderVertical` icon). The reply branch in `flattenThread` now iterates `viewModel.sortedReplies` instead of `tp.replies` so the dropdown drives row order without a refetch.

## Gotchas

- **Five options vs RN's three.** The live RN `HeaderDropdown` exposes only `top` / `oldest` / `newest`. The issue spec for #0140 — and the lexicon — admit all five, so we surface the full set per the spec (matches the same call made for #0116). Writing `hotness` or `random` from the SwiftUI dropdown is in-session only (no `putPreferences` write), so RN's normalize-to-`top` fallback never sees an unfamiliar value persisted from this control.
- **Per-session, not persisted.** Per the issue spec, sort changes here do *not* call `putPreferences`. The persisted default is owned by the Thread Preferences settings screen (#0116). On thread re-open the dropdown returns to the user's preference.
- **Client-side sort, no refetch.** V1 `app.bsky.feed.getPostThread` does not accept a `sort` parameter — RN uses `app.bsky.unspecced.getPostThreadV2` with `sort` arg, which we have not modeled. Per the issue's recommendation, sort runs over the already-loaded `thread.replies` tree. The `.hotness` comparator is therefore an approximation of the appview ranking; once V2 lands in `BlueskyCore` it should replace the client-side `hotness` path with a server refetch keyed by `sort`.
- **Random sort stability.** A naive `Array.shuffle()` would re-randomize on every `sortedReplies` access, churning the SwiftUI list. The seeded splitmix64 RNG keeps the order stable until `setSort(.random)` is called again (which also re-seeds when the user re-taps `random`).
- **One-shot preference hydration.** `getPreferences` runs only on the first `load()` (guarded by `didHydrateDefault`). A pull-to-refresh would clobber a per-session override otherwise; the user's choice persists for the lifetime of the screen.
