## Description

In the macOS app, a feed/list row shows the feed name "Popular With Friends" alongside a raw 12-character hex string `77a502dfc060`. That string is the **rkey** of an `at://` URI (e.g. the trailing path component of `at://did:plc:.../app.bsky.feed.generator/77a502dfc060`) — internal storage detail that should never be user-visible.

This is a debug leak: somewhere in the feed/list row builder a developer-debug `Text(uri.rkey)` (or equivalent) is being rendered next to the proper display name.

The React Native iOS reference shows feed rows with name + creator handle + description + like count — never the rkey.

## Steps to reproduce

1. Run the macOS SwiftUI app.
2. Open the screen that lists feeds (likely **My Feeds**, **Saved Feeds**, or the Home feed-switcher row group).
3. Find the row for the "Popular With Friends" feed (or any custom feed).
4. Observe the rkey rendered as plain text next to the name.

## Expected behavior

The row shows the feed's `displayName` (and probably creator handle + description per the existing `FeedCard` shape from Module 4). No rkey, no CID, no DID, no AT URI.

## Actual behavior

Display name + raw 12-char rkey alongside it.

## Notes

- The leaking value `77a502dfc060` is the rkey of an `app.bsky.feed.generator` record. Search the SwiftUI tree for `uri.rkey`, `"\(uri" `, or any `Text` rendering of an `ATURI` to find the offender. Likely in `MyFeedsScreen.swift`, `SavedFeedsScreen.swift`, `FeedSwitcherView.swift`, or a `FeedCard` variant.
- A trailing relative timestamp ("5d") is visible in the same screenshot — that part appears to be intentional. The rkey next to the name is the bug.
- Audit the row builder for any other internal-only fields being leaked at the same time.

## Root cause

Three sites used the AT-URI's last path component (the rkey) as a render-time fallback whenever real `GeneratorView.displayName` metadata was unavailable:

- `BlueskyFeed/FeedView.swift:347` — `pinnedFeedDisplayName(_:)` used `feed.value.components(separatedBy: "/").last` as the strip-tab label for pinned feeds. The Home tab strip never resolved the saved-feed AT-URIs, so the fallback was the *only* label users ever saw — the screenshot's `77a502dfc060` is the rkey of `at://…/app.bsky.feed.generator/77a502dfc060`. The doc-comment on the helper even acknowledged the leak ("Falls back to the rkey portion of the AT-URI…"), so this was a known shortcut waiting to be paid down.
- `BlueskyFeed/MyFeedsScreen.swift:430` — same fallback function, used as the placeholder while `SuggestedFeedsStore.resolve(uris:)` is in flight on the My Feeds screen. Less visible because resolution lands quickly, but still a brief leak.
- `BlueskyFeed/SavedFeedsScreen.swift:115` — same fallback, used for the legacy macOS Saved Feeds list rows.

RN's `HomeHeader.tsx` builds tab labels from `f.displayName` directly — it never falls back to the rkey because the saved-feeds query already returns resolved metadata. The SwiftUI port skipped that resolution step for the strip and patched the gap with the rkey, which leaked through to the UI.

## Fix

- Wired a `SuggestedFeedsStore` (`pinnedFeedsResolver`) into `FeedView`. On `.task` it resolves every pinned feed AT-URI via `app.bsky.feed.getFeedGenerator`; an `.onChange(of: pinnedFeedURIs)` and the existing `savedFeedsChanged` notification handler keep it in sync as the user pins/unpins feeds.
- `pinnedFeedDisplayName(_:)` now reads the resolved `GeneratorView.displayName` first and falls back to the generic placeholder `"Feed"` while metadata loads — never the rkey, never the AT-URI.
- `SavedFeedsScreen.displayName(for:)` and `MyFeedsScreen.displayName(for:)` (the placeholder used while the resolver hasn't returned) now switch on `feed.type` and return `"Following"`, `"List"`, or `"Feed"` — no AT-URI parsing at all.
- Audited neighbor `Text(...)` calls in the affected row builders and in `FeedCard` / `ListCard`: they only render `displayName`, `creator.handle.rawValue`, and `description`, so no other internal-only fields (DID, CID, full AT URI, raw `Date` ISO string) are reaching the UI.

## Gotchas

- The new `SuggestedFeedsStore` instance in `FeedView` is a *separate* object from the one `MyFeedsScreen` owns — both screens resolve their own pinned feeds independently. That's wasted work in theory but cheap in practice (resolve calls are idempotent and the typical pinned set is <10), and sharing would require threading a store through the environment which is out of scope for this fix.
- The `"Feed"` placeholder is briefly visible on first launch before `getFeedGenerator` returns. RN solves this by hydrating from a longer-lived feed-info cache; we accept the brief placeholder for now since the resolver fires immediately on `.task`.
- `SuggestedFeedsStore.resolve(uris:)` only issues calls for AT-URIs containing `app.bsky.feed.generator`, which protects against accidentally calling `getFeedGenerator` on list URIs that the user might pin via a future RN-parity feature.
