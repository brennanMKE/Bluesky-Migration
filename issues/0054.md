## Description

In the thread view, tapping on a reply post card does nothing. Each reply may itself have replies (sub-threads), but there is no way to drill into them because the tap gesture is not wired up.

## Steps to reproduce

1. Tap any post in the feed to open the thread view.
2. Scroll down to the replies listed below the focal post.
3. Tap on any reply post card.
4. Observe that nothing happens — no navigation occurs.

## Expected behavior

Tapping a reply navigates to that reply's own thread view, showing it as the focal post with its own replies below it.

## Actual behavior

The tap is silently ignored. Sub-thread replies are inaccessible.

## Notes

`ThreadView` renders replies as `PostCard` items with a `PostCard.Actions` that only wires `onReply`. The `onTap` closure is not set, so tapping the card body does nothing. The fix is to wire `onTap` in `ThreadView.actions(for:)` to push a new `ThreadView` via a `navigationDestination`, passing the tapped reply's URI. This requires a `@State var selectedReplyURI: ATURI?` and a `.navigationDestination(item:)` on the list.

## Root cause

`ThreadView.actions(for:)` set `a.onTap = onPostTap`, deferring navigation to the host. In `MainTabView`, `onPostTap` was `{ post in threadURI = post.uri }` — but `threadURI` was the same state binding driving the `navigationDestination(item:)` that was already presenting this `ThreadView`. SwiftUI saw the binding rebound to a value while the destination was active and declined to push another instance, so the tap appeared to do nothing.

## Fix

`ThreadView` now owns a local `@State var selectedReplyURI: ATURI?` plus a `.navigationDestination(item: $selectedReplyURI)` that pushes a fresh `ThreadView` for the tapped post. The new `actions(for:focalURI:)` ignores taps on the focal post (it is already on screen) and pushes for any ancestor or reply row. The host's `onPostTap` is no longer consulted for in-thread navigation, so re-binding the parent's URI is no longer required.

## Gotchas

- `navigationDestination(item:)` accepts any `Hashable`; `ATURI` already conforms, so no Identifiable shim was needed.
- The host's `onPostTap` is still passed down to nested `ThreadView`s so external observers (analytics, deep-link mirroring) keep firing if they rely on it — though the in-thread push is independent of that closure.
