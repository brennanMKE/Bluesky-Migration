# 0032 — Module 11 feature: image attachments and message requests inbox not implemented

## Description

The DM composer bar has no image attachment button. The message requests inbox (for messages from accounts not followed) is also absent from `ConversationListScreen`.

## Steps to reproduce

1. Open a DM conversation.
2. Look for an image attachment button in the compose bar.
3. Return to the conversation list and look for a "Message Requests" section or button.

## Expected behavior

An image picker button in the compose bar uploads a blob and sends it as an embed in the message. A message requests section shows pending DMs from unfollowed accounts.

## Actual behavior

No attachment button exists. No message requests section exists.

## Notes

Image attachment: add a `PhotosPicker` or `PHPickerRepresentable` to the compose bar; upload via `com.atproto.repo.uploadBlob`; embed in `MessageInput.embed`. Message requests: the API likely exposes a separate endpoint or filter on `chat.bsky.convo.listConvos`; needs investigation.

## Root cause

Two interrelated gaps in the BlueskyMessages module:

1. The compose bar in `MessageThreadScreen` already had a `PhotosPicker` and the `MessageThreadStore.sendImageAttachment(...)` flow was wired to `com.atproto.repo.uploadBlob` + `chat.bsky.convo.sendMessage` with an `Embed.images(...)` payload — but the corresponding **receive** side, `MessageBubble`, only rendered `message.text`, so image embeds came in over the wire and were silently dropped from the UI.
2. `MessageRequestsScreen` existed and `ConversationStore` already loaded pending convos via `chat.bsky.convo.listConvos?status=request`, but there was no way to accept or reject a request — the screen functioned as a read-only list. The `chat.bsky.convo.acceptConvo` lexicon was not wired at all.

## Fix

- **MessageBubble image rendering** — `MessageThreadScreen.MessageBubble` now inspects `MessageView.embed`. When the embed is `.images([EmbedImageView])` (or a `recordWithMedia` whose media is `.images`), each image is rendered inline above the text bubble using `AsyncImage(url: thumb)` with the lexicon's `aspectRatio` hint, capped at 220×220, rounded corners, and an `accessibilityLabel` from the alt text. Empty-text messages render image-only.
- **Accept / reject convo** — `ConversationStoring` gained `acceptRequest(_:)` and `rejectRequest(_:)`. `ConversationStore.acceptRequest` POSTs to `chat.bsky.convo.acceptConvo` and optimistically moves the convo from `requestConvos` into `convos`; failure triggers `refresh()` + `loadRequests()` to roll back. `rejectRequest` POSTs to `chat.bsky.convo.leaveConvo` (matching RN's `RejectMenu` "delete chat" action) and optimistically removes the convo from `requestConvos`. `MessagesViewModel` exposes both as thin wrappers.
- **MessageRequestsScreen rework** — the screen now owns its own `MessagesViewModel` (replacing the read-only `convos:` parameter) and runs `loadRequests()` via `.task` plus pull-to-refresh. Each row exposes trailing swipe actions for **Reject** (destructive) and **Accept** (accent tint). `ConversationListScreen` passes only `network:` and `viewerDID:` now.

Build verified with `swift build` in BlueskyKit and `xcodebuild -workspace Bluesky.xcworkspace -scheme Bluesky-SwiftUI -configuration Debug -destination 'platform=macOS' build`.

## Files changed

- `BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift` — `MessageBubble` renders `EmbedImageView` images via `AsyncImage`, handles `.images` + `recordWithMedia` shapes, image-only bubble layout.
- `BlueskyKit/Sources/BlueskyMessages/ConversationStore.swift` — adds `acceptRequest` + `rejectRequest` to protocol and store; optimistic list moves; rollback on failure.
- `BlueskyKit/Sources/BlueskyMessages/MessagesViewModel.swift` — exposes `acceptRequest` / `rejectRequest`.
- `BlueskyKit/Sources/BlueskyMessages/MessageRequestsScreen.swift` — owns its `MessagesViewModel`; `.task { loadRequests }`; trailing swipe actions Accept / Reject; refreshable.
- `BlueskyKit/Sources/BlueskyMessages/ConversationListScreen.swift` — drops obsolete `convos:` argument when navigating to `MessageRequestsScreen`.

## Gotchas

- The AT Proto chat lexicon for `chat.bsky.convo.sendMessage` only formally accepts a `record` (quoted post) embed in `MessageInput.embed`; sending an `Embed.images(...)` payload may be rejected by some PDS deployments. The compose flow optimistically appends to the local `messages` list and surfaces server errors via `errorMessage`, so any rejection will be visible at runtime even though the local UX path looks complete.
- The "Reject" action calls `leaveConvo` (matching the RN `DeleteChatButton` path), not a hypothetical separate reject lexicon. Per the RN reference, accept and "leave/delete" are the only two terminal actions for a request.
