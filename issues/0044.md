# 0044 — Feed fails to load: missing `$type` in embedded quote record

| | |
|---|---|
| **Status** | resolved |
| **Module** | BlueskyCore |
| **Platform** | All |
| **First seen** | 2026-04-29 |

## Description

The home feed fails to load with a decoding error when any post in the timeline contains a quoted post embed whose `record` object is missing the `$type` field. Because `EmbedRecordContent.init(from:)` uses a hard `try c.decode(String.self, forKey: .type)` call, a single missing `$type` in any feed item throws and aborts decoding of the entire response.

## Steps to reproduce

1. Build and run the app.
2. Sign in with a Bluesky account whose timeline contains a quoted post.
3. Observe that the feed fails to load with an error banner.

## Expected behavior

Posts with a missing `$type` in their embed record are treated as unknown and skipped; the rest of the timeline loads normally.

## Actual behavior

```
fetch error: BlueskyCore.ATError.decodingFailed("DecodingError.keyNotFound: Key '$type' not found
in keyed decoding container. Path: feed[47].post.embed.record.")
```

The entire feed response is dropped and the feed shows an error state.

## Notes

Root cause: `EmbedRecordContent.init(from:)` in `Sources/BlueskyCore/Embed.swift` line 251 uses `try c.decode(String.self, forKey: .type)` which throws on a missing key. Fix: use `decodeIfPresent` with an `"unknown"` fallback so the switch falls through to the existing `.unknown` case.

Secondary symptom: `task fired, selection=timeline` and `creating FeedViewModel for timeline` appear twice in the log, indicating `FeedView` is instantiated twice (both task firings see `viewModels[selection] == nil`). This is a separate issue.
