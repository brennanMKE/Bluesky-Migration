## Description

When a user types `@alice` or `https://example.com` or `#tech` in the React Native message composer, RN runs the same facet-builder used by the post composer (`useExtractEmbedFromFacets`) and:

- Resolves `@handle` to a DID and includes a mention facet in the sent message.
- Detects URLs and either includes a link facet, or upgrades the message to include a link-card embed.
- Highlights mentions/links inline as the user types.

The SwiftUI compose bar (`MessageThreadScreen.swift` ~lines 104–128) sends the raw text via `MessageInput.text` without facets and without inline highlighting. Recipients see plain text where mentions and links should be tappable.

## Steps to reproduce

1. Open the SwiftUI compose bar.
2. Type `Hey @alice.bsky.social check out https://example.com`.
3. Send.
4. View the sent message in the recipient's RN client.
5. Observe @alice is plain text, not a mention; the URL is plain text, not a link.

## Expected behavior

The compose bar runs `FacetBuilder` (already exists in `BlueskyComposer`) over the input text on send:

- `@handle` segments are resolved to DID and added as `mention` facets.
- URL segments are added as `link` facets.
- `#tag` segments are added as `tag` facets.
- The outgoing `MessageInput` carries `text` + `facets[]`.

For visual polish (lower priority but RN does it): live-highlight detected facets with the link color while the user is typing. This can use the same `RichTextView`-based path as the post composer, or a simpler inline regex check.

## Actual behavior

Raw plaintext is sent. Facets are never built. Inline highlighting is absent.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Messages/components/MessageInput.tsx`.
- Helper: `BlueskyKit/Sources/BlueskyComposer/FacetBuilder.swift` — already implemented for posts; reuse for messages.
- Lexicon: `chat.bsky.convo.defs#messageInput.facets` — confirm this is in `BlueskyCore/Chat.swift`. If not, add.
- Files to edit: `MessageThreadScreen.swift` (compose bar) and `MessageThreadViewModel.swift` (`sendMessage` to construct `MessageInput.facets`).

## Root cause

`MessageInput` carried only `text` + `embed`; no `facets` field, and
`MessageThreadStore.sendMessage` shipped the raw string straight through.
`FacetBuilder` did exist for posts but lived in `BlueskyComposer` (Layer 3),
which `BlueskyMessages` (also Layer 3) cannot import — strict layering forbids
sibling Layer-3 dependencies. It also lacked URL detection and any async
handle-resolution path: the post composer pre-resolves handles via the
autocomplete UI, but the DM compose bar has no autocomplete and needs the
on-the-fly resolution path RN runs through `rt.detectFacets(agent)`.

## Fix

Three coordinated changes:

1. **Move and extend `FacetBuilder`.** Relocated from
   `Sources/BlueskyComposer/FacetBuilder.swift` →
   `Sources/BlueskyKit/FacetBuilder.swift`. Both `BlueskyComposer` and
   `BlueskyMessages` already depend on `BlueskyKit`, so the post composer's
   call sites needed no change. Added URL detection (`NSDataDetector`,
   `http`/`https` only) and a new async `buildAsync(from:network:)` flavour
   that resolves each detected `@handle` through
   `com.atproto.identity.resolveHandle` in parallel via a `TaskGroup`.
   Handles that fail to resolve are silently dropped — RN's
   `stripInvalidMentions` does the same so the recipient never sees a
   broken pill.

2. **Lexicon:** added `facets: [RichTextFacet]?` to
   `BlueskyCore/Chat.swift#MessageInput`. Encoded only when non-empty so a
   plain-text DM still serializes without an empty array.

3. **Send path:** `MessageThreadStore.sendMessage` now calls
   `FacetBuilder.buildAsync` over the trimmed text and attaches the
   resulting facets to `MessageInput`. Mirrors RN's
   `MessagesList.tsx#onSendMessage` (RichText → detectFacets →
   stripInvalidMentions → `convoState.sendMessage({text, facets, …})`).

The mention regex was tightened to require a `.` (a bare `@alice` without a
domain isn't a valid AT Proto handle and the resolver would fail anyway), and
trailing dots are trimmed so `@alice.bsky.social.` resolves `alice.bsky.social`.

## Files changed

- `BlueskyKit/Sources/BlueskyKit/FacetBuilder.swift` — new file (moved from
  BlueskyComposer + extended with link detection, async build, handle
  resolution).
- `BlueskyKit/Sources/BlueskyComposer/FacetBuilder.swift` — deleted.
- `BlueskyKit/Sources/BlueskyCore/Chat.swift` — `MessageInput` gains
  `facets: [RichTextFacet]?`.
- `BlueskyKit/Sources/BlueskyMessages/MessageThreadStore.swift` — build
  facets via `FacetBuilder.buildAsync` and attach to `MessageInput`.

## Gotchas

- **Inline highlighting deferred.** RN highlights detected mentions/URLs/tags
  inline as the user types via its `RichText`-backed input. SwiftUI's plain
  `TextField` does not support attributed-string editing; an
  `AttributedString`-backed `TextEditor` with manual styling on every change
  would risk scope creep this pass. The build-on-send path is the priority
  per the issue spec and ships now; the live highlighting can be added in a
  follow-up using the same `FacetBuilder.hasDetectableFacets` /
  detection helpers — the byte-accurate ranges are already there.
- **Layer rules.** `FacetBuilder` had to leave `BlueskyComposer` because
  Layer-3 modules can't import each other. Lifting it to Layer-1
  (`BlueskyKit`) keeps a single source of truth without introducing a new
  module just for shared utilities.
- **`ResolveHandleResponse` is `nonisolated`.** BlueskyKit's
  `defaultIsolation` is `MainActor`, so the private `Decodable` response
  struct must be marked `nonisolated` to satisfy
  `NetworkClient.get<Response: Decodable & Sendable>`'s isolated-conformance
  requirement.
