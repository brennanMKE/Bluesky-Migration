## Description

The React Native composer renders the character counter as a circular progress ring (RN uses `ProgressCircle` / `ProgressPie`) that fills as the post grows toward the 300-character cap. The ring color shifts from primary to amber when fewer than 20 characters remain and to red when the post is over the limit. The numeric count appears inside or beside the ring.

The SwiftUI composer (`ComposerSheet.swift` lines 569ŌĆō574) shows only a plain `Text` count. There's no ring, no color transition, and the affordance reads as a generic counter rather than the at-a-glance gauge RN provides.

## Steps to reproduce

1. Open the composer and start typing.
2. Compare the character indicator to the RN reference.

## Expected behavior

A circular progress ring sized roughly 24ŌĆō28pt, drawn next to the Post button. As `text.count` grows toward 300:

- 0% ŌĆö empty ring outline.
- 0ŌĆō93%: ring fills primary blue.
- 93ŌĆō100% (last ~20 chars): ring fills amber + numeric remaining count appears inside the ring.
- Ōēź100%: ring fills red, count goes negative, Post button disables.

Match RN's color thresholds and stroke weight.

## Actual behavior

Plain text count. No visual gauge, no color escalation.

## Root cause

The SwiftUI composer's character counter (`charCounter` in
`ComposerSheet.swift`, lines 569ŌĆō574 at the time of filing) was a plain
`Text` showing `300 - text.count`, with only `.foregroundStyle` swapping
between secondary, orange, and red. RN renders a 30pt
`ProgressCircle`/`ProgressPie` from `react-native-progress` that fills
from primary blue toward the cap, switches to amber/red at the
thresholds, and shows the remaining count beside it
(`Bluesky-ReactNative/src/view/com/composer/char-progress/CharProgress.tsx`).
The plain-text version reads as a generic counter rather than the
at-a-glance gauge RN provides, and the additional-thread-post counter
in `threadPostSection(index:)` had the same issue.

## Fix

Added a reusable `CharProgressRing` view to `BlueskyUI` and replaced
both composer counters with it. The ring is built from
`Circle().trim(from: 0, to: clampedProgress).stroke(...)` rotated
`-90┬░` so the fill starts at 12 o'clock, with a faint
`theme.colors.border` track underneath. Implicit
`.animation(.smooth, value:)` modifiers smooth both the fill and the
color transitions as the user types.

**Color thresholds (matching RN exactly).**

- `progress < 0.93` (>20 chars remaining) ŌĆö `theme.colors.link`
  (Bluesky blue).
- `0.93 Ōēż progress Ōēż 1.0` (last 20 chars or fewer) ŌĆö `Color.orange`
  (the theme palette has no `warning` token yet; orange reads on
  light, dark, and dim surfaces).
- `progress > 1.0` (over the cap) ŌĆö `theme.colors.error`.

**Numeric display (matching RN exactly).** The remaining count is
hidden until the user is within 20 of the cap, then it appears beside
the ring (negative when over) using `theme.colors.textSecondary`,
escalating to `theme.colors.error` once over. RN does the same ŌĆö the
bare ring is shown until the warning band, at which point the numeric
context becomes useful.

The Post button's existing disabled state for `text.count > 300`
(driven by `ComposerViewModel.canPost` / `isOverLimit`) is unchanged.

Verified the previews on light, dark, and dim themes via
`#Preview` blocks in `CharProgressRing.swift` showing the four key
states (idle / mid-fill / boundary / amber / at-cap / over).

## Files changed

- `BlueskyKit/Sources/BlueskyUI/CharProgressRing.swift` ŌĆö new reusable
  ring component. Lives in `BlueskyUI` (Layer 2) so any feature module
  can adopt it; `BlueskyComposer` (Layer 3) already depends on
  `BlueskyUI`, so wiring it in required no `Package.swift` changes.
- `BlueskyKit/Sources/BlueskyComposer/ComposerSheet.swift` ŌĆö replaced
  `charCounter`'s plain `Text` with `CharProgressRing(count:max:)` and
  swapped the per-thread-post `Text` counter in `threadPostSection`
  for the same ring at a slightly smaller 20pt size.

## Gotchas

- **RN reference path.** The planning doc cited
  `view/com/composer/text-input/CharProgress.tsx`, but the live RN
  file is at `view/com/composer/char-progress/CharProgress.tsx`. Both
  paths show the same component; the planning doc is mildly stale.
- **No `warning` token in `BlueskyTheme`.** Today's palette defines
  `link`, `like`, `error`, and `success` ŌĆö no amber/warning. Used
  `Color.orange` as the fallback rather than minting a new token in
  this issue, to keep the change scoped. Issue #0040 (color palette
  refinement) is the right place to introduce a proper warning token
  if/when the design system grows one.
- **`textTertiary` is more secondary-looking than `textSecondary` on
  dim, but the ring uses `textSecondary` for the numeric.** RN uses
  the default text color; SwiftUI's `textSecondary` matches RN's read
  more closely than `textTertiary`, which is too washed out at small
  sizes against the dim background.
- **Clamped vs. raw progress.** The trim is clamped to `[0, 1]` so
  the ring tops out visually at full at 100% and stays full while
  over (matching RN's `Math.min((count - max) / max, 1)` pie behavior
  for the over state ŌĆö the visible fill never overshoots the ring).
  The color decision uses raw progress so the over-cap red kicks in
  at exactly `progress > 1.0` regardless of how far past the cap the
  user has typed.
- **Per-thread-post ring uses `size: 20`, not the default 24.** The
  per-post counter sits below a smaller `TextEditor` than the main
  one and shares row space with the remove button, so the slightly
  smaller ring keeps the row's vertical rhythm aligned with how the
  plain-text counter previously laid out.
