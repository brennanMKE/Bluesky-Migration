## Description

The React Native Language Settings screen has three distinct controls:

1. **App language** — a single-select picker for the UI's display language (locale), applied via the i18n layer.
2. **Primary post language** — single-select; the default `langs[]` value attached to new posts.
3. **Content languages** — **multi-select** of which languages the user wants to see in feeds; posts in unselected languages are filtered.

The SwiftUI `LanguageSettingsScreen` only shows a hardcoded single-select list of 12 languages used as Post Languages. App language and content-language multi-select are missing entirely.

## Steps to reproduce

1. Settings → Language.
2. Compare to RN — SwiftUI shows one list; RN shows three sections.

## Expected behavior

Refactor `LanguageSettingsScreen` into three sections:

- **App language** — picker bound to `Locale` override (applied via `.locale(...)` modifier or `Bundle.preferredLocalizations`).
- **Primary post language** — picker; this becomes the default `langs[0]` for new posts.
- **Content languages** — multi-select with checkboxes; consumed by feed-side filtering.

Persist via `UserDefaultsPreferencesStore` (locally) and `app.bsky.actor.putPreferences` (server-side, under `app.bsky.actor.defs#contentLanguagesPref`).

## Actual behavior

Single hardcoded post-languages list, no app-language picker, no content-language multi-select.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Settings/LanguageSettings/`.
- Lexicon: `app.bsky.actor.defs#contentLanguagesPref` and `#postLabelsPref`.
- File to edit: `BlueskyKit/Sources/BlueskySettings/LanguageSettingsScreen.swift`.

## Root cause

The original `LanguageSettingsScreen.swift` rendered a single hardcoded list
of 12 post-language toggles bound to `SettingsViewModel.postLanguages`. RN's
`LanguageSettings.tsx` exposes three logically distinct controls (App
language single-select, Primary post language single-select, Content
languages multi-select), and `BlueskyCore` did not yet model the two
language-related lexicon prefs (`contentLanguagesPref`, `postLanguagesPref`)
so server round-tripping was impossible. Both gaps had to be filled to reach
RN parity.

## Fix

- Added `BlueskyCore/Languages.swift` with `Language`, `AppLanguages.all`
  (40 entries mirroring RN's `APP_LANGUAGES`), and `ContentLanguages.all`
  (deduped subset of RN's `LANGUAGES` table by ISO 639-1 code, ~170 entries
  with display names matching RN). Includes `AppLanguages.sanitize` and
  `ContentLanguages.byCode` / `name(for:)` helpers.
- Extended `GetPreferencesResponse` to parse
  `app.bsky.actor.defs#contentLanguagesPref.languages` and
  `#postLanguagesPref.languages`, exposing them as `contentLanguages` and
  `postLanguages: [String]`.
- Added `PutPreferencesRequest.init(contentLanguages:)` and
  `init(postLanguages:)`, encoding the matching `$type` + `languages: []`
  payloads.
- Replaced `LanguageSettingsScreen.swift` with a three-section `Form`:
  - **App language** — `Picker` over `AppLanguages.all`, written locally
    via `PreferencesStore.set("settings.appLanguage")`.
  - **Primary language** — `Picker` over the alphabetically sorted
    `ContentLanguages.all`, written via
    `putPreferences(postLanguages: [code])` with optimistic update +
    rollback. Mirrored to `settings.primaryPostLanguage` for the composer.
  - **Content languages** — checkbox list (selected codes ∪ primary post
    language), written via `putPreferences(contentLanguages: [...])`.
- New `@Observable @MainActor` `LanguageSettingsViewModel` owning load and
  per-section writes, mirroring the pattern from `ThreadPreferencesScreen` /
  `FollowingFeedPreferencesScreen`.
- Updated `SettingsScreen` call site to pass `network` to
  `LanguageSettingsScreen`.
- Seeded `ComposerViewModel.selectedLanguage` from
  `settings.primaryPostLanguage` so new posts inherit the user's setting
  (matches RN's `useLanguagePrefs().primaryLanguage` default).

## Gotchas

- **App language doesn't take effect until next launch.** SwiftUI's
  `.environment(\.locale, ...)` only re-styles a subtree; the underlying
  Lingui-equivalent string lookups read `Bundle.main.preferredLocalizations`
  at startup. The setting is persisted under `settings.appLanguage`; the app
  shell can pick it up on `applicationDidFinishLaunching` to apply on the
  next launch (or via `AppleLanguages` user defaults). This matches RN's
  iOS behavior — RN shows an app-restart toast after changing the App
  language setting; we omit the toast for now.
- **Stale `SettingsViewModel.postLanguages`.** The legacy field on
  `SettingsViewModel` is no longer read by the screen but remains for
  backwards compat (it's still load/saved against
  `settings.postLanguages`). It is unused by the composer (which now reads
  `settings.primaryPostLanguage`). A follow-up cleanup can drop the field
  and migrate any reads — out of scope here.
- **Content-language picker shows union, not full table.** Mirroring RN's
  `LanguageSettings.tsx` `possibleLanguages` derivation, the multi-select
  shows the user's currently selected codes plus their primary post
  language. RN also adds a "Recently used" set tracked by an in-memory
  `recentLanguages` state and an "Add more languages…" dialog backed by
  `LanguageSelectDialog`. The dialog is deferred — users can add languages
  by switching the Primary picker to introduce a new code and then toggling
  it. Filing the dialog feature as a follow-up is reasonable; not blocking
  this issue.
