## Description

`NotificationSettingsScreen` is a single screen with 6 toggles (likes, reposts, follows, mentions, replies, quotes). The React Native equivalent has:

- **Push permission gate** at the top: detects iOS notification permissions; if not granted, shows a "Push notifications are off — turn them on in Settings" admonition + a deep link to system Settings.
- **Per-type sub-screens** for each notification kind (Likes, Reposts, Mentions, Replies, Quotes, Follows, Subscribed Posts, Verifications). Each sub-screen has separate toggles for **push** and **email** delivery and an "Include from" filter (Anyone / People I follow / Verified accounts).
- **Email notifications** as a separate top-level group with their own toggles (digest, security alerts).
- **DM notifications** toggle.

## Steps to reproduce

1. Settings → Notifications.
2. Compare to RN: SwiftUI is one flat screen, RN is a hub with 8+ sub-screens.

## Expected behavior

Refactor `NotificationSettingsScreen` into a hub with NavigationLinks to per-type sub-screens. Each sub-screen has push toggle, email toggle, and "Include from" picker. Add a top-level push-permission admonition that detects iOS authorization status and links to system Settings when denied.

## Actual behavior

One flat screen, six toggles, no permission detection, no email options.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Settings/NotificationSettings/` and the per-type screens.
- iOS push permission: `UNUserNotificationCenter.current().getNotificationSettings`.
- Files to edit/create: `BlueskyKit/Sources/BlueskySettings/NotificationSettingsScreen.swift` (turn into hub) + per-type sub-screens.
- Endpoints: `app.bsky.notification.putPreferences` (per-type filters live here under `app.bsky.notification.defs#preferences`).

## Root cause

`NotificationSettingsScreen` shipped as a stub during the original Settings module pass — six flat `Toggle("Likes", isOn: …)` rows wired straight to `SettingsViewModel` `Bool` properties. RN's actual screen is a hub of nine `NavigationLink` rows, each pushing a per-type sub-screen with push/email channel toggles and an "Include from" filter, plus a push-permission admonition gated on `UNUserNotificationCenter.getNotificationSettings()`. None of the hub structure or permission UI existed.

## Fix

- **`NotificationSettingsScreen.swift`** — replaced the flat toggle list with a hub. Sections in RN order: Activity (Likes, New followers, Replies, Mentions, Quotes, Reposts, Activity from others) → Verifications → Direct messages → Email (referral). Each row shows a condensed channel/filter preview ("Push, Email, People you follow", "Off", …) mirroring RN's `SettingPreview` helper.
- **Push-permission admonition** — top-of-screen banner row that reads `UNUserNotificationCenter.current().notificationSettings()` on iOS and renders only when status is `.denied` or `.notDetermined`. Tapping the row opens `UIApplication.openSettingsURLString`. Refresh fires on view appear and on `scenePhase == .active` so returning from Settings updates the gate. macOS short-circuits to `.authorized` (the system handles permission via the app delegate; the admonition has no place there).
- **`NotificationTypeSettingsScreen.swift`** (new) — single parameterized sub-screen used by every kind. Header (icon, title, subtitle), Channels section (Push + Email toggles), and an optional From section (Anyone / People you follow / Verified accounts) for kinds where `supportsIncludeFilter` is true. Filter row is disabled when both channels are off, matching RN's `disabled={channels.length === 0}`.
- **`NotificationPreferencesStore.swift`** (new) — local-only persistence on top of `PreferencesStore`. Holds `NotificationKind` enum (the source of titles, subtitles, SF Symbols, and filter eligibility), `IncludeFilter` enum, and a `NotificationTypePreference` value type (push, email, include).

## Files changed

- `BlueskyKit/Sources/BlueskySettings/NotificationSettingsScreen.swift` — refactored from flat toggles into hub.
- `BlueskyKit/Sources/BlueskySettings/NotificationTypeSettingsScreen.swift` — new per-type sub-screen.
- `BlueskyKit/Sources/BlueskySettings/NotificationPreferencesStore.swift` — new local store + model types.
- `BlueskyKit/Sources/BlueskySettings/SettingsViewModel.swift` — exposed `preferences` package-internally so the hub can build a `NotificationPreferencesStore`; documented the legacy `notify*` bools as non-authoritative mirrors.

## Gotchas

- **Local-only persistence (deferral).** `app.bsky.notification.putPreferences` and the `app.bsky.notification.defs#preferences` lexicon types are **not** modelled in `BlueskyCore` (no `Preference` / `FilterablePreference` / `Preferences` records). Per-type settings persist through `PreferencesStore` (UserDefaults under the app target). Wiring the server-side mutation is a follow-up — likely a new `NotificationPreferences` model in `BlueskyCore` plus a `putPreferences` call from the sub-screen's `onChange`.
- **DM toggle is local-only too.** Server-side, RN encodes DM gating on `chat.bsky.actor.declaration.allowIncoming`; that record write isn't plumbed yet, so the row stores a local push/email pair for now and the section footer points the user at bsky.app.
- **Email notifications (digest, security alerts) deferred.** RN reaches these through bsky.app's web settings, not a stable public lexicon. We surface a single "Email notifications" row that opens a referral sheet (matches the pattern from #0113 / #0114).
- **"Verified accounts" filter is forward-compatible only.** RN's `FilterablePreference.include` ships as `'all' | 'follows'` today. We persist `verified` locally so the radio UI can offer it, but a future server write will need lexicon support before that value is round-tripped to bsky.social.
- **macOS permission UI.** The admonition is iOS-only because macOS handles notification authorization via the app delegate prompt and a per-app pane in System Settings; surfacing a banner there would be redundant. We deep-link to the Notifications pane via `x-apple.systempreferences:` if a future change wants to expose it.
- **Default isolation = `MainActor`.** New value types (`NotificationKind`, `IncludeFilter`, `NotificationTypePreference`, `NotificationPreferencesStore`) are explicitly `nonisolated` so their `Codable` conformances satisfy the `Sendable` parameter on `PreferencesStore.set`/`get`.
