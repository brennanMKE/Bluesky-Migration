## Description

In React Native, long-pressing (or right-clicking on web) any message bubble opens a context menu with: **React** (emoji picker), **Copy**, **Translate**, **Delete** (own messages only), and **Report** (others' messages). The SwiftUI `MessageBubble` has no gesture handlers and no menu — taps and long-presses do nothing. There's no way to copy a message, delete a sent mistake, or report abuse from the thread surface.

## Steps to reproduce

1. Open any message thread in the SwiftUI app.
2. Long-press a message bubble.
3. Observe nothing happens.

## Expected behavior

Long-press (iOS) or right-click (macOS) on any bubble opens a context menu:

- **React** — opens reactions picker (covered by #0110 if filed).
- **Copy** — copies message text to pasteboard.
- **Translate** — opens system Translate sheet (iOS 17+) or a translate-to-clipboard fallback.
- **Delete** — only on own messages; calls `chat.bsky.convo.deleteMessageForSelf` (or appropriate endpoint) and removes locally.
- **Report** — only on others' messages; opens existing `ReportDialog` from `BlueskyModeration` with the message as subject.

iOS uses `.contextMenu { … }`. macOS uses the same modifier — SwiftUI bridges to AppKit's right-click automatically.

## Actual behavior

No menu at all on long-press / right-click.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Messages/components/MessageItem.tsx` (search for `MessageContextMenu`).
- Endpoints needed: `chat.bsky.convo.deleteMessageForSelf` (already in `BlueskyCore`?), translation can use `TranslationKit` on iOS 17+ / system translation services.
- File to edit: `BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift` (the `MessageBubble` view).
- Pairs naturally with reactions (#0110) since both attach to the bubble.

## Root cause

`MessageBubble` in `MessageThreadScreen.swift` had no gesture handlers and no
menu modifier — long-press on iOS and right-click on macOS were silently
absorbed by the bubble's background. The view-model also had no
`deleteMessage` plumbing and `BlueskyCore` had no request type for
`chat.bsky.convo.deleteMessageForSelf`, so even if the menu had existed,
there was no path to act on Delete.

## Fix

- Added `DeleteMessageForSelfRequest` (and a `DeleteMessageForSelfResponse`
  alias for the returned tombstone view) to `BlueskyCore/Chat.swift`.
- Added `deleteMessage(_:convoId:)` to `MessageThreadStoring` /
  `MessageThreadStore` — optimistic local removal with revert-on-failure,
  matching RN's `agent.ts#deleteMessage` shape (RN uses a `deletedMessages`
  set; ours uses straight removal since the SwiftUI rendering does not yet
  show tombstones).
- Added `MessageThreadViewModel.deleteMessage(_:)` as a thin pass-through.
- Wrapped `MessageBubble`'s root `HStack` in `.contextMenu { … }` with the
  RN ordering: **Translate**, **Copy message text**, divider, **Delete for
  me** (own messages only). The Translate item drives
  `.translationPresentation(isPresented:text:)` (Translation framework,
  available on macOS 15 / iOS 18 — both are this project's deployment
  baseline).
- Copy uses an inline `UIPasteboard` / `NSPasteboard` block that mirrors
  the existing PostCard "Copy post text" pattern (#0047).
- The delete-confirmation prompt lives at the screen level
  (`pendingDeleteMessage` state + `.confirmationDialog`), with copy that
  matches RN's `Prompt.Basic` ("Delete message" / "Are you sure you want to
  delete this message? The message will be deleted for you, but not for the
  other participants.").

## Files changed

- `BlueskyKit/Sources/BlueskyCore/Chat.swift` — new
  `DeleteMessageForSelfRequest` + `DeleteMessageForSelfResponse` typealias.
- `BlueskyKit/Sources/BlueskyMessages/MessageThreadStore.swift` — new
  `deleteMessage(_:convoId:)` in the protocol and the store.
- `BlueskyKit/Sources/BlueskyMessages/MessageThreadViewModel.swift` —
  pass-through `deleteMessage(_:)`.
- `BlueskyKit/Sources/BlueskyMessages/MessageThreadScreen.swift` —
  `.contextMenu` on `MessageBubble`, `.translationPresentation` for
  Translate, screen-level `.confirmationDialog` for Delete, `onDeleteRequested`
  callback bridging the bubble to the screen, and Translation/UIKit/AppKit
  imports.

## Gotchas

- **Report** is intentionally omitted from the menu for now. The existing
  `BlueskyModeration.ReportDialog` only accepts `account` or `record`
  subjects (`ReportSubjectKind`); chat messages need a `view: 'message'`
  subject (RN passes `{view: 'message', convoId, message}`). Wiring this
  requires a new `ReportSubjectKind.chatMessage(convoId:messageView:)`
  case, an updated `BlueskyMessages` -> `BlueskyModeration` dependency
  edge, and the `chat.bsky.convo.report` (or equivalent) endpoint plumbed
  through `MessageThreadStore`. Track as a follow-up.
- **React** (emoji reactions) is intentionally omitted — it belongs to
  #0110 (reactions). Adding a disabled stub would clutter the menu in the
  meantime.
- The Translate item is unconditional (no `#available` gate) because the
  package's deployment baseline (macOS 15 / iOS 18) already includes the
  Translation framework's `.translationPresentation` modifier (introduced
  in macOS 14.4 / iOS 17.4).
- RN deletes a message by adding the id to a `deletedMessages` set so the
  bubble can render a "Message deleted" tombstone. Our store removes the
  message outright; we will revisit if/when the SwiftUI side renders
  `DeletedMessageView` rows in-thread.
