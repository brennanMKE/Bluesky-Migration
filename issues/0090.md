## Description

The Search tab still uses a giant left-aligned "Search" headline taking ~25% of the screen — the same anti-pattern that #0072 (Home) and #0077 (Notifications) already fixed for their respective tabs. To match the React Native reference and feel consistent with Home and Notifications, Search needs the same slim top-bar treatment: hamburger / inline title / appropriate trailing action.

## Attachments

![Search screen — giant headline; needs slim top bar](search.png)

## Steps to reproduce

1. Run the SwiftUI iOS app.
2. Tap the Search tab.
3. Observe the giant "Search" headline above the search field.

## Expected behavior

A slim top bar flush below the status bar:
- **Left**: hamburger button — opens the same shared drawer used by Home (#0072) and Notifications (#0077).
- **Center**: "Search" title in a single line (regular weight; matches the wordmark sizing on Home / Notifications).
- **Right**: leave empty, or use whatever trailing action RN puts there (likely none — the search field handles all the discovery affordances).

The search field stays where it is, sitting directly below the slim top bar.

## Actual behavior

A giant `.large` "Search" navigation title takes ~25% of the vertical space above the search field.

## Implementation notes

- Reuse `BlueskyTopBar` (added in #0072) — three slots, drop the trailing slot or leave it blank.
- Wire the hamburger to the existing `showDrawer` state in `MainTabView` (the drawer is already shared across Home and Notifications).
- Search is already iOS+macOS — this change is iOS only (macOS keeps its existing toolbar). Gate with `#if os(iOS)`.

## Acceptance

- iOS Search screen has a slim top bar (hamburger / "Search" / empty) flush below the status bar.
- Hamburger opens the shared drawer.
- macOS Search unchanged.
- Combined with #0088 (top safe-area gap), the bar should fill behind the status bar.

## Root cause

`SearchScreen` declared `.navigationTitle("Search")` with no display-mode
override, so the iOS system large-title navigation bar rendered a
~25%-of-screen "Search" headline above the search field — the same
anti-pattern that #0072 (Home) and #0077 (Notifications) already removed
from their tabs. The compact iOS layout in `MainTabView` only mounted a
`BlueskyTopBar` on the Home and Notifications tabs, so Search fell back
to system chrome with no shared drawer trigger.

## Fix

Suppressed the system large title on iOS in `SearchScreen` by gating
`.navigationTitle("Search")` behind `#if os(macOS)` and applying
`.navigationBarTitleDisplayMode(.inline)` on iOS — macOS keeps its
sidebar-friendly title; iPhone and iPad regular get a flat nav bar that
the parent's slim top bar visually replaces.

In `MainTabView.iosCompactLayout`, added a `.search` case to the
top-bar `switch` that mounts a new `iosSearchTopBar` view. The Search
top bar uses the existing `BlueskyTopBar` slot API: a
`line.3.horizontal` hamburger button that toggles the same
`showDrawer` state Home and Notifications use (so the slide-in drawer
overlay added in #0072 is shared across the tab shell — no duplicate
state, no separate overlay), a centred plain `Text("Search")` in
regular weight at 20pt to match Notifications' title sizing, and an
empty trailing slot.

**Trailing-slot decision.** RN's Search shell
(`Bluesky-ReactNative/src/screens/Search/Shell.tsx`, lines 339–365)
puts a `Layout.Header.MenuButton` on the leading edge, the title
("Search" or "Explore") left-aligned, and either a
`SearchLanguageDropdown` (only when the SearchResults sub-screen is
showing filters) or an empty `Layout.Header.Slot` on the trailing
edge. For the base discovery / typeahead view this top bar covers the
trailing slot is blank in RN, so SwiftUI matches that with
`EmptyView()`. Surfacing the language-filter dropdown on the
SearchResults sub-screen is out of scope for this issue and would be
a follow-up tied to the SwiftUI equivalent of `SearchResults.tsx` if
filters land there later.

**Safe-area bleed.** With the slim bar in place, the status-bar zone
above Search now sits over the chrome's color rather than the dead
black band shown in the #0088 attachments — `BlueskyTopBar`'s
background `.ignoresSafeArea(edges: .top)` modifier (landed in #0088)
carries through automatically, no per-screen safe-area work needed.

## Files changed

- `BlueskyKit/Sources/BlueskySearch/SearchScreen.swift` — gate
  `.navigationTitle("Search")` to macOS; on iOS use
  `.navigationBarTitleDisplayMode(.inline)` so the parent's
  `BlueskyTopBar` is the visible top chrome and there's no giant
  "Search" headline.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` — added new
  `iosSearchTopBar` view (hamburger / "Search" / empty), extended the
  `iosCompactLayout` top-bar `switch` to mount it on the `.search`
  case. The hamburger binds to the same `showDrawer` state Home and
  Notifications use.

## Gotchas

- **Trailing slot uses `EmptyView()`, not a hidden / spacer view.**
  `BlueskyTopBar` allots a fixed `slotWidth` to the trailing slot
  regardless of content, so the title remains centred even with an
  empty view in that position. Swapping to a `Spacer()` or an explicit
  blank frame is unnecessary — `EmptyView` is the idiomatic SwiftUI
  way to declare "no content here" inside a builder.
- **Drawer state was already shared — no lifting needed.** `showDrawer`
  already lives on `MainTabView` from #0072, the drawer overlay is
  attached once at the `iosCompactLayout` level (not per-tab), and
  the drawer's "Search" row already routes via `selectedTab = .search`.
  The Search hamburger just had to bind the same `@State`.
- **Centre text styling matches #0077, not #0072.** `BlueskyWordmark`
  is the heavy brand-blue mark reserved for Home; per-screen titles on
  Notifications and Search use a plain `Text` at 20pt regular in
  `theme.colors.textPrimary` so the bar reads as a screen title, not a
  logo. This keeps the visual weight of the bar consistent across
  tabs.
