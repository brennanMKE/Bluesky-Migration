## Description

The React Native client lets a user reset a forgotten password directly inside the app: **`ForgotPasswordForm`** (enter email, request reset token), **`SetNewPasswordForm`** (enter token + new password) and **`PasswordUpdatedForm`** (confirmation). The entry point is a "Forgot?" link inline with the password field in `LoginForm.tsx` (lines 256–266).

The SwiftUI `LoginView` has no "Forgot password?" link and no recovery screens at all. A user who has forgotten their password is locked out unless they leave the app and reset on bsky.app.

## Steps to reproduce

1. Launch the SwiftUI app on the login screen.
2. Look for a "Forgot password?" / "Forgot?" link near the password field.
3. Observe there is none.

## Expected behavior

- A "Forgot?" link to the right of the password field's label (or below the password field), matching RN's placement.
- Tapping it opens the recovery flow:
  1. **Request screen**: email field + hosting provider picker → calls `com.atproto.server.requestPasswordReset`.
  2. **Set new password screen**: token field (the user pastes the code from the email) + new-password field with confirmation → calls `com.atproto.server.resetPassword`.
  3. **Confirmation screen**: "Password updated" message + "Sign in now" CTA returning to `LoginView` with the email pre-filled.
- Each screen exposes inline error display for invalid tokens / weak passwords / unknown emails.

## Actual behavior

No "Forgot password?" link on `LoginView`; no recovery screens exist anywhere in `BlueskyAuth`.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Login/ForgotPasswordForm.tsx`, `SetNewPasswordForm.tsx`, `PasswordUpdatedForm.tsx`. `index.tsx` orchestrates the form transitions.
- AT Proto endpoints needed: `com.atproto.server.requestPasswordReset`, `com.atproto.server.resetPassword`. Lexicons may not exist yet in `BlueskyCore/Auth.swift`.
- File to edit: `BlueskyKit/Sources/BlueskyAuth/LoginView.swift` (add the link + sheet/navigation), and create new `ForgotPasswordView.swift` / `ResetPasswordView.swift`.

## Root cause

The SwiftUI port of the login screen never exposed a way to recover a
forgotten password. RN ships a three-screen flow inside the same Login
orchestrator (`ForgotPasswordForm`, `SetNewPasswordForm`,
`PasswordUpdatedForm`) reached via a "Forgot?" link beside the password
field; the Swift `LoginView` shipped without the link and without any of
the recovery screens, so a locked-out user had to leave the app and
reset on bsky.app.

The two AT Protocol lexicons that drive the flow
(`com.atproto.server.requestPasswordReset` and
`com.atproto.server.resetPassword`) were also not yet defined in
`BlueskyCore/Auth.swift`.

## Fix

- Added `RequestPasswordResetRequest` / `ResetPasswordRequest` to
  `BlueskyCore/Auth.swift`. Both endpoints return empty bodies on
  success, so callers reuse the existing `EmptyResponse` /
  HTTP-validation path.
- Added `PasswordResetValidation.checkAndFormatResetCode` mirroring RN's
  base32 (A-Z, 2-7) `XXXXX-XXXXX` formatter, including the auto-insert
  of the hyphen when the user pastes a 10-character compact code.
- Extended `SessionManager` with
  `requestPasswordReset(serviceURL:email:)` and
  `resetPassword(serviceURL:token:password:)`. Both go straight to
  `URLSession`, matching the unauthenticated `describeServer` /
  `createAccount` pattern (the user is signed out at this point, so the
  authenticated `NetworkClient` is wrong).
- New `ForgotPasswordFlowView` orchestrator in `BlueskyAuth` driven by
  an `@Observable @MainActor ForgotPasswordModel`, with three step views
  (`ForgotPasswordRequestView`, `ForgotPasswordSetView`,
  `ForgotPasswordUpdatedView`) that mirror the RN form-by-form layout —
  hosting provider picker, email field, reset-code field, new-password
  field, "Sign in now" CTA. Inline field-level error highlighting via
  `fieldErrorKey` mirrors RN's red-message-under-field pattern.
- `LoginView` gained a "Forgot?" caption-weight link inline with the
  password label (matching RN's `LoginForm.tsx` placement at lines
  256–266) and a `.sheet` that hosts `ForgotPasswordFlowView`. On
  completion the email used for the reset is fed back into the sign-in
  form so the user can sign in immediately.

## Files changed

- `BlueskyKit/Sources/BlueskyCore/Auth.swift` — new request types and
  `PasswordResetValidation`.
- `BlueskyKit/Sources/BlueskyAuth/SessionManager.swift` — new
  `requestPasswordReset` / `resetPassword` methods.
- `BlueskyKit/Sources/BlueskyAuth/ForgotPasswordFlowView.swift` — new
  orchestrator + three step views + `ForgotPasswordModel` (new file).
- `BlueskyKit/Sources/BlueskyAuth/LoginView.swift` — inline "Forgot?"
  link, sheet presentation, completion handler that pre-fills the email
  back into the login form.

## Gotchas

- Reset codes are RFC 4648 **base32** (A-Z, 2-7) — not arbitrary
  alphanumerics. Using `[A-Z0-9]` would silently accept invalid codes
  and reject valid ones containing `0`, `1`, `8`, or `9`. Confirmed
  against RN's `lib/strings/password.ts`.
- `BlueskyAuth` only depends on `BlueskyKit` and `BlueskyCore`; it does
  not depend on `BlueskyUI` (Layer 2 sibling), so the `.blueskyTheme()`
  environment helper isn't available here. The flow uses the same
  `.quaternary.opacity(0.5)` field background style established in
  `SignupStepInfoView` to stay visually consistent with the existing
  signup flow.
- The reset endpoints are unauthenticated, so they must go through a
  bare `URLSession` call rather than the `NetworkClient` (which
  attaches the access-token bearer). This is the same pattern login
  itself uses.
