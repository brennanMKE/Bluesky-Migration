## Description

The Notifications tab currently shows a giant left-aligned "Notifications" headline that takes ~25% of the screen above the list — same anti-pattern as the Home tab. The React Native reference uses a slim top bar with a hamburger button (left), centered "Notifications" title, and a gear icon (right) that opens the notification settings screen.

This is a follow-up to the iOS Notifications parity audit in #0069.

## Scope

- **Drop the large "Notifications" navigation title** on iOS. macOS layout is unchanged.
- **Add a slim top bar** flush below the status bar:
  - **Left**: hamburger button (opens the same drawer added in #0072).
  - **Center**: "Notifications" title in a single line.
  - **Right**: gear icon (`gearshape`) → pushes notification settings.
- The notification settings screen already exists somewhere in `BlueskySettings`; if not, file a follow-up to build it. For now the gear can route to a placeholder if the destination isn't ready, so this issue isn't blocked.

## Implementation notes

- Reuse the `BlueskyTopBar` component introduced in #0072.
- The hamburger should open the same drawer instance as Home — share the drawer presentation state across the app shell.

## Root cause

`NotificationsScreen` declared `.navigationTitle("Notifications")` with
no display-mode override, so the iOS system large-title navigation bar
rendered a ~25%-of-screen headline above the notification list — the
same anti-pattern that #0072 removed from Home. The compact iOS layout
in `MainTabView` only mounted a `BlueskyTopBar` on the Home tab, so the
Notifications tab fell back to the system chrome with no shared drawer
trigger and no gear shortcut.

## Fix

Suppressed the system large title on iOS in `NotificationsScreen` by
gating `.navigationTitle("Notifications")` behind `#if os(macOS)` and
applying `.navigationBarTitleDisplayMode(.inline)` on iOS — macOS keeps
its sidebar-friendly title; iPhone and iPad regular get a flat nav bar
that the parent's slim top bar visually replaces.

In `MainTabView.iosCompactLayout`, switched the top-bar mount from a
single `if activeTab == .home` block to a `switch` that mounts
`iosHomeTopBar` for Home and the new `iosNotificationsTopBar` for
Notifications. The Notifications top bar uses the existing
`BlueskyTopBar` slot API: a `line.3.horizontal` hamburger button that
toggles the same `showDrawer` `@State` Home uses (so the slide-in drawer
overlay added in #0072 is shared across the tab shell — no duplicate
state, no separate overlay), a centred plain `Text("Notifications")` in
regular weight at 20pt to match the wordmark sizing, and a `gearshape`
trailing button that flips a new `showNotificationSettings` flag.

The flag drives a `.navigationDestination` attached to the
`.notifications` case in `tabContent(...)`. The destination uses the
existing `placeholderScreen("Notification Settings", systemImage: "bell.badge")`
helper because `BlueskySettings.NotificationSettingsScreen` is
module-internal — exposing it publicly is a follow-up. Added the new
flag to `resetTransientNavState()` so it clears when the user switches
tabs.

## Files changed

- `BlueskyKit/Sources/BlueskyNotifications/NotificationsScreen.swift` —
  gate `.navigationTitle("Notifications")` to macOS; on iOS use
  `.navigationBarTitleDisplayMode(.inline)` so the parent's
  `BlueskyTopBar` is the visible top chrome and there's no giant
  "Notifications" headline.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` — added
  `showNotificationSettings` state, new `iosNotificationsTopBar` view,
  switched the compact-layout top-bar mount from an `if` to a `switch`
  that handles `.home` and `.notifications`, attached
  `.navigationDestination(isPresented: $showNotificationSettings)` to
  the `.notifications` case routing to `placeholderScreen`, and added
  the new flag to `resetTransientNavState()`.

## Gotchas

- **Gear destination is a placeholder, not the real settings screen.**
  `BlueskySettings.NotificationSettingsScreen` exists but is declared
  `internal` and is constructed inside `SettingsScreen` from a
  `SettingsViewModel` that's not exposed at the module boundary either.
  Wiring the gear directly into it would require widening API surface
  in `BlueskySettings` (and threading a fresh `SettingsViewModel` down
  through `MainTabView`). The dispatch instructions explicitly forbid
  building the settings screen as part of this issue, so the gear
  routes to the existing `placeholderScreen("Notification Settings", systemImage: "bell.badge")`
  helper. Follow-up: file a separate issue to make
  `NotificationSettingsScreen` public (or to add a thin
  `NotificationSettingsEntry` wrapper that constructs its own
  `SettingsViewModel` on demand) and swap the placeholder for that.
- **Drawer state was already shared — no lifting needed.** `showDrawer`
  already lives on `MainTabView` from #0072, and the drawer overlay
  (`iosDrawerOverlay`) is attached once at the `iosCompactLayout`
  level (not per-tab), so the Notifications hamburger just had to bind
  the same `@State` and the slide-in works the same way. No
  refactor / `EnvironmentObject` was required.
- **Centre text styling deliberately diverges from `BlueskyWordmark`.**
  `BlueskyWordmark` is brand-blue, heavy-weight "Bluesky" branding —
  appropriate for Home but visually loud for a per-screen title. The
  Notifications top bar uses a plain `Text("Notifications")` at 20pt
  regular in `theme.colors.textPrimary` so the bar reads as a screen
  title, not a logo. This matches the dispatch's explicit
  "regular weight; match the Home wordmark sizing" guidance and keeps
  the visual weight of the bar consistent with Home's chrome.
- **`navigationDestination` lives on the `.notifications` case, not at
  layout root.** Attaching the destination at `iosCompactLayout` level
  would require iOS-only viewmodifiers that fight the existing
  `.id(activeTab)` reset; placing it inside `tabContent(.notifications)`
  follows the pattern Home uses for `showSavedFeeds` / `showMyFeeds`
  and integrates cleanly with the existing per-tab navigation reset.
