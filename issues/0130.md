## Description

`VideoFeedView` renders a vertical paging `TabView` of `AVPlayer`-backed videos with an avatar, handle, bio, and post text overlay. It is missing the interactive controls the React Native video feed has (TikTok-style):

- **Mute toggle** (top-right or bottom-right) — videos default muted; users tap to unmute.
- **Like / repost / reply / share** action buttons stacked vertically on the right edge, with counts.
- **Comments / replies** entry — opens the post's thread.
- **Avatar/handle** is tappable to navigate to the author's profile.
- **Scrubber** at the bottom showing playback progress.
- **Tap to pause** in the center — RN supports a long-press / tap-to-pause gesture overlay.

Without these, the video feed is a passive viewer with no engagement affordances.

## Steps to reproduce

1. Open the video feed.
2. Look for like / comment / share buttons or a mute toggle.
3. Observe none.

## Expected behavior

Add to `VideoPostPage`:

- A muted/unmuted icon button (top-right corner) tied to the player's `isMuted`.
- A right-aligned vertical stack of action buttons (like, repost, comment count, share) with counts; reuse the post mutation paths from `FeedViewModel`.
- Avatar + handle row tappable into `ProfileScreen`.
- Bottom scrubber bar showing video progress (read `player.currentTime` via `addPeriodicTimeObserver`).
- Single-tap on the player surface toggles play/pause.

## Actual behavior

Player + static avatar/text only; no interactive overlays.

## Root cause

`VideoPostPage` was a thin wrapper around `VideoPlayer` with a single
avatar/handle/caption strip layered on top. The RN immersive video feed
(`Bluesky-ReactNative/src/screens/VideoFeed/index.tsx`, `Overlay`) layers a
gradient with the author header, the standard post action controls, a
mute toggle, a tap-to-pause hit area, and a thin scrubber driven by the
player's `timeUpdate` events. None of those affordances had a SwiftUI
counterpart, so the feed was a passive viewer with no engagement paths.

## Fix

Rewrote `VideoPostPage` as a `ZStack` whose lowest layer is the
`VideoPlayer` and whose upper layers carry:

- A top-trailing mute button bound to a `@Binding` so the preference
  persists across pages within the session (`speaker.slash.fill` /
  `speaker.fill`, RN parity for icon set).
- A right-edge action stack ordered comments → repost → like → share to
  match RN's `PostControls`. Like and repost route through
  `FeedViewModel.toggleLike(_:)` / `toggleRepost(_:)` so the regular feed
  and the video feed share optimistic state.
- An author/caption overlay in the bottom-left whose tappable region
  invokes a new `onProfileTap: (DID) -> Void` callback so the parent
  navigation stack drives `ProfileScreen`.
- A comments entry that calls `onPostTap: (PostView) -> Void` to push
  `ThreadView` for the post.
- A 2 pt full-width scrubber backed by `AVPlayer.addPeriodicTimeObserver`
  (10 Hz). The token is captured in `@State` and removed in
  `onDisappear` so the closure does not outlive the view.
- A full-bounds tap gesture toggles play/pause and briefly flashes a
  centered play / pause SF Symbol, mirroring RN's tap feedback.
- Looping playback via `AVPlayerItemDidPlayToEndTime` (RN sets
  `player.loop = true`).
- A new `autoplay: Bool = true` init parameter so the parent (or the
  Settings hub via `SettingsViewModel.autoplayVideo`) can gate auto-start.

The new `progress` state is mutated from the periodic observer's main
queue closure inside a `Task { @MainActor in … }` to satisfy strict
concurrency without coupling the closure to actor isolation up front.

## Files changed

- `BlueskyKit/Sources/BlueskyFeed/VideoFeedView.swift` — `VideoPostPage`
  reworked into a layered `ZStack` with mute, like, repost, comments,
  share, scrubber, tap-to-pause, autoplay gating, and avatar/comment
  navigation callbacks (`onProfileTap`, `onPostTap`).

## Gotchas

- `AVPlayer.addPeriodicTimeObserver(forInterval:queue:)` returns a
  `protocol-typed` token; it must be stored and passed back to
  `removeTimeObserver(_:)`. Forgetting that retains the observer's
  closure (and the view's state) for the lifetime of the player.
- Looping HLS assets on AVPlayer is hand-rolled — observe
  `AVPlayerItemDidPlayToEndTime` and `seek(to: .zero)`. The RN reference
  gets this for free from `expo-video`'s `player.loop = true`.
- The system `VideoPlayer` shows its own playback controls; we set
  `.disabled(true)` so our tap gesture and overlays own the interaction.
