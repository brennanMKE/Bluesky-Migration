## Description

The React Native About screen contains, in addition to the version and legal links the SwiftUI version has:

- **System Info** section: device model, OS version, build number, build channel (release / TestFlight / dev), session DID — copyable for support tickets.
- **Status Page** link to the Bluesky status site.
- **Cache management**: "Clear cache" button that wipes the cache store (matches "Clear data" on Android / "Reset" on iOS).
- **Dev / Debug tools** (only visible in dev builds): toggles for verbose logging, force a specific feature flag, etc.
- **Build details** beyond version: build hash, build date.

The SwiftUI `AboutScreen` shows only Version, ToS, Privacy, Community Guidelines, Source Code.

## Steps to reproduce

1. Settings → About.
2. Compare to RN.
3. Observe missing sections.

## Expected behavior

Extend `AboutScreen.swift`:

- Add **System Info** section (read-only rows with the values listed above; long-press to copy).
- Add **Status Page** link.
- Add **Clear Cache** action that calls `cacheStore.evictAll()`.
- Gate dev tools behind a build configuration check (e.g. `#if DEBUG`).

## Actual behavior

Minimal About screen with version + 4 legal links.

## Notes

- RN reference: `Bluesky-ReactNative/src/screens/Settings/AboutSettings.tsx`.
- Build info: `Bundle.main.infoDictionary` for version + build; `ProcessInfo.processInfo.operatingSystemVersionString` for OS; `UIDevice.current.model` (iOS) / `Host.current().localizedName` (macOS) for device.
- File to edit: `BlueskyKit/Sources/BlueskySettings/AboutScreen.swift`.

## Root cause

`AboutScreen` was a stub: only the version label and four legal links. RN's `AboutSettings.tsx` additionally exposes the build identity (so users can attach it to support tickets), the public Status Page, an image-cache clear action, and a developer-mode toggle gated to internal builds. None of that was ported when BlueskySettings was first scaffolded.

## Fix

Rewrote `AboutScreen` to add four new sections while keeping the existing legal links intact:

- **System Info** — read-only rows for App Version, Build, Build Channel (Debug / Release), OS Version, Device, and Account DID (only when `currentAccount` is non-nil). Each row is a tappable button that copies its value to the clipboard via `UIPasteboard` (iOS) or `NSPasteboard` (macOS), with a transient toast confirmation. A "Copy all system info" row copies a multi-line summary suitable for support tickets.
- **Status Page** — `Link` row to `https://status.bsky.app/` (matches RN's `STATUS_PAGE_URL` from `lib/constants.ts`), grouped alongside the existing ToS / Privacy / Community Guidelines / Source Code rows.
- **Clear cache** — destructive button that opens a `confirmationDialog`; on confirm it calls the injected `CacheStore.evictAll()`, shows an in-flight `ProgressView`, and toasts on success or surfaces an error footer on failure. RN's parallel action only clears the image disk cache, but the SwiftUI cache layer is unified, so the action wipes the entire `SwiftDataCacheStore`.
- **Developer** — gated behind `#if DEBUG`, exposes a `Verbose logging` `Toggle` backed by `@AppStorage("debug.verboseLogging")`. RN gates similar tools behind a runtime dev-mode flag; we use the build configuration so the section never ships in TestFlight / App Store builds.

To make the cache available to the screen, `SettingsScreen.init` gained an optional `cache: (any CacheStore)? = nil` parameter (placed before `currentAccount` to keep callers stable) which is forwarded to `AboutScreen`. `MainTabView` in the app target now passes `env.cache` when constructing `SettingsScreen`.

## Files changed

- `BlueskyKit/Sources/BlueskySettings/AboutScreen.swift` — rewritten with new sections, copy-to-clipboard helper, clear-cache confirmation dialog, debug-only developer section.
- `BlueskyKit/Sources/BlueskySettings/SettingsScreen.swift` — added `cache` parameter and forwarded it to `AboutScreen`.
- `Bluesky-SwiftUI/Bluesky-SwiftUI/MainTabView.swift` — passes `env.cache` when constructing `SettingsScreen`.

## Gotchas

- `Section("Developer") { ... } header: { ... } footer: { ... }` does not compile because the string-label initializer is a separate overload that doesn't take trailing `header` / `footer` closures. Use `Section { ... } header: { Text("Developer") } footer: { ... }` instead.
- `UIPasteboard` exists on iOS / iPadOS / tvOS / visionOS but **not** watchOS, so the copy helper guards on `canImport(UIKit) && !os(watchOS)` and falls back to `NSPasteboard` on macOS.
- `Host.current().localizedName` is optional on macOS; the device-model accessor coalesces to `"Mac"` so support strings always have a value.
