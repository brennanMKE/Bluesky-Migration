## Description

The Posts tab on a profile should pin the user's pinned post (if any) at the top of the feed with a `📌 Pinned` indicator above the byline. The React Native reference does this; the SwiftUI client currently shows the user's posts in pure chronological order with no pinned-post handling.

This is a follow-up to the iOS Profile parity audit in #0070.

## Attachments

![Reference: RN profile feed with "📌 Pinned" indicator](profile-rn.png)
![Current: SwiftUI profile feed with no pinned-post handling](profile-swiftui.png)

## Scope

- **Read the user's pinned post** from the profile metadata. AT Proto stores the pinned post AT-URI on the profile record (`app.bsky.actor.profile`'s `pinnedPost` field, or via `app.bsky.feed.getAuthorFeed` if it surfaces a pin marker per item).
- **Render the pinned post first** in the Posts tab, above the chronological feed. The pinned post may also appear later in chronological order — that's expected and matches RN; do not de-duplicate.
- **Pin indicator**: a small `📌 Pinned` badge above the post byline (use `pin.fill` SF Symbol + a `.caption` text). Match RN positioning.
- **Other tabs** (Replies / Media / Videos / Likes / Feeds / Lists): no pin handling.
- **Own profile pin management** (allow user to pin/unpin their own post via the post's ellipsis menu): out of scope for this issue — file a follow-up if needed.

## Implementation notes

- Code: `BlueskyKit/Sources/BlueskyProfile/ProfileScreen.swift` (Posts tab) and/or `ProfileFeedStore`.
- Fetch the pinned-post URI as part of `getProfile` and render it as a separate item at the top of the Posts tab list. Use the same `PostCardView` component with the pin indicator overlaid.
- If the AT Proto API doesn't expose a separate pinned-post fetch, look at how `app.bsky.feed.getAuthorFeed` surfaces it — RN uses a per-item `feedContext`/`feedSource` flag to identify the pinned item.

## Root cause

The Profile Posts tab in SwiftUI hit `app.bsky.feed.getAuthorFeed` with
`filter=posts_no_replies` and no `includePins` flag. That filter happens to be
the wrong one to pair with pinned-post handling — and the missing
`includePins=true` would have suppressed the pinned item even if the right
filter were in place. As a result, the pinned post never came back as a
distinct feed entry, and there was no UI affordance for the pin reason
either: `FeedReason` only decoded `app.bsky.feed.defs#reasonRepost`, falling
through everything else (including `reasonPin`) to `.unknown`, and `PostCard`
only switched on the repost case to draw a banner.

## Fix

Path **(b)** chosen: `getAuthorFeed` itself surfaces the pinned post inline,
flagged per-item via `reason` of type `app.bsky.feed.defs#reasonPin`. This
matches the RN reference exactly. The RN feed API
(`src/lib/api/feed/author.ts`) sets `includePins = (filter ===
'posts_and_author_threads')` and the Profile screen's Posts tab keys off
`author|<did>|posts_and_author_threads`; the PostFeedReason component
(`src/view/com/posts/PostFeedReason.tsx`) draws the "Pinned" label whenever
`AppBskyFeedDefs.isReasonPin(reason)` returns true. Path (a) — reading
`pinnedPost.uri` from `app.bsky.actor.defs#profileViewDetailed` and issuing a
follow-up `getPosts` — would have worked but is **not** the path RN uses, so
implementing it here would have introduced a needless extra round-trip and a
divergence from the upstream behavior.

Three small changes carry the fix:

1. `FeedReason` in `BlueskyCore/Post.swift` gains a `.pin` case that decodes
   the `app.bsky.feed.defs#reasonPin` discriminator. The lexicon currently
   has no fields, so the case carries no payload.
2. `ProfileStore.fetchTab(.posts, …)` switches the filter from
   `posts_no_replies` to `posts_and_author_threads` and adds
   `includePins=true` — exactly matching RN's request shape.
3. `PostCard.body` now switches on all `FeedReason` cases and renders a new
   `pinnedBanner` view (a `pin.fill` SF Symbol plus the word "Pinned" in the
   tertiary text color, indented to align with the byline) above the
   author header when `reason == .pin`. Layout matches the existing
   `repostBanner` so the two never visually collide; the AT Proto feed never
   returns both reasons on the same item.

The pinned post may also appear later in chronological order — that is
expected and matches RN. We deliberately do not de-duplicate.

## Files changed

- `BlueskyKit/Sources/BlueskyCore/Post.swift` — `FeedReason` gains `.pin`
  case; encodes/decodes the `app.bsky.feed.defs#reasonPin` discriminator.
- `BlueskyKit/Sources/BlueskyProfile/ProfileStore.swift` — Posts tab now
  requests `filter=posts_and_author_threads&includePins=true` against
  `app.bsky.feed.getAuthorFeed`, the same shape the RN client uses.
- `BlueskyKit/Sources/BlueskyUI/PostCard.swift` — `body` switches over the
  full `FeedReason` enum; new `pinnedBanner` view draws the
  pin.fill + "Pinned" caption above the byline. Same layout slot as the
  existing `repostBanner`.

## Gotchas

- **Path (b), not (a).** The original issue listed both source-of-truth
  options. RN uses (b): `getAuthorFeed` returns the pinned item inline with
  `reason: ReasonPin` whenever `includePins=true` accompanies a
  `posts_and_author_threads` filter. Path (a) — reading `pinnedPost.uri`
  from the detailed profile and calling `getPosts` separately — would have
  worked but is the wrong shape: extra request, no pagination guarantee,
  and divergence from the live RN behavior. Path (b) is "free" once the
  filter and flag are right.
- **The Posts tab filter changed.** `posts_no_replies` becomes
  `posts_and_author_threads`. The RN reference treats the latter as the
  Posts tab's canonical filter, and `includePins=true` is **only** honored
  alongside that filter — pairing `includePins` with `posts_no_replies`
  would not surface the pin. The author-thread filter does include the
  actor's replies that are part of their own thread chains, which RN
  considers correct Posts-tab content.
- **No de-duplication.** The pinned post can show up twice — once at the
  top with the pin banner, once in its chronological slot below. That is
  the RN behavior and the issue spec explicitly calls it out as expected.
- **Pin/unpin management still out of scope.** The ellipsis menu does not
  yet expose a "Pin to profile" / "Unpin from profile" affordance for the
  viewer's own posts. RN does this via `usePinnedPostMutation` writing to
  the `app.bsky.actor.profile.pinnedPost` field; that's a separate feature
  and the issue scope explicitly defers it.
